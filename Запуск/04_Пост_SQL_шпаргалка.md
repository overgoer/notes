## Вариант А: "Как я нашёл баг в API через SQL (и почему Postman бы не помог)"

Был у меня эндпоинт `GET /orders?status=pending`.
Документация: "возвращает заказы со статусом pending".

Я дёргаю — приходит 5 заказов. Всё ок.

Но что-то мне подсказывало проверить иначе.
Я открыл базу и выполнил:

```sql
SELECT status, COUNT(*) FROM orders GROUP BY status;
```

Результат:
| status  | count |
|---------|-------|
| pending | 12    |
| paid    | 45    |
| shipped | 23    |

В API — 5. В базе — 12. Куда делись 7 заказов?

Оказалось, что разработчик добавил фильтр `WHERE user_id = ?` — но не в API, а в сервисный слой.
Эндпоинт должен был отдавать заказы для текущего пользователя, а отдавал для всех.
Но в тестовой БД были только мои заказы — поэтому я не заметил.

**SQL здесь — не замена API-тестированию.**
SQL — это детектор лжи.
Когда API говорит одно, а база — другое, ты хотя бы знаешь, в какую сторону копать.

## Вариант Б: "3 SQL запроса, которые я использую каждый день"

Короткая шпаргалка, без истории.

**1. Проверка, что API не врёт**
```sql
SELECT COUNT(*) FROM users;
```
vs `GET /api/users` → `total`. Если числа не совпадают — пагинация или фильтры сломаны.

**2. Проверка, что DELETE работает**
```sql
SELECT * FROM users WHERE id = 42;
```
Выполнить `DELETE /api/users/42`.
Проверить — удалился? А каскады сработали?

**3. Проверка дублей**
```sql
SELECT email, COUNT(*) FROM users GROUP BY email HAVING COUNT(*) > 1;
```
API может вернуть unique constraint error. А может — молча создать дубль. SQL покажет.
