Джуны, которые на собеседовании показывают «пет-проект» с идеальным API погоды — мимо. Работодатель видит: ты не сталкивался с реальным говном.

Practicum API — учебный стенд, где специально заложены баги. Это твой шанс показать, что умеешь не просто дёргать 200, а искать нарушения RFC и логические дыры.

Вот три кейса, которые выстрелят:

**1. DELETE возвращает 200 OK вместо 204 No Content.**  
Нарушение RFC 7231. Клиент ждёт пустой ответ, а получает тело. Ломает кеширование и логику фронта. Проверить одной командой:  
`curl -X DELETE -w "%{http_code}" https://practicum-api.test/api/users/1`

**2. Повторный POST с тем же email — 500 Internal Server Error.**  
Должно быть 409 Conflict. 500 — необработанная ошибка в БД. Это баг, а не фича. Покажи, что отличаешь бизнес-логику от паники сервера.

**3. GET /orders с чужим userId — возвращает чужие заказы.**  
Классическая BOLA-уязвимость (Broken Object Level Authorization). Три теста: свои данные, чужие данные (должен быть 403), без токена (401). Если есть этот баг — ты нашёл дыру в безопасности.

Такой проект на порядок сильнее, чем абстрактные автотесты на абстрактный API. Ты показываешь поиск багов, оформление репортов, знание протокола. И да — никакой AI не заменит понимания, почему 200 на DELETE — зашквар.
