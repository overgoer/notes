# 5 SQL-запросов, которые сэкономили мне часы жизни (и нервы)

Тестируешь API с базой? Эти 5 запросов я использую каждый день в Practicum. Сохраняй — будешь благодарить 👇

**1. Найти дубли по email за 2 секунды**
Когда после регресса в БД 1000+ юзеров, а баг говорит «можно создать дубль»:
```sql
SELECT email, COUNT(*) as dup_count
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```
⭐ Офигеть-момент: не надо листать 50 страниц в админке — один запрос и готово.

**2. Все записи с багами в данных (пробелы/типы)**
В Practicum баг — name приходит с пробелами, age — строкой. Ловится мгновенно:
```sql
SELECT id, name, age
FROM users
WHERE name != TRIM(name)      -- лишние пробелы слева/справа
   OR age::text ~ '\D';        -- age не число
```
⭐ Без этого ты увидишь баг только когда пользователь упадёт с ошибкой на фронте.

**3. Сравнение V1 (бажная) vs V2 (починенная) — утерянные записи**
Когда DELETE в V1 не удаляет по-настоящему, а только делает вид:
```sql
SELECT u.id FROM api_v1.users u
EXCEPT
SELECT u.id FROM api_v2.users u;
```
⭐ Если разница непустая — баг не починили. В Practicum этот запрос раскрыл, что V1 DELETE просто SELECT'ит данные 😱

**4. Первые N «проблемных» пользователей (кто ломает ПРОД)**
Кто отправил больше всего некорректных данных — кандидаты на баг:
```sql
SELECT user_id, COUNT(*) as bad_requests, SUM(amount) as total
FROM api_logs
WHERE status_code >= 400
GROUP BY user_id
ORDER BY bad_requests DESC
LIMIT 10;
```
⭐ Я перестал гадать «почему упало» — сразу вижу, кто и сколько раз сломал.

**5. Реальный размер базы (спойлер: тестовая БД никогда не чистится)**
```sql
SELECT pg_size_pretty(pg_database_size('practicum'));
```
А потом:
```sql
SELECT COUNT(*) FROM users;
```
⭐ Если у тебя 10 тестовых юзеров, а БД весит гигабайт — ищи WHERE без индекса.
