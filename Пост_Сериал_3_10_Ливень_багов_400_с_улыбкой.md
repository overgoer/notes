# Сериал 3/10: Ливень багов — 400 с улыбкой

**Статус:** ✅ Готов к публикации
**Рейтинг GA:** MUST POST
**Формат:** Обучение / Сериал
**Время публикации:** 18:30 МСК

---

Представь: шлёшь валидный JSON на POST /api/products, а в ответ 400 с текстом "Bad Request — smile :)". Смешно? Только если это не твой прод.

Это классическая заглушка — разработчик захардкодил `return BadRequest("...")` вместо вызова валидатора и забыл убрать. Любой запрос (даже корректный) возвращает одно и то же. А бизнес-логика создания продукта вообще не реализована — данные не сохраняются.

## Как отличить настоящую валидацию от мусора?

Вот чек-лист из 5 признаков, что перед тобой заглушка:

1. **Однообразный ответ** — на любой запрос один и тот же код и текст.
2. **Неформальный текст** — смайлики, "null", "stub", "TODO", "placeholder". Нормальная API так не отвечает.
3. **Мгновенный ответ** — замерь через `time curl`: если <1ms, валидация не выполняется.
4. **Нет бизнес-эффекта** — после POST сделай GET /api/products. Если товар не появился — endpoint пустышка.
5. **Отсутствие структуры** — вместо JSON с полями `code`, `message`, `details` — просто строка.

Этот баг — красный флаг для тестировщика. Он не только нарушает RFC 7231 (400 только для клиентских ошибок), но и маскирует неработающий функционал. В проде такие заглушки ломают интеграции и убивают UX.

**Мораль:** всегда читай тело ответа на ошибку и проверяй, что после POST данные реально сохранились. Иначе рискуешь принять "улыбку" за легитимный баг.
