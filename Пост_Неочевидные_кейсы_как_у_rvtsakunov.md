# Что реально работает в проде, а не на курсах

**Статус:** ✅ Готов к публикации
**Рейтинг GA:** MUST POST
**Формат:** Обучение / Жизненные кейсы
**Инспирировано:** @rvtsakunov

---

Три кейса, которые не объясняют ни в одной школе тестирования.

**Кейс 1: «Это фича» — через год падает прод**
В легаси-функции форматирования даты при null выдавалось "01.01.1970". Дев: «так задумано». Через год мигрировали данные, null стали частыми — прод упал с ошибкой валидации.
**Как спасаться:** пиши property-based тест на краевые случаи (null, мусор). Если тест падает — значит контракт нарушен. Добавь мониторинг аномальных входов через OpenTelemetry.

**Кейс 2: Баг-призрак, который не воспроизводится**
Раз в неделю на проде ловили duplicate primary key при бронировании. В тестовой среде — тишина. Оказалось, гонка данных, которая проявляется только при параллельных запросах в продакшен-нагрузке.
**Решение:** добавили unique constraint в БД как защитный слой. Для теста — параллельные потоки с CountDownLatch (JUnit) и логи MySQL. Запускали 1000 раз с разными задержками.

**Кейс 3: «Докажи метрикой, а не эмоциями»**
Стратегия «доказательство от метрик»: не «это баг», а «вот assertion, который ловит NPE на проде». Зафиксируй спецификацию, напиши юнит-тест на погранслучаи, покажи разработчику лог с реальным падением.

Эти ситуации не учат на курсах — их даёт опыт. И если ты столкнулся с таким — ты не плохой тестировщик, ты просто входишь в реальность 😅
