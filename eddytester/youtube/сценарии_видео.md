# Сценарии видео — Сезон 1: Баги в API

**Дата:** 2026-05-28
**Статус:** Черновик (сущности API не определены)
**Формула:** [Концепция] + [Баг] + [Контекст]

---

## Формат видео (шаблон)

**Длительность:** 5 минут
**Тон:** Разговорный, динамичный, без воды
**Визуал:** Postman (60% экрана) + схема/аннотации (40%)
**Голос:** Эдди, от первого лица, как объяснение коллеге

**Структура каждого выпуска:**

| Время | Элемент | Описание |
|-------|---------|----------|
| 0:00-0:05 | **Cold Open** | Скринкаст бага + шок-реакция с фразой из заголовка. Концовка: "Но самое страшное — в конце видео. Досмотри до конца." |
| 0:05-0:10 | **Переход** | Чёрный экран / склейка |
| 0:10-0:25 | **Interactive Hook** | Скриншот ответа API на экране. "Пауза. Найди баг за 3 секунды. Прокрути назад если не успел. А теперь смотрим." |
| 0:25-1:00 | **Введение** | Что это за штука, как должна работать, как тестировать |
| 1:00-2:15 | **Демо бага** | Плохие и хорошие примеры в Postman |
| 2:15-3:00 | **Реальный мир** | "А что в реале?" — ссылка на инцидент в известном API |
| 3:00-3:45 | **Разбор** | Почему это баг, чем опасен, почему 90% пропускают |
| 3:45-4:30 | **Чек-лист** | Как тестировать системно |
| 4:30-4:45 | **Правило Эдди** | blind spot phrase + Memeable Rule |
| 4:45-5:00 | **Аутро** | Тизер следующего выпуска + Cheat Sheet на 3 сек (стоп-кадр) |

**Постоянные элементы:**
- Cold open: обязательно фраза из заголовка + "Но самое страшное — в конце видео"
- Interactive hook: "Найди баг за 3 секунды. Пауза."
- Реальный мир: отсылка к известному инциденту
- Blind spot: "90% тестировщиков это пропускают. Теперь ты — нет."
- Правило: уникальное для каждого выпуска
- Cheat Sheet: финальный кадр со сводным чек-листом
- На протяжении: ссылка на практикум (сайт для тестирования)

## Эпизод 1: POST, который создаёт два объекта

**Cold Open (0:00-0:05):** Экран: Postman, отправка POST, ответ 201. Быстрая проверка GET /objects — в ответе два одинаковых объекта. Голос: "Смотри! Один POST! А в базе — два объекта! Дубликаты! Валидация у API сломана! Но самое страшное — в конце видео. Досмотри до конца."

**Interactive Hook (0:10-0:25):** Скриншот ответа POST — 201 Created, тело: `{"id": 42, "name": "Анна", "age": 25}`. "Пауза. Найди баг за 3 секунды. С виду всё чисто. Но я отправил один запрос. Прокрути назад если не успел. А теперь смотрим."

**Хук:** "Ты отправил один запрос, а база раздувается в два раза. Где искать причину?"

**Боль зрителя:** Новичок пишет POST -> видит 201 -> радуется. Но не проверяет, сколько объектов реально создалось. В продакшене такие баги живут годами.

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход | — |
| 0:25 | Интро: схема нормального POST vs бажного | "Один POST-запрос. 201 Created. А в базе — два объекта. Это не магия, это баг. И он встречается чаще, чем ты думаешь." |
| 1:00 | Схема: нормальный POST vs бажный | "Нормальное API: получил запрос -> создал объект -> вернул 201. Бажное: получил запрос -> создал объект -> ещё раз создал -> вернул 201." |
| 1:30 | Postman: отправляем POST | "Ок, отправляю. 201, всё красиво. Тело — один объект. Если не копать — я бы поверил." |
| 2:00 | GET /objects — их два | "А теперь GET на список. Вижу два одинаковых объекта. Вот она, аномалия." |
| 2:30 | Реальный мир | "А что в реале? В 2022 году Stripe API: при создании платежа под нагрузкой иногда создавалось два списания. Баг жил в проде 6 месяцев." |
| 3:00 | Анализ | "90% тестировщиков проверяют только статус — и пропускают это. Дубликаты заказов, двойные списания — клиент платит дважды." |
| 3:45 | Чек-лист | "Правильный тест POST: 1. Статус 201. 2. Тело ответа. 3. GET и проверка, что объект один. 4. Поля совпадают." |
| 4:30 | **Правило Эдди: "После POST — всегда GET. ППВГ"** | "Проверь состояние после мутирующего запроса. POST -> GET. ППВГ. 90% пропускают. Теперь ты — нет. Следующий выпуск: DELETE, который говорит да, а делает нет." |
| 4:45 | Cheat Sheet | Стоп-кадр: чек-лист POST — статус, тело, GET, сравнение полей |

**Ключевой инсайт:** Статус 201 не гарантирует корректность. Проверяй состояние после запроса.

**Правило Эдди:** После POST — всегда GET. ППВГ

**Теги:** `api тестирование`, `postman`, `баги api`, `rest api`, `qa`, `тестирование api для начинающих`, `post запрос`

## Сезон 2: Альтернативные форматы

**Базовые метрики (из вордстата + ютуб-исследования):**
- "сайт для тестирования api" — спрос 24 (люди ищут полигон)
- "чек лист тестирование api" — спрос 20 (шаблоны, не теория)
- "qa собеседование вопросы" — смежная тема с высоким спросом
- YouTube: видео 20-40 мин (freeCodeCamp, SDET QA) набирают 1-4М просмотров — аудитория готова к длинному формату
- Короткие explainer (3-9 мин, Exponent, ByteByteGo) — 900К-2.6М просмотров
- **Гэп:** нет формата "тестируем конкретное API в реальном времени"

---

### Формат A: «Тестируем [название API] за 20 минут»

**Гипотеза:** Люди ищут "сайт для тестирования api" (спрос 24) — им нужен не урок, а полигон. Мы даём и то, и другое: разбираем реальное API, показываем где искать баги.

**Содержание выпуска:**
1. Берём открытое API — GitHub, Telegram, Stripe, OpenWeather
2. 5 мин: что за API, какая бизнес-логика, документация
3. 10 мин: тестируем в Postman — happy path, граничные, негативные
4. 5 мин: разбор найденного + баг-репорт

**Фишки удержания:**
- В начале: "Этот API использует каждая вторая компания. Сейчас найдём, что сломано."
- Интерактив: зритель тестирует вместе — API открытый, можно повторять
- В конце: "Слайды со схемой тестирования — в телеграме"

**Формат:** YouTube + стрим

---

### Формат B: «Найди 3 бага за 10 минут» (челлендж)

**Гипотеза:** Вордстат показывает запрос "сайт для тестирования api" — люди хотят тренажёр. Игровой формат с таймером конвертирует пассивного зрителя в активного участника.

**Содержание выпуска:**
0:00-0:30 — Cold open: "У тебя 10 минут. Три бага. Поехали."
0:30-10:00 — Экран: демо API, Postman открыт. Зритель ищет баги.
10:00-15:00 — Разбор: я показываю все баги, которые были заложены
15:00-17:00 — Таблица: кто какие баги нашёл (из комментов)

**Фишки удержания:**
- Азарт + комментарии: "Я нашёл 2/3, а ты?"
- Таймер на экране весь раунд — визуальное напряжение
- Результаты в следующем видео: "Топ-5 подписчиков, которые нашли все баги"
- Для шортс/инсты: нарезка "3 бага за 30 секунд"

**Формат:** YouTube (основной) + Shorts/Reels/TikTok (нарезка)

---

### Формат C: «Баг-репорт, который примут» (туториал)

**Гипотеза:** "чек лист тестирование api" (спрос 20) — люди ищут шаблоны. 90% новичков пишут "не работает" в баг-репорте. На курсах этому не учат. Мы даём готовый шаблон — виральность в телеграм-каналах.

**Содержание выпуска:**
1. Берём один баг из Сезона 1 (любой)
2. Показываем плохой баг-репорт: "не работает, почините"
3. Показываем хороший: заголовок, шаги, окружение, ожидание/реальность, скриншоты/curl, severity
4. Даём шаблон: PDF / Google Docs / Notion — всё в описании

**Фишки удержания:**
- Шаблон можно скачать только по ссылке в телеграме — конвертация в канал
- Пример из жизни: "Этот баг-репорт сэкономил команде 2 дня"
- В конце: чек-лист из 10 пунктов "Проверь свой баг-репорт"

**Формат:** YouTube 5-7 мин + PDF-шаблон в телеграме

---

### Формат D: «Собеседование: ответ, который выделит тебя»

**Гипотеза:** ЦА — QA 60-80К, хотят расти и менять работу. Запрос "как тестировать api" + "qa собеседование" — смежные боли. Видео отвечает на конкретный вопрос с собеседования с демонстрацией навыка.

**Содержание выпуска:**
1. Вопрос с реального собеседования: "Как ты тестировал API в прошлом проекте?"
2. Плохой ответ: "Ну, отправлял запросы, смотрел ответы"
3. Хороший ответ: структура — какие эндпоинты, какие методы, какие кейсы, какие баги нашёл, как оформлял
4. Демо: показываем портфолио-баг (тот же Сезон 1)

**Фишки удержания:**
- Формат "до/после" — плохой vs хороший ответ — легко запомнить
- В конце: шпаргалка для собеседования (3 пункта)
- Навигация в комментах: "Был на собеседовании? Напиши свой вопрос"
- Естественный репост: те, кто ищет работу, шлют друзьям

**Формат:** YouTube 5-7 мин

---

### Формат E: «Стрим — тестируем API подписчика»

**Гипотеза:** Никто в ру-нише это не делает. Английские стримеры (ThePrimeagen и др.) собирают 10-50К зрителей на code review. У нас — API review. Лояльность + контент без подготовки.

**Содержание стрима:**
1. Кто-то из комментов оставляет ссылку на своё тестовое API / пет-проект
2. Открываем Postman, тестируем в прямом эфире
3. Находим баги — обсуждаем, как править
4. В конце: результаты + чек-лист для владельца API

**Фишки удержания:**
- Азарт: чей API будут тестировать в следующем стриме? (оставляй ссылку в коментах)
- Драма: "ой, а тут у него авторизация не работает... публичный API без ключа"
- Обучение: зритель видит процесс мышления тестировщика в реальном времени
- Пост-контент: нарезка лучших моментов на YouTube

**Формат:** Twitch / YouTube Live (1 час) + нарезка 10-15 мин

## Стримы и эфиры: когда Сезон 1 запущен

### Стрим F: «Пишем демо API с багами за час»

**Что делаем:** Открываем редактор, пишем Express-сервер, намеренно закладываем баги. Зритель видит, как баги рождаются — и как их найти.

**Почему это контент:** Процесс создания API никто не стримит на русском. А зритель, который видел баг в видео и увидел, как он закладывается в код, запоминает навсегда.

**Структура:**
- 0-10 мин: каркас сервера
- 10-30 мин: пишем эндпоинты с багами (1-2 бага за стрим)
- 30-50 мин: тестируем вместе, кто найдёт — приз
- 50-60 мин: Q&A

**Конверсия:** Ссылка на то же демо API — зритель может протестировать сам после стрима.

---

### Эфир G: «Живой разбор багов»

**Формат:** 30-40 мин, YouTube Live, без подготовки.

**Что делаем:** Открываем чат, берём вопросы. "А почему этот тест упал?", "А как тестировать PATCH?", "А что если API без документации?" — отвечаем и тут же показываем в Postman.

**Почему работает:** Лайв = алгоритмы YouTube дают дополнительный буст. Вопросы из чата = контент, который реально нужен аудитории.

**Расписание:** Раз в 2 недели, анонс в телеграме за день.

---

### Воркшоп H: «10 багов за 2 часа» (закрытый)

**Формат:** Закрытый эфир для подписчиков телеграма / Patreon / Boosty.

**Что внутри:**
- Ссылка на демо API с 10 багами
- 2 часа совместного тестирования
- В конце: чек-лист всех багов + правильные ответы
- Запись остаётся у участников

**Монетизация:** 300-500 руб за участие. 10 человек = 3-5К с одного воркшопа. При 1000 подписчиков в телеграме конверсия 1-2%.

### Креаторская жилка: что делать, когда просмотров мало

**Реальность:** Первые 10-20 видео увидят 50-200 человек. Не 5000. Это норма.

**Почему не надо бросать:**
- **Ниша пустая.** 0 конкурентов в формате "реальные баги + разбор". Любой контент, который появится первым, закрепится в выдаче на годы.
- **Каждый баг — evergreen.** Видео про off-by-one не устареет. Через год его будут смотреть так же, как сегодня.
- **Сериал работает на ретро.** Когда выйдет 20+ видео, новый зритель придёт за одним — останется на 10. DCR (discovery-to-content ratio) растёт с каждой серией.

**Что делать вместо просмотров:**

1. **Снимать для 10 человек, а не для 10000.** Если хотя бы 10 QA-новичков скажут "спасибо, помогло" — контент состоялся. Охват придёт потом.

2. **Каждый выпуск = кусочек портфолио.** Ты не просто снимаешь — ты строишь доказательство: "я автор единственного русскоязычного курса по API-тестированию в формате практических багов". Это актив, который работает на тебя через год.

3. **Низкий барьер первого видео.** Первый выпуск не обязан быть шедевром. Сними на телефон. Сделай монтаж в CapCut. Выпусти. Второе будет лучше.

4. **Short как тест-драйв.** Не уверен в теме? Сделай 30-секундный шортс. Если залетает — делай полное видео. Если нет — не делай. Shorts = дешёвая проверка гипотез.

5. **Телеграм — твой HDD.** YouTube — витрина. Телеграм — доказательство, что есть комьюнити. Каждый зритель с ютуба должен уйти в телеграм. Каждый пост в телеграме — подогрев перед следующим видео.

**Главное:** 100 видео по 100 просмотров = 10000 просмотров суммарно + 200 постоянных зрителей, которые придут на стрим. Через год 100 видео в нише без конкурентов = ты главный эксперт рунета по API-тестированию. Вопрос не в таланте, а в количестве выпусков.

| **Апселл на телеграм** | В каждом выпуске | "Чек-лист только в телеге. Бесплатно. Ссылка в описании." |
| **Кросс-ссылка на другой выпуск** | 1 раз в видео | В аутро: ссылка на предыдущий. В середине: "я об этом рассказывал в выпуске про [название]" |
| **Пин предыдущего видео** | В первые сутки после выхода | Закреплённый коммент: "Не пропусти предыдущий выпуск: [ссылка]" |
| **Shorts/Reels** | Через 2-3 дня после YouTube | Нарезка самого сочного момента + ссылка на полное видео |

### 4. Что упоминать в кадре (визуальные якоря)

Чтобы зритель не пропустил ссылки — они должны быть на экране в ключевые моменты:

| Момент | На экране | Как |
|--------|-----------|-----|
| 0:10 | Логотип канала / ник | Угол экрана, полупрозрачный |
| Первое упоминание практикума | Ссылка на сайт | Всплывающая плашка на 3 сек |

| Правило Эдди | Текст правила | Анимация текста на весь экран на 3 сек |
| Cheat Sheet в конце | Текст "ЧЕК-ЛИСТ В ТЕЛЕГРАМЕ" | Поверх стоп-кадра |
| Аутро | Плейлист на экране | Карточка с предыдущим и следующим видео |

### 5. Упоминание продуктов — по градации жёсткости

| Уровень | Где | Текст | Частота |
|---------|-----|-------|---------|
| **Мягкий** | В контексте | "Вот такое же демо API стоит на моём сайте — ссылка в описании" | Каждое видео |
| **Средний** | Аутро | "Если хочешь попрактиковаться — ссылка на сайт с демо API в описании" | Каждое 2-3 видео |
| **Жёсткий** | Отдельный блок | "Кстати, у меня есть практикум по API-тестированию. Там 30+ багов, изолированная база, сертификат. Ссылка в описании." | Каждое 5-е видео |
| **Прямая продажа** | Аутро / описание | "Подписка на практикум — [ссылка]. Промокод EDDYTESTER на скидку." | Анонсы, релизы |

**Правило:** мягкий — всегда, средний — регулярно, жёсткий — редко.
Зритель, который пришёл за контентом, не должен чувствовать, что ему продают на каждом видео.

### 6. Telegram + Instagram — кросс-платформенная стратегия

**Основной принцип:** YouTube — алгоритм роста. Telegram — ретеншн и комьюнити. Instagram — виральность.
Не каннибализируем платформы. Каждая получает свой формат.

---

#### 6.1. Telegram: правила работы

**Жёсткий запрет:** не постить прямые ссылки на YouTube в ленту.
Прямая ссылка в ленте Telegram = открытие во встроенном браузере = 5 сек просмотра = убитые метрики YouTube.

**Что постим вместо ссылок:**
- Нативное видео 30-40 сек (загружено прямо в Telegram) — самая сочная часть бага
- Текст: "Баг за 30 секунд. Полный разбор — в закрепе."
- В закреплённом сообщении канала: ссылка на плейлист YouTube + ссылка на последнее видео
- Кто хочет — осознанно идёт в закреп, открывает YouTube в нормальном клиенте

**Формат нативного видео для Telegram:**
- Обрезка самого яркого момента из выпуска (cold open + открытие бага)
- Можно без монтажа — просто экспорт таймлайна с 0:00 по 0:40
- Размер: вертикальный квадрат 1:1 или 16:9 с субтитрами
- Длительность: строго 30-40 сек
- В конце: "Полный выпуск — ютуб, ссылка в закрепе"

**Это же видео идёт в Instagram Reels — без изменений.**

---

#### 6.2. Контент-план Telegram

| Когда | Пост | Формат | Цель |
|-------|------|--------|------|
| Пн (день до видео) | Тизер: текст + картинка "Завтра разберу баг, который 90% пропускают" | Текст | Подогрев |
| Вт (выход видео) | Нативное видео 30 сек + "Полный разбор в закрепе" | Видео + текст | Охват в Telegram |
| Чт | Чек-лист PDF из видео | PDF | Эксклюзив, репосты |
| Сб | Текстовый пост: Правило Эдди (коротко, цитата) | Текст | Охват, запоминание |
| Раз в 2 нед | Анонс стрима / эфира | Текст | Посев |
| Раз в месяц | Пост-знакомство: "кто я, чем занимаюсь, куда веду" | Текст | Доверие |

**Важно:** чек-листы PDF только в Telegram, нигде больше. Это причина подписаться.

---

#### 6.3. Instagram: что и как

**Формат:** Reels 30-40 сек — те же нарезки, что в Telegram.

**Постить:** 2-3 раза в неделю.
- 1 Reel = нарезка свежего видео
- 1 Reel = правило Эдди текстом на фоне Postman
- 1 Reel = вопрос/челлендж: "Найди баг за 10 секунд"

**Связка с YouTube:** В описании Reels — ссылка на YouTube (через "ссылка в шапке профиля").
Не ждать конверсии из Instagram в YouTube. Instagram = охват + узнаваемость.

---

#### 6.4. Почему это работает

- **Telegram видео** не конкурирует с YouTube: короткий формат для тех, кто не пойдёт на YouTube
- **YouTube не страдает** от плохих метрик: прямой ссылки в ленте нет
- **Instagram Reels** — кросспостинг без доп.работы: то же видео, та же длительность
- **Чек-листы PDF** — причина подписаться на Telegram, а не просто зайти разово
- **Закреп** — единственное место ссылки на YouTube, но оно всегда перед глазами подписчика

### 8. Что НЕ делать (частые ошибки новичков)

- **Не глуши контентом.** Одна ссылка за видео — нормально. Три ссылки — спам.
- **Не начинай с продажи.** Первые 2 минуты зритель решает, остаться или уйти. Продажа в начале = уход.
- **Не ставь ссылки в середину речи.** "Зайди на сайт practicum.example.com" — сбивает. Ссылки в описании. В речи — только "ссылка в описании".
- **Не забывай про закреп.** Первые сутки 80% комментов под новым видео — от тебя. Закрепи ссылку на плейлист сразу.
- **Не пости ссылку на YouTube в ленту Telegram.** Открывается во встроенном браузере — зритель смотрит 5 секунд, метрики падают. Только нативное видео 30-40 сек.
- **Не пости в Telegram "просто видео".** К каждому видео — текст 2-3 предложения + призыв в закреп. Голое видео не конвертирует.
- **Не копируй одинаковый контент в Instagram и Telegram.** Один ролик — да, но описание и призыв к действию — разные, под платформу.
- **Не жди быстрых результатов в Instagram.** Reels работают на отложенный охват. Первые 10 Reels может увидеть 100 человек. 20-й — 10000.

## Рост Telegram и воронка продаж

### 1. Как растить Telegram канал

**YouTube -> Telegram (основной канал трафика)**
Это самый эффективный источник. Человек посмотрел видео — захочет чек-лист — идёт в Telegram.
- В видео: 1 упоминание в середине + визуальный якорь в конце чек-листа
- В описании: "Чек-лист к этому выпуску — в телеграме"
- Конверсия: 3-8% от просмотров YouTube в подписку Telegram (в нише QA)
- С каждого видео с 1000 просмотров — 30-80 новых подписчиков в Telegram

**Instagram -> Telegram (дополнительный)**
- Reels не конвертят напрямую. Ссылка только в шапке профиля
- Конверсия: 0.5-1% от охвата Reels
- Работает на узнаваемость, не на подписку

**Telegram Ads (платный рост)**
- Telegram Ads позволяет таргетироваться на каналы конкурентов
- Цель: каналы Artsiom Rusau QA Life, The Testing Academy, QA Tech и другие QA/тестировщики
- Минимальный бюджет: 2-3 EUR за 1000 показов
- Реклама выглядит как нативное сообщение в списке каналов — не бесит пользователей
- Оценивать CPA (cost per subscriber): при 3-5 EUR за подписчика — норм, выше — стоп

**Кросс-промо (бесплатный рост)**
- Договориться с другими QA-каналами о взаимном пиаре
- Формат: "Друзья из канала [name] тоже делают контент про API — рекомендую"
- Конверсия: зависит от пересечения аудитории, но подписчики качественные

**SEO Telegram**
- Название канала должно содержать ключевые слова: "тестирование API", "QA", "баги"
- Описание канала: полное, с ключевиками (поиск в Telegram работает по описанию)
- Аватар: узнаваемый (твоё лицо или логотип)

**Итого по приоритетам роста Telegram:**
1. YouTube -> Telegram (контент) — бесплатно, основной драйвер
2. Telegram Ads (платный) — когда есть что продавать
3. Кросс-промо — точечно
4. Instagram -> Telegram — приятный бонус, но не опора

## Чек-листы: карта всех тем

**Стратегия:** 1 мастер (лид-магнит, 30 пунктов) + 5 эпизодных (к выпускам Сезона 1) + тематические (контент в TG между видео).

### Мастер-чек-лист "30 проверок одного API-запроса"

**Боль:** "Я не знаю, что именно проверять в ответе. Статус зелёный — иду дальше."

**Структура (30 пунктов, 6 блоков):**

**1. Структура запроса (4)**
- Метод: тот ли используется (POST вместо PUT?)
- URL: путь с id, query-параметры
- Headers: Content-Type, Accept, Authorization
- Тело: формат (JSON/XML/form-data), обязательные поля

**2. Статус-код (4)**
- Успех: 200 vs 201 vs 204 — правильный под код
- Ошибка клиента: 400/404/422 — не 500
- Ошибка сервера: 500/502/503 — не маскировка
- Нестандартный статус: API не врёт

**3. Тело ответа (5)**
- Формат: JSON/XML — соответствует Content-Type
- Обязательные поля: все есть
- Типы данных: число vs строка vs null vs boolean
- Формат: дата ISO, id uuid, enum из документации
- Лишние поля: есть то, чего нет в контракте?

**4. Валидация и граничные (7)**
- Числа: отрицательное, ноль, переполнение, строка, null
- Строки: пустая, пробелы, максимум, спецсимволы, null
- Даты: часовой пояс, невалидная, 29 фев, null
- Boolean: true/false/1/0/null/отсутствие
- Enum: невалидное значение, регистр
- Массив: пустой, null, один элемент
- Отсутствие обязательного поля: 400, не 500

**5. Идемпотентность и состояние (4)**
- POST: повторный -> дубликат?
- DELETE: повторный -> 404 или 200?
- PATCH: повторно те же данные -> изменений нет?
- GET после мутирующего: состояние изменилось?

**6. Авторизация и безопасность (4)**
- Без токена: 401, не 200
- Чужой токен: 403, не данные другого пользователя
- IDOR: /users/1 -> /users/2 с тем же токеном
- Rate limiting: headers, превышение, блокировка

**Плюс 2 на засыпку**
- Заголовки ответа: Content-Type, Cache-Control, Retry-After
- Сетевые ошибки: таймаут, закрытие соединения, перезапуск

- PATCH/PUT: 6 — null затирает, частичное, без обязательных, конфликт, версионирование, идемпотентность
- Загрузка файлов: 5 — без файла, пустой, большой, не тот тип, multipart

**Контракт (3):**
- Статус-коды: 8 — маппинг сценарий->статус, частые ошибки
- Схема JSON: 4 — обязательные, типы, nullable, лишние поля
- Заголовки: 4 — Content-Type, CORS, Cache, Rate-Limit

**Общие (2):**
- API без документации: 5 шагов — DevTools, сниффер, читать код, спросить dev, что тестить в первую очередь
- Новичку с нуля: roadmap — структура запроса -> Postman -> GET -> POST -> статусы -> валидация -> коллекция
