## Эпизод 1: POST, который создаёт два объекта

**Cold Open (0:00-0:05):** Экран: Postman, отправка POST, ответ 201. Быстрая проверка GET /objects — в ответе два одинаковых объекта. Голос: *«Смотри! Один POST! А в базе — два объекта! Дубликаты! Валидация у API сломана!»*

**Хук:** «Ты отправил один запрос, а база раздувается в два раза. Где искать причину?»

**Боль зрителя:** Новичок пишет POST → видит 201 → радуется. Но не проверяет, сколько объектов реально создалось. В продакшене такие баги живут годами.

**Формат:** 5 минут, экран: Postman → сервер → база данных

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход: "Ок, давай с начала" | — |
| 0:10 | Интро: схема нормального POST vs бажного | "Один POST-запрос. 201 Created. А в базе — два объекта. Это не магия, это баг. И он встречается чаще, чем ты думаешь. Давай разберёмся, что такое валидация POST-запроса, как она работает и как тестировать, чтобы не пропустить такие баги." |
| 0:40 | Схема: как выглядит нормальный POST vs бажный | "Нормальное API: получил запрос → создал объект → вернул 201. Бажное: получил запрос → создал объект → ещё раз создал → вернул 201. Где-то в коде висит лишний вызов." |
| 1:15 | Postman: отправляем POST, проверяем ответ | "Ок, отправляю. 201, всё красиво. Теперь смотрю body ответа — там один объект. Если не копать — я бы поверил." |
| 1:45 | GET /objects — показываем, что их два | "А теперь делаю GET на список. И вижу — два одинаковых объекта. Разные id, но поля совпадают. Вот она, аномалия." |
| 2:30 | Анализ: как такой баг влияет | "В продакшене такой баг означает: дубликаты заказов, двойные списания, задвоенные тикеты. Клиент получает два счёт за один товар." |
| 3:15 | Как НЕ надо тестировать (только статус) | "80% новичков проверяют только статус. 201 — значит ок. Но статус — это не ответ. Тело ответа — не вся картина. Состояние системы — вот что важно." |
| 3:45 | Чек-лист: что проверить | "Правильный тест POST: 1. Статус 201. 2. Тело ответа. 3. GET запрос и проверка, что объект один. 4. И что поля совпадают с ожиданием." |
| 4:30 | Аутро | "Запомни: POST проверить мало — проверь, что он создал ровно то, что нужно, и ровно один раз. В следующем выпуске: DELETE, который говорит 'да', а делает 'нет'." |

**Ключевой инсайт:** Статус 201 не гарантирует, что запрос обработался корректно. Надо проверять состояние системы после запроса.

**Теги:** `api тестирование`, `postman`, `баги api`, `rest api`, `qa`, `тестирование api для начинающих`, `post запрос`

## Эпизод 2: DELETE возвращает 200, но ничего не удаляет

**Cold Open (0:00-0:05):** Экран: DELETE /objects/42 → 200 OK. Тут же GET /objects/42 — объект на месте. Голос: *«DELETE — 200 OK! А объект на месте! API врёт! DELETE не работает!»*

**Хук:** «DELETE → 200 OK. Объект на месте. Твой API врёт тебе в лицо.»

**Боль зрителя:** QA смотрит на зелёный статус и идёт дальше. Баг остаётся в системе. Классический fake success.

**Формат:** 5 минут

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход: "Ок, погнали с начала" | — |
| 0:10 | Интро: DELETE → 200, затем GET — объект жив | "DELETE. 200 OK. Всё зелёное. А теперь GET того же объекта. Он жив. API сказал 'да', но ничего не сделал. Что такое DELETE в REST API, как он должен работать и как тестировать, чтобы не пропустить fake success — смотрим на реальном примере." |
| 0:50 | Схема: что должно происходить vs что происходит | "Нормальный DELETE: удалил объект → вернул 200 или 204. Бажный: вошёл в метод → вернул 200 → даже не тронул базу." |
| 1:20 | Postman: демонстрация | "Отправляю DELETE на /objects/42. Ответ: 200 OK. Тело пустое. Визуально — всё ок. Теперь GET /objects/42 — и объект на месте." |
| 2:00 | Почему это опаснее, чем 500 | "500 — ты хотя бы знаешь, что что-то сломалось. А 200 с невыполненным действием — это тихий баг. Автотесты пройдут, CI зелёный, а данные висят." |
| 2:45 | Как найти такой баг | "Правило: после мутирующего запроса — всегда читающий запрос. POST → GET. PUT → GET. DELETE → GET. Проверяй состояние." |
| 3:30 | Почему такое вообще происходит | "Причины: разработчик забыл добавить вызов БД, middleware перехватил запрос, ошибка в транзакции, которую съел try-catch." |
| 4:00 | Демонстрация: 204 vs 200 | "Кстати, правильный ответ для DELETE без тела — 204 No Content. Если видишь 200 — задай вопрос. А что именно удалилось?" |
| 4:30 | Аутро | "Мораль: зелёный статус — не повод расслабляться. Проверяй состояние. Следующий выпуск: GET, который возвращает чужой объект." |

**Ключевой инсайт:** DELETE с 200 OK — это красный флаг, а не зелёный свет.

**Теги:** `api тестирование`, `delete запрос`, `баги api`, `postman`, `qa`, `rest api`, `тестирование api с нуля`, `статус 200`

## Эпизод 3: GET по id возвращает не тот объект

**Cold Open (0:00-0:05):** Экран: GET /objects/3 в Postman. Ответ приходит с id=2. Голос: *«GET /objects/3 — получил объект #2! Off-by-one! API возвращает не те данные!»*

**Хук:** «GET /objects/3 вернул объект #2. Off-by-one — баг, который живёт в каждом втором API.»

**Боль зрителя:** Ты протестировал создание, удаление, а простейший GET по id даёт не те данные. И найти это можно только случайно.

**Формат:** 5 минут, акцент на внимательность к деталям

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход: "Давай по порядку" | — |
| 0:10 | Интро: GET /objects/3 → объект #2 | "GET /objects/3. Ответ: {... id: 2 ...}. Я просил третий — получил второй. Что такое off-by-one, почему GET-запросы этим страдают и как тестировать, чтобы найти — разбираем на реальном примере." |
| 0:40 | Что такое off-by-one | "Ошибка на единицу. В коде написали `id - 1` или `index = id - 1` вместо `index = id`. И каждый запрос смещается на один." |
| 1:00 | Демо: проверяем несколько id | "GET /objects/1 → объект 0 (или ошибка). GET /objects/2 → объект 1. GET /objects/3 → объект 2. Закономерность видна, когда проверишь несколько." |
| 1:45 | Ошибка новичка: проверить один id | "Новичок создаст объект, получит id, сделает GET по этому id — и всё сойдётся. Потому что `id == index` только для последнего созданного. А для всех остальных — мимо." |
| 2:30 | Сетка тестов: как найти системно | "Проверь минимум 3 разных id. Не только 'свой'. Создай 5 объектов, удали один, проверь GET соседних. Проверь id=0, id=MAX, id=-1." |
| 3:15 | Почему это опасно | "Представь: API банка. GET /accounts/102 → баланс клиента #101. Ты видишь чужие деньги. Или API документов — ты читаешь чужой договор." |
| 4:00 | Boundary values | "Это классика boundary value analysis. id=1, последний id, id+1. Если тестируешь только 'среднее' — пропустишь off-by-one." |
| 4:30 | Аутро | "GET — самый простой запрос. Но именно в нём прячется off-by-one. Правило: проверяй минимум 3 точки — начало, конец, середина. Следующий выпуск: POST, который принимает age=-5." |

**Ключевой инсайт:** Достаточно проверить больше одного id — и баг найдется. Не тестируй только "свой" объект.

**Теги:** `api тестирование`, `get запрос`, `off by one`, `баги api`, `boundary values`, `qa для начинающих`, `rest api`

## Эпизод 4: POST, который пускает age=-5 (или падает с 500)

**Cold Open (0:00-0:05):** Экран: Postman, POST c body `{"age": -5}`. Ответ 201 Created. Снизу — запись в БД с age=-5. Голос: *«Возраст -5! 201 Created! Валидация у API сломана! Впускаем отрицательных пользователей!»*

**Хук:** «Я отправил возраст -5. API ответил 201 Created. Теперь в системе живет пользователь, который родился через 5 лет.»

**Боль зрителя:** Новички тестируют только happy path. А дыры в валидации — это то, что роняет продакшн.

**Формат:** 5 минут, акцент на валидацию

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход: "А теперь по фактам" | — |
| 0:10 | Интро: age=-5 → 201 Created | "POST /objects, body: { age: -5 }. 201 Created. Минус пять лет. API принял это как норму. Что такое валидация в POST-запросах, как она должна работать и как тестировать, чтобы не пропустить — на реальных плохих и хороших примерах." |
| 0:40 | Ещё граничные случаи | "А теперь age=0. 200. age=999. 200. age='abc'. А вот и 500 Internal Server Error — приложение упало." |
| 1:00 | Классификация: два типа багов | "Тут два бага в одном: 1. Отсутствие валидации — пускает отрицательные значения. 2. Отсутствие обработки ошибок — падает с 500." |
| 1:45 | Демо: тестируем валидацию системно | "Тестируем: отрицательное, ноль, граничное (0 и 1), очень большое, строка, null, пустое тело. Семь тестов — три бага." |
| 2:30 | Почему 500 — это не просто "упало" | "500 в ответ на 'abc' — это stack trace наружу. Разработчик не обернул парсинг числа в try-catch. Если это продакшен — пользователь видит жёлтый экран смерти." |
| 3:15 | Как правильно должно быть | "Правильный API: age < 0 → 400 Bad Request + сообщение 'age must be positive'. age='abc' → 400 + 'age must be a number'. Никаких 500." |
| 3:50 | Чек-лист для тестирования любого поля | "Для любого числового поля проверь: отрицательное, ноль, граничное значение, очень большое, нечисловое, null, отсутствие поля." |
| 4:30 | Как оформить баг-репорт | "Баг-репорт: 'API принимает age=-5 и age=999. Ожидаемо: 400 Bad Request с описанием, что age должен быть в диапазоне 0-150.'" |
| 5:00 | Аутро | "Валидация — это первая линия обороны. Если API не валидирует вход — жди беды. Следующий выпуск: поиск, который работает только с большой буквы." |

**Ключевой инсайт:** 7 простых тестов на одно поле — и уже нашёл баг в валидации. Happy path тестирования недостаточно.

**Теги:** `api тестирование`, `валидация api`, `post запрос`, `баги api`, `error handling`, `400 bad request`, `тестирование с нуля`

## Эпизод 5: GET /search?q=john не находит John (case-sensitive)

**Cold Open (0:00-0:05):** Экран: два запроса в Postman — GET /search?q=john → пустой массив. GET /search?q=John → 10 результатов. Голос: *«Ищу john — пусто! Ищу John — 10 результатов! Поиск сломан! Регистр букв решает всё!»*

**Хук:** «Ты ищешь 'john' — пустой ответ. Ищешь 'John' — 10 результатов. Добро пожаловать в case-sensitive баги.»

**Боль зрителя:** Тестировщик проверил поиск по одному значению — всё работает. А регистр букв не учёл — и пользователи не находят то, что ищут.

**Формат:** 5 минут

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход: "Ок, с начала" | — |
| 0:10 | Интро: два поиска рядом | "Один и тот же запрос. Отличается только буква — регистр решил судьбу поиска. Что такое case-sensitive поиск, почему это проблема и как тестировать фильтры, чтобы пользователи находили то, что ищут — на реальном примере." |
| 0:40 | Как должно работать vs как работает | "В реальном мире пользователь вбивает 'john' с маленькой. Потому что лень нажимать Shift. И получает пустоту." |
| 1:00 | Postman: демонстрация | "John → 10 результатов. john → 0. J0HN → 0 (ноль вместо буквы o). john@email → 0. Только точное совпадение по регистру проходит." |
| 1:45 | Почему это баг, а не фича | "Фича — если это поиск по коду, где A и a разные переменные. Баг — если это поиск по имени, названию, адресу. Пользователь не знает, в каком регистре данные в БД." |
| 2:30 | Как тестировать поиск | "Минимальный чек-лист: точное совпадение, lowercase, UPPERCASE, Mixed Case, частичное совпадение, спецсимволы, пробелы в начале/конце." |
| 3:15 | Пример из жизни | "В одном проекте поиск товаров не находил 'чайник' — только 'Чайник'. Три дня пользователи писали в поддержку 'у вас нет чайников'. А они были." |
| 4:00 | Ожидаемое поведение | "Правильный API: ILIKE или LOWER() в БД, toLowerCase() на бэке, или full-text search. Ответ одинаковый для 'john' и 'John'." |
| 4:30 | Аутро | "Проверь поиск в нижнем регистре. Если результатов нет — это баг. Один символ регистра отличает работающий поиск от сломанного." |

**Ключевой инсайт:** Поиск надо проверять минимум в трёх регистрах. Если lower case не работает — это баг, а не особенность.

**Теги:** `api тестирование`, `поиск api`, `case sensitive`, `баги api`, `get запрос`, `qa`, `rest api`, `тестирование поиска`

## Формат видео (шаблон)

**Длительность:** 5 минут
**Тон:** Разговорный, динамичный, без воды
**Визуал:** Postman (60% экрана) + схема/аннотации (40%)
**Голос:** Эдди, от первого лица, как объяснение коллеге

**Структура каждого выпуска:**
0. **Cold Open** (0:00-0:05) — скринкаст обнаружения бага + голосовая реакция с фразой из заголовка. Сразу в гущу событий.
1. **Переход** (0:05-0:10) — чёрный экран / монтажная склейка. «Ок, давай с начала»
2. **Введение** (0:10-0:40) — что это за штука (POST/DELETE/GET/search), как должна работать, как тестировать. Плавный старт с контекстом.
3. **Демо бага** (0:40-2:00) — показываем в Postman, что именно сломано, на плохих и хороших примерах
4. **Разбор** (2:00-3:30) — почему это баг, как возникает, чем опасен в продакшене
5. **Чек-лист** (3:30-4:30) — как тестировать этот случай системно
6. **Аутро** (4:30-5:00) — вывод + тизер следующего выпуска + призыв подписаться

**Постоянные элементы:**
- Cold open: обязательно фраза из заголовка + слово "сломано/не работает/баг"
- На протяжении: ссылка на практикум (сайт для тестирования)
- В конце: "А ты встречал такой баг? Пиши в комментарии"
