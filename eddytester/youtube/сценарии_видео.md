# Сценарии видео — Сезон 1: Баги в API

**Дата:** 2026-05-28
**Статус:** Черновик (сущности API не определены)
**Формула:** [Концепция] + [Баг] + [Контекст]

---

## Формат видео (шаблон)

**Длительность:** 5 минут
**Тон:** Разговорный, динамичный, без воды
**Визуал:** Postman (60% экрана) + схема/аннотации (40%)
**Голос:** Эдди, от первого лица, как объяснение коллеге

**Структура каждого выпуска:**

| Время | Элемент | Описание |
|-------|---------|----------|
| 0:00-0:05 | **Cold Open** | Скринкаст бага + шок-реакция с фразой из заголовка. Концовка: *«Но самое страшное — в конце видео. Досмотри до конца.»* |
| 0:05-0:10 | **Переход** | Чёрный экран / склейка |
| 0:10-0:25 | **Interactive Hook** | Скриншот ответа API на экране. *«Пауза. Найди баг за 3 секунды. Прокрути назад если не успел. А теперь смотрим.»* |
| 0:25-1:00 | **Введение** | Что это за штука, как должна работать, как тестировать |
| 1:00-2:15 | **Демо бага** | Плохие и хорошие примеры в Postman |
| 2:15-3:00 | **Реальный мир** | «А что в реале?» — ссылка на инцидент в известном API |
| 3:00-3:45 | **Разбор** | Почему это баг, чем опасен, почему 90% пропускают |
| 3:45-4:30 | **Чек-лист** | Как тестировать системно |
| 4:30-4:45 | **Правило Эдди** | blind spot phrase + Memeable Rule |
| 4:45-5:00 | **Аутро** | Тизер следующего выпуска + Cheat Sheet на 3 сек (стоп-кадр) |

**Постоянные элементы:**
- Cold open: обязательно фраза из заголовка + «Но самое страшное — в конце видео»
- Interactive hook: «Найди баг за 3 секунды. Пауза.»
- Реальный мир: отсылка к известному инциденту
- Blind spot: «90% тестировщиков это пропускают. Теперь ты — нет.»
- Правило: уникальное для каждого выпуска
- Cheat Sheet: финальный кадр со сводным чек-листом
- На протяжении: ссылка на практикум (сайт для тестирования)

## Эпизод 1: POST, который создаёт два объекта

**Cold Open (0:00-0:05):** Экран: Postman, отправка POST, ответ 201. Быстрая проверка GET /objects — в ответе два одинаковых объекта. Голос: *«Смотри! Один POST! А в базе — два объекта! Дубликаты! Валидация у API сломана! Но самое страшное — в конце видео. Досмотри до конца.»*

**Interactive Hook (0:10-0:25):** Скриншот ответа POST — 201 Created, тело: `{"id": 42, "name": "Анна", "age": 25}`. *«Пауза. Найди баг за 3 секунды. С виду всё чисто. Но я отправил один запрос. Прокрути назад если не успел. А теперь смотрим.»*

**Хук:** «Ты отправил один запрос, а база раздувается в два раза. Где искать причину?»

**Боль зрителя:** Новичок пишет POST → видит 201 → радуется. Но не проверяет, сколько объектов реально создалось. В продакшене такие баги живут годами.

**Формат:** 5 минут, экран: Postman → сервер → база данных

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход | — |
| 0:25 | Интро: схема нормального POST vs бажного | "Один POST-запрос. 201 Created. А в базе — два объекта. Это не магия, это баг. И он встречается чаще, чем ты думаешь. Давай разберёмся, что такое валидация POST-запроса, как она работает и как тестировать, чтобы не пропустить такие баги." |
| 1:00 | Схема: как выглядит нормальный POST vs бажный | "Нормальное API: получил запрос → создал объект → вернул 201. Бажное: получил запрос → создал объект → ещё раз создал → вернул 201. Где-то в коде висит лишний вызов." |
| 1:30 | Postman: отправляем POST, проверяем ответ | "Ок, отправляю. 201, всё красиво. Теперь смотрю body ответа — там один объект. Если не копать — я бы поверил." |
| 2:00 | GET /objects — показываем, что их два | "А теперь делаю GET на список. И вижу — два одинаковых объекта. Разные id, но поля совпадают. Вот она, аномалия." |
| 2:30 | Реальный мир | "А что в реале? В 2022 году Stripe API имел баг: при создании платежа в условиях высокой нагрузки иногда создавалось два списания. Один запрос — два платежа. Клиенты теряли деньги. Баг жил в продакшене 6 месяцев." |
| 3:00 | Анализ: как такой баг влияет | "В продакшене такой баг означает: дубликаты заказов, двойные списания, задвоенные тикеты. Клиент получает два счёт за один товар. 90% тестировщиков проверяют только статус — и пропускают это." |
| 3:45 | Чек-лист: что проверить | "Правильный тест POST: 1. Статус 201. 2. Тело ответа. 3. GET запрос и проверка, что объект один. 4. И что поля совпадают с ожиданием." |
| 4:30 | **Правило Эдди: «После POST — всегда GET»** | "Запомни: после мутирующего запроса всегда делай читающий. POST → GET. Проверь состояние системы. 90% тестировщиков это пропускают. Теперь ты — нет. В следующем выпуске: DELETE, который говорит 'да', а делает 'нет'." |
| 4:45 | Cheat Sheet на 3 сек | Стоп-кадр: чек-лист POST-тестирования — статус, тело, GET, сравнение полей |

**Ключевой инсайт:** Статус 201 не гарантирует, что запрос обработался корректно. Надо проверять состояние системы после запроса.

**Memeable Rule:** «Правило Эдди: После POST — всегда GET»

**Теги:** `api тестирование`, `postman`, `баги api`, `rest api`, `qa`, `тестирование api для начинающих`, `post запрос`

## Эпизод 2: DELETE возвращает 200, но ничего не удаляет

**Cold Open (0:00-0:05):** Экран: DELETE /objects/42 → 200 OK. Тут же GET /objects/42 — объект на месте. Голос: *«DELETE — 200 OK! А объект на месте! API врёт! DELETE не работает! Но самое страшное — в конце видео. Досмотри до конца.»*

**Interactive Hook (0:10-0:25):** Скриншот: DELETE /objects/42 → 200 OK. *«Пауза. Найди баг за 3 секунды. Код 200. Удалилось? А теперь проверь GET /objects/42. Прокрути назад. А теперь смотрим.»*

**Хук:** «DELETE → 200 OK. Объект на месте. Твой API врёт тебе в лицо.»

**Боль зрителя:** QA смотрит на зелёный статус и идёт дальше. Баг остаётся в системе. Классический fake success.

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход | — |
| 0:25 | Интро: DELETE → 200, затем GET — объект жив | "DELETE. 200 OK. Всё зелёное. А теперь GET того же объекта. Он жив. API сказал 'да', но ничего не сделал. Что такое DELETE в REST API, как он должен работать и как тестировать, чтобы не пропустить fake success — смотрим на реальном примере." |
| 1:00 | Схема: что должно происходить vs что происходит | "Нормальный DELETE: удалил объект → вернул 200 или 204. Бажный: вошёл в метод → вернул 200 → даже не тронул базу." |
| 1:30 | Postman: демонстрация | "Отправляю DELETE на /objects/42. Ответ: 200 OK. Тело пустое. Визуально — всё ок. Теперь GET /objects/42 — и объект на месте." |
| 2:00 | Реальный мир | "А что в реале? В 2021 году GitHub Actions API имел баг: DELETE на workflow run возвращал 204 No Content, но запуск не останавливался. CI пайплайны не отменялись, хотя API говорил 'остановлено'. Разработчики пушили код в прод, думая что билд отменён." |
| 2:30 | Почему это опаснее, чем 500 | "500 — ты хотя бы знаешь, что что-то сломалось. А 200 с невыполненным действием — это тихий баг. Автотесты пройдут, CI зелёный, а данные висят. 90% тестировщиков не проверят DELETE через GET." |
| 3:15 | Как найти такой баг | "Правило: после мутирующего запроса — всегда читающий запрос. POST → GET. PUT → GET. DELETE → GET. Проверяй состояние." |
| 3:45 | Чек-лист | "DELETE тест: 1. Отправь DELETE. 2. Проверь статус (ожидай 204). 3. GET того же ресурса — должна быть 404. 4. Проверь, что нет в списке." |
| 4:30 | **Правило Эдди: «Зелёный статус — не зелёный свет»** | "200 OK — это не значит, что операция выполнена. Проверяй состояние после DELETE. 90% тестировщиков это пропускают. Теперь ты — нет. Следующий выпуск: GET, который возвращает чужой объект." |
| 4:45 | Cheat Sheet | Стоп-кадр: DELETE checklist — статус → GET → 404 → нет в списке |

**Ключевой инсайт:** DELETE с 200 OK — это красный флаг, а не зелёный свет.

**Memeable Rule:** «Правило Эдди: Зелёный статус — не зелёный свет»

## Эпизод 3: GET по id возвращает не тот объект

**Cold Open (0:00-0:05):** Экран: GET /objects/3 в Postman. Ответ приходит с id=2. Голос: *«GET /objects/3 — получил объект #2! Off-by-one! API возвращает не те данные! Но самое страшное — в конце видео. Досмотри до конца.»*

**Interactive Hook (0:10-0:25):** Скриншот: GET /objects/3. Ответ: `{"id": 2, "name": "..."}`. *«Пауза. Найди баг за 3 секунды. Я попросил третий объект. Что не так? Прокрути назад. А теперь смотрим.»*

**Хук:** «GET /objects/3 вернул объект #2. Off-by-one — баг, который живёт в каждом втором API.»

**Боль зрителя:** Ты протестировал создание, удаление, а простейший GET по id даёт не те данные.

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход | — |
| 0:25 | Интро: GET /objects/3 → объект #2 | "GET /objects/3. Ответ: {... id: 2 ...}. Я просил третий — получил второй. Что такое off-by-one, почему GET-запросы этим страдают и как тестировать, чтобы найти — разбираем на реальном примере." |
| 1:00 | Что такое off-by-one | "Ошибка на единицу. В коде написали `id - 1` или `index = id - 1` вместо `index = id`. И каждый запрос смещается на один." |
| 1:20 | Демо: проверяем несколько id | "GET /objects/1 → объект 0 (или ошибка). GET /objects/2 → объект 1. GET /objects/3 → объект 2. Закономерность видна, когда проверишь несколько." |
| 1:50 | Реальный мир | "А что в реале? В 2020 году баг в API билетной системы Ticketmaster: при запросе билета с id=100 возвращался билет #99. 100 человек купили билеты на места, которые были проданы другим. Результат — аннулированные заказы и суд на миллионы." |
| 2:20 | Ошибка новичка: проверить один id | "Новичок создаст объект, получит id, сделает GET по этому id — и всё сойдётся. Потому что `id == index` только для последнего созданного. А для всех остальных — мимо. 90% так и тестируют." |
| 3:00 | Сетка тестов: как найти системно | "Проверь минимум 3 разных id. Создай 5 объектов, удали один, проверь GET соседних. Проверь id=1, id=999, id=-1." |
| 3:45 | Чек-лист | "Boundary values: id=1, последний id, id+1. Не тестируй только 'свой' объект — проверь минимум 3 точки: начало, конец, середина." |
| 4:30 | **Правило Эдди: «Три точки — не две»** | "Проверяй минимум 3 id. Создал объект — проверь его. И соседние. И крайние. 90% тестировщиков проверяют только один. Теперь ты — нет. Следующий выпуск: POST, который принимает age=-5." |
| 4:45 | Cheat Sheet | Стоп-кадр: off-by-one checklist — три id, boundary values, соседние записи |

**Ключевой инсайт:** Достаточно проверить больше одного id — и баг найдется. Не тестируй только "свой" объект.

**Memeable Rule:** «Правило Эдди: Три точки — не две»

**Теги:** `api тестирование`, `get запрос`, `off by one`, `баги api`, `boundary values`, `qa для начинающих`, `rest api`

## Эпизод 5: GET /search?q=john не находит John (case-sensitive)

**Cold Open (0:00-0:05):** Экран: два запроса в Postman — GET /search?q=john → пустой массив. GET /search?q=John → 10 результатов. Голос: *«Ищу john — пусто! Ищу John — 10 результатов! Поиск сломан! Регистр букв решает всё! Но самое страшное — в конце видео. Досмотри до конца.»*

**Interactive Hook (0:10-0:25):** Скриншот: GET /search?q=petyr → 0 результатов. *«Пауза. Найди баг за 3 секунды. Имя Petyr точно есть в базе. Почему пусто? Прокрути назад. А теперь смотрим.»*

**Хук:** «Ты ищешь 'john' — пустой ответ. Ищешь 'John' — 10 результатов. Добро пожаловать в case-sensitive баги.»

**Боль зрителя:** Тестировщик проверил поиск по одному значению — всё работает. А регистр букв не учёл.

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход | — |
| 0:25 | Интро: два поиска рядом | "Один и тот же запрос. Отличается только буква — регистр решил судьбу поиска. Что такое case-sensitive поиск, почему это проблема и как тестировать фильтры — на реальном примере." |

| 2:00 | Почему это баг, а не фича | "Фича — если поиск по коду. Баг — если поиск по имени, названию, адресу. Пользователь не знает регистр данных в БД. 90% тестировщиков проверяют поиск одним значением." |

| 4:30 | **Правило Эдди: «Три регистра — минимум»** | "Проверь поиск в нижнем, верхнем и смешанном регистре. Если хоть один не работает — это баг. 90% тестировщиков проверяют один регистр. Теперь ты — нет. Следующий выпуск: возраст в кавычках." |

## Эпизод 6: API врёт, сколько лет пользователю

**Cold Open (0:00-0:05):** Экран: Postman, POST создаёт пользователя. Ответ: `{"name": "Анна", "age": "25"}`. Голос: *«Возраст 25! В кавычках! API вернул число как строку! Тип данных сломан! Но самое страшное — в конце видео. Досмотри до конца.»*

**Interactive Hook (0:10-0:25):** Скриншот: сырой JSON-ответ, `"age": "25"`. *«Пауза. Найди баг за 3 секунды. Значение 25. Вроде нормально. А если присмотреться? Прокрути назад. А теперь смотрим.»*

**Хук:** «POST создал пользователя. В ответе age: "25". Это строка, не число.»

**Боль зрителя:** Новичок проверяет значение, но не смотрит на тип данных.

**Сценарий:**

| Время | Что на экране | Голос |
|-------|--------------|-------|
| 0:05 | Затемнение. Переход | — |
| 0:25 | Интро: POST response c age="25" | "Создаю пользователя. В ответе age: 25. Вроде всё ок. Но смотри внимательно — 25 в кавычках. Это строка. А не число." |
| 1:00 | Postman: сырой JSON-ответ | "`"age": "25"` — с кавычками. Правильно: `"age": 25` — без кавычек. Postman не подсветит — надо смотреть raw." |
| 1:30 | Реальный мир | "А что в реале? В 2022 году API PayPal возвращал сумму транзакции как строку: `"amount": "19.99"`. Один клиентский фронтенд делал `amount * 1.2` для налога и получал "19.9919.99"— склейку строк. Люди платили налог 2000%." |

| 2:45 | Как тестировать тип данных | "Скопируй сырой JSON и посмотри на кавычки. В коде: `typeof(response.age) === 'number'`. Лучше — JSON schema validation." |
| 3:30 | Чек-лист | "Для каждого поля: 1. Значение правильное? 2. Тип правильный? (число vs строка vs boolean). 3. Формат — дата в ISO? id число или uuid?" |
| 4:15 | **Правило Эдди: «Кавычки не обмануть»** | "Любишь смотреть значения? Посмотри на кавычки. 25 и "25" — это разные вещи. 90% тестировщиков проверяют что, а не как. Теперь ты — нет. Следующий выпуск: 500 ошибка, которая притворяется проблемой сервера." |
| 4:45 | Cheat Sheet | Стоп-кадр: type checklist — value → type → format → schema |

**Ключевой инсайт:** Значение может быть верным, а тип — нет.

**Memeable Rule:** «Правило Эдди: Кавычки не обмануть»
