## Формат №1: «Сериал: тестируем банковский API»

### Суть
Сквозной сюжет на 10 серий. Один проект, одна команда, один API. Аудитория следит за историей и ждёт продолжение.

### Механика
Сериал. Каждый пост — серия. Интрига в конце. Завязка → баги → конфликты → кульминация → финал.

### Зачем
Вовлечение через нарратив. Аудитория ждёт следующий пост. Комментарии — обсуждение сюжета.

### Когда использовать
1 серия в день, 10 дней подряд. Раз в 1-2 месяца.

### CTA
«➕ если ждёте продолжение» — в каждом посте. В финале — прямая ссылка на бот Practicum.

### Готовые посты (4 из 10)

**Серия 1/10: «Мне дали API банка. Я в панике»**

Утро. Понедельник. Тикет: «Проверить новый API платёжного шлюза. Дедлайн — пятница».

Документация — PDF на 47 страниц. Авторизация, переводы, балансы, история. Выглядит как свадебный альбом — красиво, но бесполезно.

Открываю Postman. Первый запрос: `GET /health`.

`200 OK`. Живой. Начинаем.

В этом сериале: 10 серий, 1 API банка, куча багов.

➕ если ждёте продолжение

**Серия 2/10: «Первый баг за 3 минуты»**

`POST /api/v1/transfer`. Документация: `{ "from": "uuid", "to": "uuid", "amount": 100.50 }`

Отправляю: `{ "from": "alice-uuid", "to": "bob-uuid", "amount": -5000 }`

Сервер: `{ "status": "success", "transaction_id": "tx-123" }`

Я перевёл -5000 рублей. Отрицательную сумму. Сервер сказал «успешно».

Для банка — не баг, а дыра. Можно «отрицательным переводом» накрутить баланс.

Как думаете — баг или фича?

**Серия 7/10: «400 Bad Request с улыбкой»**

День четвёртый. API ломается каждые 2 запроса.

`POST /api/v1/pay` с правильным телом. Сервер:

`{ "error": "Bad Request", "message": "Invalid request" }`

Какое поле невалидное? Не сказано. Что не так? Непонятно.

Смотрю код — `message` хардкодом стоит для любого 400.

Ребята, если ваш API возвращает 400 — скажите, **что именно** не так. Не «Invalid request». А `"field: amount, reason: exceeds daily limit"`.

Бывает такое?

**Серия 10/10: «23 бага за 5 дней»**

Финальная серия.

За 5 дней тестирования одного API банка:
- 23 бага (8 security, 7 functional, 5 validation, 3 документация)
- 1 мог привести к потере денег (отрицательный перевод)
- Авторизация: токен живёт 7 дней даже после logout
- Rate limiting: 10000 запросов в минуту — DDoS делай

Что вынес:
1. Документация врёт — проверяй каждый пункт
2. Security-баги — самые тихие
3. 400 без описания — не баг, а преступление

Хотите такой же опыт? В Practicum — 15+ багов в живом API. 2 задачи бесплатно. Ссылка в шапке.

## Формат №2: «API-шпаргалка: 3 строки, как в проде»

### Суть
Одна операция API → три рабочие команды curl/jq/grep. Скопировал → вставил → запустил.

### Механика
Минимум текста. Три команды + пояснение. Без Postman.

### Зачем
Рабочий инструмент, а не теория. Можно сохранить себе.

### Когда использовать
2-3 раза в неделю.

### CTA
«Кидайте свои — соберу большую шпаргалку»

### Готовые посты (4 шт.)

**Пост №1: «Как проверить, жив ли API, за 3 секунды»**

Не открывай браузер. Не логинься. Одна команда:

```bash
curl -s -o /dev/null -w "%{http_code}" https://api.example.com/health
```

Вернул `200` — жив. `503` — мёртв. `000` — сервер не отвечает.

А если API не отдаёт status endpoint:

```bash
curl -s -I https://api.example.com/health | grep "200"
```

Для тех, кто любит цифры:

```bash
curl -s -o /dev/null -w "Time: %{time_total}s, Code: %{http_code}\n" https://api.example.com/health
```

Покажет и статус, и сколько сервер думал.

Скопируйте и запустите на своём API. Кидайте свои команды.

**Пост №2: «Как проверить, врёт ли сервер про Content-Type»**

Было сто раз: фронт ждёт JSON, а приходит HTML.

```bash
curl -s -I https://api.example.com/users/1 | grep -i "content-type"
```

Должно быть `application/json`. Если `text/html` — сервер врёт.

А теперь с телом запроса — когда сервер должен вернуть ошибку:

```bash
curl -s -H "Content-Type: application/json" -X POST https://api.example.com/login -d '{"u":"a","p":"b"}' | head -c 100
```

Если кусок ответа начинается с `<html>` — проблема на бэкенде.

Кидайте свои команды — соберу шпаргалку.

**Пост №3: «Rate limiting: с какого запроса тебя забанят»**

```bash
curl -s -I https://api.example.com/endpoint | grep -i "rate"
```

Видишь `X-RateLimit-Limit: 100` — можешь дёргать 100 раз в минуту. Нет заголовка — лимит есть, но о нём молчат.

Проверь границу:

```bash
for i in {1..120}; do curl -s -o /dev/null -w "$i: %{http_code}\n" https://api.example.com/endpoint; done
```

Смотри номер, после которого код сменился с `200` на `429`. Это твой лимит.

А `Retry-After` есть?

```bash
curl -s -I https://api.example.com/endpoint | grep -i "retry"
```

Скопируйте, запустите.

**Пост №4: «Как найти баг в CORS за 10 секунд»**

Фронт не достучаться до API? 90% — CORS.

```bash
curl -s -H "Origin: https://evil-site.com" -I https://api.example.com/endpoint | grep -i "access-control"
```

`Access-Control-Allow-Origin: *` — твой API может читать любой сайт. Security-баг.

Нет заголовка вообще — CORS не настроен.

Проверка сложного сценария:

```bash
curl -s -H "Origin: https://example.com" -H "Access-Control-Request-Method: DELETE" -X OPTIONS https://api.example.com/endpoint | grep -i "access-control"
```

Нет `DELETE` в `Allow-Methods` — разработчики забыли.

Скопируйте, запустите.

## Формат №3: «Тестировщик VS Разработчик: чья версия правда?»

### Суть
Кейс из прода/собеса → две версии → доказательство данными.

### Механика
Конфликт и разбор. Показать, как доказывать правду инструментами.

### Зачем
Снимает синдром самозванца. Холивар в комментариях. Шоу-донт-телл.

### Когда использовать
1-2 раза в неделю.

### CTA
«А у вас такое было?» / «Кидайте кейсы»

### Готовые посты (4 шт.)

**Пост №1: «401 Unauthorized — и кто прав?»**

Кейс: POST /api/login, токен есть, API возвращает 401.

**Тестировщик:** «Это баг. Мы авторизованы».
**Разработчик:** «401 — стандартный код. Всё ок».

**Доказательство:**

1. `curl -v -H "Authorization: Token abc123" https://api.example.com/protected`
   → В логах: `token = null`

2. Код бэкенда:
   ```python
   auth_header = request.headers.get("Authorization")
   token = auth_header.replace("Bearer ", "")
   ```
   — Парсер ждёт `Bearer`, фронт шлёт `Token`.

3. `git blame` — 2 года назад переписали формат, парсер не обновили.

**Вердикт:** Прав разработчик. 401 корректный. Но баг есть — несоответствие формата.

**Пост №2: «500 на логине — чей баг?»**

Кейс: POST /api/login, пустое тело → 500.

**Тестировщик:** «Баг. Нет обработки пустого тела».
**Разработчик:** «Кто отправляет пустое тело? Фронт должен валидировать».

**Доказательство:**

1. `curl -v -X POST https://api.example.com/login -d '{}'`
   → 200. Пустой JSON съел.

2. `curl -v -X POST https://api.example.com/login -d ''`
   → 500. Логи: `NullPointerException: 'email' is null`.

3. Должен быть: пустое тело → 400.

**Вердикт:** Прав тестировщик. Сервер должен возвращать 400, а не 500. Никто не гарантирует, что клиент не отправит пустоту.

**Пост №3: «DELETE возвращает 200 — баг или стиль?»**

Кейс: DELETE /api/users/123 → 200 OK, пустое тело.

**Тестировщик:** «RFC 7231: DELETE должен возвращать 204».
**Разработчик:** «Работает же. 200 — успех».

**Доказательство:**

1. `curl -v -X DELETE https://api.example.com/users/123`
   → 200, тело пустое.

2. `curl -v -X DELETE https://api.example.com/users/999`
   → 200. Несуществующий — тоже 200.

3. RFC 7231 §4.3.5: сервер ДОЛЖЕН отправить 204.

**Вердикт:** Прав тестировщик. 200 вместо 204 маскирует проблемы — например, DELETE несуществующего.

**Пост №4: «Age = 150 — почему это баг, а не фича?»**

Кейс: POST /api/users, `age: 150` → 200 OK.

**Тестировщик:** «Баг. 150 — невалид».
**Разработчик:** «Может, возраст организации. У нас нет ограничения».

**Доказательство:**

1. `curl -s -X POST /api/users -d '{"age":150}'` → 200, создан.
2. `curl -s -X POST /api/users -d '{"age":-5}'` → 200, создан.

Отрицательный возраст прошёл. Это уже не «организация».

3. Продакт: «Максимум 150, но мы не проверяем».

**Вердикт:** Баг тестировщика. Валидация границ — базовая история. Сервер должен проверять: age >= 0 и age <= 150.

## Формат №4: «Угадай баг» / «Две правды, одна ложь»

### Суть
3 утверждения — 2 правдивых, 1 ложный. Аудитория угадывает.

### Механика
Пост с А, Б, В. Один — баг. Через 2 часа — правильный ответ.

### Зачем
Низкий порог входа. Холивар. Обучение через игру.

### Когда использовать
1 раз в неделю.

### CTA
«Пишите букву» / «🔥 если знали, ➕ если нет»

### Готовые посты (4 шт.)

**Пост №1: «Какой из этих статус-кодов — баг?»**

[A] POST /api/users с правильным телом → `201 Created`

[Б] GET /api/users/999 (не существует) → `200 OK`, тело = `{}`

[В] DELETE /api/users/123 → `204 No Content`

Какой — баг? Пишите букву.

Правильный ответ через 2 часа.

**Пост №2: «Три запроса, один баг»**

[A] `curl -X POST /api/transfer -d '{"amount": 100}'` → 200, деньги переведены

[Б] `curl -X POST /api/transfer -d '{"amount": -100}'` → 200, деньги... переведены?

[В] `curl -X POST /api/transfer -d '{"amount": 0}'` → 400

Где баг? Подсказка: один ответ правильный, но не все.

**Пост №3: «Кто из трёх разработчиков прав?»**

API возвращает 500 на пустое тело.

Разработчик А: «Баг. Сервер падает».
Разработчик Б: «Фича. Клиент должен слать валидные данные».
Разработчик В: «500 — ок, просто логи добавь».

Кто прав? Кто нет? Почему?

**Пост №4: «Найди баг в документации»**

Документация API:
- POST /users → 201
- GET /users/:id → 200
- DELETE /users/:id → 200
- PATCH /users/:id → 200

Где баг? Пишите.

## Формат №5: «Исправь меня»

### Суть
Намеренно неполный/неверный чек-лист. Аудитория дописывает.

### Механика
Чек-лист с одной ошибкой/пропуском. Аудитория находит и дописывает.

### Зачем
Ego boost — люди любят указывать на ошибки. Краудсорсинг знаний.

### Когда использовать
1 раз в 2 недели.

### CTA
«Что я упустил?» / «Дополните» / «Полный чек-лист — в боте Practicum»

### Готовые посты (4 шт.)

**Пост №1: «Чек-лист POST /users — что я упустил?»**

Вот как я проверяю создание пользователя:
1. POST с валидным телом → 201
2. POST без авторизации → 401
3. POST с пустым телом → 400

Что я пропустил? 👇
*Пропущено: дубликат email (409), XSS в name, граничные age=150.*

**Пост №2: «Чек-лист DELETE — что я упустил?»**

Как я проверяю удаление:
1. DELETE /users/123 → 204
2. DELETE /users/999 → 404
3. DELETE без авторизации → 401

Что я пропустил? 👇
*Пропущено: GET после DELETE (реально ли удалён), каскадное удаление.*

**Пост №3: «Чек-лист пагинации — что я упустил?»**

Как я проверяю пагинацию:
1. GET /users?limit=10 → 10 элементов
2. GET /users?limit=10&offset=10 → следующие 10
3. GET /users?limit=-1 → ошибка

Что я пропустил? 👇
*Пропущено: дублирование на стыке страниц, limit=0, total в метаданных.*

**Пост №4: «Чек-лист авторизации — что я упустил?»**

Как я проверяю авторизацию:
1. Без токена → 401
2. Невалидный токен → 401
3. Токен без прав → 403

Что я пропустил? 👇
*Пропущено: expired токен, токен от другого пользователя, токен с неверным форматом.*

## Формат №6: «Да/Нет»

### Суть
Простой вопрос с ответом Да/Нет + объяснение.

### Механика
Провокационный вопрос. Ответ — 2 секунды. Холивар если неочевидно.

### Зачем
Низкий порог входа. Обучение через провокацию.

### Когда использовать
1 раз в 2 недели.

### CTA
«🔥 если знал, ➕ если узнал новое»

### Готовые посты (4 шт.)

**Пост №1: «GET может иметь тело? Да/Нет»**
*Ответ: Формально да, HTTP не запрещает. Практически — curl отбросит, Nginx тоже. Лучше не использовать.*

**Пост №2: «DELETE должен возвращать тело? Да/Нет»**
*Ответ: По спецификации — 204 No Content без тела. DELETE с телом — странно.*

**Пост №3: «PUT можно использовать вместо PATCH? Да/Нет»**
*Ответ: Нет. PUT заменяет весь ресурс. PATCH — частично. Если не передать поле — станет null.*

**Пост №4: «201 Created всегда создаёт ресурс? Да/Нет»**
*Ответ: Нет. 201 — «создано по мнению сервера». Всегда проверяй GET после POST.*

## Формат №7: «Вопрос без ответа» (холивар)

Вопрос без однозначного ответа. Без правильного ответа.

### Механика
Дилемма. Автор не занимает сторону.

### Зачем
Самый длинный тред в комментариях. Активирует молчаливых.

### Когда использовать
1 раз в 2 недели.

### CTA
«Пишите свою версию»

### Готовые посты (4 шт.)

500 — сервер упал, тест упал, ты видишь.
200 — сервер сказал «ок», но данные не пришли. Тест прошёл.
Где потеряешь больше времени? Правильного ответа нет.

**Пост №2: «Должен ли junior QA знать SQL с первого дня?»**
Одни говорят: «SQL — база, без него не возьмут».
Другие: «В первую неделю научится».
А что думаете вы? 👇

**Пост №3: «Что лучше: curl или Postman?»**
curl — быстрый и гибкий. Postman — наглядный.
А ты что используешь? 👇

**Пост №4: «Документация врёт, а код делает по-своему. Кому верить?»**
Документация обещает одно. Код делает другое.
Кого фиксить первым? 👇

## Формат №8: «Разбор комментария»

### Суть
Реальный комментарий под постом → отдельный пост-разбор.

### Механика
Цитата комментария + разбор от Эда.

### Зачем
Человек чувствует, что его услышали. Лояльность.

### Когда использовать
1 раз в 1-2 недели.

### CTA
«Есть ещё? Пишите» / «Кидайте кейсы»

### Готовые посты (3 шт.)

**Пост №1: «"А как тестировать API без документации?"»**

«У нас нет OpenAPI. Разработчики говорят "смотри код". Я только начал».

Начинай с HTTP-методов. Какие есть? GET, POST, PUT, DELETE — уже понимаешь логику.
Потом — статусы. Отправь запрос, смотри что вернулось: 200, 400, 500.
Потом — curl -v. Заголовки: Content-Type, Authorization, Cache-Control.

Без документации ты не слепой. Ты исследователь.

**Пост №2: «"Сколько искать первый баг?"»**

«Второй день тыкаю Postman. Ни одного бага. Это нормально?»

Нормально. Первый баг — самый сложный.
Возьми один POST и отправь: пустое тело, невалидный JSON, отрицательные числа, строку в 10К символов, спецсимволы.
Если API не упадёт хотя бы на одном — я удивлюсь.

Напиши что нашёл 👇

**Пост №3: «"Как убедить лида, что баг — это баг?"»**

«Нашёл баг. Разработчик сказал "так задумано". Лид поверил. Что делать?»

Покажи данные. Не «я думаю это баг», а «curl показал то, документация обещает это, а в коде — третье».
Приложи: вывод curl, выдержку из документации, строку из кода.
Три источника — не поспоришь.

В Practicum есть модуль «Как доказать баг данными». Как раз про это.

## Формат №9: «Скринкаст: ход мыслей»

### Суть
2-минутное видео без монтажа. Открываешь API, тестируешь в реальном времени, комментируешь вслух.

### Механика
Показать процесс, а не результат. Аудитория видит, как ты думаешь.

### Зачем
Шоу-донт-телл в видеоформате. Максимальная прозрачность.

### Когда использовать
2 раза в неделю. Видео в TG (галерея) + текстовый пересказ.

### CTA
«А ты как тестируешь? Покажи свой процесс» / «Повтори в Practicum — 2 задачи бесплатно»

### Готовые посты (3 шт.)

**Пост №1: «Смотрю на API в первый раз. Мои первые 2 минуты»**

[Видео 2:00]

Открываю Practicum. Первый эндпоинт: GET /api/users.
Смотрю документацию: «Возвращает список пользователей».
Отправляю запрос. Смотрю тело. Смотрю заголовки.

За 2 минуты:
- Content-Type: text/plain (а должен быть application/json)
- Cache-Control: нет (данные кэшируются браузером)

Два бага за 2 минуты без подготовки.

Хотите так же? В Practicum — 15+ багов. 2 задачи бесплатно.

**Пост №2: «Тестирую DELETE. Что я делаю не так?»**

[Видео 2:00]

DELETE /api/users/1 → 200 OK. По RFC — 204.
Повторный DELETE → 200 OK. А должен быть 404.
GET после DELETE → 200 OK. Пользователь... не удалился?

Это не DELETE. Это PUT с пустым телом.

А вы проверяете, что DELETE реально удаляет? Или просто смотрите статус?

**Пост №3: «Ищу баги в авторизации. Метод — перебор»**

[Видео 2:00]

Шаг 1: GET /api/users без токена → 401. Ок.
Шаг 2: GET /api/users с битым токеном → 401. Ок.
Шаг 3: GET /api/users с токеном user1, запрашиваю user2 → ...

Спойлер: данные пришли. Authorization bypass.

Сервер проверил, что токен есть, но не проверил — твой ли это токен.

В Practicum такой баг тоже есть. Найдёте?

## Формат №10: «Короткие баг-репорты»

### Суть
800 символов. Один баг. Одна мысль. Никакой воды.

### Механика
Что делал → что ожидал → что получил → вывод.

### Зачем
Аудитория читает по диагонали. 2000 символов — пролистывают. 800 — дочитывают.

### Когда использовать
2-3 раза в неделю. Между большими постами.

### CTA
«➕ если сталкивались» / «Найди такой же в Practicum — ссылка в шапке»

### Готовые посты (6 шт.)

**Пост №1: «POST /users: email не уникален»**

Делаю: POST /api/users `{"email": "test@test.com", "name": "Alice"}` → 201.
Ещё раз: POST /api/users `{"email": "test@test.com", "name": "Bob"}` → 201.

В базе — два пользователя с одинаковым email.
Документация: «email — уникальный идентификатор».
Реальность: уникальность не проверяется.
Ожидал: 409 Conflict.

Вывод: уникальность на бэкенде — не опция, а обязательное условие.

➕ если сталкивались

**Пост №2: «DELETE /users/999 → 200 OK»**

Делаю: DELETE /api/users/999 (пользователя не существует).
Сервер: 200 OK, тело пустое.

Ожидал: 404 Not Found.
Получил: 200 на то, чего нет.

Вывод: если DELETE несуществующего возвращает 200 — ты не узнаешь, что удалил что-то не то.

**Пост №3: «PATCH: обновляю age, name сбрасывается»**

Делаю: PATCH /api/users/1 `{"age": 25}`.
Сервер: `{"id": 1, "name": null, "age": 25}`.

PATCH обновил age, сбросил name в null.
Ожидал: name остался как был.
Получил: PATCH, работающий как PUT.

Вывод: PATCH — частичное обновление. Если сервер перезаписывает весь объект — это баг.

**Пост №4: «Content-Type: text/plain на JSON-эндпоинте»**

Делаю: GET /api/users → `Content-Type: text/plain`.
Тело — валидный JSON. Заголовок — text/plain.

Фронт упадёт? Нет, браузер догадается. Но при строгой проверке — всё сломается.

Вывод: заголовки должны соответствовать содержимому. Не text/plain для JSON.

**Пост №5: «Rate limit: 0 requests per minute»**

Делаю: GET /api/users 50 раз подряд.
49 первых — 200. 50-й — 429.

Ок, лимит есть. Пауза на минуту.
Снова: GET /api/users → 429.

Лимит не сбрасывается. Никогда.

Вывод: rate limiting без сброса — это «заблокировано навсегда».

➕ если видели такое

**Пост №6: «500 на специальные символы»**

Делаю: POST /api/users `{"name": "<script>alert(1)</script>"}`.
Сервер: 500 Internal Server Error.

Ожидал: 400 с сообщением «Недопустимые символы».
Получил: падение.

Вывод: если спецсимволы валят сервер — это security-баг. XSS-защита на бэкенде.

## Принципы (коротко)

5. **6 разных форматов в неделю**, не повторять один и тот же.
6. **Шоу-донт-телл.** Покажи баг, не рассказывай о нём. Покажи curl, не говори «это плохая практика».

| «Каждый тестировщик должен знать» | Усиливает синдром самозванца |

- «вы обязаны» → «попробуйте / я рекомендую»
