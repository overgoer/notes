## eddytester/65

Если у вас SOAP: Отправка XML запросов в Postman 🚀 Postman умеет не только HTTP запросы отправлять, но и работать с протоколом SOAP, в котором тело запросов имеет XML форматирование. Отправить такой запрос довольно просто: 1: Создай нового запрос Кликни на кнопку "New" в верхнем левом углу и выбери "Request". 2: Выбери метод запроса В новом запросе, определи метод запроса, например, "POST", для отправки SOAP/XML запроса. 3: Вставь URL Скопируй и вставь URL-адрес, на который хочешь отправить запрос. Убедись, что URL верен. 4: Добавь заголовки (по желанию) Если необходимы дополнительные HTTP-заголовки, добавь их в разделе "Headers" запроса. 5: Вставь XML в тело запроса Перейди на вкладку "Body" и выбери "raw". После этого вставь свой XML-запрос в текстовое поле. 6. Отправь запрос Нажми на кнопку "Send" (Отправить) и дождись ответа от сервера. Готово 🥳 Подробная инструкция в великолепной доке Postman Кстати можно и с gRPC , но об этом позже 👌 Тестируй въедливо 🧪 🔍

Запуск коллекции без переменных newman run my-collection.json Запуск с переменными окружения: newman run my-collection.json -e dev-env.json Запуск с URL (если коллекция опубликована): newman run https://www.postman.com/collections/<collection-id> Запуск онлайн-коллекции с переменными окружения newman run "https://api.postman.com/collections/<collection-id>" -e dev-env.json Для приватной коллекции нужно указать API-ключ: newman run "https://api.postman.com/collections/<collection-id>" &#092; -e dev-env.json &#092; --apiKey <your-postman-api-key> ⚡️ Полезные флаги которые реально пригодятся в работе -n <число> Сколько раз прогнать коллекцию (по дефолту 1) --delay-request <ms> Задержка между запросами (по дефолту 0) --bail Остановка при первом упавшем тесте (по дефолту false) -r cli,json Отчеты в консоль + JSON файл (по дефолту только cli) --color off Убрать цветной вывод (по дефолту on) 🤓 ПОЛЕЗНО ЗНАТЬ: 1. Запуск с глобальными переменными: newman run "https://api.postman.com/collections/<collection-id>" &#092; -e dev-env.json &#092; -g globals.json 2. Запуск с несколькими окружениями: newman run "https://api.postman.com/collections/<collection-id>" &#092; -e dev-env.json &#092; -e common-env.json 3. Переопределение переменных: newman run "https://api.postman.com/collections/<collection-id>" &#092; -e dev-env.json &#092; --env-var "API_KEY=newkey" &#092; --env-var "BASE_URL=http://newurl" 4. Запуск нескольких коллекций newman run collection1.json && newman run collection2.json 5. Сохранение результатов в файл newman run collection.json -r cli,json --reporter-json-export ./results.json 6. Полезные алиасы для терминала # .bashrc или .zshrc alias nrun='newman run' alias ntest='newman run my-collection.json -e dev-env.json --bail' alias nreport='newman run collection.json -r cli,json --reporter-json-export'

## eddytester/398

⚡️ Кэширование в API: Шпаргалка для QA Кэширование - это сохранение копии ответа сервера, чтобы при следующем запросе не тратить время на долгие вычисления или походы в базу данных. 🔍 Два главных состояния (из словаря профи) 🔘 Cache HIT (Попадание) Данные нашлись в кэше. Ответ прилетел мгновенно. 🔘 Cache MISS (Промах) В кэше пусто. Сервер идет в базу данных, берет данные и по пути «закидывает» их в кэш для следующего раза. 💻 Как это выглядит в коде Программист пишет простую логику: «Если ответ уже есть в "коробке" с кэшем - отдай его. Если нет - сходи в базу данных и положи результат в коробку». // Пример логики (псевдокод): if (cache.exists("product_10")) { return cache.get("product_10"); // Cache HIT } else { data = database.find(10); // Cache MISS cache.put("product_10", data); return data; } 🛰 Как это выглядит в запросе В DevTools (вкладка Network) смотри Response Headers: cache-control: public, max-age=31536000 - «Храни это год&#33;». age: 69459 - файл лежит в кэше уже 20 часов. ETag: "vflset/lottie-light.js" - «отпечаток пальца» файла. Если он не изменился, сервер не будет слать файл заново. Важно: статус-код 304 Not Modified Если ты видишь такой статус ответа - это победа. Сервер проверил ETag и сказал: «У тебя в браузере и так всё свежее, не буду тратить трафик». Тело ответа в этом случае - 0 байт&#33; ⚠️ Где искать баги? (Инвалидация) Инвалидация - это очистка кэша. Самые «мистические» баги тут: 1. «Проблема старых данных»: Цену поменяли, но в API висит старая, так как кэш еще «живой». 2. Эффект «Только у меня»: Один коллега видит баг, другой - нет. Кто-то попал на старый кэш, а кто-то пробил его. 3. Утечка приватности: Кэш по ошибке сохранил данные одного юзера и отдал их другому. 🛠 Чек-лист: Что проверить тестировщику? 🔘 Заголовки: Если видишь no-store - данные не должны кэшироваться (важно для корзин и паролей). 🔘 Обновление (POST/PUT/DELETE): Проверь, что после изменения данных GET-запрос сразу отдает новое значение (кэш должен сброситься). 🔘 Статус 304: Убедись, что при повторном заходе браузер использует кэш и не качает тяжелые картинки/скрипты заново. 🔘 Сброс кэша: Узнай у разработчиков, как принудительно очистить кэш (например, через Redis), чтобы проверить «чистое» состояние. Лайфхак: Если баг «плавает» - добавь в URL параметр ?v=1. Это заставит систему думать, что запрос новый, и она пойдет в базу в обход кэша. Если баг исчез - ты локализовал проблему&#33; 📚 Полезные ссылки: 1. What is Caching? (Postman) - лучший визуальный гид. 2. MDN: HTTP Caching - база про заголовки.
