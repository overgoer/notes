# 🔍 Как протестировать API получения баланса

## Сценарий
API `GET /api/v1/accounts/{id}/balance` возвращает JSON:
```json
{
  "accountId": "acc-123",
  "balance": 1500.50,
  "currency": "RUB",
  "timestamp": "2026-04-01T12:00:00Z"
}
```

Сервис — банковский (Тинькофф/Сбер/Озон), критичен к точности и доступности.

## Задача
Опиши, **как ты будешь тестировать это API** как Senior QA (на 2–3 страницы). Укажи:
- Какие тест-кейсы ты придумаешь и почему;
- Какие инструменты используешь (и почему);
- Как проверишь качество тестов.

---

## ✅ Ответ: полный тест-план

### 1. Тест-кейсы — по категориям

| Категория | Примеры кейсов | Почему | Цель |
|-----------|----------------|--------|------|
| **✅ Позитивные** | `id = acc-123` → возвращает `200 OK`, корректный `balance`, `currency = RUB` | Базовый сценарий | Убедиться, что сервис работает |
| **⚠️ Негативные** | `id = acc-999` → `404 Not Found`; `id = ""` → `400 Bad Request` | Проверка обработки ошибок | Избежать падений в проде |
| **🔒 Безопасность** | `GET` с `X-Auth-Token: invalid` → `401 Unauthorized`; `GET` без токена → `401` | Защита от анонимных запросов | Соответствие политикам безопасности |
| **⏱️ Производительность** | Time < 200ms при нагрузке 100 RPS (JMeter) | Банковские сервисы требуют низкой задержки | Удовлетворение SLA |
| **🔄 Граничные значения** | `timestamp` в будущем / прошлом → валидация формата ISO-8601 | Ошибки времени — частая причина багов | Стабильность в разных часовых поясах |
| **🔍 Стабильность** | 100 запросов подряд → `200 OK`, одинаковые `balance` | Проверка кеширования и idempotency | Убедиться, что баланс не скачет |

### 2. Инструменты и почему

| Инструмент | Как используется | Почему именно он |
|------------|------------------|------------------|
| **REST Assured** | Пишем автотесты на Java: `given().auth().oauth2(token).when().get(«/balance»).then().statusCode(200)` | Интегрируется с JUnit/TestNG, поддерживает DSL, идеален для Java QA. |
| **Postman** | Ручное тестирование, коллекции, документирование API | Удобен для быстрой проверки и передачи разработчикам. |
| **JMeter** | Нагрузочное тестирование: 100 RPS, 5 мин, 95-й перцентиль < 200 ms | Open-source, масштабируем, поддерживает кластер, используется в BigTech. |
| **Allure Reports** | Собираем отчёты из REST Assured + JMeter → `allure serve` | Генерирует красивые отчёты с шагами, скриншотами, временем. Используется в Авито/Озон. |

### 3. Как проверишь качество тестов?

- **Код-ревью тестов** — как и код: смотрим на покрытие, отсутствие хардкодов, читаемость.
- **Mutation Testing** с PITest: `mvn pitest:mutationCoverage` → если тесты «не убивают мутантов» (например, не проверяют `currency`), значит — покрытие слабое.
- **Реальные баги**: если за последний месяц 3 бага в API были найдены разработчиками — значит, тесты не покрывают критичные сценарии.
- **Состав команды**: если QA пишет тесты, но разработчик не может их править — значит, тесты «закрыты» для команды. Хорошие тесты написаны так, что разработчик — тоже их запускает и правит.

---

## 📌 Вопрос на интервью
> «Как ты убедишь разработчика, что не все тест-кейсы нужны?»

**Ответ:** «Я не убеждаю — я показываю данные. Показываю, какие кейсы нашли баги в прошлом, какие из них регулярно падают из-за флапов, а какие не запускались 3 месяца. Я предлагаю приоритизировать не по «всё должно быть», а по риску: баг в балансе — критичнее, чем баг в timestamp. А приоритезация — это язык, понятный и QA, и разработчику».
