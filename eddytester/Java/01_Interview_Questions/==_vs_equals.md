# `==` vs `equals()`

## Вопрос
> В чём разница между `==` и `equals()` в Java? Когда что использовать?

## Ожидаемый ответ

### 1. `==` (оператор сравнения)
- **Сравнивает примитивы** по значению.
- **Сравнивает ссылки** (reference types) — проверяет, указывают ли две ссылки на один и тот же объект в памяти.

### 2. `equals()` (метод класса `Object`)
- **Определяет семантическое равенство** объектов.
- По умолчанию (в классе `Object`) реализован как `==` (сравнение ссылок).
- **Переопределяется** в классах (`String`, `Integer`, `LocalDate` и др.) для сравнения содержимого.

### 3. Ключевые различия

| Критерий | `==` | `equals()` |
|----------|------|------------|
| **Тип** | Оператор | Метод |
| **Применимость** | Примитивы и ссылки | Только объекты (не примитивы) |
| **Сравнение** | Значение (примитивы) или идентичность ссылок | Семантическое равенство (зависит от реализации) |
| **Переопределение** | Нельзя | Можно (и нужно для value-классов) |

### 4. Примеры

```java
// Примитивы — только ==
int a = 5;
int b = 5;
System.out.println(a == b); // true

// Ссылки — == сравнивает ссылки
String s1 = new String("hello");
String s2 = new String("hello");
System.out.println(s1 == s2);      // false (разные объекты)
System.out.println(s1.equals(s2)); // true  (одинаковое содержимое)

// Строковый пул (string pool)
String s3 = "hello";
String s4 = "hello";
System.out.println(s3 == s4);      // true (один объект в пуле)
```

### 5. Контракт `equals()`
При переопределении `equals()` необходимо:
1. **Рефлексивность:** `x.equals(x)` → true
2. **Симметричность:** `x.equals(y)` → true ⇔ `y.equals(x)` → true
3. **Транзитивность:** `x.equals(y)` и `y.equals(z)` → `x.equals(z)`
4. **Консистентность:** многократные вызовы дают одинаковый результат
5. **Сравнение с null:** `x.equals(null)` → false

### 6. Правила использования
- **Для примитивов** — всегда `==`.
- **Для ссылочных типов:**
  - Если нужно проверить **один и тот же объект** — `==`.
  - Если нужно проверить **логическое равенство** — `equals()`.
- **Всегда переопределяйте `hashCode()`**, если переопределяете `equals()` (контракт `hashCode()`).

## Частые ошибки кандидатов

1. **Использование `==` для сравнения строк** (кроме литералов).
2. **Незнание, что `equals()` по умолчанию работает как `==`**.
3. **Путаница с автоупаковкой:**
   ```java
   Integer a = 127;
   Integer b = 127;
   System.out.println(a == b); // true (кеширование -128..127)
   
   Integer c = 200;
   Integer d = 200;
   System.out.println(c == d); // false (разные объекты)
   ```
4. **Отсутствие проверки на `null` перед вызовом `equals()`:**
   ```java
   String s = null;
   s.equals("test"); // NullPointerException
   // Правильно:
   "test".equals(s); // false, но не исключение
   ```

## Оценка

- **Junior:** знает, что `==` сравнивает ссылки, `equals()` — содержимое.
- **Middle:** может объяснить строковый пул, автоупаковку, контракт `equals()`.
- **Senior:** приводит примеры переопределения `equals()`/`hashCode()`, обсуждает immutable-объекты, знает про `Objects.equals()`.
