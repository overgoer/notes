# Пустое тело, нет тела — какая разница?

Разработчик сказал: «это одно и то же». Я показал два curl'а и открыл RFC.

---

Прилетает баг-репорт: POST /api/users падает с 500, если отправить без тела.

Я пишу разработчику: «Лови curl: `curl -v -X POST /api/users -H "Content-Type: application/json"` — падает».

Он: «Так ты тело не отправил. Конечно падает. Это не баг — так и должно быть для POST без данных».

Я: «Ок, лови ещё: `curl -v -X POST /api/users -H "Content-Type: application/json" -d ''` — падает так же».

Он: «Так это одно и то же. Пустое тело, нет тела — какая разница?»

Разница есть. И она принципиальная.

**Без тела:**
→ Заголовка `Content-Length` нет вообще. Сервер получает POST-запрос и не знает, сколько данных ожидать. Некоторые серверы на это отвечают 411 Length Required, другие — 500.

**С пустым телом (`-d ''`):**
→ `Content-Length: 0`. Сервер знает: тело есть, оно нулевой длины. JSON-парсер получает пустую строку. Результат зависит от фреймворка: где-то `{}`, где-то ошибка парсинга.

На проде оба кейса ведут себя по-разному. Я это доказал двумя curl'ами за 30 секунд:

```
# Разница — в заголовке Content-Length
curl -v -X POST /api/users -H "Content-Type: application/json"
curl -v -X POST /api/users -H "Content-Type: application/json" -d ''
```

Спойлер: после этого разговора в тест-кейсы добавили оба варианта.
