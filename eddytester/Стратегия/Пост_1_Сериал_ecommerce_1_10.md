# Сериал 1/10: Мне дали API интернет-магазина -- API ecommerce стартапа

**в 2022 году Знакомые предложили потестировать ecommerce стартап из США. Part time. Конечно я согласился.**

Первый день на проекте. Мне дали одно из API интернет-магазина. --для ecommerce апи в 3 эндпоинта мало реалистично. надо подумать как лаконично но точно показать реальную картину. напрмиер сказать что это было одно из их апи, небольшая часть и тп.

Документация: одна страница в Confluence. Три эндпоинта. Без примеров.

Коллега: «Там всё просто. GET /products, POST /orders, DELETE /cart. Дерзай».

Через час я нашёл первый баг. Через два — ещё три.

**Спойлер:** за неделю я нашёл 23 бага в этом «простом» API.

Этот сериал — хроника той недели. Без прикрас, без выдумок. Только реальные баги, реальные разговоры с разработчиками и реальные решения. Все за-NDA-ено, но баги реальные.

**Выпуск 1: «GET /products — пагинация без границ»**

Открываю Postman. GET /products. Жму Send.

200 OK. 10 товаров. Отлично.

GET /products?page=2 → 10 товаров. Работает.

GET /products?page=100 → 200 OK. 0 товаров.

GET /products?page=-1 → 200 OK. Те же 10 товаров.

GET /products?page=abc → 500 Internal Server Error.

Три бага в одной пагинации:
1. Отрицательный номер страницы — молча сбрасывается на первую
2. Несуществующая страница — пустой массив вместо 404
3. Невалидный page — сервер падает

**Итог:** 3 бага за 5 минут. Хорошее начало недели.

**Bizzy:** Эдди, лови мои мысли по посту ↓

**1. Про «почему».** Сейчас баги перечислены, но нет объяснения — а в сериал отлично заходит фраза «я полез смотреть код и понял, в чём дело». 

**2. Про фреймворк пагинации.** Ты спрашиваешь — есть ли универсальный инструмент? Короткий ответ: нет одного фреймворка, но есть **keyset (курсорная) пагинация** — `WHERE id > ? LIMIT 10`. Она не сдвигается при вставках. Для сериала можно одной фразой: *«Разработчик сказал: „мы юзаем OFFSET, че не так“. Я показал ему курсоры. На следующий день он переписал пагинацию»*. Без погружения в технику — просто жизненный момент.

**3. Про баг с фильтрами.** Классная идея для выпуска 2 или 3 — показать continuity: в первом эпизоде нашли баги пагинации, а через пару дней вылез тот самый баг с фильтрами, который не обнуляет page. Это прям сериальный ход.

**4. Мой вариант — что добавить в этот пост** (буквально 2-3 предложения после списка багов):

> — Заглянул в код. Там был обычный `LIMIT 10 OFFSET (page-1)*10`. Без сортировки, без валидации. OFFSET на отрицательном page просто срабатывал как OFFSET 0. Пустая страница — тоже особенность OFFSET: он не знает, сколько всего записей. А `page=abc` падал, потому что никто не проверял тип параметра.
> — Через пару дней я вернусь к этой же пагинации — и найду ещё один баг, связанный с фильтрами. Но об этом в другом выпуске.

Это добавляет «жизненности» (заглянул в код, поговорил с девом) и подогревает к следующей серии.

**5. Что думаю в целом:** Пост живой, читается легко. Историю про 23 бага за неделю и спойлер в начале — отлично заходят. Добавь чуть больше «запаха» реальности (диалог с разработчиком, его реакция) — и будет огонь.
