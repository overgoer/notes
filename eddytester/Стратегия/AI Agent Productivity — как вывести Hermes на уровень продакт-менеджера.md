# AI Agent Productivity — как вывести Hermes на уровень продакт-менеджера

> **Цель:** превратить Hermes-агента (меня) из «собеседника» в автономного продакт-менеджера, который сам берёт задачи, решает их и отчитывается.

---

## Проблема

Текущий режим работы:
- Я жду твоего сообщения
- Ты описываешь задачу
- Я делаю, но с постоянными уточнениями
- На простые вещи уходит 2 дня (Cloudflare)
- Нет проактивности

---

## Решение: 4 системы Hermes, которые уже есть

### 1. Cron Jobs — расписание и регулярные задачи

У Hermes есть встроенная система cron. Команды:

`cronjob(action='create', prompt='...', schedule='every 1h')`

Что можно:
- Ежедневная проверка здоровья сервисов
- Автоматический деплой по расписанию
- Мониторинг бэклога и напоминания
- Ночной пайплайн анализа

**Как применить у нас:**
- Каждое утро я проверяю сервисы (api-practicum-bot, food API, БД) и пишу тебе статус
- Каждый вечер — дайджест что сделано
- Автоматический sync Obsidian → Git

### 2. Kanban — многоагентная оркестрация

Hermes Kanban — доска задач, где я могу:
- Создавать задачи
- Разбивать большие на подзадачи
- Назначать их разным профилям (или самому себе)
- Ставить зависимости (эту — после той)
- Блокировать, если нужно твоё решение
- Возвращать результат

**Как применить у нас:**
- Ты пишешь задачу в Obsidian inbox
- Я создаю Kanban-карточку
- Разбиваю на подзадачи
- Делаю их
- Когда нужно твоё решение — блокирую с вопросом
- Когда готово — отчитываюсь

### 3. Subagent Delegation — параллельная работа

`delegate_task` — я могу запускать параллельных агентов для независимых задач.

**Как применить у нас:**
- Один агент фиксит баг в API
- Второй агент пишет пост для Telegram
- Третий проверяет, что прод не упал
- Всё параллельно, ты получаешь один суммарный отчёт

### 4. Skills — обучаемая память процедур

Я сохраняю повторяющиеся процедуры как skills:

- **skill:** развернуть новый поддомен на Timeweb
- **skill:** деплой food-delivery-api через Git
- **skill:** проверить статус всех сервисов
- **skill:** создать пост для Telegram канала

После первого выполнения я запоминаю и больше не спрашиваю.

## Что вне Hermes (сторонние сервисы)

### Devin (cognition.ai) — $500/мес

Полноценный AI-инженер:
- Имеет IDE, терминал, браузер
- Пишет код, деплоит, фиксит баги
- Отчитывается в Slack/Telegram

Для нас — overkill. У нас уже есть сервер и доступ.

### Factory AI — $400-1000/мес

Агент для codebase-работы. Может:
- Рефакторить код
- Добавлять фичи
- Проводить code review

Но: не знает нашу инфраструктуру, не умеет в прод.

**Вывод:** своих денег не стоит, Hermes + навыки перекрывают.

### Сервисы для observability (мониторинг)

Стоит подключить для самодиагностики:
- **Uptime Kuma** (self-hosted) — простой мониторинг, что сервисы живы
- **Healthchecks.io** — если я запускаю cron задачу, она должна «чекиниться»

## План действий — как внедрить

### Фаза 1: База (сделать сейчас)

1. **Настроить cron на healthcheck** — каждое утро я проверяю сервисы и пишу тебе статус
2. **Kanban-доска** — создаём доску `eddytester`, я переношу туда задачи из бэклога
3. **Skill: деплой** — запоминаю процедуру деплоя food-delivery-api, больше не спрашиваю
4. **Obsidian → task** — ты пишешь в `inbox.md`, я вижу и беру в работу

### Фаза 2: Автономия

5. **Ночной пайплайн** — я сам проверяю сервисы, логи, занятость диска, пишу утренний дайджест
6. **Proactive fix** — если сервис упал, я чинить сам, не дожидаясь твоего сообщения
7. **Параллельные задачи** — сложные фичи я разбиваю на подзадачи и исполняю параллельно

### Фаза 3: Полный PM

8. **Планирование** — я сам смотрю на бэклог, приоритизирую, предлагаю, что делать сегодня
9. **Content pipeline** — посты в Telegram генерируются по расписанию, ты только аппрувишь
10. **Product decisions** — я предлагаю решения (не подходы, а решения), ты говоришь «да» или «нет»
