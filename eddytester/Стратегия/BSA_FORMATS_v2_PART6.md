# BSA_FORMATS_v2 — Часть 6: Форматы №9, №10 (из трёх кардинальных изменений)

## Формат №9: «Скринкаст: ход мыслей»

**Механика:** 2-минутное видео без монтажа. Ты открываешь API, тестируешь в реальном времени, комментируешь вслух.

**Зачем:** Показать процесс, а не результат. Аудитория видит, как ты думаешь — это не заменяет текст.

**Когда:** 2 раза в неделю. Видео в TG (галерея) + текстовый пересказ.

**Структура:**
- 0:00–0:30 — что тестирую и почему
- 0:30–1:30 — сам процесс (запрос → ответ → анализ)
- 1:30–2:00 — вывод

**CTA:** «А ты как тестируешь? Покажи свой процесс» / «Повтори в Practicum → ссылка»

---

### Пост 1: «Смотрю на API в первый раз. Мои первые 2 минуты»

[Видео 2:00 — скринкаст экрана]

Я открываю Practicum. Первый эндпоинт: `GET /api/users`.

Смотрю документацию: «Возвращает список пользователей».

Отправляю запрос. Смотрю тело. Смотрю заголовки.

За 2 минуты я нашёл:
- Content-Type: text/plain (а должен быть application/json)
- Cache-Control: нет (данные кэшируются браузером)

Всё. Два бага за 2 минуты. Без подготовки.

### Пост 2: «Тестирую DELETE. Что я делаю не так?»

[Видео 2:00]

DELETE /api/users/1 → 200 OK.
Хм. По RFC — 204.

Проверяю повторный DELETE того же пользователя → 200 OK.
А должен быть 404.

Проверяю GET после DELETE → 200 OK.
Пользователь... не удалился?

Стоп. Это не DELETE. Это PUT с пустым телом.

За 2 минуты нашёл маскировку метода.

А вы проверяете, что DELETE реально удаляет? Или просто смотрите статус?

### Пост 3: «Ищу баги в авторизации. Метод — перебор»

[Видео 2:00]

Шаг 1: GET /api/users без токена → 401. Ок.
Шаг 2: GET /api/users с битым токеном → 401. Ок.
Шаг 3: GET /api/users с токеном user1, запрашиваю данные user2 → ...

Спойлер: данные пришли. Authorization bypass.

Один из самых частых security-багов в API. Сервер проверил, что токен есть, но не проверил, что это твой токен.

## Формат №10: «Короткие баг-репорты»

**Механика:** 800 символов. Один баг. Одна мысль. Никакой воды.

**Зачем:** Аудитория читает по диагонали. 2000 символов — пролистывают. 800 — дочитывают.

**Когда:** 2–3 раза в неделю. Можно между большими постами.

**Структура:**
- Что делал: `POST /api/users с email test@test.com`
- Что ожидал: `409 Conflict — email уже существует`
- Что получил: `200 OK, второй пользователь с тем же email`
- Вывод в одну строку: Уникальность должна проверяться на бэкенде, а не только на фронте.

**CTA:** «Найди такой же в Practicum → ссылка на бот» / «➕ если сталкивался»

---

### Пост 1: «POST /users: email не уникален»

Делаю: `POST /api/users {"email": "test@test.com", "name": "Alice"}` → 201 Created.

Делаю ещё раз: `POST /api/users {"email": "test@test.com", "name": "Bob"}` → 201 Created.

В базе — два пользователя с одинаковым email.

Документация: «email — уникальный идентификатор».
Реальность: уникальность не проверяется.
Ожидал: 409 Conflict.

Уникальность на бэкенде — не опция, а обязательное условие. Фронт может врАТь.

➕ если сталкивались

---

### Пост 2: «DELETE /users/999 → 200 OK»

Делаю: `DELETE /api/users/999` (пользователя не существует).

Сервер: `200 OK`, тело пустое.

Ожидал: 404 Not Found или 204 No Content.
Получил: 200 на то, чего нет.

Вывод: Если DELETE несуществующего возвращает 200 — ты не узнаешь, что удалил что-то не то.

---

### Пост 3: «PATCH: обновляю age, name сбрасывается»

Делаю: `PATCH /api/users/1 {"age": 25}`.
Сервер: `200 OK`, `{"id": 1, "name": null, "age": 25}`.

PATCH обновил age, но сбросил name в null.

Ожидал: `{"id": 1, "name": "Alice", "age": 25}` — name остался.
Получил: PATCH, работающий как PUT.

Вывод: PATCH — это частичное обновление. Если сервер перезаписывает весь объект — это баг.

---

### Пост 4: `Content-Type: text/plain` на JSON-эндпоинте»

Делаю: `GET /api/users` → `Content-Type: text/plain; charset=utf-8`.

Тело — валидный JSON: `[{"id":1,"name":"Alice"}]`.
Заголовок — text/plain. Фронт ждёт application/json.

Фронт падает при парсинге? Нет, браузер сам догадается. Но если кто-то использует строгую проверку Content-Type — всё сломается.

Вывод: Заголовки должны соответствовать содержимому. Не text/plain для JSON. Не application/json для HTML.

---

### Пост 5: «Rate limit: 0 requests per minute»

Делаю: `GET /api/users` 50 раз подряд.

Первые 49 — 200 OK.
50-й — 429 Too Many Requests.

Ок, лимит есть. Делаю паузу на минуту.

Снова: `GET /api/users` → 429.

Лимит не сбрасывается. Никогда.

Вывод: Rate limiting без сброса — это «заблокировано навсегда». Пользователь не может вернуться.

➕ если видели такое

---

### Пост 6: «500 на специальные символы»

Делаю: `POST /api/users {"name": "<script>alert(1)</script>"}`.

Сервер: `500 Internal Server Error`.

Ожидал: 400 Bad Request с сообщением «Имя содержит недопустимые символы».
Получил: падение сервера.

Вывод: Если спецсимволы валят сервер — это security-баг. XSS-защита должна быть на бэкенде.
