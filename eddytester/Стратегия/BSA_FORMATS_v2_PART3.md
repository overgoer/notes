# BSA_FORMATS_v2 — Часть 3: Формат №3 «Тестировщик VS Разработчик»

## Формат №3: «Тестировщик VS Разработчик: чья версия правда?»

**Механика:** Кейс → две версии → доказательство данными.
**Когда:** 1–2 раза в неделю.

---

### Пост 1: «401 Unauthorized — и кто прав?»

Кейс: POST /api/login, токен есть, но API возвращает 401.

**Тестировщик:** «Это баг. Мы авторизованы, но сервер не понимает токен».

**Разработчик:** «401 — стандартный код. Сервер говорит "не знаю тебя". Это ожидаемое поведение».

**Как я доказал правду:**

1. `curl -v -H "Authorization: Token abc123" https://api.example.com/protected`
   → В логах: `token = null`

2. Открыл код бэкенда, нашёл парсер заголовка:
   ```python
   auth_header = request.headers.get("Authorization")
   token = auth_header.replace("Bearer ", "")
   ```
   — Оно! Парсер ждёт `Bearer`, а фронт шлёт `Token`.

3. Проверил `git blame` на эту строку — оказалось, 2 года назад переписали формат токена, но парсер не обновили.

**Вердикт:** Прав разработчик. 401 — корректный код. Но баг есть — несоответствие формата токена между фронтом и бэкендом.

А у вас такое было? Кидайте кейсы.

### Пост 2: «500 на логине — чей баг?»

Кейс: POST /api/login, пустое тело → 500 Internal Server Error.

**Тестировщик:** «Баг. Сервер падает. Нет обработки пустого тела».

**Разработчик:** «А кто отправляет пустое тело? Фронт должен валидировать. 500 — это ок, потому что клиент дурак».

**Как я доказал правду:**

1. `curl -v -X POST https://api.example.com/login -H "Content-Type: application/json" -d '{}'`
   → 200 OK. Сервер съел пустой JSON.

2. `curl -v -X POST https://api.example.com/login -H "Content-Type: application/json" -d ''`
   → 500 Internal Server Error.

3. Посмотрел логи: `NullPointerException: 'email' is null`.

4. Написал тест: `POST с пустым телом → 400 Bad Request`. Баг.

**Вердикт:** Прав тестировщик. Сервер должен возвращать 400, а не 500. Никто не гарантирует, что клиент не отправит пустое тело — ты не контролируешь клиента.

А у вас сервер падает на пустых запросах? Кидайте в комменты.

### Пост 3: «DELETE возвращает 200 — баг или стиль?»

Кейс: DELETE /api/users/123 → 200 OK с пустым телом.

**Тестировщик:** «Баг. RFC 7231 говорит: DELETE должен возвращать 204 No Content, если тело пустое».

**Разработчик:** «Какая разница? Работает же. 200 — это успех».

**Как я доказал правду:**

1. `curl -v -X DELETE https://api.example.com/users/123`
   → `HTTP/1.1 200 OK`, тело пустое.

2. `curl -v -X DELETE https://api.example.com/users/999`
   → `HTTP/1.1 200 OK`. Несуществующий пользователь — тоже 200.

3. Проверил RFC 7231 §4.3.5: «If a DELETE method is successfully applied... the origin server SHOULD send a 204 (No Content)».

4. Сделал `curl -v -X DELETE https://api.example.com/users/999`
   → По RFC, если пользователя нет — должен быть 404.

**Вердикт:** Прав тестировщик. 200 вместо 204 — не критично, но маскирует реальные проблемы (например, DELETE несуществующего ресурса).

А у вас 200 или 204 на DELETE? Кидайте.

### Пост 4: «Age = 150 — почему это баг, а не фича?»

Кейс: POST /api/users, поле `age: 150` → 200 OK, пользователь создан.

**Тестировщик:** «Баг. Возраст 150 — это невалидное значение. Сервер должен вернуть 400».

**Разработчик:** «А кто сказал, что 150 — невалид? Может, это возраст организации. У нас нет ограничения».

**Как я доказал правду:**

1. `curl -s -X POST https://api.example.com/users -H "Content-Type: application/json" -d '{"name":"Test","age":150}'`
   → `{"id": 42, "name": "Test", "age": 150}`

2. `curl -s -X POST https://api.example.com/users -H "Content-Type: application/json" -d '{"name":"Test2","age":-5}'`
   → `{"id": 43, "name": "Test2", "age": -5}`

3. Отрицательный возраст прошёл. Это уже не «может быть организацией».

4. Спросил у продакта: «Какой максимальный возраст пользователя?» — «150, но мы не проверяем».

**Вердикт:** Баг тестировщика. Валидация граничных значений — базовая история. Сервер должен проверять: age >= 0 и age <= 150.

А у вас есть API, который принимает любые числа? Кидайте.
