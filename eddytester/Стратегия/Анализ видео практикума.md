# Анализ курса «API Практикум» — 16 видео

## 1. Структура курса

Курс построен по классической схеме **«теория → практика → закрепление»** с последовательным усложнением:

**Блок 1: Введение и настройка (видео 1–3)**
- Введение — что такое практикум, сколько багов (~15+), структура, обратная связь
- Установка Postman — скачивание, импорт коллекции, настройка авторизации (API key)
- Первые запросы (GET all users, POST create user) + метод очистки базы (Clear Database)
- План практикума — roadmap по тестированию API

**Блок 2: Теория (видео 4–7)**
- **HTTP** — протокол, клиент-сервер, структура запроса/ответа, методы GET/POST/PUT/PATCH/DELETE
- **API** — отличие от HTTP, CRUD-операции, работа с документацией, пример неверного endpoint (404)
- **Классы эквивалентности** — разбиение значений возраста (minor 0-17, candidate 18-65, retired 66+), принцип «одно значение на класс»
- **Граничные значения** — проверка границ между классами (-1/0, 17/18, 65/66), золотое правило «две стороны каждой границы»

**Блок 3: Практика по методам (видео 8–14)**
- **GET all users** (теория + практика) — список тест-кейсов, разбор одного по шаблону, нахождение первого бага (Content-Type: text/plain вместо application/json)
- **Bug Report** — оформление баг-репорта по шаблону (заголовок, объект, приоритет, шаги, артефакты из Postman console)
- **Bug Fix + справочник** — импорт V2 коллекции, проверка исправления, справочник багов (HTML-файл)
- **GET by ID** — endpoint с ID в пути, безопасный метод (safe), проверка точности данных
- **POST** — создание кандидата, валидация полей (name, age), уникальность имени, фаззинг
- **PATCH** — изменение кандидата (реализован как full update), проверка смены статуса при изменении возраста
- **DELETE** — удаление по ID, идемпотентность/неидемпотентность, проверка через GET и Clear Database

**Блок 4: Финал (видео 15–16)**
- Итоги — что пройдено, обратная связь, чек-лист для любого метода
- *(предположительно 16-е видео — бонус/обзор эфиров)*

**Логика progression:** настройка инструментов → HTTP/API теория → тест-дизайн → практика на каждом методе (от простого GET к сложному DELETE) → оформление багов → проверка фиксов.

## 2. Основные концепты

### Протокол HTTP
- Определение (HyperText Transfer Protocol), клиент-серверная архитектура
- Структура запроса: метод, endpoint, host, заголовки, тело
- Структура ответа: статус-код, заголовки, тело
- HTTP методы: GET (получение), POST (создание), PUT (полная замена), PATCH (частичное/полное изменение), DELETE (удаление)
- Статус-коды: 200 (OK), 400 (Bad Request), 404 (Not Found), 500 (Server Error)
- Версии HTTP, заголовки (Content-Type, X-FixBug, Cache-Control, X-Content-Type-Options, X-Frame-Options)

### API
- Отличие API от HTTP: HTTP — глобальный стандарт, API — локальный договор для конкретного сервиса
- CRUD-операции (Create, Read, Update, Delete)
- Endpoint / «ручка» — конкретный адрес ресурса на сервере
- Параметры запроса (query params) — отделяются `?`, несколько через `&`
- Авторизация через заголовок (API key)
- Форматы данных: JSON (объекты, массивы), типы (int, string, enum)

### Тест-дизайн
- **Классы эквивалентности** — разбиение входных данных на группы с одинаковым поведением
- **Граничные значения** — проверка значений на границах классов (обязательно с двух сторон)
- Принцип «одно значение внутри класса» — экономия тест-кейсов
- Спор в сообществе — обязательность проверки границ vs. необязательность проверки внутри класса

### Практические навыки
- Работа в Postman: коллекции, переменные, импорт, заголовки, тело запроса, консоль, сохранение response
- Составление тест-кейсов по шаблону (заголовок, объект, приоритет, предусловия, шаги, ожидаемый результат)
- Оформление баг-репортов (заголовок, объект, автор, приоритет/критичность, предусловия, шаги воспроизведения, фактический результат, артефакты)
- Проверка фиксов (V2 коллекция)
- Работа со справочником багов
- Фаззинг — отправка невалидных данных (строка вместо числа, массив, спецсимволы)

## 3. Что есть в теории, чего нет в API/bot

### Что покрыто в видео, но НЕ реализовано в боте/API-шлюзе

| Концепт | Где в видео | Почему нет в API/bot |
|---|---|---|
| **Несколько версий API (V1 → V2)** | Видео 11 — импорт Candidates V2 для проверки фикса | В боте одна версия, нет механики переключения версий |
| **Clear Database (служебный метод)** | Видео 3 — метод очистки тестовых данных | В реальном API нет такого метода; в боте нет изолированной тестовой среды |
| **Справочник багов (HTML)** | Видео 11 — скачиваемый HTML со списком всех багов | Нет аналога; баги хранятся разрозненно |
| **Postman как единственный инструмент** | Весь курс основан на Postman | В реальной работе используют curl, Insomnia, Bruno, или пишут автотесты |
| **Отправка баг-репортов на email** | Видео 10 — отправка на testsaggmail.com | Нет интеграции с трекерами (Jira, YouTrack и т.д.) |
| **Коллекция с шаблонными запросами** | Импорт коллекции из файла | В боте нет готовой коллекции; пользователь создаёт запросы сам |
| **Чек-лист для всех методов** | Упоминается в нескольких видео как универсальный чек-лист | Не реализован как отдельный артефакт |
| **Прямые эфиры / обратная связь** | Видео 1, 15 — обещаны эфиры и разборы | Нет аналога в боте; только асинхронное взаимодействие |
| **Обсуждение фаззинга** | Видео 12 — рекомендация почитать про фаззинг | В боте нет модуля тестирования с автоматическим фаззингом |
| **SPOILER-баны в чате** | Видео 11 — обещание банить за спойлеры багов | Нет такой механики в боте |

### Что в теории есть, но могло бы быть в боте
- **Статус-коды** — видео объясняет 200, 400, 404, 500, но в боте нет модуля с их перечнем и значениями
- **HTTP методы** — подробно разобраны, но в боте нет интерактивной шпаргалки
- **Типы данных (int, string, enum)** — объяснено на примере JSON схемы, можно было бы сделать визуальную карточку
- **Концепция idempotent/non-idempotent** — упомянута для DELETE, можно развить

## 4. Стиль подачи

Эд (автор курса) использует **максимально практичный и дружелюбный стиль**:

### Характеристики
- **Разговорный, простой язык** — без излишней академичности, как будто разговаривает с другом
- **Постоянные отсылки к практике** — «давай сейчас сделаем», «проверим это на практике», «откроем Postman и посмотрим»
- **Поощрение самостоятельности** — «попробуй сам», «составь свой список тест-кейсов», «не буду спойлерить»
- **Забота о студенте** — «если застрял — пиши в чат», «если не получается — не волнуйся, разберёшься»
- **Методичность** — каждый урок имеет чёткую структуру: теория → пример → практика → ссылки на допматериалы
- **Транслитерация и оговорки** — текст субтитров содержит характерные «э-э», «м-м», паузы — это живая речь
- **Использование примеров из жизни** — Level.Travel для объяснения query params, Wikipedia для гипертекста, Obsidian
- **Профессиональные инсайты** — объясняет, как «в реальных компаниях» работают с DELETE (soft delete, is_deleted), как спорят про классы эквивалентности
- **Юмор и лёгкость** — «баню за спойлеры», «нашими данными торгуют», «мало кто доживает до 1000 лет»

### Оценка стиля
Техничность: **средняя** (концепты объясняются без глубокого погружения в детали протокола)
Простота: **высокая** (понятно новичкам)
Примеры: **очень много** (каждый концепт сопровождается практической демонстрацией)

Идеально для начинающих тестировщиков. Для более опытных — слишком много воды.

## 6. Ключевые выводы

### Что я узнал нового

1. **HTTP — это просто текст с определённым форматом.** Не магия, не чёрный ящик. Метод + endpoint + host + заголовки + тело. Всё.

2. **API ≠ HTTP.** HTTP — глобальный стандарт, API — локальный договор. Можно отправить технически верный HTTP-запрос (200, 404 etc.), но получить 404 от API, потому что endpoint не существует.

3. **Классы эквивалентности ≠ граничные значения.** Это связанные, но разные техники. Классы — разбиваем данные на группы. Границы — проверяем стыки групп. Границы важнее.

4. **Золотое правило: каждую границу проверять с двух сторон.** -1/0, 17/18, 65/66 — обе точки должны быть протестированы. Разработчик буквально пишет эти числа в коде, и там легко ошибиться.

5. **Внутри класса достаточно одного значения.** Спорное утверждение (сообщество делится), но экономит массу времени. Если ошибка есть — она проявится на границе, а не в середине класса.

6. **Content-Type: text/plain вместо application/json — классический баг.** Простой, но может сломать парсинг у клиента. И обнаруживается только если проверять заголовки ответа в консоли Postman.

7. **Postman Console (View → Show Raw Log) — must-know.** Без неё не вытащить полный HTTP-лог для баг-репорта.

8. **PATCH как full update — антипаттерн, но встречается.** В документации честно написано «реализован как полный апдейт». Нужно быть внимательным к таким отклонениям от стандартов.

9. **DELETE в реальных компаниях ≠ удаление в БД.** Soft delete (is_deleted = true) — стандарт индустрии. Метод может не удалять данные, а просто скрывать доступ.

10. **Clear Database — служебные эндпоинты для тестировщиков — нормальная практика.** На тестовых средах часто есть методы, которые не тестируются как бизнес-функционал.

11. **Фаззинг — отправка невалидных данных (строка вместо числа, массив, спецсимволы) — эффективный способ найти баги**, о которых даже не подумаешь, читая документацию.

12. **Структура хорошего баг-репорта:** заголовок (что + где + что не так), объект, приоритет, шаги воспроизведения с curl/Postman, ожидаемый vs фактический результат, артефакты (raw log).

### Чего не знал раньше
- Концепцию Clear Database как служебного endpoint для тестировщиков
- Детали про Content-Type: text/plain — «запах жареного» при тестировании API
- Что query params в URL (всё после `?`) — это get-параметры, а не часть endpoint
- Метод POST может не только создавать, но и «перетирать» данные в БД, ломая другие запросы
- Важность проверки заголовков ответа (не только тело) — заголовки = часть контракта

### Резюме
Курс даёт **практическую базу для junior manual QA**, сфокусированную на REST API тестировании через Postman. Теория (HTTP, классы эквивалентности, граничные значения) подаётся ровно в том объёме, чтобы применить на практике. Основная ценность — не в глубине теории, а в **навыке мыслить как тестировщик**: читать документацию, выделять классы, проверять границы, оформлять баги, проверять фиксы.
