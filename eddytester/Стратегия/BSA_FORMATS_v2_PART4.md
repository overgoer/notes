# BSA_FORMATS_v2 — Часть 4: Форматы №4, №5

## Формат №4: «Угадай баг»

**Механика:** 3 скриншота/утверждения → 2 правды, 1 ложь → аудитория угадывает.
**Когда:** 1 раз в неделю.

---

### Пост 1: «Какой из этих статус-кодов — баг?»

[A] POST /api/users с правильным телом → `201 Created`

[Б] GET /api/users/999 (не существует) → `200 OK`, тело = `{}`

[В] DELETE /api/users/123 → `204 No Content`

Какой из них — баг? Пишите букву в комменты.

Правильный ответ через 2 часа.

🔥 если знали, ➕ если нет

---

### Пост 2: «Что здесь не так? Три запроса, один баг»

[A] `curl -X POST /api/transfer -d '{"amount": 100}'` → 200 OK, деньги переведены

[Б] `curl -X POST /api/transfer -d '{"amount": -100}'` → 200 OK, деньги... переведены?

[В] `curl -X POST /api/transfer -d '{"amount": 0}'` → 400 Bad Request

Где баг? Пишите.

Подсказка: один из ответов — правильный, но только один.

---

### Пост 3: «Кто из этих трёх разработчиков прав?»

Ситуация: API возвращает `500` на запрос с пустым телом.

Разработчик А: «Это баг. Сервер падает».
Разработчик Б: «Это фича. Клиент должен отправлять валидные данные».
Разработчик В: «500 — это ок, просто логи добавь».

Кто прав? Кто не прав? Почему?

Пишите. Через 2 часа разберу.

---

### Пост 4: «Найди баг в документации»

Документация API:
- `POST /users` — создаёт пользователя. Возвращает 201.
- `GET /users/:id` — возвращает пользователя. Возвращает 200.
- `DELETE /users/:id` — удаляет пользователя. Возвращает 200.
- `PATCH /users/:id` — обновляет пользователя. Возвращает 200.

Вопрос: где в документации баг?

Пишите. Через час ответ.
