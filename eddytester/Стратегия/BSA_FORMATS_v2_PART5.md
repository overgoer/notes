## Формат №8: «Разбор комментария»

**Механика:** Реальный комментарий подписчика → разбор в отдельном посте.
**Когда:** 1 раз в 1-2 недели.

---

### Пост 1: «Подписчик спросил: как тестировать PATCH, если он обновляет не все поля?»

«@user123: Я тестирую PATCH /users/:id. Отправляю только name — он обновляется. Но если отправляю только age — возвращается старый name. Это баг?»

Отвечаю:

PATCH по RFC должен обновлять только те поля, которые ты отправил. Если ты шлёшь `{ "age": 25 }`, сервер должен обновить только age. name — остаться как был.

Но есть нюанс: многие разработчики пишут PATCH как PUT — перезаписывают весь объект. Тогда:
- PATCH с `{ "age": 25 }` → name становится null
- PATCH с `{ "name": "new" }` → age становится null

Это баг. Проверяется так:

```bash
curl -X PATCH /api/users/1 -d '{"name": "test1"}'  # → name=test1, age=old
curl -X PATCH /api/users/1 -d '{"age": 99}'       # → age=99, name=test1 (должен)
# Если name стал null — это баг PATCH, работающего как PUT
```

Спасибо за вопрос. Есть ещё? Пишите.

### Пост 2: «Подписчик: "Сервер возвращает 200 на всё. Даже на 404"»

«@user456: Я тестирую API. Любой запрос — GET, POST, DELETE на несуществующий ресурс — возвращает 200 OK. Разработчик говорит: "Так проще. Фронт всё равно смотрит тело". Я считаю — баг. Кто прав?»

Отвечаю:

Разработчик неправ. 200 на DELETE несуществующего ресурса — это баг.
Во-первых, мониторинг: если у тебя 100% 200-х ответов, ты не увидишь, что что-то сломалось.
Во-вторых, кэширование: 200 может закэшироваться там, где не нужно.
В-третьих, это просто непрофессионально.

Ты прав. Держись.
