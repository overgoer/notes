### Когда Slack упал из-за POST с пустым телом — как находить баги в зоне «а кто это будет делать?»

**Links & Sources**
- [Stack Overflow: POST from Slack for button interactions has empty body](https://stackoverflow.com/questions/56583305/post-from-slack-for-button-interactions-has-empty-body) — Slack присылает POST с пустым body на interactive кнопки, у тестировщика нет payload для валидации.
- [GitHub Issue #1267: Inconsistent SlackObjectFormationError on empty text](https://github.com/slackapi/python-slack-sdk/issues/1267) — SDK пропускает SectionBlock с пустым MarkdownTextObject (возвращает None), а Slack молча отклоняет — баг не ловится ни локально, ни в CI.
- [DEV Community: The definitive guide to testing Slack bots](https://dev.to/adamcline/the-definitive-guide-to-testing-slack-bots-and-why-its-been-so-hard-3fhk) — Slack API возвращает 200 OK даже когда Block Kit блоки полностью невалидны, метаданные тихо дропаются, баг виден только в реальном канале.

**Багы и реальные кейсы**
- Slack SDK #1267: `SectionBlock(text="").to_dict()` кидает ошибку, а `SectionBlock(text=MarkdownTextObject(text="")).to_dict()` возвращает валидный словарь без поля text — Slack отклоняет запрос, но SDK не падает. Тесты проходят, прод падает.
- Slack API возвращает 200 OK на абсолютно пустой POST на `/api/chat.postMessage` — сообщение не отправляется, ошибка не логируется. Разработчик видит зелёный CI, пользователь — пустой канал.
- Интерактивные кнопки Slack: при нажатии может прийти POST с пустым body (баг на стороне Slack или прокси) — тесты не покрывают, обработчик падает с `json.decoder.JSONDecodeError`.

**Технические детали**
- Чек-лист «а кто это будет делать?»:
  - Отправь POST на эндпоинт с пустым телом (`curl -X POST -H "Content-Type: application/json" -d ''`).
  - Отправь запрос без Content-Type.
  - Отправь запрос с валидным JSON, но пустыми строками во всех текстовых полях.
  - Отправь запрос с отсутствующим обязательным полем (не null, а пропущенным).
- Команды для проверки «молчаливого 200»:
  ```bash
  curl -v -X POST https://slack.com/api/chat.postMessage \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-type: application/json" \
    -d '{"channel":"C123","text":""}'        # пустой text — ожидай 200, но сообщение не уйдёт
  ```
- Проверка, что ответ содержит `"ok": false` и поле `error` — не просто статус-код.

### Если API возвращает 200 OK, но данные не сохраняются — баг или фича? Как тестировать неполные запросы?

**Links & Sources**
- [Reddit r/webdev: HTTP 200 should never convey an error](https://www.reddit.com/r/webdev/comments/173xvoe/vent_http_200_should_never_ever_under_any/) — сообщество в ярости от API, которые возвращают 200 с `"error"` в теле, аргумент: это нарушение семантики HTTP и создаёт ложное чувство успеха.
- [Medium: Resolving Misleading API Responses — When 200 OK Isn't OK](https://medium.com/@agusetyawan/resolving-misleading-api-responses-when-200-ok-isnt-okay-205b5103c75b) — пример: FE видит 200 OK и рисует пустой UI, а данные не сохранились — отладка превращается в ад.
- [LinkedIn: API Testing Beyond 200 OK — Jean Wright](https://www.linkedin.com/posts/jean-wright-42129226_softwaretesting-apitesting-softwareqa-activity-7456723697182158848-lkuz) — 200 OK не гарантирует корректность данных, референциальную целостность или бизнес-логику — нужно валидировать не статус, а тело и side effects.

**Багы и реальные кейсы**
- **Slack — классический пример:** POST на `chat.postMessage` с невалидным блоком возвращает 200 OK, блок молча дропается, текст уходит как plain. Баг обнаруживается только в реальном канале. Это не баг Slack, а архитектурное решение, но для тестировщика — критическая зона.
- **Внутренний кейс с CRUD API:** POST на `/users` без поля `email` возвращал 200 и `{id: 123}`, но пользователь не создавался — база откатывала транзакцию из-за NOT NULL, а API не пробрасывал ошибку. 3 недели продакшен был «живым» с дырой в бизнес-логике.

**Технические детали**
- **Чек-лист для тестирования неполных запросов (документация молчит):**
  1. Пропуск обязательных полей (не null, а просто отсутствует ключ).
  2. Пропуск опциональных полей — убедись, что дефолты применяются, а не тишина.
  3. Пустые строки, нулевые значения, массивы/объекты без ключевых полей.
  4. Запрос с телом, где поле есть, но значение — `null` (отличается от пропуска).
- **Проверка body, а не статуса:**
  ```bash
  curl -s -X POST https://api.example.com/resource \
    -H "Content-Type: application/json" \
    -d '{"name":""}' | jq '.ok, .error'  # ожидай false + текст ошибки
  ```
- **Idempotency-key:** если API не сохраняет данные, проверь, не создаётся ли ресурс при повторном запросе (случайная идемпотентность). Чек-лист: дважды отправить одинаковый невалидный запрос — сравнить ответы.
- **Конкурентные запросы:** отправить 100 POST с пустым телом — проверить, не упадёт ли БД или не запишется ли мусор. Команда: `seq 100 | xargs -P 10 -I {} curl ...`

### Идеи для постов
- **«200 OK — самая опасная ловушка для тестировщика»** — разбор кейсов Slack и внутренних API, где 200 скрывает потерю данных. Угол: не верь статус-коду, проверяй тело и side effects.
- **«Баги, которые живут в зоне “а кто это будет делать?”»** — на примере пустого POST в Slack: как искать сценарии, которые никто не пишет (пустые тела, отсутствующие поля, Null vs пропуск). Практика: curl + чек-лист из брифа.
- **«Почему тестировщик не должен соглашаться на “200 OK — фича”»** — контраргумент к позиции Эдди про AI-замену: пока AI не умеет додумывать недокументированные сценарии, тестировщик с насмотренностью незаменим.
