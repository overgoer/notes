### Как найти «тихие» баги API при лишних полях в JSON?

**Links & Sources**
- [Avoiding silent errors in your Stripe integration | Stripe Dev Blog](https://stripe.dev/blog/avoiding-silent-errors) — молчаливые ошибки возникают из-за невалидных данных, которые не выбрасывают исключений, а просто игнорируются или приводят к неверным расчётам.
- [Error handling – Stripe Docs](https://docs.stripe.com/error-handling) — официальное руководство по обработке ошибок: как принудительно вызывать failure‑сценарии (например, declined payment) и проверять ответ.
- [Stripe webhook pattern looks correct but silently breaks – Reddit](https://www.reddit.com/r/webdev/comments/1sukfbe/this_stripe_webhook_pattern_looks_correct_but/) — failure‑handler часто впервые тестируется в продакшене, потому что код не предусматривает обработку ошибочных вебхуков.

**Багы и реальные кейсы**
- Передача лишнего поля `"foo": "bar"` в `POST /v1/charges` не вызывает ошибку Stripe — API молча игнорирует неизвестные поля. Но бэкенд компании, парсящий ответ, может ожидать строгую схему и упасть с NPE при отсутствии ожидаемого поля.
- В одном проекте Stripe‑вебхук `payment_intent.succeeded` не обрабатывался, если в payload был лишний `"extra_metadata": {}` — бэкенд рвал JSON, и статус платежа оставался `processing`, хотя деньги списались.

**Технические детали**
- curl с лишним полем:  
  `curl -X POST https://api.stripe.com/v1/charges -u sk_test_... -d "amount=2000¤cy=usd&source=tok_visa&extra_field=ignore_me"`
- Чек-лист для ручного QA:
  1. Добавить одно неизвестное поле в каждый запрос → убедиться, что ответ 200 и бизнес-логика не сломана.
  2. Добавить поле с null-значением → проверить, что не выбрасывается 500.
  3. Добавить поле с неверным типом (строка вместо числа) → ожидать ошибку валидации, а не молчаливое игнорирование.
  4. Проверить, что Stripe возвращает сам факт игнорирования в `warning` или `extra_info` (документация не гарантирует, но можно залогировать).
- Использовать `Stripe CLI` для симуляции событий: `stripe trigger payment_intent.succeeded` с кастомным payload, содержащим лишние поля.

### Как не пропустить 500 при пустом теле запроса?

**Links & Sources**
- [Fixing a 500 internal server error response – Postman support](https://support.postman.com/hc/en-us/articles/6018809487255-Fixing-a-500-internal-server-error-response) — пошаговая диагностика 500: проверить тело, заголовки, окружение.
- [Request body is empty when posting form‑data – Stack Overflow](https://stackoverflow.com/questions/70250715/request-body-is-empty-when-posting-a-postman-form-data) — отсутствие middleware или парсера тела приводит к пустому `req.body` и крашу.
- [Empty body after send request block – Postman Community](https://community.postman.com/t/empty-body-after-send-request-block/50037) — ошибка при логировании в Flow: тело пустое, хотя запрос успешен — путает тестировщика и маскирует реальную проблему.

**Багы и реальные кейсы**
- Бэкенд на Express.js без `body-parser` (или с неправильным `Content-Type`) — на пустой POST сервер получает `req.body === undefined` и падает с 500, потому что пытается обратиться к `req.body.field`.
- Тестировщик через Postman всегда отправлял `{ "data": "test" }` — всё работало. Продакшен упал, когда мобильный клиент прислал `POST /api/order` без тела (Content‑Type: application/json, тело пустое). Причина: бэкенд не проверяет наличие тела перед десериализацией.

**Технические детали**
- curl с пустым телом:  
  `curl -X POST https://example.com/api/endpoint -H "Content-Type: application/json" -d ""`
- Postman: создать запрос с пустым `Body` (выбрать `none` или `raw` с пустой строкой). Добавить тест:  
  ```javascript
  pm.test("Status is not 500", () => { pm.response.to.not.have.status(500); });
  ```
- Чек-лист:
  1. Для каждого POST/PUT/PATCH: отправить запрос без тела → ожидать 400 (Bad Request) или 422, а не 500.
  2. Отправить тело с `null` / `{}` / `""` — проверить валидацию.
  3. Сменить `Content-Type` на `text/plain` при JSON-эндпоинте — бэкенд может вернуть 415, но не 500.
  4. Проверить логи сервера на наличие `TypeError: Cannot read properties of undefined`.
- Если бэкенд на Spring Boot: активировать `spring.mvc.throw-exception-if-no-handler-found=true` и добавить глобальный обработчик `@ExceptionHandler(HttpMessageNotReadableException.class)`.

### Идеи для постов
1. **«Тихие» баги в API: почему ИИ их не найдёт, а ручной тестировщик обязан»** — разбор кейса с лишним полем в Stripe и пустым телом. Контраргумент к хайпу про замену QA: бизнес-логика не укладывается в промпты.
2. **«Postman — ловушка для самоуверенного тестировщика»** — как слепая вера в «зелёный» ответ маскирует проблемы с пустым телом или игнорируемыми полями. Привести пошаговый чек-лист.
3. **«Ты не аудитор, пока не проверил граничные случаи API»** — перекликаясь с позицией Эдди о росте роли тестировщика до аудитора процесса. Показать, что ручные тесты на «тихие» ошибки — единственный способ гарантировать надёжность без автотестов.
