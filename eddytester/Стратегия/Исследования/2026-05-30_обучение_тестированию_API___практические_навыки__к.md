### Когда API банка вернул 200 OK с телом `{status: "error"}` вместо 422 — как тестировать валидацию кода ответа, а не только формат?

**Links & Sources**
- [StackOverflow: статус для validation errors](https://stackoverflow.com/questions/3290182/which-status-code-should-i-use-for-failed-validations-or-invalid-duplicates) — рекомендовано 422 для ошибок валидации, 409 для дубликатов.
- [Medium: Status Code & Response Validation](https://sureshparimi.medium.com/post-1-status-code-response-validation-in-api-testing-fde0aa529a5f) — общий принцип: проверять не только код, но и смысл ошибки.
- [Merge.dev: API response codes](https://www.merge.dev/blog/api-response-codes) — разбор стратегий обработки ошибок, включая костыли с 200+body.
- [MDN: HTTP status codes](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status) — полная классификация кодов.
- [Scoop Labs: Why Junior QAs Fail](https://scooplabs.in/blog/why-junior-qas-fail-first-api-testing-task) — причина: отсутствие визуальных подсказок, джуны не различают код и тело ответа.

**Багы и реальные кейсы**
- Банковский API на невалидный номер карты возвращал 200 OK с `{"status":"error","message":"invalid card"}`. Фронтенд не обрабатывал тело, пользователь видел успех, а деньги не списывались. Настоящий баг: код ответа должен был быть 422.
- Есть API, который все ошибки (включая 401, 404) отдаёт с 200 и полем `error`. Это ломает стандартные HTTP-клиенты и кэширующие прокси.

**Технические детали**
- `curl -w "%{http_code}" -X POST -d '{"amount":-1}' https://api/transfer` — проверка кода отдельно от тела.
- Чек-лист по негативным сценариям: для каждого предусмотреть ожидаемый `4xx`, а не только `200` с `error`.
- Скрипт на Python:  
  ```python
  resp = requests.post(url, json={"amount": -1})
  assert resp.status_code == 422, f"Expected 422, got {resp.status_code}"
  assert resp.json().get("status") == "error"  # попутно проверить тело
  ```
- Идея Эдди: AI-агенты часто генерят тесты только на 200/404 и пропускают такие семантические ошибки — подтверждает тезис о «бесполезности ИИ» в сложной валидации.

### Если в документации указано «поле author — uuid» а реально возвращается `author: 12345` — как не пропустить несоответствие типов на уровне интеграции?

**Links & Sources**
- [GitHub Issue: UUID not properly supported (int assumption)](https://github.com/awcodes/filament-curator/issues/690) — реальный баг: код ожидал `int`, а не UUID.
- [StackOverflow: Как валидировать UUID](https://stackoverflow.com/questions/7905929/how-to-test-valid-uuid-guid) — regex и методы проверки.
- [Pydantic: Validation Errors для int vs UUID](https://pydantic.dev/docs/validation/latest/errors/validation_errors/) — примеры ошибок несоответствия типа.

**Багы и реальные кейсы**
- Микросервис интеграции возвращал `author: 12345` вместо строки UUID. Клиент падал, потому что в коде было `str(uuid)` — пришёл int.
- В JSON-сериализаторе библиотеки случайно сконвертировали UUID в int (реальный баг из PHP-проекта Filament Curator). Фронтенд сломался: ожидал дефисный формат.

**Технические детали**
- Проверять тип поля через `jq`:  
  `curl -s https://api/resource | jq '.author | type'` — должно вернуть `"string"`.
- Использовать JSON Schema:  
  ```json
  {
    "type": "object",
    "properties": {
      "author": {"type": "string", "pattern": "^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$"}
    }
  }
  ```
- В тестах на Python:  
  ```python
  import uuid
  resp = requests.get(url).json()
  assert isinstance(resp["author"], str), "author должен быть строкой"
  uuid.UUID(resp["author"])  # вызовет ValueError если невалидный
  ```
- Мнение Эдди: ИИ-помощники часто не различают типы, если не задать явный промпт — это аргумент против полной замены тестировщика. **Контраргумент**: но если настроить pipeline с JSON Schema, то такая проверка делается автоматом – ИИ тут не нужен, а человек должен знать, что schema нужно писать.

### Идеи для постов

- **Почему 200 OK с error в теле — это баг, а не фича** — кейс банковского API, разница между HTTP-протоколом и бизнес-логикой.
- **UUID или int? Как джуны пропускают несоответствие типов** — реальный баг из Filament Curator, подходы к валидации (JSON Schema, тесты на тип).
- **AI-агенты для тестирования API: 1500 агентов сожгли $25k, а баги остались** — отсылка к твиту Эдди, показать, что ИИ пока не ловит семантические ошибки (200-error, type mismatch).
