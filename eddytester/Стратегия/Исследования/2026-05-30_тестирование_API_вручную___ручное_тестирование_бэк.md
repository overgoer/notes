# Research: тестирование API вручную — ручное тестирование бэкенда, что проверять в API без автотестов, ручные техники

**Date:** 2026-05-30

### Когда ручной тестер случайно положил продакшен, отправив DELETE без idempotency-key

**Links & Sources**
- [REST API Tutorial: Idempotency](https://restfulapi.net/idempotent-rest-apis/) — идемпотентность означает, что повторные одинаковые запросы не меняют состояние ресурса; DELETE по HTTP‑спецификации идемпотентен, но на практике многие API требуют idempotency‑key для защиты от дублей.
- [Medium: API Idempotency Testing — The Most Misunderstood API Problem](https://medium.com/@gunashekarr11/api-idempotency-testing-the-most-misunderstood-api-problem-b817ef3534f9) — разбирает кейс: если DELETE‑эндпоинт не проверяет idempotency‑key, повторная отправка того же DELETE может привести к удалению соседнего ресурса (если ключ не хранится).
- [Google Cloud: What is Idempotency?](https://cloud.google.com/discover/idempotency) — ключевой принцип: повторные запросы должны давать тот же эффект, что и первый; без идемпотентности, одна капризная сеть приводит к двойным списаниям или удалениям.

**Багы и реальные кейсы**
- Баг: тестер на staging отправил `DELETE /orders/123` без idempotency‑key. Система не хранила ключи — повторный запрос из Postman (из‑за таймаута) ушёл как новый и удалил `order/124` (сдвиг индекса или race condition). На проде то же самое — потеря заказов.
- Другой случай: ручной тестер во время теста деплоя отправил `DELETE /users/42` в цикле (скрипт без идемпотентности), и API последовательно удалял пользователей, пока не упал с 500 — потому что DELETE без ключа трактовался как «удалить следующий».

**Технические детали**
- Проверка идемпотентности DELETE вручную:
  ```bash
  # 1. Создать ресурс (POST) и запомнить его ID
  curl -X POST https://api.example.com/items -H 'Content-Type: application/json' -d '{"name":"test"}'

  # 2. Отправить DELETE с заголовком Idempotency-Key
  curl -X DELETE https://api.example.com/items/123 \
    -H 'Idempotency-Key: uuid-1'

  # 3. Отправить тот же DELETE повторно с тем же ключом
  curl -X DELETE https://api.example.com/items/123 \
    -H 'Idempotency-Key: uuid-1'
  # Ожидание: статус 200/204 (как в первый раз), тело то же или пустое, ресурс не удаляется дважды.

  # 4. Отправить DELETE без ключа
  curl -X DELETE https://api.example.com/items/456
  # Если API без ключа отклоняет запрос (409 Conflict или 400 Bad Request) — хорошо. Если принимает и удаляет — это баг.
  ```
- Чек‑лист ручной проверки:
  - Удалить ресурс, потом повторить DELETE с тем же idempotency‑key → ответ должен быть одинаковым, ресурс должен оставаться в состоянии «удалён» (не создавать новый, не выдавать ошибку).
  - Повторить DELETE без ключа после успешного удаления → должен прийти 404 (Not Found) или идемпотентный ответ 200/204? Спорно: по HTTP‑спецификации DELETE после удаления может вернуть 404, но это нормально; главное — без side‑effects.
  - Отправить DELETE с невалидным ID и заведомо новым ключом → получить 404.

### Как вручную проверить, что повторный идемпотентный запрос возвращает одинаковый ответ

**Links & Sources**
- [Medium: Idempotent APIs: A Deep Dive for Backend Engineers](https://medium.com/@shubhamkchoudhary07/idempotent-apis-how-a-booking-bug-made-me-obsess-over-them-a-deep-dive-for-backend-engineers-9a846006c5b3) — практический случай: payment API дважды списал деньги из‑за того, что повторный POST не проверили. Автор рекомендует для ручного теста: 3 одинаковых запроса подряд и сравнение статуса, тела, заголовков (особенно `idempotent-replayed`).
- [Stack Overflow: How to keep an API idempotent while receiving multiple requests](https://stackoverflow.com/questions/31621970/how-to-keep-an-api-idempotent-while-receiving-multiple-requests-with-the-same-id) — решение: distributed lock по request ID; для ручной проверки достаточно сериализовать запросы и смотреть, не создаётся ли дубль.
- [LinkedIn post: Idempotency in API Design](https://www.linkedin.com/posts/prerna-saxena3_backenddevelopment-apidesign-systemdesign-activity-7429381595398144000-gqt1) — говорит, что идемпотентный PUT с теми же данными должен возвращать один и тот же HTTP‑статус и тело при каждом вызове.

**Багы и реальные кейсы**
- Баг: вручную тестировали PUT /users/42 с полем `role=admin`. Первый запрос вернул `200 { "status": "updated" }`. Второй — `200 { "status": "already updated" }`. Разные ответы — нарушение идемпотентности. На проде клиенты кешировали первый ответ и падали на втором.
- Баг: POST /payments с idempotency‑key при повторной отправке возвращал `201 Created` (как новый) вместо `200 OK` с тем же ID транзакции — клиент создавал дубликат платежа.

**Технические детали**
- Ручной тест на идемпотентность ответа:
  ```bash
  # Выполнить POST с ключом и запомнить ответ
  RESP1=$(curl -s -w "%{http_code}" -X POST https://api.example.com/orders \
    -H 'Idempotency-Key: uuid-test-1' \
    -H 'Content-Type: application/json' \
    -d '{"product":"laptop"}')
  echo "$RESP1"

  # Повторить тот же запрос
  RESP2=$(curl -s -w "%{http_code}" -X POST https://api.example.com/orders \
    -H 'Idempotency-Key: uuid-test-1' \
    -H 'Content-Type: application/json' \
    -d '{"product":"laptop"}')
  echo "$RESP2"

  # Сравнить статус, заголовки (особенно Content-Location, Location), тело
  diff <(echo "$RESP1") <(echo "$RESP2")
  ```
- Чек‑лист:
  - **Статус‑код**: должен совпадать (например, 200OK для повторных и 201 для первого? Спорно. По‑хорошему первый создаёт ресурс (201), повторные возвращают 200, но некоторые API возвращают 200 всегда. Важно, чтобы клиент не ломался. Ручная проверка: читать документацию и сверять.)
  - **Тело ответа**: для GET, PUT, DELETE тело должно быть идентичным (для POST — как минимум не содержать новый ID ресурса). Использовать `curl` с флагами `--write-out '%{http_code}' --output /tmp/resp1.json` и затем `md5sum`.
  - **Заголовки**: `Idempotent-Replayed: true` (опционально), `Cache-Control`, `Date` (допускается разница).
  - **Side‑effects**: проверить в базе (через отдельный GET), что количество записей не увеличилось.
