## Когда DELETE возвращает 200 с пустым телом — почему ломается идемпотентность и как ловить

**Links & Sources**
- [RFC 7231 — HTTP Semantics (DELETE)](https://tools.ietf.org/html/rfc7231#section-4.3.5) — спецификация: DELETE должен возвращать 202 (принято), 204 (успешно, без тела) или 200 (с телом). Повторный DELETE к тому же ресурсу должен возвращать 404, если ресурс удалён.
- [HTTP Status Codes by Mozilla](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status) — 204 No Content не должен содержать тела, 200 OK может, но для идемпотентности различие критично.

**Багы и реальные кейсы**
- Баг в API интернет-магазина: первый DELETE возвращал 200 с пустым JSON `{}`, повторный — тоже 200. Клиент не понимал, удалён ли товар. На самом деле сервер просто не проверял существование ресурса. Пользователи видели «удалено» даже для уже удалённого — потеряли контроль над корзиной.
- Кейс с микросервисом заказов: DELETE заказа возвращал 200 с пустым телом, но не удалял запись в БД (только мягкий флаг deleted_at). Повторный DELETE = та же 200, но данные оставались. Баг нашли только когда статистика считала удалённые заказы как активные.

**Технические детали**
- Команды для проверки:
  ```bash
  # Первый DELETE
  curl -X DELETE http://api.example.com/users/1 -v
  # Ожидаем 204 (без тела) или 200 с телом
  # Повторный DELETE
  curl -X DELETE http://api.example.com/users/1 -v
  # Ожидаем 404 Not Found
  ```
- Чек-лист:
  1. Проверить, что сервер не возвращает 200 на повторный DELETE, если ресурс уже удалён.
  2. Проверить, что тело ответа не пустое при 200, если сервер решает вернуть 200 — должно описать результат.
  3. Проверить, что идемпотентность сохраняется: повторный запрос не создаёт сайд-эффектов (например, повторное удаление из БД).
  4. Проверить поведение при DELETE несуществующего ресурса: должен быть 404, а не 200.
- Если API возвращает 200 с пустым телом — считай багом, пиши в репорт с указанием RFC.

## Кэш на 10 минут — как воспроизвести баг с устаревшими данными после обновления

**Links & Sources**
- [MDN: HTTP Caching](https://developer.mozilla.org/en-US/docs/Web/HTTP/Caching) — как работают Cache-Control, ETag, Last-Modified.
- [RFC 7234 — HTTP Caching](https://tools.ietf.org/html/rfc7234) — правила инвалидации кэша при мутациях.
- [Статья про stale cache bugs в микросервисах (eng)](https://ably.com/blog/caching-strategies-microservices) — конкретный пример: при обновлении пользователя не инвалидируется CDN-кэш, данные устаревают на часы.

**Багы и реальные кейсы**
- Типовой баг: фронтенд кэширует список юзеров на 10 минут. Бекенд при PATCH /users/1 не очищает кэш. Юзер меняет имя — в списке остаётся старое. Воспроизвести: сделать PATCH, затем сразу GET /users/1 — может прийти старый ответ из кэша.
- Кейс с ролью админа: админ меняет права юзера, но кэш отдаёт старые права. Атакующий юзер может 10 минут пользоваться устаревшим доступом. Баг пропустили, потому что тест не проверял порядок: UPDATE → GET.

**Технические детали**
- Команды для воспроизведения:
  ```bash
  # Устанавливаем заголовки кэша (если тестируем вручную)
  GET /users/1
  # Ответ приходит с Cache-Control: max-age=600
  # Обновляем юзера
  PATCH /users/1 -d '{"name":"new"}'
  # Тот же GET
  GET /users/1
  # Если пришло старое имя — баг.
  ```
- Чек-лист тестирования кэша:
  1. Для каждого read-ендпоинта (GET) проверить наличие Cache-Control, ETag.
  2. После мутирующих запросов (POST/PUT/PATCH/DELETE) проверить, что повторный GET возвращает актуальные данные.
  3. Использовать `curl -H "Cache-Control: no-cache"` чтобы гарантированно получить свежий ответ от сервера.
  4. В автоматизации: замерить время — если кэш не сброшен, второй GET может быть мгновенным и с устаревшими данными.
  5. Проверить разные кэш-слои: браузерный, CDN, серверный (Redis, Varnish).
- Утилита для принудительного сброса: `curl -X PURGE http://example.com/users/1` (если настроено Varnish) или запрос с `Cache-Control: max-age=0`.
