### POST и Content-Type: сервер вернул XML, а клиент ждал JSON

**Links & Sources**
- [Postman Community: «content-type of the request should be text/xml»](https://community.postman.com/t/content-type-of-the-request-should-be-text-xml-error-when-connecting-to-salesforce-via-soap/26478) — причина: Postman не устанавливал нужный Content-Type, сервер отвечал 500.
- [GitHub Issue #327: выбор JSON в Postman не проставляет Content-Type](https://github.com/postmanlabs/postman-app-support/issues/327) — баг Postman: остаётся `application/xml` при выборе JSON.
- [StackOverflow: запрос из Postman не отправляет JSON](https://stackoverflow.com/questions/62617615/post-request-from-postman-seems-to-not-be-sending-any-json-data) — нужно использовать body-parser, иначе данные не передаются.

**Багы и реальные кейсы**
- Джун отправил POST с телом JSON, но не проверил **Content-Type ответа**. Сервер вернул `text/xml` (например, SOAP-заглушка). Клиент на фронте упал с ошибкой парсинга. В Postman видно на вкладке `Headers` ответа — `Content-Type: text/xml`, а не `application/json`.

**Технические детали**
- **Как проверить Content-Type ответа в Postman:**
  - Вручную: после отправки — раздел `Headers` ответа, строка `Content-Type`.
  - Скрипт в `Tests`:
    ```javascript
    pm.response.to.have.header("Content-Type");
    pm.expect(pm.response.headers.get("Content-Type")).to.include("application/json");
    ```
- **Как правильно задать Content-Type запроса:**
  - Вручную добавить заголовок `Content-Type: application/json` (если авто-подстановка не сработала).
  - Чек-лист для POST:
    - ✅ Проверить, что в запросе указан `Content-Type`.
    - ✅ Проверить, что в ответе `Content-Type` соответствует ожидаемому.
    - ✅ Если ответ XML — искать баг в настройках сервера или заголовке запроса.

**Связь с позицией Эдди**  
Эдди в заметках говорит, что AI воспринимается как «дешёвый читкод». Здесь контраргумент: даже если AI генерирует тесты, джун обязан вручную убедиться, что Content-Type корректен — AI может не проверить заголовки. Это подтверждает необходимость ручного тестирования как самостоятельного навыка, а не «лишнего слоя».

### DELETE: 200 OK с пустым телом против 204 No Content

**Links & Sources**
- [Medium: REST Response Code for DELETE — 200 vs 204](https://medium.com/@sameeranel/rest-response-code-for-delete-operation-200-or-204-ee9c72a2dcb2) — RFC 7231: при успешном удалении без тела надо возвращать 204.
- [LinkedIn: Delete with REST — 200 or 204](https://www.linkedin.com/pulse/delete-rest-200-204-response-code-abhijit-mazumder) — цитата из RFC: сервер **SHOULD** отдавать 204 (или 202, или 200 с телом).
- [StackOverflow: HTTP status for update and delete](https://stackoverflow.com/questions/2342579/http-status-code-for-update-and-delete) — консенсус: 204 для DELETE, 200 только если есть тело ответа.

**Багы и реальные кейсы**
- API возвращает `200 OK` и пустое тело `{}`. Фронтенд на `axios` обрабатывает 200 как успех, но при попытке разобрать `response.data` получает `undefined` — падает. Если бы был `204`, клиент бы просто завершил запрос без парсинга тела. Реальный кейс: джун написал тест на `pm.response.to.have.status(200)` — тест прошел, но баг остался.

**Технические детали**
- **Как отличить в Postman:**
  - Статус-код: 204 — «No Content», тело ответа пустое (даже без `{}`).
  - 200 — тело может быть пустым, но это нарушение конвенции для DELETE.
- **Скрипт для проверки:**
  ```javascript
  pm.test("Status is 204 No Content", function () {
      pm.response.to.have.status(204);
      pm.expect(pm.response.text()).to.equal('');
  });
  ```
- **Чек-лист для DELETE-запросов:**
  - ✅ Проверить, что статус **204** (не 200, не 202, не 404).
  - ✅ Если статус 200 — убедиться, что тело содержит информацию об удалённом ресурсе (не пустое).
  - ✅ Проверить, что повторный DELETE на тот же ресурс возвращает **404** (ресурс удалён).

**Связь с позицией Эдди**  
Эдди пишет, что «специализация тестировщика растет вширь» и он должен давать втык разработчику. Этот кейс — именно такой: джун, зная RFC, может указать разработчику на несоответствие статуса. AI-агенты из его твита не смогут этого сделать, если не обучены на конвенциях REST. Значит, роль тестировщика-аудитора не заменима.
