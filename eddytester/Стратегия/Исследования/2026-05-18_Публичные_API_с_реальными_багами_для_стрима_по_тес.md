### Swagger Petstore: duplicate ID возвращает 200 вместо 409

**Links & Sources**  
- [Swagger Petstore](https://petstore.swagger.io) — демонстрационное API, где POST /pet не проверяет уникальность id, возвращает 200 при дублировании, документация не описывает ограничение.  
- [Swagger Editor](https://editor.swagger.io) — инструмент для просмотра спецификации Petstore; спецификация не указывает на запрет дублирования.

**Багы и реальные кейсы**  
- Баг: отправка POST /pet с уже существующим id (например, 1) возвращает 200 OK, а не 409 Conflict. Запись перезаписывается новыми данными.  
- Кейс: тестировщик создаёт питомца с id=1, затем создаёт другого с тем же id — система не сообщает об ошибке, данные первого питомца теряются.  

**Технические детали**  
- Первый запрос:  
  `curl -X POST 'https://petstore.swagger.io/v2/pet' -H 'Content-Type: application/json' -d '{"id":1,"name":"doggie"}'`  
- Второй запрос:  
  `curl -X POST 'https://petstore.swagger.io/v2/pet' -H 'Content-Type: application/json' -d '{"id":1,"name":"cat"}'`  
- Проверка: `curl -X GET 'https://petstore.swagger.io/v2/pet/1'` — вернёт `name: cat` (перезапись).  
- Уровень сложности: низкий.

### JSONPlaceholder: POST /posts возвращает 201, но данные не сохраняются

**Links & Sources**  
- [JSONPlaceholder](https://jsonplaceholder.typicode.com) — фиктивный API, все ответы статичны; созданные записи не персистентны.  
- По этому углу информации почти нет (результаты поиска ушли на скачивание Chrome).  

**Багы и реальные кейсы**  
- Баг: POST /posts возвращает 201 с фиктивным id (например, 101), но последующий GET /posts/101 даёт 404 — данные не сохраняются.  
- Кейс: разработчик получает 201 и игнорирует проверку — на production при повторном чтении ресурс отсутствует.  

**Технические детали**  
- Создание: `curl -X POST 'https://jsonplaceholder.typicode.com/posts' -H 'Content-Type: application/json' -d '{"title":"foo","body":"bar","userId":1}'`  
- Ответ: `201, id: 101`.  
- Проверка: `curl -X GET 'https://jsonplaceholder.typicode.com/posts/101'` — 404.  
- Чек-лист: всегда делать GET после POST; проверять наличие заголовка Location и возвращённого id.  
- Уровень сложности: низкий.

### OpenWeatherMap: несуществующий город возвращает 200 с дефолтными данными

**Links & Sources**  
- [OpenWeatherMap API](https://openweathermap.org/api) — документация; для неверного названия должен быть 404, на практике возвращается 200.  
- По этой теме в результатах поиска информации нет (ссылки на LinkedIn).  

**Багы и реальные кейсы**  
- Баг: запрос `/weather?q=несуществующийгород` возвращает 200, поле `cod: 200`, а в теле — данные другого города (или пустые).  
- Кейс: пользователь вводит неправильный город, приложение отображает погоду для Лондона — аномалия на production.  

**Технические детали**  
- Команда:  
  `curl 'http://api.openweathermap.org/data/2.5/weather?q=asdfghjkl&appid=YOUR_KEY'`  
- Ожидаемый ответ: `404, {"cod":404,"message":"city not found"}`.  
- Фактический ответ: `200, {"cod":200,"name":""}` или название другого города.  
- Чек-лист: проверять не только статус, но и поля `cod` и `name` в теле; сравнивать `name` с запрошенным.  
- Уровень сложности: средний.

### GitHub API: POST /user/repos без токена даёт 404 вместо 401

**Links & Sources**  
- [GitHub API Documentation](https://docs.github.com/en/rest) — описание эндпоинта и требований аутентификации.  
- [GitHub Blog](https://github.blog) — статьи о безопасности, в т.ч. намеренный возврат 404 для скрытия ресурсов.  

**Багы и реальные кейсы**  
- Баг: запрос POST /user/repos без токена возвращает 404 Not Found, хотя по REST должен быть 401 Unauthorized.  
- Кейс: пентест ожидает 401 для проверки аутентификации — получает 404 и ошибочно считает, что эндпоинт не существует.  

**Технические детали**  
- Команда:  
  `curl -X POST -H "Accept: application/vnd.github.v3+json" https://api.github.com/user/repos -d '{"name":"test"}'`  
- Ответ: `404, {"message":"Not Found"}`.  
- Ожидаемый: `401, {"message":"Bad credentials"}`.  
- Чек-лист: тестировать аутентификацию на всех эндпоинтах; документировать расхождение как баг безопасности (хоть и намеренное).  
- Уровень сложности: средний.

### Restful Booker: двойное бронирование возвращает 200

**Links & Sources**  
- [Restful Booker](https://restful-booker.herokuapp.com) — популярное API для тестирования; предоставляет бронирование комнат.  
- По этому углу информации из поиска нет (ссылки на врачей).  

**Багы и реальные кейсы**  
- Баг: два одинаковых POST /booking с одними датами возвращают 200 OK и разные bookingid — отсутствует проверка уникальности временного слота.  
- Кейс: система брони может дважды забронировать одну комнату на одно время — двойной счёт и конфликт.  

**Технические детали**  
- Первое бронирование:  
  `curl -X POST 'https://restful-booker.herokuapp.com/booking' -H 'Content-Type: application/json' -d '{"firstname":"John","lastname":"Doe","totalprice":150,"depositpaid":true,"bookingdates":{"checkin":"2025-06-01","checkout":"2025-06-05"}}'`  
- Второе бронирование (точно такие же данные): снова 200, новый bookingid.  
- Чек-лист: проверять идемпотентность — повторный запрос должен возвращать существующую бронь или 409; тестировать параллельные запросы.  
- Уровень сложности: средний.
