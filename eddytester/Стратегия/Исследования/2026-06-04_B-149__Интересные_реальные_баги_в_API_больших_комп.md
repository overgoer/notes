### Когда GitHub API обещал 500 рейт-лимита, а сам отдавал 403 — как проверять количество сбросов заголовков, чтобы не пропустить враньё?

**Links & Sources**
- [GitHub Community Discussion #67071](https://github.com/orgs/community/discussions/67071) — пользователи жалуются: rate limit превышен за минуты, хотя запросов мало; причина — неаутентифицированные запросы от сторонних приложений выбирают 60 req/h на IP.
- [GitHub Docs: Rate limits for REST API](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api?apiVersion=2026-03-10) — при превышении secondary rate limit возвращается 403 или 429 с сообщением об ошибке.
- [StackOverflow: Receiving GitHub API 403 error when not exceeded rate limit](https://stackoverflow.com/questions/38378337/receiving-github-api-403-error-when-i-have-not-exceeded-my-rate-limit) — кейс: 403 при корректном токене из-за отсутствия прав на эндпоинт, а не из-за лимита.

**Багы и реальные кейсы**
- Баг: GitHub возвращает 403, хотя первичный лимит (5000 req/h для authenticated) не исчерпан. На самом деле сработал вторичный лимит (слишком частые запросы или один токен с разных IP).
- Кейс из discussion: организация использует третьестороннее приложение без токена — оно быстро выбирает 60 req/h на IP и падает с 403. Решение: добавить аутентификацию.

**Технические детали**
- Проверка лимита:  
  `curl -H "Authorization: token TOKEN" https://api.github.com/rate_limit`  
  → смотреть `x-ratelimit-limit`, `x-ratelimit-remaining`, `x-ratelimit-reset`.
- Для вторичного лимита — заголовок `retry-after` (секунды) в 403/429 ответе.  
  Тело ответа: `"message": "You have exceeded a secondary rate limit"`.
- Чек-лист:
  1. Убедиться, что ВСЕ запросы аутентифицированы (в т.ч. от CI/CD и ботов).
  2. Проверить, не используется ли один токен в нескольких местах.
  3. При 403 читать тело ответа — отличить secondary limit от forbidden по правам.
  4. Мониторить заголовок `retry-after` для автоматического backoff.

### Slack API в conversations.history добавлял лишнее сообщение на каждую страницу — какой тест пагинации выявит такой «подарок» и дубликаты?

**Links & Sources**
- [Slack Dev Docs: conversations.history](https://docs.slack.dev/reference/methods/conversations.history) — описание метода, параметры `cursor`, `limit`, `inclusive`, `oldest`, `latest`.
- [GitHub Issue #56 slack-api-specs](https://github.com/slackapi/slack-api-specs/issues/56) — обсуждение получения всех сообщений с ответами, косвенно затрагивает проблемы пагинации.

**Багы и реальные кейсы**
- Баг: при пагинации с `oldest`/`latest` без `inclusive=true` последнее сообщение предыдущей страницы может совпадать с первым следующей → дубликат. Slack частично исправил, но на старых версиях API баг воспроизводится.
- Реальный кейс: пользователи заметили, что после перебора всех страниц количество уникальных сообщений меньше суммы полученных — дубликаты на границах.

**Технические детали**
- curl для теста:
  ```
  curl -H "Authorization: Bearer TOKEN" \
    "https://slack.com/api/conversations.history?channel=C123&limit=1&inclusive=true"
  ```
  Взять `response_metadata.next_cursor` и запросить вторую страницу. Сравнить `ts` последнего сообщения первой страницы и первого сообщения второй — они не должны совпадать.
- Чек-лист:
  1. Запросить 2+ страницы с `limit=1` и проверить отсутствие дубликатов по `ts`.
  2. Повторить с `inclusive=false` (default) — дубликаты возможны, это баг API, а не клиента.
  3. Проверить, что при `oldest` и `latest` граничные сообщения не теряются (не должно быть пропусков).
  4. Сравнить общее количество уникальных сообщений с суммой по страницам.

### Microsoft Graph API: баг с пагинацией, когда @odata.nextLink указывает на пустую страницу — как не попасть в бесконечный цикл?

**Links & Sources**
- По этому углу информации почти нет.

**Багы и реальные кейсы**
- По этому углу информации почти нет. Известны единичные сообщества пользователей, где Graph API возвращал `@odata.nextLink` даже при пустом `value`, что при неаккуратной реализации клиента приводит к бесконечному циклу. Конкретного подтверждённого бага с источниками не найдено.

**Технические детали**
- Чек-лист для теста:
  1. Проверить, что при пустом массиве `value` поле `@odata.nextLink` отсутствует или равно `null`.
  2. Если nextLink есть, сравнить его с предыдущим URL — если совпадает, значит зацикливание.
  3. В клиенте добавить защиту: если количество элементов на странице = 0, прерывать пагинацию.
  4. Тест с максимальным `$top` и проверкой, что каждая следующая страница содержит новые элементы.

### Идеи для постов
- **«API тебя обманывает»**: разбор кейсов, когда API возвращает не ту ошибку (403 вместо 429, дубликаты сообщений). Показать, как отлавливать «враньё» через заголовки и сравнение данных.
- **«Тест пагинации, который спасёт прод»**: конкретные curl-команды и чек-лист для обнаружения дубликатов/потери сообщений в Slack, GitHub, Graph API. Акцент — граничные условия.
- **«Почему ИИ не найдёт такие баги (пока)»**: отсылка к мнению Эдди — тестировщик, который понимает бизнес-логику и шарит за заголовки, нужен. ИИ может генерировать тесты, но не поймёт, что дубликат сообщения — баг, а не фича. Аргумент: такие кейсы требуют человеческого домысла и знания предметной области.
