# Research: Автотестерам тоже конец? Как ИИ меняет QA

**Date:** 2026-06-03

### Когда Copilot генерирует тест, который всегда зелёный — как проверить, что тест не тестирует сам себя?

**Links & Sources**
- [GitHub Docs: Writing tests with Copilot](https://docs.github.com/en/copilot/tutorials/write-tests) — советуют ревьюить сгенерированные тесты, но не дают конкретных методик выявления «циклических» тестов.
- [YouTube: Testing with GitHub Copilot](https://www.youtube.com/watch?v=smdBqEu7fx4) — демо, как Copilot создаёт юнит-тесты, но не затрагивает проблему ложноположительных результатов.
- [Community discussion #152916](https://github.com/orgs/community/discussions/152916) — участники жалуются, что Copilot иногда генерирует тесты, которые проверяют только вызов функции, а не её поведение.

**Баги и реальные кейсы**
- Реальный баг: разработчик попросил Copilot сгенерировать тест для метода `calculateDiscount()`. Модель выдала тест, который сравнивал результат с `calculateDiscount()` внутри самого теста (хардкод логики), а не с эталоном. Тест всегда проходил зелёным, но при изменении формулы скидки тест не падал.
- Другой кейс: Copilot сгенерировал юнит-тест на API, который отправлял запрос и проверял, что ответ не пустой (assertNotNull). При этом эндпоинт возвращал `{}` с кодом 200, тест проходил, но бизнес-логика не работала.

**Технические детали**
- **Что проверять на код-ревью:**
  1. Тест не должен содержать копию тестируемой логики (например, не пересчитывать скидку внутри теста).
  2. Ожидаемый результат должен быть константой или эталоном, полученным из другого источника (ручной расчёт, зафиксированный пример).
  3. Если тест использует mock/stub — проверьте, что mock действительно имитирует правильное поведение, а не пропускает все вызовы.
- **Команды для статического анализа:**
  ```bash
  # Поиск подозрительных сравнений с самим собой (Python)
  grep -rn "assertEqual.*result.*function_under_test" tests/
  # Поиск тестов, которые не используют assert вообще
  grep -rn "def test_" tests/ | grep -v "assert"
  ```
- **Чек-лист для ревью автотестов Copilot:**
  - [ ] Ожидаемое значение задано явно (не вычисляемая переменная из тестируемой функции).
  - [ ] Тест не использует ту же функцию для подготовки входных данных.
  - [ ] Есть негативные сценарии (ошибки, исключения).

### ИИ-агент сам правит баги и перезапускает тесты — кто отвечает за регрессию при удалении нужной функциональности?

**Links & Sources**
- [YouTube: Будущее для Manual QA: как AI переизобретает подходы в тестировании](https://www.youtube.com/watch?v=it9L2gAkWWQ) — обсуждается, что AI-агенты могут изменять код, но ответственность за регрессию остаётся на человеке.
- [Reddit: How AI changed the way we learn testing?](https://www.reddit.com/r/QualityAssurance/comments/1ny6wqk/do_you_think_ai_changed_the_way_we_learn_testing/) — пользователи жалуются на отсутствие должного регрессионного покрытия после автоправок ИИ.

**Баги и реальные кейсы**
- Кейс из open-source: AI-агент, подключённый к CI, «чинил» упавший тест, удаляя проверку на пустой ответ из функции обработки данных. Тест стал проходить, но на проде начали падать запросы с null-значениями.
- Другой пример: ИИ-агент для исправления уязвимости удалил проверку авторизации в middleware, решив, что она избыточна. Регрессионные тесты авторизации не были написаны — взлом стал возможен.

**Технические детали**
- **Как защититься:**
  1. Запретить AI-агентам напрямую коммитить в `main` — только через PR с обязательным код-ревью.
  2. Добавить метрики регрессии: до и после изменений запускать полный набор тестов, включая интеграционные и E2E.
  3. Использовать инструменты вроде `git diff` с проверкой на удаление проверок или условий:
     ```bash
     git diff HEAD~1 -- '*.py' | grep -E '^\-.*(assert|if|raise|return False)'
     ```
- **Чек-лист для аудита правок от AI-агента:**
  - [ ] Каждое удаление условия должно быть явно объяснено в описании коммита.
  - [ ] Тесты, которые падали до правки, — убедиться, что они не замокали проблему.
  - [ ] Запуск тестов с случайным порядком (pytest-randomly) для выявления race condition после изменений.

### Почему ИИ-автотесты пропускают граничные кейсы (пустые ответы, race condition) — как ловить на код-ревью без запуска?

**Links & Sources**
- [Medium: Why AI-Generated Test Cases Miss Edge Cases](https://harshasuraweera.medium.com/why-ai-generated-test-cases-miss-edge-cases-and-how-to-fix-it-562eaa6a1836) — причина: модель обучается на «счастливых путях» и не видит паттернов редко встречающихся ошибок.
- [SoftwareSeni: Testing and Debugging AI-Generated Code](https://www.softwareseni.com/testing-and-debugging-ai-generated-code-systematic-strategies-that-work/) — перечислены приоритеты: граничные тесты часто на 5-м месте, хотя должны быть на 1-2.
- [VirtuosoQA: Edge Case Testing Explained](https://www.virtuosoqa.com/post/edge-case-testing) — методика ручного определения границ.

**Баги и реальные кейсы**
- Баг: Copilot сгенерировал тесты для функции `getUser(id)`. Все тесты проверяли `id` типа int, но не проверили `null` и `string`. В проде пришёл запрос с `id=""` — упала вся страница.
- Race condition: ИИ написал тест для многопоточного кэша, который запускал один поток. Тест проходил, но на реальной нагрузке с 10 потоками происходил data race — ИИ не знает о необходимости тестировать одновременность.

**Технические детали**
- **Как ловить на код-ревью без запуска:**
  1. Составить mind map входных данных: что кроме «правильного» может прийти (None, пустая строка, отрицательное число, очень длинная строка).
  2. Проверить наличие тестов на состояния гонки: если код содержит `thread`, `lock`, `async` — должны быть тесты с параллельными вызовами.
  3. Использовать статический анализ: Python — `pytest-randomly`, `pytest-xdist` для stress-тестов.
- **Команды для быстрой проверки:**
  ```bash
  # Найти тесты, которые не используют параметризацию (потому что граничные часто параметризованы)
  grep -rn "def test_" tests/ | grep -v "pytest.mark.parametrize"
  # Проверить, что есть тесты на None/null
  grep -rn "None" tests/ | grep "test_"
  ```
- **Чек-лист при ревью AI-тестов:**
  - [ ] Есть ли тесты с `@pytest.mark.parametrize` с крайними значениями?
  - [ ] Есть ли негативные сценарии (исключения, ошибки)?
  - [ ] Для конкурентного кода — тесты с `threading.Barrier` или `concurrent.futures`.
