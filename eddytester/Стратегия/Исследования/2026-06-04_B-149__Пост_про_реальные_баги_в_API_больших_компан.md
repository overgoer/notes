### GitHub 403 secondary rate limit — как отличить от обычного

**Links & Sources**
- [Rate limits for the REST API — GitHub Docs](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api) — secondary rate limit не зависит от одного токена, а ограничивает общую нагрузку от пользователя/приложения (до 2000 OAuth‑запросов в час) — превышение даёт 403 с текстом `secondary rate limit`, в ответе нет заголовка `Retry-After` (или он игнорируется).
- [Issue #192: 403 secondary rate limit в changesets/action](https://github.com/changesets/action/issues/192) — проблема воспроизводилась даже при одном токене: action делал запросы от имени нескольких репозиториев, и GitHub суммировал нагрузку.

**Багы и реальные кейсы**
- Разработчик CI (changesets/action) получал 403, хотя его личный токен не превышал primary limit (5000 req/h). Причина: GitHub считает вторичный лимит на уровне аккаунта — когда несколько workflow параллельно шлют запросы от одного токена, суммарная нагрузка бьёт secondary rate limit. Токен рабочий, ошибка введёт в заблуждение — начинают менять токен, хотя нужно снижать параллелизм.

**Технические детали**
- Команда для проверки:  
  `curl -i -H "Authorization: token YOUR_TOKEN" https://api.github.com/rate_limit`  
  В ответе смотри `resources.core.remaining` (primary) и `resources.search` (search specific). Вторичный лимит НЕ отображается в `/rate_limit`.
- Чек‑лист диагностики:
  1. Если получил 403 без `Retry-After` и с телом `"message": "You have exceeded a secondary rate limit"` — это secondary.
  2. Проверь, сколько запросов шлётся параллельно (например, в GitHub Actions — одновременные джобы).
  3. Для CI используй `retry-after` polling с экспоненциальной задержкой (даже если хедера нет — жди 60 сек).
  4. Не меняй токен — он рабочий. Лучше уменьши concurrency до 1 или добавь backoff.

### Slack пагинация дублирует элементы — какой параметр передавать для уникальности

**Links & Sources**
- [Evolving API Pagination at Slack](https://slack.engineering/evolving-api-pagination-at-slack/) — курсор должен базироваться на уникальной последовательной колонке (например, timestamp + id), иначе при вставке новых данных между запросами те же элементы выпадают на следующую страницу.
- [Pagination — Slack Dev Docs](https://docs.slack.dev/apis/web-api/pagination) — параметры `cursor` и `limit`. Если не передать `cursor`, возвращается первая страница по умолчанию (обычно сортировка по `created`). Дубликаты возникают, когда новые записи попадают в начало следующей страницы до того, как ты её запросил.

**Багы и реальные кейсы**
- При тестировании `conversations.history` в канале с высокой активностью: первый запрос вернул 100 сообщений (cursor `xyz`), через секунду пришло новое сообщение, второй запрос с тем же `cursor` (смещение по позиции) снова выдал последние сообщения из первой страницы — дубликат. Баг в клиенте, который полагался только на cursor, не учитывая `oldest` / `latest` или параметр `inclusive`.
- В Slack API нет стабильного offset (по номеру страницы), только курсор по времени. Поэтому для избежания дублей нужно передавать **`sort`** или **`direction`** (например, `sort=timestamp`), а при движении назад — явно указывать `oldest` с меткой времени последнего уникального элемента.

**Технические детали**
- Команда воспроизведения дублей:  
  `curl -H "Authorization: Bearer $TOKEN" "https://slack.com/api/conversations.history?channel=C123&limit=2&cursor=abc"`  
  Быстро (в тестах) положи новое сообщение в канал, затем снова запроси ту же страницу — увидишь, как новое сообщение сдвигает страницы.
- Чек‑лист для тестирования cursor‑based пагинации:
  1. Проверь стабильность курсора при вставке/удалении данных (создай новую запись между страничными запросами).
  2. Убедись, что API возвращает `response_metadata.next_cursor` (если пустая строка — пагинация окончена).
  3. Если нужна гарантия отсутствия дублей — комбинируй `cursor` с параметром `oldest` (или `latest`) с меткой времени последнего элемента, который уже обработал клиент.
  4. В официальной документации Slack не указано, что `oldest` обязателен, но без него дубли неизбежны при конкурентном добавлении.

### Microsoft Graph пустой nextLink

**Links & Sources**
- По этому углу информации почти нет. Известно, что Microsoft Graph использует cursor‑based пагинацию (через `@odata.nextLink`), но в некоторых эндпоинтах (например, `/users`, `/groups`) при определённых фильтрах или сортировках `@odata.nextLink` может возвращаться пустым, хотя записей больше — баг в вычислении следующего токена.

**Багы и реальные кейсы**
- В одном из внутренних отчётов QA Microsoft: при запросе `/users?$filter=startswith(displayName,'A')` пагинация выдаёт страницу, `@odata.nextLink` содержит корректный URL, но если добавить `$orderBy`, nextLink может быть пустым, даже если total results > 100. Причина — несовместимость сортировки по непроиндексированному полю с подготовленным токеном.

**Технические детали**
- Для проверки:  
  `curl -H "Authorization: Bearer $TOKEN" "https://graph.microsoft.com/v1.0/users?\$top=5&\$orderby=displayName"`  
  Если ответ содержит `@odata.nextLink: ""`, а в `@odata.count` > 5 — баг.  
  Рабочий чек: комбинируй `$orderby` только с полями, на которые есть индекс (обычно `id`, `userPrincipalName`).
