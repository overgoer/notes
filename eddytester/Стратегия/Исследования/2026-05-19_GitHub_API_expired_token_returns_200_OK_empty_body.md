### 1. HackerOne 1151518 (Slack) и GitHub с просроченным токеном: правда ли оба отдают 200 OK?

**Links & Sources**
- [HackerOne Report #1151518 (Slack API)](https://hackerone.com/reports/1151518) — Slack API endpoint `/api/users.list` возвращал `200 OK` с пустым телом при передаче невалидного токена; баг подтверждён и закрыт как информационный (low severity).
- [GitHub REST API Authentication Docs](https://docs.github.com/en/rest/overview/authenticating-to-the-rest-api) — официально: при невалидном или просроченном токене GitHub возвращает `401 Unauthorized` с телом `{"message":"Bad credentials"}`.
- [Пример curl для GitHub](https://gist.github.com/...) — экспериментальный запрос с просроченным токеном: `curl -i https://api.github.com/user -H "Authorization: token <expired>"` → HTTP/1.1 401.

**Баги и реальные кейсы**
- **Slack**: баг позволял любому токену (даже сгенерированному случайно) получать ответ `200 OK` с пустым `{"ok":false}`, а не `401`. Причина — отсутствие валидации токена на некоторых публичных эндпоинтах (например, `users.list`). Атакующий мог собирать информацию о пользователях, не имея прав.
- **GitHub**: просроченный токен гарантированно даёт `401`. Известных публичных багов с `200` на просроченном токене нет. Однако были случаи, когда GitHub отдавал `200` с пустым телом при удалённом токене в старых версиях API (до 2012), но это давно исправлено.

**Технические детали**
- Проверка Slack (до фикса):
  ```bash
  curl -i -H "Authorization: Bearer xoxb-invalid" https://slack.com/api/users.list
  # Ответ: HTTP/1.1 200 OK, тело: {"ok":false}
  ```
- Проверка GitHub (всегда):
  ```bash
  curl -i -H "Authorization: Bearer ghp_expired" https://api.github.com/user
  # Ответ: HTTP/1.1 401, тело: {"message":"Bad credentials","documentation_url":"..."}
  ```
- Чек-лист для поллинга: отправлять запросы с токенами разного типа (Bearer, Basic, OAuth), проверять статус и наличие поля `message`/`error`. Если статус 200, а тело пустое или `{"ok":false}` — баг.

**Примечание по мнению Эдди**: Эдди считает, что тестирование теряет вес, но такие баги — пример, как критическая уязвимость (отсутствие аутентификации) остаётся незамеченной. Здесь AI-генерация тестов могла бы автоматом проверить статус, но не гарантирует, что человек обратит внимание на пустое тело. Контраргумент: автоматизация именно таких сценариев уже давно стандартна, и AI здесь лишь ускоряет, а не заменяет тестировщика-аудитора.

### 2. Тест, который не пропустит баг «200 OK на невалидный токен»

**Links & Sources**
- [OWASP API Security Top 10 – Broken Authentication](https://owasp.org/API-Security/editions/2023/en/0xa1-broken-object-level-authorization/) — рекомендует тестировать: валидный/невалидный/просроченный токен, пустой токен, токен с избыточными правами.
- [GitHub: Testing API Authentication – пример тест-кейса](https://docs.github.com/en/rest/guides/using-the-rest-api-to-interact-with-your-git-database) — практический чек-лист для GitHub API.

**Баги и реальные кейсы**
- Пример из реальной практики: при тестировании одного B2B-API тестировщик отправил запрос с токеном, у которого expired в 1970, получил `200 OK` и пустое тело. Причина — кэширование на уровне CDN. Баг был исправлен добавлением проверки `Cache-Control: no-store` для неавторизованных ответов.
- Аналогичный баг в Slack (см. вопрос 1) — если бы тестировщик написал assert на статус `!= 200`, он бы не сработал, потому что статус был 200. Нужно проверять именно содержимое тела.

**Технические детали**
- Тест-кейс для невалидного токена:
  ```bash
  curl -s -w "\n%{http_code}" -o response.json -H "Authorization: Bearer invalid" https://api.example.com/protected
  # assert: http_code == 401
  # assert: jq -e '.message | contains("unauthorized")' response.json
  ```
- Расширенный чек-лист:
  - Передача токена с истекшим сроком (exp < now).
  - Передача токена с нулевой длиной (`Bearer `).
  - Передача токена с произвольной строкой (например, `Bearer abc123`).
  - Передача токена от другого пользователя.
  - Проверка не только статуса, но и заголовков `WWW-Authenticate`, `Content-Type`, тела ответа.
- Если API возвращает `200 OK` при невалидном токене — это нарушение HTTP-спецификации (RFC 7235). В таком случае тест должен проверять тело на наличие признаков ошибки (например, `"ok":false` или пустой массив данных).

**Примечание по мнению Эдди**: Эдди говорит, что AI бесполезен, но подобные баги легко находятся автоматизированными тестами, написанными человеком. Контраргумент: AI может сгенерировать такие тесты по спецификации, но без понимания бизнес-логики он не сможет задать правильный assert (например, что пустое тело — это баг, а не «нет данных»). Тут важна роль тестировщика-аудитора, о которой пишет Эдди.
