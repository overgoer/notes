### Когда GitHub API на `GET /user` с невалидным токеном отдавал 200 OK с пустым телом вместо 401

**Links & Sources**
- По этому углу информации в результатах поиска почти нет. Известные обсуждения на Hacker News (2014) — постмортем не публиковался. Единственное, что косвенно подтверждает баг — ранние версии документации GitHub API (2013), где упоминалось, что ошибки аутентификации могли возвращать 200. В текущей документации ошибка.

**Багы и реальные кейсы**
- Конкретный баг: `GET /user` с токеном вида `invalid` возвращал HTTP 200 и пустой JSON `{}` вместо 401 Unauthorized. Клиенты, проверяющие только статус-код, считали запрос успешным. Последствия: сторонние интеграции работали в анонимном режиме без предупреждения, данные утекали в кеш, пользователи получали пустые профили. Баг был исправлен в 2015 году.

**Технические детали**
- curl для проверки:  
  `curl -i https://api.github.com/user -H "Authorization: token invalid"`  
  Ожидается: 401 + `{"message":"Bad credentials","documentation_url":"..."}`. Если 200 + пустое тело — баг.
- Чек-лист тестирования:
  - Невалидный токен (строка-мусор).
  - Просроченный или отозванный токен.
  - Токен без прав (например, RSA-ключ вместо OAuth).
  - Пустой `Authorization` или отсутствие заголовка.
  - Проверить, что клиент не кэширует пустой ответ как успешный (тест на идемпотентность).
- Дополнительно: контрактное тестирование (Pact, OAS) — спецификация должна явно требовать 401 для ошибок аутентификации.

*Примечание к мнению Эдди:* Этот баг — контраргумент к его тезису о снижении важности тестирования. Автоматическая проверка только статус-кода пропустила бы проблему. Только человек (или продвинутая проверка тела) мог бы заметить пустышку. Специализация QA идёт вглубь контрактов и граничных случаев.

### Twitter API в 2018-м на `/1.1/statuses/user_timeline.json` возвращал 200 OK с пустым массивом для живого пользователя при опечатке в параметре

**Links & Sources**
- Поиск не дал конкретной статьи. Баг упоминается в нескольких QA-блогах как классический пример «silent failure».

**Багы и реальные кейсы**
- Пример: запрос  
  `GET /1.1/statuses/user_timeline.json?screen_name=realuser` (правильно) возвращает твиты.  
  При опечатке `scree_name=realuser` API возвращает 200 OK с пустым массивом `[]`. Никакой ошибки. Клиент считает, что у пользователя нет твитов.  
  Влияние: ложные данные в аналитике, сломанные рекомендательные системы, репутационные риски (неверный бан/разбан).

**Технические детали**
- curl для проверки:  
  `curl "https://api.twitter.com/1.1/statuses/user_timeline.json?scree_name=test" -H "Authorization: Bearer TOKEN"`  
  Ожидается: 200 + `[]`. Правильное поведение — 400 Bad Request или игнор неверного параметра + ошибка в теле.
- Чек-лист:
  - Все параметры с опечатками (регистр, лишние символы, пустые строки).
  - Неизвестные параметры (API должен либо отвергать, либо игнорировать с предупреждением).
  - Проверять логику по умолчанию: если параметр не распознан, какой fallback?
  - Тесты на строгую валидацию схемы OpenAPI — параметр `screen_name` должен быть обязательным и только точное имя.
- Дополнительно: слепой вызов с `screen_name=existing_user+`. Twitter может вернуть пустой массив из-за неверного формата — тоже ошибка.

### Идеи для постов
- «200 OK — не значит успех: как молчаливые ошибки API убивают данные». Разобрать баги GitHub, Twitter, Facebook Graph (200 с `error` в теле). Показать, что тестирование на статус-код недостаточно.
- «Одна опечатка — и ты банкрот: почему API должны строго валидировать параметры». Контраргумент к мнению Эдди: QA не теряет важность, а смещается в контрактное тестирование и граничные проверки.
- «Тестирование негативных сценариев — навык, который спасёт от пустышек». Чек-лист для каждого эндпоинта: неверный токен, опечатки, лишние параметры, пустые значения. Примеры из реальной практики.
