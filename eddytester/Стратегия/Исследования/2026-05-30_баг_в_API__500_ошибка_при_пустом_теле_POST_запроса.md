# Research: баг в API: 500 ошибка при пустом теле POST запроса — разбор инцидента с practicum API

**Date:** 2026-05-30

---

### Когда пустое тело POST уронило сервер с 500 — как такое пропускают на code review?

**Links & Sources**
- [Stack Overflow: REST API error code 500 handling](https://stackoverflow.com/questions/27826491/rest-api-error-code-500-handling) — ключевой принцип: если ошибка вызвана клиентским вводом — это 4xx, а не 500, сервер не должен падать из-за некорректного запроса.
- [GitHub Issue: API returns 500 error when request body is empty](https://github.com/KarthikRelevantz/PZPOCDemo/issues/1) — конкретный баг в Node.js API: при пустом теле POST вылетает необработанное исключение, сервис не валидирует вход.
- [Reddit: HttpClient giving 500 Server Error response](https://www.reddit.com/r/dotnet/comments/14zgm5l/httpclient_giving_500_server_error_response_while/) — в .NET аналогичная проблема: пустое тело POST приводит к 500 из-за отсутствия проверки Content-Length и парсинга.

**Багы и реальные кейсы**
- Practicum API (предположительно образовательный проект): реальный инцидент — POST запрос без тела приводит к 500. Причина: код обработчика пытается десериализовать null/пустую строку в объект, выбрасывается исключение, которое не перехватывается middleware.
- Аналогичный баг в PZPOCDemo (Node.js): при пустом теле или пропущенных полях — 500 вместо 400. Заведен issue с лейблом bug.

**Технические детали**
- curl для воспроизведения: `curl -X POST https://api.example.com/resource -H "Content-Type: application/json" -d ''`
- Что должно быть: ответ 400 `{"error": "Request body is required"}`.
- Что происходит: ответ 500 с HTML/стектрейсом (или пустым телом).
- Чек-лист для code review:
  - Есть ли middleware валидации тела запроса (например, express-validator, Joi, FluentValidation)?
  - Обрабатывается ли случай `req.body === null || req.body === undefined`?
  - Есть ли глобальный обработчик ошибок (error handler middleware), который ловит исключения и отдает 500? Без него 500 — это нормальное поведение, но плохая практика.
  - Проверка Content-Type и Content-Length перед парсингом.

### Почему 500 на пустое тело — признак отсутствия middleware валидации payload?

**Links & Sources**
- [GitHub Issue #1 (тот же)](https://github.com/KarthikRelevantz/PZPOCDemo/issues/1) — автор прямо пишет: «API should validate input and return a meaningful error response». Отсутствие валидации — корень.
- [Stack Overflow: Why do I get 500 when sending empty POST request?](https://stackoverflow.com/questions/30398499/why-do-i-get-internal-error-500-when-sending-empty-post-request) — Rails 4.0: пустое тело вызывает ошибку парсинга JSON, фреймворк не умеет отдавать 400 по умолчанию для такого случая, всплывает до уровня Rack middleware как 500.
- [GitLab Issue: Code Suggestions API returns 500 for payloads ≥100k](https://gitlab.com/gitlab-org/gitlab/-/issues/585841) — хоть и про размер, но суть та же: отсутствие валидации на граничных значениях приводит к 500.

**Багы и реальные кейсы**
- В том же инциденте Practicum: разработчик не добавил проверку на пустое тело в контроллер. Фреймворк (предположительно Express) при попытке разобрать пустую строку как JSON кидает SyntaxError, который не перехвачен — отдаётся 500.
- В Rails 4.0 (стек 2014) аналогично: пустой POST ломает парсер JSON, и приложение падает. Проблема решается добавлением middleware для валидации JSON или оборачиванием в rescue.

**Технические детали**
- Команда для проверки middleware: `curl -v -X POST -H "Content-Type: application/json" -d '' http://localhost:3000/api/resource`
- Что должно быть в логах: `UnhandledPromiseRejection` или `SyntaxError: Unexpected end of JSON input`.
- Чек-лист для аудита middleware:
  1. Есть ли централизованный валидатор тела (например, `express.json()` с `verify` callback)?
  2. Есть ли глобальный error-handler, который различает ошибки парсинга (400) и прочие исключения (500)?
  3. Добавлен ли middleware, который проверяет `req.body` на `null`/`undefined` сразу после парсера?
  4. Используются ли библиотеки вроде `celebrate` (Joi) для схем валидации? Если да — проверяют ли они обязательность тела целиком?

### В идеале POST с пустым телом должен возвращать 400 с json-ошибкой, но почему фреймворки не делают это по умолчанию?

**Links & Sources**
- [Good practice for JSON error response for 400 and 500 — Problem Details for HTTP APIs](https://datatracker.ietf.org/doc/html/rfc7807) — стандарт RFC 7807 предлагает единый формат ошибок, но фреймворки его не реализуют по умолчанию, оставляя это разработчику.

**Багы и реальные кейсы**
По этому углу информации почти нет. Можно лишь отметить, что большинство популярных фреймворков (Express, Spring Boot, Django REST) отдают 500 при невалидном JSON по умолчанию, потому что парсер тела — это часто внешний модуль, который кидает исключение, а стандартный обработчик ошибок не различает типы исключений.

**Технические детали**
- Как проверить поведение фреймворка по умолчанию:
  - Express: `node -e "require('express')().post('/test', (req,res) => res.json({ok:1})).listen(3000)"` -> `curl -X POST -H 'Content-Type: application/json' -d '' localhost:3000/test` — получите 500 (если нет error-handler).
  - Spring Boot: `curl -X POST -H 'Content-Type: application/json' -d '' localhost:8080/test` — получите 400 только если включена валидация (`@Valid`).
- Чек-лист:
  - Проверить документацию фреймворка: какое поведение по умолчанию при пустом теле?
  - Если фреймворк не обрабатывает эту ситуацию — это осознанное решение (разработчик должен добавить middleware).
  - Рекомендация: всегда добавлять глобальный фильтр, который ловит исключение парсинга JSON и преобразует в 400.

### Идеи для постов
- «500 на пустое тело — это не баг, а архитектурный симптом: как отсутствие единого middleware валидации подрывает доверие к API». Угол — не про тестирование, а про инженерную культуру.
- «Почему половина API в интернете падает с 500 на пустой POST — и при чём тут AI-кодеры?» Связать с заметкой Эдди про AI: если AI пишет код без middleware, такие баги будут тиражироваться. Но контраргумент: AI может и писать middleware, если промпт правильный.
- «CLI-чеклист для code review: 5 curl-команд, которые убивают невалидный API за минуту». Показать, как быстро тестировать граничные случаи (пустое тело, невалидный JSON, лишние поля).
