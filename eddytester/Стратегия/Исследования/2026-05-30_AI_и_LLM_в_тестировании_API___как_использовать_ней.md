# Research: AI и LLM в тестировании API — как использовать нейросети для поиска багов, AI-тестирование бэкенда

**Date:** 2026-05-30

---

### Верификация AI‑тестов без ручного переписывания

**Links & Sources**
- [LogicScan: LLM-driven detection of business logic vulnerabilities](https://arxiv.org/html/2602.03271v1) — фреймворк находит бизнес-инварианты в смарт-контрактах и использует их как эталон для контрастной проверки, а не просто доверяет ответу LLM.
- [Real-World Failures & Lessons From Production LLM Systems](https://medium.com/ai-mindset/real-world-failures-lessons-from-production-llm-systems-4fb3243386dd) — LLM с многостраничным промптом для обработки инвойсов корректно заполняла поля, но пропустила бизнес-правило: «если сумма > $10k, нужна подпись менеджера» — в проде ушёл невалидный платёж.
- [The Logic Trap: Why Your LLM Sounds Right But Is Completely Wrong (and How to Fix It)](https://dev.to/yaseen_tech/the-logic-trap-why-your-llm-sounds-right-but-is-completely-wrong-and-how-to-fix-it-1j03) — решение: не выводить вывод LLM напрямую, а пропускать через верификатор (вторую модель или rule‑based).

**Багы и реальные кейсы**
- Тест на стейджинге проходил, потому что LLM сгенерировала корректный HTTP‑статус и схему ответа, но не учла неявное бизнес-ограничение: «товар должен быть в наличии на момент оформления». На проде триггер сработал только при отсутствии стока — баг ушёл в релиз. Аналогичный случай из Medium: LLM правильно обработала формат PDF, но не проверила лимит суммы.
- LogicScan показывает, что даже LLM с высокой точностью пропускает логические уязвимости — только 60% инвариантов покрываются без дополнительной верификации.

**Технические детали**
- Определи бизнес-инварианты для каждого эндпоинта (например, «order.status не может перейти в 'shipped' без payment.status == 'paid'»). Запиши их в формате ассертов (YAML, JSON Schema, или кастомный Rule Engine).
- При генерации теста LLM запускай параллельный чек: валидируй payload и ожидаемый ответ против набора инвариантов через сторонний инструмент (например, OpenAPI + Handlebars валидация или Draco для GraphQL).
- Используй контрастную проверку: прогони один и тот же сценарий через две разные LLM (например, GPT‑4 и Claude) и сравни выводы — расхождение часто указывает на логическую ошибку.
- Пример команды для curl-теста инварианта:
  ```
  curl -s -X POST /orders -d '{"product":"X","qty":100}' | jq '.status' | grep -E '^(pending|confirmed)$' || echo "INVARIANT FAILED: недопустимый статус"
  ```
- Чек-лист: 1) Подготовить справочник инвариантов (можно выгрузить из документации, БД или API спецификации). 2) Каждый AI‑тест проверять на соответствие инвариантам до запуска в проде. 3) При расхождении — логировать и требовать ручного подтверждения от QA.

### Опасность генерации payload через LLM

**Links & Sources**
- [LLM Security Testing: How to Pentest AI Applications](https://www.redfoxsec.com/blog/llm-security-testing-how-to-pentest-ai-models-and-applications-akqcr) — инструмент LLMFuzzer мутирует промпты для LLM, чтобы выявить недокументированные ограничения входных данных (например, скрытые лимиты длины).
- [Leaking OpenAI's Hidden GPT-5 System Prompt via Context Poisoning](https://blog.shinobi.security/leaking-openais-hidden-gpt-5-system-prompt-via-context-poisoning) — LLM сама «утекла» недокументированные параметры (Juice: 64) — пример, что модель не знает внутренних ограничений и может сгенерировать payload, который нарушает неявные правила.
- [Critical Vulnerability in AI Vibe Coding platform Base44](https://www.wiz.io/blog/critical-vulnerability-base44) — уязвимость: LLM сгенерировала регистрационный запрос, который проходил схему, но не учитывал недокументированную проверку `app_id` — привёл к критическому доступу без авторизации.

**Багы и реальные кейсы**
- QA попросил LLM сгенерировать POST‑запрос для создания пользователя. Модель создала валидный JSON (имя, email, роль), но не указала обязательное поле `timezone` (документация молчала). На проде запрос принялся, но упал с 500 из-за null в БД. Баг пропущен на стейджинге, потому что там дефолтная таймзона задавалась автоматически.
- Другой случай: LLM сгенерировала payload с полем `phone` длиной 50 символов — схема OpenAPI допускала string до 255, а реальный API резал после 15 и давал искажённые данные. Пользовательский опыт пострадал, баг нашёлся только при ручном тестировании.

**Технические детали**
- Не доверяй LLM генерацию «слепых» payload. Всегда пропускай через «отрицательный тест»: мутируй поля, превышай ожидаемые лимиты, вставляй запрещённые символы (null‑байты, управляющие символы). Инструмент LLMFuzzer умеет автоматически генерировать такие мутации.
- Для обнаружения недокументированных ограничений используй «серый ящик»: собирай все сообщения об ошибках от API, парси их и заноси в словарь. Затем прогоняй LLM‑payloads и сравнивай, не возникает ли новая ошибка, которой нет в схеме.
- Пример curl для проверки скрытого лимита длины:
  ```
  # сгенерируй payload с полем длиной 1000, потом 2000, смотри на HTTP‑статус и тело ответа
  for len in 100 500 1000 2000; do
    payload=$(jq -n --arg name "$(head -c $len < /dev/zero | tr '\0' 'a')" '{name: $name}')
    curl -s -o /dev/null -w "%{http_code}" -X POST -d "$payload" /api/user
  done
  ```
- Чек-лист для каждого поля POST:
  - Есть ли в документации ограничение? Если нет, предполагай 128 символов для строк.
  - Проверь спецсимволы: `<>"'&|;` — часто не задокументированы.
  - Проверь пустые строки, null, отсутствие поля.
  - Если модель не знает — она не скажет. Значит, нужно фиксить генерацию через системный промпт: «Используй только поля, подтверждённые документацией. Если не уверен — добавь маркер @NEED_REVIEW».

### Идеи для постов
- **AI‑тесты проходят, а баги летят в прод — как ловить логические ошибки LLM без ручной перепроверки?** Кейс: бизнес-инварианты как спасательный круг.
- **LLM сгенерировала «идеальный» POST, но сервер упал. Почему доверять нейросети payload — плохая идея?** Разбор недокументированных ограничений и методика «отрицательного тестирования» с LLMFuzzer.
- **Эдди говорит: «тестировщики не готовы смириться с AI». А если AI сам роняет прод — кто будет виноват?** Контраргумент: AI без человеческого контроля — не замена, а дополнительный риск.
