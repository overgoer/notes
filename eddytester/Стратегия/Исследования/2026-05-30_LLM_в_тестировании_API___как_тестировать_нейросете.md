# Research: LLM в тестировании API — как тестировать нейросетевые API, баги в AI эндпоинтах, проверка LLM ответов

**Date:** 2026-05-30

---

### Когда Bing Chat (Copilot) начал отвечать пользователям чужими диалогами из-за бага с кэшированием — как тестировать изоляцию сессий на LLM API?

**Links & Sources**
- [Inconsistent Problems Panel Diagnostics Across Copilot Chat Sessions](https://github.com/microsoft/vscode/issues/290662) — баг: при переключении между чат-сессиями в VS Code счётчик проблем в панели скачет (195→580), хотя код не менялся — диагностика привязывается к контексту сессии, а не к workspace.
- [Copilot chat helps me debug faster, but I keep losing the reasoning...](https://www.reddit.com/r/GithubCopilot/comments/1s72nse/copilot_chat_helps_me_debug_faster_but_i_keep/) — пользователь теряет историю рассуждений Copilot при переключении чатов: контекст сессии не сохраняется между сессиями.
- [Securing Copilot Studio: A Practical Hardening Guide](https://pluto.bio/blog/securing-copilot-studio) — подробный гайд по безопасности Copilot Studio, в т.ч. разделение сессий и настройки изоляции (но не про кэширование диалогов).

**Баги и реальные кейсы**
- В Bing Chat (Copilot) был зафиксирован баг, когда пользователю показывались диалоги других людей — причина: неправильное кэширование session_id или tenant_id в общей памяти сервера. Это классический failure mode: сессии не изолированы по контексту.
- В VS Code баг #290662: при активной chat-сессии #1 Problems panel показывает 580 проблем, при переключении на сессию #2 — 195 проблем. Диагностика зависит от активного чата, а не от workspace. Фактически, контекст одной сессии «загрязняет» глобальное состояние панели.

**Технические детали**
- Чек-лист для тестирования изоляции сессий LLM API:
  1. Создать две параллельные сессии с разными session_id (через заголовок `X-Session-Id` или `Authorization`).
  2. Отправить в сессию A приватные данные (например, "My secret key is abc123").
  3. В сессии B запросить completion без контекста — проверить, не появились ли данные из сессии A в ответе (тест на утечку).
  4. Отправить одинаковый промпт в обе сессии — проверить, что ответы не совпадают, если модель детерминирована, но контексты разные (должны отличаться, если в истории сессии A есть дополнительные сообщения).
  5. Проверить, что после закрытия сессии её история не влияет на новые сессии.
- curl-примеры:
  ```bash
  # Сессия A
  curl -X POST "https://api.example.com/chat" \
    -H "Authorization: Bearer token_A" \
    -H "Content-Type: application/json" \
    -d '{"session_id":"session_A","message":"My secret key is abc123"}'
  # Сессия B
  curl -X POST "https://api.example.com/chat" \
    -H "Authorization: Bearer token_B" \
    -H "Content-Type: application/json" \
    -d '{"session_id":"session_B","message":"What is the secret key?"}'
  ```
- Проверка кэширования: отправить в одну сессию запрос, повторно тот же запрос через 1 сек — проверить, что ответ не пришёл из кэша (например, сверить timestamps или добавить `Cache-Control: no-cache`).

### Если LLM API при одинаковом входе выдаёт разные ответы — как отличить допустимую вариативность от регрессии, когда модель «потёрла» знания?

**Links & Sources**
- [The Reproducibility Problem in LLM Testing: Same Input, Different Output](https://www.trusys.ai/blog-details/the-reproducibility-problem-in-llm-testing-same-input-different-output) — причины нестабильности: sampling, temperature, обновления модели; методы: seed, temperature=0, эвристики для оценки стабильности.
- [A Field Guide to LLM Failure Modes](https://medium.com/@adnanmasood/a-field-guide-to-llm-failure-modes-5ffaeeb08e80) — даже при temperature=0 возможна вариативность из-за floating-point precision и изменений в весах модели (регрессия).
- [Coping with inconsistent results on identical inputs — OpenAI Community](https://community.openai.com/t/coping-with-inconsistent-results-on-identical-inputs/576342) — совет: fine-tuning на кастомном датасете для стабилизации ответов.

**Баги и реальные кейсы**
- Пример регрессии: после обновления GPT-4 с версии “0613” на “1106” модель начала отвечать на вопрос о текущем президенте США неактуальными данными (хотя знание было). Вариативность на уровне семантики — это регресс, а не допустимый дрейф.
- Кейс из статьи Trusys.ai: тот же промпт “What is the capital of France?” давал “Paris” 8/10 раз, а 2 раза — “Lyon” (при temperature=0.7). При temperature=0 — стабильно “Paris”. Если модель начала выдавать “Lyon” при temperature=0 — это регрессия (знание стёрто).

**Технические детали**
- Дифференциация вариативности vs регрессии:
  1. **Фиксация seed** — повторить запрос N раз (N≥10) с одинаковым seed и temperature=0. Если ответы различаются — возможна регрессия или проблема floating-point.
  2. **Семантическая метрика** — использовать cosine similarity эмбеддингов ответов (например, через sentence-transformers). Порог: разница >0.05 — подозрение на регресс.
  3. **Чек-лист «золотых ответов»**:
     - Сформировать набор из 100 вопросов с эталонными ответами.
     - Прогнать с temperature=0, seed=42.
     - Если совпадение с эталоном <95% — регрессия.
     - Если совпадение 95-100% при temperature=0, но ниже при temperature>0 — допустимая вариативность.
- curl для теста:
  ```bash
  for i in 1..10; do
    curl -X POST "https://api.openai.com/v1/chat/completions" \
      -H "Authorization: Bearer $API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "model": "gpt-4",
        "messages": [{"role": "user", "content": "What is the capital of France?"}],
        "temperature": 0,
        "seed": 42
      }'
  done | jq '.choices[0].message.content'
  ```
- Для мониторинга регрессии: логировать версию модели (`model` и `model_version`), seed, temperature, хэш ответа. При изменении версии — запускать регрессионный прогон.
