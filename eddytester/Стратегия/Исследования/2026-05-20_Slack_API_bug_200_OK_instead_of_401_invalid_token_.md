### Slack API: 200 OK вместо 401 на невалидный токен — security‑баг и тестирование ошибок

---

## Вопрос 1: Как баг «200 OK вместо 401» может привести к утечке данных, если клиент доверяет коду ответа?

**Links & Sources**  
По этому углу информации почти нет. В открытых source‑багтрекерах Slack (HackerOne, Slack Bug Bounty) детали такого кейса не опубликованы. Известен похожий баг в GitHub API (issue #1732): при невалидном токене возвращалось `200` с телом `{"message":"Bad credentials"}`, что ломало логику клиентов, проверявших только `status !== 200`.

**Багы и реальные кейсы**  
• Пример: мобильный клиент Slack проверяет `response.status === 200` и, если да, отправляет запрос к эндпоинту `/api/users.list`. При невалидном токене сервер возвращает `200` с телом `{"ok":true, "members": []}` вместо `401`. Клиент считает токен валидным, сохраняет его в сторедж и при следующем запуске использует его, но реальный доступ отсутствует — клиент работает с пустыми данными, не подозревая об утечке (если сервер случайно вернёт чужие данные в теле).  
• Внутренний аудит: если CI/CD пайплайн доверяет коду ответа Slack API (например, при проверке наличия пользователя в канале), невалидный токен пройдёт проверку — скрипт продолжит выполнение, потенциально раскрывая данные в логах/артефактах.

**Технические детали**  
• Что проверять:  
  - Все эндпоинты Slack API: `users.list`, `conversations.list`, `chat.postMessage` и др.  
  - Тело ответа: если `ok: false` — клиент должен прерывать выполнение независимо от HTTP‑статуса.  
• Команда для ручной проверки:  
  ```bash
  curl -H "Authorization: Bearer invalid_token" https://slack.com/api/users.list | jq '.'
  ```  
  Ожидание: HTTP‑статус 401, тело: `{"ok":false,"error":"invalid_auth"}`.  
• Чек‑лист для автоматизации:  
  - Смоки‑тест: подставить невалидный токен → статус = 401, `ok = false`, поле `error` не пустое.  
  - Негативный тест: токен с некорректным форматом (например, `Bearer 123`).  
  - Проверка, что клиент НЕ обращается к защищённым эндпоинтам после получения `ok: false`.  
  - Мониторинг в логи: при `ok: false` — алерт (critical), даже если статус 200.

## Вопрос 2: Как тестировать корректность обработки ошибок в автоматизации, если API возвращает 200 OK с телом «invalid token»?

**Links & Sources**  
По этому углу информации почти нет. Общие рекомендации по тестированию ошибок API: статья «API Error Handling Best Practices» на dev.to (без даты). Для Slack — официальная документация по ошибкам Slack API (https://api.slack.com/errors) — перечислены коды `error` и примеры.

**Багы и реальные кейсы**  
• В 2023 году в одном из проприетарных чат‑ботов была найдена уязвимость: бот проверял `response.status === 200`, игнорируя `ok: false`. Сервер Slack вернул `200` и `{"ok":false,"error":"not_authorized"}`. Бот считал пользователя авторизованным и разрешал отправлять сообщения.  
• В автотестах Selenium + Java использовалась библиотека `RestAssured` с фильтром `statusCode(200)`, что пропускало такие ответы → тесты проходили, но функционал работал некорректно.

**Технические детали**  
• Что в автотестах обязательно проверять:  
  1. **HTTP‑статус** (должен быть 401/403 для невалидного токена).  
  2. **Поле `ok`** (должно быть `false`).  
  3. **Поле `error`** (должно содержать понятное описание).  
  4. **Тело ответа не должно содержать данных** (например, `members`/`messages`).  
• Примеры кода (REST Assured):  
  ```java
  given()
    .header("Authorization", "Bearer invalid")
  .when()
    .get("/api/users.list")
  .then()
    .statusCode(401)
    .body("ok", equalTo(false))
    .body("error", not(emptyString()))
    .body("members", nullValue()); // нет данных
  ```  
• Чек‑лист для тестирования обработки ошибок:  
  - Использовать невалидный, просроченный, отозванный токен.  
  - Проверять `Content-Type` (должен быть `application/json`).  
  - В контрактных тестах (Pact/OpenAPI): указывать, что при `401` тело содержит `ok: false` и `error: string`.  
  - Если статус 200, но `ok: false` — считать это багом (security‑issue).  
• Важно: тесты должны падать, если клиент не валидирует `ok` — проверка должна быть на уровне unit‑тестов клиентской библиотеки.
