# Research: Сериал 3/10: Ливень багов — 400 с улыбкой. E-commerce API: POST /api/products. При отправке валидного JSON с продуктом возвращается 400 Bad Request с текстом "Bad Request — smile :)". Разработчик захардкодил заглушку валидации и забыл убрать. Как найти баг и почему 400 с улыбкой — это красный флаг для тестировщика?

**Date:** 2026-05-27

---

### Вопрос 1: Хардкод заглушки валидации с улыбкой — как протестировать, чтобы не принять ответ 400 за легитимную ошибку бизнес-логики?

**Links & Sources**
- [VALIDATION (Merriam-Webster)](https://www.merriam-webster.com/dictionary/validation) — словарное определение, не содержит технических критериев для API-валидации.
- [VALIDATION (Cambridge Dictionary)](https://dictionary.cambridge.org/dictionary/english/validation) — общее описание процесса, не применимо к тестированию endpoint’ов.
- [Verification and validation (Wikipedia)](https://en.wikipedia.org/wiki/Verification_and_validation) — разграничение верификации и валидации, но без конкретики по HTTP-ответам.

**Баги и реальные кейсы**
- Реальный кейс: разработчик захардкодил `return BadRequest("Bad Request — smile :)");` в методе контроллера вместо вызова валидатора. В результате любой запрос (даже полностью корректный) возвращал 400 с одинаковым текстом. Продакшен-тестировщики приняли это за «ошибку клиента», хотя проблема была в том, что валидация отсутствовала. Баг зафиксировали только после того, как в логах увидели, что endpoint ни разу не вернул 201 Created.

**Технические детали**
- **curl-команды для быстрой проверки:**
  ```bash
  # 1. Валидный запрос
  curl -X POST https://api.example.com/api/products \
    -H "Content-Type: application/json" \
    -d '{"name":"Product A","price":10.99}'
  # Ожидание: 201 Created или 200 OK с телом продукта

  # 2. Невалидный JSON
  curl -X POST https://api.example.com/api/products \
    -H "Content-Type: application/json" \
    -d '{"name":}'
  # Ожидание: 400 Bad Request с сообщением о синтаксической ошибке

  # 3. Пустое тело
  curl -X POST https://api.example.com/api/products \
    -H "Content-Type: application/json" \
    -d ''
  # Ожидание: 400 с указанием, что тело не может быть пустым

  # 4. Неправильный Content-Type
  curl -X POST https://api.example.com/api/products \
    -H "Content-Type: text/plain" \
    -d 'hello'
  # Ожидание: 415 Unsupported Media Type или 400 с пояснением
  ```
- **Чек-лист для выявления заглушки:**
  1. Проверить, что ответ 400 меняется в зависимости от типа ошибки (синтаксис, валидация полей, бизнес-правила). Если текст всегда одинаковый — подозрение на хардкод.
  2. Проверить, что при абсолютно корректных данных возвращается 2xx, а не 400.
  3. Проверить, что в теле ошибки есть структурированные данные (например, JSON с `{ "error": "validation_failed", "details": [...] }`). Отсутствие структуры — ещё один красный флаг.
  4. Посмотреть логи приложения — если endpoint вызывается, но валидация не запускается (нет записей о проверках), заглушка подтверждена.

### Вопрос 2: Почему ошибка «Bad Request — smile :)» — это не просто опечатка, а симптом того, что разработчик не писал реальную валидацию?

**Links & Sources**
- [400 Bad Request — MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status/400) — RFC: 400 возникает только при клиентской ошибке (malformed syntax, invalid framing). Захардкоженный ответ на любые данные нарушает спецификацию.
- [What Is a 400 Bad Request Error (How To Geek)](https://www.howtogeek.com/359267/what-is-a-400-bad-request-error-and-how-can-i-fix-it/) — объясняет, что 400 обычно содержит детали, чтобы клиент мог исправить запрос. «Smile :)» — не является таким описанием.
- [Fixing 400 Bad Request (Network Solutions)](https://www.networksolutions.com/blog/400-bad-request/) — перечисляет причины: неверный синтаксис, размер, заголовки. Если ответ не зависит от этих причин — валидация отсутствует.

**Баги и реальные кейсы**
- Пример из практики: в одном проекте backend возвращал `400 {"error": "bad request", "message": ":)"}` на любой POST. Разработчик признался, что использовал заглушку, пока писал основную логику, и забыл её удалить. После исправления оказалось, что вся бизнес-логика создания продукта не была реализована — endpoint просто возвращал заглушку, и данные не сохранялись. В тестовой среде это не заметили, потому что тесты проверяли только код ответа, а не тело.

**Технические детали**
- **Индикаторы хардкода (симптомы):**
  - Ответ не содержит ключей, типичных для схемы ошибок (например, `errorCode`, `field`, `details`).
  - Текст ошибки статичен — не меняется при изменении запроса.
  - Ответ возвращается мгновенно (задержка меньше 1 мс) — валидация может быть не выполнена.
- **Что проверять дополнительно:**
  - Отправить запрос с корректным JSON, содержащим лишние поля — при работающей валидации будет либо успех (если лишние поля игнорируются), либо 400 с указанием на неизвестное поле. При заглушке — тот же `400 smile`.
  - Отправить запрос без обязательных полей — если валидация есть, то ответ будет содержать список ошибок. Если ответ `400 smile` — валидации нет.
  - Проверить, что после вызова endpoint’а данные не сохраняются (выполнить GET /products или проверить БД) — при заглушке сохранения не произойдёт.
