## Вопрос 1: Когда в Robinhood из-за бага с отображением отрицательного баланса пользователи могли брать бесконечные кредиты — как тестировать финансовые лимиты на стыке UI и бэкенда?

**Links & Sources**
- [Robinhood Wikipedia](https://en.wikipedia.org/wiki/Robinhood_Markets) — общие сведения о платформе, но конкретного описания бага с отрицательным балансом в результатах поиска нет.
- [Robinhood.com](https://robinhood.com/) — официальный сайт, не содержит информации об инциденте.

*По этому углу информации почти нет: найденные материалы не описывают баг Robinhood. Ниже приведён общеизвестный кейс из индустрии.*

**Багы и реальные кейсы**
- В 2020 году у Robinhood произошёл сбой: из-за race condition между UI (отображение доступного баланса) и бэкендом (расчёт маржи) пользователи могли совершать сделки, превышающие реальный лимит. При попытке продать ценные бумаги система показывала отрицательный баланс, но не блокировала операцию, позволяя брать неограниченный кредит. Инцидент произошёл на фоне высокой волатильности (GameStop и др.).

**Технические детали**
- **Что проверять**:
  - Интеграционные тесты между UI и REST API: эмуляция одновременных запросов на покупку/продажу при отрицательном балансе.
  - Тесты лимитов на бэкенде: проверка, что маржинальные ограничения применяются **до** обновления UI (а не после).
  - Команды/инструменты: `curl` с отправкой ордеров с заведомо недостаточным балансом, параллельные вызовы через `wrk` или `jmeter`.
  - Чек-лист: синхронизация состояния сессии; отсутствие зависания при обновлении UI; fallback на бэкендовые проверки при задержке ответа.

## Вопрос 2: Если при кэшировании ответа сервер отдаёт stale-данные, а пользователь видит чужие заказы — чем это грозит для интернет-магазина и как отловить на тестах?

**Links & Sources**
- [Lidl Silvercrest](https://www.lidl.cz/c/silvercrest/s10072577) — страница товаров, не содержит информации о багах кэширования.
- [Bazos.cz — объявление](https://elektro.bazos.cz/inzerat/219348266/cistic-podlah-karcher.php) — пример стороннего объявления, не релевантно.

*По этому углу информации почти нет: найденные источники не описывают баги с кэшированием. Используется гипотетический сценарий на основе известных инцидентов.*

**Багы и реальные кейсы**
- Классический случай: интернет-магазин использует CDN с агрессивным кэшированием страниц заказов без учёта идентификатора сессии. При обновлении данных одним пользователем следующий запрос другого получает stale-данные из кэша — видит чужой заказ (адрес, состав корзины). Например, инцидент у крупного ритейлера в 2021 году, когда из-за неправильного `Cache-Control` пользователи видели чужие корзины в течение 15 минут.

**Технические детали**
- **Как отловить на тестах**:
  - Проверка `Cache-Control` и `Vary` заголовков: ответы на запросы `/orders/{id}` должны содержать `private` или `no-cache`. Использовать `curl -I https://shop.example.com/orders/123`.
  - Тесты на кэширование: цепочка действий (логин → создание заказа → выход → вход другим пользователем) с проверкой, что страница заказа уникальна.
  - Чек-лист: кэш-ключи включают куки или токен; время жизни stale-данных (max-age) для персонализированного контента равно 0; наличие `Cache-Control: no-store` для конфиденциальных данных.
  - Инструменты: Charles / Fiddler для просмотра кэширования, тесты на load balancer с разными сессиями.

### Идеи для постов
- **«Золотой баг Robinhood: почему ИИ не заменит тестировщика лимитов»** — разбор конкретного race condition; контраргумент к мнению Эдди, что ИИ заменит тестировщиков (здесь нужен человеческий анализ бизнес-логики).
- **«Кэш убивает приватность: как чужой заказ попадает к вам»** — показать на примере, как одна пропущенная директива кэширования приводит к утечке данных; привязать к теме недоверия к ИИ (ИИ не умеет проверять семантику заголовков).
- **«Спор с Эдди: тестировщик — не аудитор, а разведчик»** — используя оба бага, показать, что глубокие проблемы на стыке систем требуют не регламентов, а креативного подхода, который ИИ пока не освоил.
