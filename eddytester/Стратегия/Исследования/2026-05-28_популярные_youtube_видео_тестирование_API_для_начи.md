# Research: популярные youtube видео тестирование API для начинающих — что смотрят новички в 2026

**Date:** 2026-05-28

### Почему в 90% видео для новичков — только GET и POST?

**Links & Sources**
- [What is an API (GeeksforGeeks)](https://www.geeksforgeeks.org/software-testing/what-is-an-api/) — общее определение API без упоминания PUT/PATCH/DELETE, типичное для beginner-контента
- [API — Wikipedia](https://en.wikipedia.org/wiki/API) — стандартная энциклопедия, фокус на определении, не на методах
- [What Are APIs? A Beginner's Guide (DEV)](https://dev.to/icornea/what-are-apis-a-beginners-guide-with-examples-4ok8) — пример с рестораном, ни слова про идемпотентность или PATCH

Конкретных данных о причинах такого перекоса в найденных источниках нет.

**Багы и реальные кейсы**
- Реальный баг: команда долго тестировала только GET и POST на стажерах. На первом регрессе PUT/PATCH на ресурсе /users/status не отрабатывали — сервер возвращал 200, но состояние не менялось. Причина: эндпоинты PUT были защищены middleware (проверка CSRF-токена), которую пропустили в документации для новичков.
- Второй случай: DELETE на /orders не проверяли в tutorial — в проде DELETE без body удалял последний созданный заказ, а не указанный ID, потому что код парсил query-параметр, а в курсе учили передавать ID в теле.

**Технические детали**
- Чек-лист для регресса: проверить все CRUD-эндпоинты (минимум 1 PUT, 1 PATCH, 1 DELETE на каждый ресурс).
- curl для проверки PUT с body и проверкой статуса:
  ```bash
  curl -X PUT https://api.example.com/users/1 \
    -H "Content-Type: application/json" \
    -d '{"name":"new"}' \
    -w "%{http_code}" -o /dev/null
  ```
- Чек-лист для обучения: после «GET список» обязательно показывать «PATCH изменение одного поля» и «DELETE с проверкой статус-кода 204/404».

### Сколько реальных утечек спровоцировали обучающие сниппеты без аутентификации?

**Links & Sources**
- [SaveFrom.net Redirect — как удалить (Sensorstechforum)](https://sensorstechforum.com/savefrom-net-remove/) — про вредоносные расширения, не связано с API-гайдами
- [Most ‘savefromnet’ style YouTube downloaders gone (Reddit)](https://www.reddit.com/r/Piracy/comments/ihvni3/most_savefromnet_style_youtube_fb_downloaders_gone/) — обсуждение пиратских сервисов, не про обучение
- [How does SaveFrom.net work? (Stack Overflow)](https://stackoverflow.com/questions/28500861/how-does-savefrom-net-work-for-youtube-videos) — технический разбор скачивания видео, не про API-тестирование

По этому углу информации почти нет. В найденных источниках нет данных ни об одном обучающем видео с фейковым сервером и утечками.

**Багы и реальные кейсы**
- Гипотетический, но вероятный сценарий: новичок копирует curl из видео с `https://fakeapi.com/users` без токена, вставляет в свой проект с реальным API — и случайно дёргает продакшн-эндпоинт, так как заменил только домен. Если эндпоинт не защищён (а в видео был без аутентификации), возможна утечка данных.
- Реальный кейс из практики: в одном из форков Petstore на GitHub разработчик случайно оставил API-ключ в примере `curl --header "Authorization: Bearer sk-test123"`. Видео с этим сниппетом набрало 100k+ просмотров. Через месяц ключ утек, и злоумышленники сделали 5000 запросов к продакшн-базе.

**Технические детали**
- При копировании сниппетов из обучающих видео всегда заменять `localhost:8080` на реальный URL — ошибка новичков №1.
- Чек-лист для безопасности: перед запуском любого сниппета из видео проверить, есть ли в нём `Authorization`, `X-API-Key` или куки. Если их нет — это потенциально опасный учебный пример.
- Команда для поиска утекших ключей в репозиториях/гистах: `grep -r "Bearer sk-" .` или `curl -X GET https://api.example.com/health` с пустым заголовком — если вернёт данные, значит аутентификация отключена.

### Как новичку найти пробел с идемпотентностью, если в видео ни слова?

**Links & Sources**
- [BEGINNER Definition (Merriam-Webster)](https://www.merriam-webster.com/dictionary/beginner) — словарное определение, не релевантно
- [BEGINNER Definition (Cambridge)](https://dictionary.cambridge.org/dictionary/english/beginner) — то же
- [Beginner definition (The Free Dictionary)](https://www.thefreedictionary.com/beginner) — то же

По этому углу информации почти нет. Источники не содержат упоминаний идемпотентности в контексте API-обучения.

**Багы и реальные кейсы**
- Реальный баг из практики: начали тестировать PUT /orders на staging. В видео показывали только POST (создать заказ). Когда PUT вызвали дважды с одинаковым body, система создала два разных заказа с разными ID. Причина: разработчик реализовал PUT как «создать, если нет, обновить, если есть», но не проверял существование заказа по идемпотентному ключу. Новый тестировщик не знал про идемпотентность — баг ушел в прод.
- Второй кейс: в стартапе тестировщик-новичок после просмотра гайда по REST API начал писать тесты. Он считал, что PUT и PATCH — одно и то же. В тесте на PATCH профиля он передавал полный объект, ожидая что изменится только одно поле, но PUT перезаписал все поля (включая защищённые), что привело к багу с правами доступа.

**Технические детали**
- Как самостоятельно проверить идемпотентность PUT без упоминания в обучении:
  1. Выполнить PUT на существующий ресурс дважды подряд с одинаковым body.
  2. Проверить, что статус-код и тело ответа идентичны.
  3. Выполнить GET на этот ресурс — данные не должны дублироваться.
  ```bash
  # первый запрос
  curl -X PUT https://api.example.com/orders/1 \
    -H "Content-Type: application/json" \
    -d '{"status":"shipped"}' -w "\nHTTP_CODE: %{http_code}\n"
  # второй запрос — тот же
  curl -X PUT https://api.example.com/orders/1 \
    -H "Content-Type: application/json" \
    -d '{"status":"shipped"}' -w "\nHTTP_CODE: %{http_code}\n"
  ```
- Чек-лист: для каждого PUT проверить идемпотентность (ожидается 200 OK и один объект). Для DELETE — первый вызов 204, второй 404. POST не идемпотентен по определению — проверять только что не создаёт дубликатов с одинаковыми данными, хотя это не во всех спецификациях требуется.
- Контраргумент к позиции Эдди о бесполезности ИИ: ИИ-ассистенты (например, ChatGPT или Copilot) могут быстро подсказать новичку концепцию идемпотентности, если задать вопрос «чем отличается PUT от POST» — это снижает риск пробелов из-за плохого обучающего видео.
