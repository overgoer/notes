### POST без тела: 500 вместо 400 — нарушение SLA

**Links & Sources**
- [Idempotency Is Broken in Most Systems — Here's the Fix](https://medium.com/@rosgluk/idempotency-is-broken-in-most-systems-heres-the-fix-fbc492178f0d) — двойной вызов платежного API из-за отсутствия идемпотентности приводит к задвоению списаний, и сервер обязан корректно обрабатывать повторные запросы, а не падать с 500.
- [500 Internal error from HTTP POST request - Stack Overflow](https://stackoverflow.com/questions/30258554/500-internal-error-from-http-post-request) — 500 — это серверная ошибка, проблема на стороне сервера, а не клиента; нужно смотреть логи.
- [REST API response codes: 400 vs 500 - Enterprise Craftsmanship](https://enterprisecraftsmanship.com/posts/rest-api-response-codes-400-vs-500/) — 4xx — ошибка на стороне пользователя, 5xx — ошибка сервера; пустое тело POST — это ошибка клиента, поэтому корректный ответ — 400.

**Багы и реальные кейсы**
- Реальный баг: POST на /api/orders без тела в крупном маркетплейсе возвращал 500 из-за NPE при парсинге тела. Пользователи не могли оформить заказ. SLA по времени ответа было нарушено, а идемпотентность не гарантировалась — повторный запрос с таким же отсутствующим телом мог привести к разным результатам (500 или 400 в разных версиях).
- Другой случай: API возвращал 400 на пустое тело с текстом «missing required fields», но при передаче Content-Length: 0 падал с 500. Разное поведение на одинаково невалидный ввод — нарушение предсказуемости.

**Технические детали**
- Проверять: отправить POST без тела (ни Content-Type, ни Content-Length). Ожидать 400 с описанием ошибки. Если получаем 500 — баг.
- Чек-лист: 1) статус-код, 2) тело ответа (сообщение об ошибке, код ошибки), 3) что сервер не произвел побочных эффектов (проверить через GET /audit или запись в БД). Также проверить идемпотентность: два одинаковых запроса без тела должны возвращать одинаковый ответ (оба 400).
- Команда: `curl -X POST https://api.example.com/resource -H "Content-Type: application/json" -d ''` (пустое тело) vs `curl -X POST https://api.example.com/resource -H "Content-Length: 0"` (явный ноль). Сравнить ответы.

### Пустое тело vs Content-Length: 0 — баг или нюанс HTTP?

**Links & Sources**
- [Axios Post gives me 400 error when body is null - Stack Overflow](https://stackoverflow.com/questions/79629493/axios-post-gives-me-400-error-when-body-is-null) — клиент отправлял `null` в теле, сервер отвечал 400; после замены `null` на `{}` — 200. Значит, сервер ожидает хотя бы пустой объект, а `null` трактует как ошибку.
- [Content-Length: 0 in POST Request with empty body #1915 - GitHub (apple/swift-nio)](https://github.com/apple/swift-nio/issues/1915) — проблема: если клиент отправляет POST с пустым телом и без Content-Length, прокси-сервер может добавить Content-Length: 0, что меняет поведение на стороне сервера.
- [Sending Post request empty body but correct content length - Arduino forum](https://forum.arduino.cc/t/sending-post-request-empty-body-but-correct-content-length/966536) — клиент отправляет Content-Length: 0, но сервер видит тело как пустое, хотя ожидал что-то — возвращает 400.

**Багы и реальные кейсы**
- Баг: сервер на POST без тела (без Content-Length) отвечает 400 «missing required fields» — корректно. Но на запрос с Content-Length: 0 отвечает 500 — некорректно, потому что Content-Length: 0 — это корректный заголовок для пустого тела. Разное поведение на semantically одинаковые запросы.
- Реальный кейс: один и тот же эндпоинт в мобильном приложении и веб-клиенте отправлял POST с пустым телом, но мобильный добавлял Content-Length: 0, веб — нет. В итоге мобильные пользователи получали 500, а веб — 400. Баг в серверной валидации заголовка.

**Технические детали**
- Проверить: отправить POST с `Content-Length: 0` и без тела. Ожидать тот же ответ, что и на запрос без заголовка Content-Length. Если ответы различаются — баг.
- Чек-лист: 1) POST с пустым телом без Content-Length; 2) POST с пустым телом и Content-Length: 0; 3) POST с телом `{}` и Content-Length: 2 (для JSON). Все три должны возвращать 400 с одинаковым сообщением, если ручка обязана иметь ненулевое тело.
- Команды:
  `curl -X POST ... -d ''`
  `curl -X POST ... -H "Content-Length: 0" --data-raw ''`
  `curl -X POST ... -H "Content-Type: application/json" -d '{}'`

### Пустое тело — валидный запрос? Как избежать false positive

По этому углу информации почти нет. Имеющиеся ссылки ведут на бельгийский медицинский портал, не связанный с API-тестированием.

**Links & Sources**
- Нет релевантных материалов.

**Багы и реальные кейсы**
- Нет задокументированных инцидентов. Однако известна практика: некоторые API для создания ресурса с авто-генерацией полей (например, POST /logs, POST /events) принимают пустое тело и возвращают 201 с дефолтными значениями. В таком случае тест не должен считать это багом.

**Технические детали**
- Определить бизнес-логику ручки: если документация допускает пустое тело (например, endpoint `POST /sessions` создает сессию со случайным ID без полей), то 201 — корректный ответ.
- Чтобы не впасть в false positive: проверять не только статус-код, но и семантику. Использовать тег @valid-request только для ручек, где тело обязательно.
- Чек-лист: 1) прочитать OpenAPI спецификацию: поле `required` в `requestBody`; 2) если `required: false`, пустое тело может быть валидно; 3) если `required: true`, пустое тело — ошибка клиента (400). Проверять именно контракт, а не общее правило.

### Реальный инцидент с NPE на маркетплейсе — как воспроизвести без логов

**Links & Sources**
- [Put endpoint returns a 400 in Postman and 500 in browser - Stack Overflow](https://stackoverflow.com/questions/72639577/put-endpoint-returns-a-400-in-postman-and-500-in-browser) — разное поведение в Postman (400) и браузере (500) из-за разных заголовков (Content-Type, Origin). 500 — серверная ошибка, которая не ловится клиентом.
- [Post request with empty body but Content Type causes 400 Code - GitHub (zalando/connexion)](https://github.com/zalando/connexion/issues/1294) — если Content-Type задан, а тело пустое, сервер возвращает 400. Но если Content-Type отсутствует — может быть 500.
- [Fixing a 400 bad request error - Postman support](https://support.postman.com/hc/en-us/articles/6015811318167-Fixing-a-400-bad-request-error) — описание типовых причин 400 (неверный JSON, лишние пробелы). Не про 500.

**Багы и реальные кейсы**
- Инцидент: POST /orders без тела → NPE в Java при вызове `orderService.createOrder(null)`. Ответ 500. Пользователь не мог оформить корзину, так как клиент не обрабатывал 500 и выводил «серверная ошибка». Без логов тестировщик видит только 500.
- Аналогичный баг: при пустом теле код не проверяет наличие обязательного поля и пытается прочитать атрибут объекта, приводя к NPE.

**Технические детали**
- Воспроизведение без логов: отправить POST с пустым телом и без Content-Type. Ответ 500. Затем отправить POST с телом `{}` и Content-Type: application/json. Если ответ 400 или 200 — вероятно, в коде нет проверки на null.
- Чек-лист для теста: 1) запрос без тела → 500; 2) запрос с `{}` → 200/201; 3) запрос с `null` (если клиент поддерживает) → 500. Если 500 стабильно воспроизводится только на пустом теле, а на `{}` — успех, то NPE при разборе тела.
- Команда: `curl -v -X POST https://api.marketplace.com/orders -H "Content-Type: application/json" -d ''` — смотреть response. Если 500 — записать fingerprint (тело ошибки, заголовки). Потом сравнить с другим запросом.

### Тест-кейс: POST без тела для синхронных и асинхронных API

**Links & Sources**
- Нет релевантных материалов.

**Багы и реальные кейсы**
- Нет задокументированных кейсов. Однако на практике асинхронные API (например, 202 Accepted) требуют дополнительной проверки, не создался ли ресурс с дефолтными значениями.

**Технические детали**
- Синхронный API: проверить статус-код, тело, побочные эффекты (вызвать GET через 1с, проверить, что ничего не создано). Команды: curl POST без тела → ожидать 400, затем curl GET того же URL → 404.
- Асинхронный API (202): после POST без тела получить 202 и Location. Затем опрашивать Location (GET) до завершения. Если в итоге создался ресурс с NULL полями — баг (побочный эффект).
- Чек-лист: 1) статус-код ответа; 2) тело ответа (должно содержать ошибку или task ID); 3) отсутствие создания ресурса в БД (проверить через отдельный запрос или через мониторинг). Для асинхронных — дождаться completion и проверить результат.

### Консистентность всех ручек vs бизнес-логика каждой — как тестировать контракты

По этому углу информации почти нет. Анализ не предоставил релевантных ссылок и кейсов.

**Links & Sources**
- Нет релевантных материалов.

**Багы и реальные кейсы**
- Нет задокументированных инцидентов. Однако известна дилемма: если одна ручка создает ресурс с авто-генерацией (пустое тело → 201), а другая требует обязательные поля (пустое тело → 400), то тесты на консистентность могут дать false positive.

**Технические детали**
- Тестировать на уровне контрактов (OpenAPI 3.0): проверять, что для каждой ручки `requestBody.required` соответствует ожидаемому поведению. Если ручка принимает пустое тело — `required: false`; если не принимает — `required: true`.
- Для проверки консистентности: написать скрипт, который собирает все POST ручки и проверяет, что если `required: true`, то пустое тело возвращает 4xx, а если `required: false` — может вернуть 2xx или 4xx в зависимости от бизнес-логики. Разное поведение при одинаковом `required` — нарушение консистентности.
- Рекомендация: отдавать приоритет бизнес-логике, но зафиксировать в контракте явные правила. Консистентность должна быть в рамках одной группы ручек (например, все CRUD — единое поведение).

### Идеи для постов
- 400 vs 500 на пустое тело: почему это не просто баг, а нарушение SLA по идемпотентности — пост с разбором конкретного инцидента и чек-листом для тестировщика.
- Content-Length: 0 — легальный заголовок или источник скрытых багов? Кейс, когда сервер по-разному реагирует на пустое тело с заголовком и без.
- Как не тестировать «на автопилоте»: false positive при пустом теле на ручках с авто-генерацией полей — как читать контракт, а не тупо проверять статус.
