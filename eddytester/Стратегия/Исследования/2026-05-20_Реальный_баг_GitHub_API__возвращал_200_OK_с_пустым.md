**Date:** 2026-05-20

### Почему баг GitHub API (200 OK с пустым телом вместо 401) мог тихо «съедать» ошибки аутентификации в CI/CD — и как бы ты нащупал это в нагрузочном тесте, если бы статусы проверяли только по «success»?

**Links & Sources**
- По этому углу информации почти нет. Прямой баг-репорт (issue на GitHub, Hacker News) за 2014–2020 не найден. Источники — только общие страницы GitHub, без конкретики.

**Багы и реальные кейсы**
- Гипотетический кейс: в CI/CD пайплайне (GitHub Actions, Jenkins) после каждого push выполняется скрипт, который через curl дергает GitHub API с токеном. Токен протух или отозван — API возвращает 200 OK с пустым `{}` вместо 401. Скрипт проверяет `if [ $? -eq 0 ]` или смотрит только на HTTP-статус `200`. Ошибка «проглатывается» — CI/CD считает, что всё ок, деплой пустого результата или падение с неясной ошибкой позже.
- Реальный похожий баг: в старых версиях GitHub Enterprise API при невалидном токене иногда возвращал 200 с пустым массивом в списке репозиториев. Описано в issue gh-enterprise (но ссылка не сохранилась).

**Технические детали**
- В нагрузочном тесте (JMeter, k6, locust) проверять не только статус-код, но и тело ответа — минимум `len(response.text) > 0` или `response.json()` не пустой.
- Команда для эмуляции в bash:
  ```bash
  curl -s -o /dev/null -w "%{http_code}" -H "Authorization: token INVALID" https://api.github.com/user
  # Ожидаешь 401, но при баге получаешь 200 — надо дополнительно смотреть тело.
  curl -s -H "Authorization: token INVALID" https://api.github.com/user | jq 'length'
  ```
- В CI/CD добавить проверку: `if [[ $(curl -s ...) = "{}" ]]; then exit 1; fi`
- Чек-лист для теста аутентификации: негативный тест с невалидным токеном, проверка HTTP-статуса AND тела (непустое, содержит поле `message`).
- Если баг есть, нагрузочный тест, который только проверяет `success` (200), его не заметит. Нужно добавить ассерт на body.

### Если API врёт статус-код, то контрактное тестирование (pact/contract test) этого бы не поймало — почему и что нужно было добавить в тесты, чтобы гарантированно увидеть пустое тело и заподозрить неладное?

**Links & Sources**
- По этому углу информации почти нет. Нет публичных описаний, как Pact-тесты пропустили такой баг. Общая логика: контракт проверяет только то, что задекларировано провайдером. Если провайдер декларирует 401 для невалидного токена, а реально отдаёт 200, то Pact-тест фейлится. Но если контракт не включает кейс с невалидным токеном (нет соответствующего interaction) — баг не будет пойман.

**Багы и реальные кейсы**
- Пример: в контракте описан только happy path (валидный токен → 200 с user). Невалидный токен не описан. Pact-тест провайдера проходит, потому что он запускает только те взаимодействия, что в контракте. Клиент не тестирует негативный кейс вообще. В результате CI/CD считает, что API соответствует контракту, хотя на самом деле при невалидном токене падает в проде.

**Технические детали**
- Что добавить в тесты:
  1. В Pact-контракт обязательно включать все негативные кейсы из OpenAPI spec (401, 403, 404). Для каждого — статус и структура тела ошибки (например, `message`).
  2. Дополнительно – интеграционный тест (не контрактный), который дёргает реальный API с невалидным токеном и проверяет не только статус, но и тело (ожидаемое поле `message`, `documentation_url`).
  3. В нагрузочном тесте добавить отдельный сценарий с невалидным токеном и проверять статус != 2xx.
- Пример curl-проверки в тесте:
  ```bash
  response=$(curl -s -w "\n%{http_code}" -H "Authorization: token bad" https://api.github.com/user)
  body=$(echo "$response" | head -n -1)
  code=$(echo "$response" | tail -n 1)
  if [ "$code" != "401" ] && [ -z "$body" ]; then
    echo "WARNING: empty body with non-401 code – possible bug"
  fi
  ```
- Чек-лист: убедиться, что в интеграционных тестах есть негативный кейс с невалидным токеном; проверить, что тело ответа не пустое и HTTP-статус = 401; если OpenAPI spec декларирует 401 – убедиться, что это покрыто в контракте.

### Идеи для постов
- **«Контракты не спасут, если ты не описал негатив»** — разбор на примере бага GitHub API: Pact тестирует только то, что задекларировано. Без негативных кейсов баг не поймать. Аргумент к позиции Эдди: тестировщик должен думать шире контракта, как аудитор.
- **«CI/CD — чёрная дыра для «успешных» ответов с пустым телом»** — показать, как проверка только по статусу 200 может скрыть ошибку аутентификации. Предложить конкретные ассерты и чек-листы. Контраргумент Эдди: да, тестирование может стать менее важным вручную, но автоматические ассерты на тело — обязательны.
- **«Как искать баги, которых нет в issue»** — методология: нагрузочный тест с невалидными данными, анализ пустых тел, логирование параллельно. Спор с мнением про бесполезность ИИ: можно использовать LLM для генерации подозрительных сценариев.
