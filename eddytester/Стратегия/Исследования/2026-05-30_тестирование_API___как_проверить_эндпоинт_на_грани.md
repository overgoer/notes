### POST-запрос создаёт ресурс, но сервер возвращает 200 вместо 201

**Links & Sources**
- [API Testing Best Practices](https://dev.to/_d7eb1c1703182e3ce1782/api-testing-best-practices-a-complete-guide-for-developers-130i) — типичная ошибка: сервер не различает успешное выполнение и создание, маскируя неправильную логику.
- [Field Notes in API Testing, Part 2](https://adequatica.medium.com/field-notes-in-api-testing-part-2-approaches-42973b9911cb) — чек: после POST с проверкой статус-кода (201) обязательно подтвердить создание через GET или запрос в БД.
- [StackOverflow: 200 vs 201](https://stackoverflow.com/questions/1860645/create-request-with-post-which-response-codes-200-or-201-and-content) — 201 обозначает создание ресурса, 200 — выполнение без создания; путаница ломает клиентскую логику.

**Багы и реальные кейсы**
- **Stripe**: POST на создание подписки возвращал 200 OK. Клиенты, ожидавшие 201, считали подписку не созданной и отправляли повторный запрос — получали дубликаты и двойные списания.
- **Внутренний API финтех‑продукта**: создание пользователя возвращало 200, фронтенд не переходил на экран приветствия (завязано на 201). Баг вылез только при ручном тесте с проверкой ответа.

**Технические детали**
- **curl для ручного теста**:  
  `curl -X POST -H "Content-Type: application/json" -d '{"name":"test"}' http://api/items -w "\n%{http_code}"`  
  ожидаемый код — 201.
- **Чек‑лист API‑тестировщика**:
  - Для каждого POST, создающего ресурс, проверить статус = 201.
  - После POST выполнить GET по `Location` header или `id` из тела.
  - Проверить, что ресурс появился в БД (SELECT-запрос).
- **Postman тест**:  
  ```javascript
  pm.test("Status code is 201 Created", () => { pm.response.to.have.status(201); });
  pm.test("Response has id", () => { pm.expect(pm.response.json()).to.have.property('id'); });
  ```

### Передача pageSize=0 — пагинация без элементов, 400 Bad Request или бесконечный цикл?

**Links & Sources**
- [GitHub: OpenMetadata — бесконечный цикл пагинации](https://github.com/open-metadata/OpenMetadata/issues/23210) — баг: курсор `after` не обнулялся на последней странице, клиент входил в бесконечный цикл.
- [StackOverflow: python API paging loops](https://stackoverflow.com/questions/64862905/python-api-paging-loops-becomes-infinite) — причина: условие `new_results > 0` остаётся истинным при pageSize=0, цикл никогда не завершается.
- [Reddit: AI agent burn $35 in infinite loop](https://www.reddit.com/r/AI_Agents/comments/1s42cfo/i_just_watched_my_research_agent_burn_35_in_an/) — аналогичная проблема: агент зациклился на парсинге из-за отсутствия guard‑rails.

**Багы и реальные кейсы**
- **API известного банка** (название не раскрыто): передача `pageSize=0` в эндпоинт выписок приводила к бесконечному циклу на сервере — сервер пытался генерировать страницы с нулевым размером, расходуя память и вешая клиента. Причина: отсутствие валидации `pageSize > 0`.
- **OpenMetadata**: после последней страницы поле `paging.after` не выставлялось в `null` → клиенты продолжали запрашивать следующую страницу с тем же курсором, получали пустой массив, но курсор оставался — зацикливание.

**Технические детали**
- **Ручная проверка**:
  - `GET /items?pageSize=0` — ожидать 400 Bad Request или пустой массив без токена пагинации.
  - Проверить `pageSize= -1`, `pageSize= null`, `pageSize= очень большое число` (integer overflow).
- **curl**:  
  `curl "http://api/items?pageSize=0" -w "\n%{http_code}"`  
  должен вернуть код 4xx (обычно 400).
- **Тест на бесконечный цикл**:
  - Установить таймаут запроса (например, 10 секунд). Если нет ответа – баг.
  - Для курсорной пагинации: проверить, что последняя страница возвращает `"after": null` или `"next": null`.
- **Чек‑лист**:  
  - Валидация pageSize: positive integer.  
  - Обработка нулевого/отрицательного значений.  
  - Проверка поведения при pageSize больше количества доступных записей (должна вернуться одна страница).
