# Research: гайд по тестированию API — чек-лист ручного тестирования бэкенда, что проверять в API, практическое руководство

**Date:** 2026-05-30

---

### Почему нужно тестировать Retry-After и X-RateLimit-* даже на публичных эндпоинтах

**Links & Sources**
- [GitHub API Rate Limiting Documentation](https://docs.github.com/en/rest/overview/resources-in-the-rest-api#rate-limiting) — описание лимитов: 5000/час для авторизованных, 60 для неавторизованных; заголовки X-RateLimit-Limit, -Remaining, -Reset.
- [GitHub incident on decreased unauthenticated rate limit (2018)](https://github.blog/news-insights/product-news/rate-limiting-public-api/) — в 2018 GitHub временно снизил лимит для неавторизованных с 5000 до 60/час, что сломало интеграции, не ожидавшие такого.
- [Postman Blog: Why You Should Always Test Rate Limits](https://blog.postman.com/why-you-should-always-test-rate-limits/) — кейсы, когда отсутствие проверки Retry-After приводило к блокировке клиентов.

**Баги и реальные кейсы**
- В 2018 GitHub изменил лимит для неавторизованных с 5000 до 60/час. CI/CD скрипты, не проверявшие X-RateLimit-* и Retry-After, начали падать с 403/429. Некоторые клиенты не обрабатывали 429 и Retry-After, продолжая слать запросы — это приводило к полной блокировке IP.
- Реальный баг: приложение мониторинга репозиториев не читало Retry-After, после получения 429 сразу же повторяло запрос, попадая в бесконечный цикл ошибок. Исправление — чтение Retry-After и пауза.

**Технические детали**
- curl для проверки заголовков:
  ```bash
  curl -I https://api.github.com/users/octocat
  ```
  Ответ: `X-RateLimit-Limit: 60`, `X-RateLimit-Remaining: 59`, `X-RateLimit-Reset: 1717000000`, при 429 — `Retry-After: 3600`.
- Чек-лист:
  - [ ] Отправить запрос без авторизации → 200 + заголовки rate limit.
  - [ ] Превысить лимит → 429 + Retry-After.
  - [ ] После паузы на Retry-After запрос успешен.
  - [ ] Разные лимиты для разных эндпоинтов (если документировано).
  - [ ] При авторизации лимит выше (например, 5000).

### Почему отсутствие проверки Content-Type — это дыра, а не просто баг

**Links & Sources**
- [Rapid7: XSS in JSON — Old-School Attacks for Modern Applications](https://www.rapid7.com/blog/post/2022/05/04/xss-in-json-attacks-in-modern-applications/) — если сервер принимает JSON, но не валидирует Content-Type, пейлоад с text/html может выполниться в браузере при ответе.
- [Medium: Bypassing JSON Content-Type via OPTIONS Method](https://medium.com/@ansarirehan1230786/bypassing-json-content-type-executing-xss-via-options-method-trick-9ab716caf1cd) — OPTIONS запрос возвращает данные без Content-Type, что позволяет внедрить скрипт.
- [Security StackExchange: JSON response flagged as XSS](https://security.stackexchange.com/questions/231930/json-api-response-flagged-as-xss-by-vulnerability-scanner-is-this-a-false-posit) — объяснение: без `X-Content-Type-Options: nosniff` браузер может интерпретировать JSON как HTML.

**Баги и реальные кейсы**
- Сервер принимает `POST /api/feedback` с `Content-Type: application/json`, но не проверяет, что запрос действительно JSON. Злоумышленник шлет `Content-Type: text/html` с телом `<script>alert(1)</script>`. Сервер возвращает ошибку с этим телом и `Content-Type: text/html`. Без `nosniff` браузер выполняет скрипт — полноценная XSS.
- HackerOne кейс: API возвращал 400 Bad Request с телом ошибки, содержащим ввод пользователя. Content-Type ответа был `text/html`. Отправка JSON с XSS-пейлоадом внутри приводила к выполнению скрипта.

**Технические детали**
- curl проверка:
  ```bash
  curl -X POST https://example.com/api/endpoint \
    -H "Content-Type: text/html" \
    -d "<script>alert('XSS')</script>"
  ```
  Проверить: ответ содержит скрипт, Content-Type ответа — `text/html` (или отсутствует `nosniff`).
- Чек-лист:
  - [ ] Отправить запрос с неправильным Content-Type (text/plain, text/html) — валидируется ли?
  - [ ] Ответ успешного запроса содержит `Content-Type: application/json`.
  - [ ] Ответ содержит `X-Content-Type-Options: nosniff`.
  - [ ] Если API возвращает ошибку с вводом пользователя — проверено экранирование/кодирование.
  - [ ] Метод OPTIONS: если возвращает данные — проверить Content-Type и nosniff.
