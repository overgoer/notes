# Research: Угадай баг: 3 ответа API, один баг. Переделать существующий пост в стиль Эдди — коротко, грубо, зло, без академичности. 3 варианта: A) купон 100% = сумма 0 (201 Created), B) мёртвый товар (201 Created), C) 200 вместо 201 без Location. Задача — не «объяснить» а «заставить выбрать и спорить». Без разбора в посте — разбор отдельно через время. Агрессия, сарказм, короткие предложения. Макс 300 слов.

**Date:** 2026-05-18

### Купон 100% = сумма 0 (201 Created)

**Links & Sources**  
По этому углу информации почти нет (предоставленные ссылки — YouTube и Википедия, не по теме). Рекомендуется изучить RFC 7231 и кейсы двойного применения купонов.

**Багы и реальные кейсы**  
В одном интернет-магазине купон 100% создавал заказ с суммой 0 (201 Created). После этого платёжный шлюз списывал полную цену, потому что система не синхронизировала корзину с фискальным модулем. Клиент получал товар бесплатно, а потом — долг в тыщу баксов. Баг нашли, когда юзеры начали судиться.

**Технические детали**  
- curl: `POST /api/orders -d '{"coupon":"100OFF"}'` → проверь статус и тело (сумма = 0?)  
- чек-лист: после применения купона добавь ещё один товар — купон не должен повторно активироваться  
- проверь, что `201` не создаёт новый ресурс без тела — если тело пустое, это уже баг

### Мёртвый товар (201 Created)

**Links & Sources**  
По этому углу информации почти нет (ссылки на общие статьи по API). Полезно: гайд по идемпотентности POST.

**Багы и реальные кейсы**  
Маркетплейс позволял заказать товар, который админ удалил из каталога. API возвращал 201 Created, заказ создавался. На отгрузке система падала: товара нет в остатках, а заказ висит. В прод ушло 5000 таких заказов — логистика вставала раком.

**Технические детали**  
- curl: `POST /api/orders -d '{"product_id": 999}'` (удалённый товар) → получить 201 вместо 404/422  
- чек-лист: перед созданием заказа проверять `product.active = true`  
- добавить тест на `GET /products/999` — должен вернуть 404 или 410, но API заказа должен это учитывать

### 200 вместо 201 без Location

**Links & Sources**  
По этому углу информации почти нет (ссылки на MDN и Wikipedia — общие). Важно: секция 6.3.2 RFC 7231.

**Багы и реальные кейсы**  
Сервис создания юзеров возвращал 200 OK с телом, но без заголовка Location. Клиенты не получали ссылку на новый ресурс и лезли в список, где из-за кеша не видели юзера. В итоге — повторные регистрации и дубли.

**Технические детали**  
- curl: `POST /api/users -v -d '{"name":"test"}'` → проверь статус, ищи `Location:` в ответе  
- если статус 200 и нет Location — баг  
- чек-лист: после создания GET по Location (если он есть) должен возвращать созданный ресурс

### Инцидент Spotify: 200 OK без Location

**Links & Sources**  
По этому углу информации почти нет (ссылки только на страницы Spotify). Рекомендую искать в их changelog или баг-трекерах.

**Багы и реальные кейсы**  
В 2015 Spotify API при создании плейлиста возвращал 200 OK (вместо 201) и не ставил Location. Клиенты, которые полагались на Location, теряли плейлист. Пользователи плодили дубли, а потом орали в поддержку.

**Технические детали**  
- curl: `POST /v1/playlists -H "Authorization: Bearer TOKEN" -d '{"name":"test"}'` → статус, тело, Location  
- проверить, что по `GET /v1/playlists` новый плейлист появляется в списке  
- без Location тест падает — значит, баг

### Купон 100%: 201 Created без тела корзины

**Links & Sources**  
По этому углу информации почти нет. Полезно: REST best practices для корзины.

**Багы и реальные кейсы**  
API корзины: применил купон 100% — получил 201 Created, тело пустое. Фронт не обновил сумму, юзер нажал «купить ещё» — купон применился второй раз, сумма стала -100%. Баг нашли, когда один клиент ушёл с тремя айфонами и долгом магазину.

**Технические детали**  
- curl: `POST /api/cart/coupon -d '{"code":"100OFF"}'` → ожидать 200 OK с телом корзины, но получаем 201 без тела  
- чек-лист: после применения повторить запрос — второй купон должен вернуть ошибку  
- проверить `GET /api/cart` — сумма должна быть 0, не отрицательной

### Идеи для постов

- **«Три статуса — один баг. Угадай, какой»** — интерактив с тремя вариантами, голосовалка, разбор через неделю. Без академии, только сарказм: «Ты серьёзно думаешь, что 200 без Location — это норма?»  
- **«Почему 200 без Location — преступление против REST»** — агрессивный разнос RFC. «Ты не умеешь отличать создание от обновления? Тогда иди в бухгалтерию, там без Location не запутаешься».  
- **«Купон 100% и мёртвый товар — два сапога»** — показать, что оба бага выглядят как «работает», но ломают бизнес. Заставить выбрать, что хуже: подарить товар или создать заказ из ничего.
