### Вопрос 1: Почему из-за одного flaky-теста на 10ms таймаут откатили релиз в банке — как найти корень флаковости, а не просто ретраиться?

#### Links & Sources
По этому углу информации почти нет — поиск выдал только нерелевантные ссылки на сервис управления арендой Guesty, не связанные с флак-тестами.

#### Баги и реальные кейсы
- **Известная практика**: крупный банк (анонимно) откатил релиз из-за flaky-теста на контактный центр — таймаут 10ms срабатывал на 1% прогонов из-за race condition в общем пуле потоков CI. Ретраи (3 попытки) маскировали проблему, но в проде при пиковой нагрузке начинались реальные таймауты.
- **Корень**: тест не изолировал shared state (кэш токенов), а «просто ретрай» скрывал настоящий race.

#### Технические детали
- **Чек-лист поиска корня**:
  - Собрать полные логи теста с точностью до миллисекунды.
  - Проверить CPU/Memory/IO метрики на момент сбоя.
  - Запустить тест 100 раз в отдельном контейнере (без параллельных тестов).
  - Использовать `curl -w "%{time_total}"` для замера времени эндпоинта в изоляции.
- **Команды**:
  ```bash
  # Замерить latency конечной точки
  for i in $(seq 1 100); do curl -o /dev/null -s -w "%{time_total}\n" http://test-endpoint; done | sort
  ```
  - Включить в CI обязательный шаг: `--flaky-test-attempts=1` (без ретраев) и прогон 3 раза с разным seed параллельности.

### Вопрос 2: Если сервис при каждом GET создаёт новый idempotency_key, то тест на идемпотентность PUT проходит, а в продакшне — дубли. Как поймать в CI до релиза?

#### Links & Sources
- [Idempotence - Wikipedia](https://en.m.wikipedia.org/wiki/Idempotence) — общее определение и примеры.
- [Idempotence Explained with Real-World Examples (freeCodeCamp)](https://www.freecodecamp.org/news/idempotence-explained) — показаны кнопки лифта как идемпотентные операции.
- [What is Idempotency? (Google Cloud)](https://cloud.google.com/discover/idempotency) — архитектурный гайд по идемпотентности API.

#### Баги и реальные кейсы
- **Кейс**: команда написала тест на PUT /orders — каждый запуск генерировал случайный `idempotency_key` через `UUID.randomUUID()`. Тест проверял только код ответа, но idempotency не была реализована по-настоящему: второй PUT с тем же телом создавал дубль заказа.
- **Причина**: тест не использовал фиксированный ключ, поэтому не проверял, что повторный запрос с тем же ключом возвращает результат первого применения. В проде клиент посылал одинаковый ключ для ретраев — получал дубли.

#### Технические детали
- **Как поймать в CI**:
  - В тесте на идемпотентность:
    1. Отправить PUT с фиксированным `idempotency_key` (например, `"test-idemp-key-1"`).
    2. Проверить, что ответ — 200 (или 201 при первом).
    3. Отправить **повторный** PUT с тем же `idempotency_key` и тем же телом.
    4. Проверить, что статус и тело ответа идентичны (body должен содержать `order_id`, который не меняется). **Ошибка** — если возвращается 201 (создание нового заказа) или меняется `order_id`.
  - Дополнительно: проверить, что количество сущностей в БД не увеличилось.
- **Curl-пример**:
  ```bash
  # Этап 1: первый запрос
  curl -X PUT "https://api.example.com/orders" \
       -H "Idempotency-Key: fixed-key-123" \
       -H "Content-Type: application/json" \
       -d '{"item":"test"}' -w "\n%{http_code}"
  # Этап 2: повторный запрос — должен вернуть те же самый ответ
  ```
- **Ошибка проектирования**: генерация `idempotency_key` должна быть **на стороне клиента**, а не на сервере при каждом GET. В CI тест обязан имитировать поведение реального клиента: ключ задаётся один раз и повторяется при ретрае.

### Идеи для постов
- **Почему `retry` в тестах — это зло**. История банка с таймаутом 10ms: ретраи скрывают реальный баг, а в проде он вылезает с утроенной силой. Угол: «Не ретрайте flaky-тесты, а убивайте их в корне».
- **Тест на идемпотентность, который пропустил дубли**. Разбор кейса с `idempotency_key`: как неправильно написанный тест создаёт ложную уверенность. Практический рецепт: фиксированный ключ, проверка статуса и тела, проверка количества записей в БД.
- **Дискуссия: AI может написать такой тест?** Мнение Эдди — AI-генерация тестов часто создаёт иллюзию качества. Контраргумент: AI, обученный на реальных кейсах, не спроектирует тест с рандомным ключом. Но пока без человека — дыра. (Учитываем скепсис Эдди к AI, но не соглашаемся — аргументированные примеры, где AI не справился бы.)
