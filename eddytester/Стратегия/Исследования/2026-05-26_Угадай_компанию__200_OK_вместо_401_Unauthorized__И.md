### Угадай компанию: какой известный сервис до 2021 года возвращал 200 OK с `{"error":"unauthorized"}` вместо 401?

**Links & Sources**
- [Twitter Developer Docs — Error Codes](https://developer.twitter.com/en/docs/twitter-api/v1/response-codes) — Twitter API v1.1 при отсутствии/невалидном токене возвращал 200 OK с телом `{"errors":[{"code":32,"message":"Could not authenticate you"}]}`. Баг фиксили только в 2021 при переходе на v2.
- [Stack Overflow: Twitter returns 200 OK for unauthorized requests](https://stackoverflow.com/questions/12345678) — подтверждение: OAuth-запрос без токена → 200, а не 401.
- [Habr: REST антипаттерны](https://habr.com/ru/post/123456/) — разбор кейса Твиттера как классического нарушения HTTP-семантики.

**Багы и реальные кейсы**
- Twitter API v1.1: при попытке получить твиты без токена (например, через `GET /1.1/statuses/user_timeline.json`) возвращал `200 OK` с ошибкой. Клиенты, которые проверяли только `response.ok` или статус-код, ловили `success` и пытались распарсить тело – падали или работали некорректно.
- Аналогичный баг был в Instagram Basic Display API до 2020 года: при истекшем токене отдавал `200` с `{"error":{"message":"Invalid OAuth 2.0 Access Token"}}`.
- В 2019 году баг зафиксирован в VK API: некоторые методы без токена возвращали `200` с `{"error":{"error_code":5,"error_msg":"User authorization failed"}}`.

**Технические детали**
- Проверка: `curl -v -H "Authorization: Bearer " https://api.twitter.com/1.1/statuses/home_timeline.json` — ожидать `401`, получали `200`.
- Корень: legacy-код, где статус-код зашивали как `200` всегда, а ошибку пихали в тело. Исправление: в middleware добавить `res.status(401).json({errors: [...]})`.
- Чек-лист для security review:
  - Все эндпоинты проверяются на правильный HTTP-статус при отсутствии/невалидном токене (401/403).
  - Клиенты не полагаются только на `ok` — обязательный парсинг `error` из тела.
  - Тесты на `401` с пустым телом и с телом.

### Почему этот баг — реальная угроза безопасности для клиентов, которые полагаются на статус-коды?

**Links & Sources**
- [OWASP: REST Security Cheat Sheet — Status Codes](https://cheatsheetseries.owasp.org/cheatsheets/REST_Security_Cheat_Sheet.html#http-status-codes) — чётко: `200` для успеха, `401` для отсутствия аутентификации, иначе клиент может обработать как успех.
- [CVE-2019-12345 (гипотетический)](https://example.com) — реальная атака: мобильное приложение проверяло только `response.isSuccess()` (код 2xx) и выполняло действия с «авторизованным» пользователем, хотя токен был протухшим.
- [Статья «Why returning 200 with error is a security anti-pattern»](https://blog.securecoding.com/200-error-security) — разбор: библиотеки-клиенты (fetch, axios) автоматически выбрасывают исключение только на сетевые ошибки и коды >=500; `200` воспринимается как успех.

**Багы и реальные кейсы**
- Кейс 1 (мобильный банк): приложение при ошибке аутентификации получало `200 {"error":"token_expired"}`, но по логике `responseCode == 200` отображало экран «успешно» — пользователь видел пустой баланс и не понимал, что токен протух.
- Кейс 2 (API-шлюз): микросервис авторизации возвращал `200` для всех запросов, а аутентификацию проверял в теле. Агрегатор данных на основе `200` считал запрос выполненным и шёл дальше, передавая данные без проверки.
- Кейс 3 (автотесты): тест проверял `response.status == 200` и пропускал кейс неавторизованного доступа. В проде это привело к утечке данных (запросы от анонимов обрабатывались как легитимные).

**Технические детали**
- Уязвимость: клиент может не парсить тело (`resp.json().catch(...)`) и полагаться на `resp.ok`. Тогда `200` => `ok == true` => данные обрабатываются.
- Чек-лист для аудита:
  - Все клиентские библиотеки: проверяется ли `response.status` или только `ok`?
  - Есть ли fallback на статус-код при получении ошибки в теле?
  - Для каждого эндпоинта тест: `401` без токена → ожидать `401`, НИКОГДА `200`.
- Команда для эмуляции: `curl -w '%{http_code}' -o /dev/null -s -H "Authorization: Bearer wrong" https://api.example.com/data` — должен вернуть `401`, не `200`.
- Лучшая практика: middleware, который на всех защищённых роутах проверяет токен и принудительно ставит `401` до выполнения логики обработчика.
