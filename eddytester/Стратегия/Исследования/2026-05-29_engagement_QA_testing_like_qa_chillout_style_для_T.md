### Если баг с двойным списанием в Stripe случился из-за того, что PUT не был идемпотентным — как ты проверишь, что твой платёжный эндпоинт выдержит повторный запрос от клиента с плохим интернетом?

**Links & Sources**
- [Stripe: Idempotent Requests](https://stripe.com/docs/api/idempotent_requests) — Stripe требует передавать `Idempotency-Key` в POST/PUT, иначе при повторном запросе может создаться новый charge.
- [REST API Idempotency: What, Why, How](https://restfulapi.net/idempotent-rest-apis/) — PUT по определению идемпотентен, но на практике без ключа при повторной отправке с теми же данными возможна двойная запись.

**Багы и реальные кейсы**
- Финтех-стартап: клиент с нестабильным интернетом дважды отправил PUT на `payment_intents/pi_123` без idempotency_key. Сервер создал два отдельных charge. Пользователь увидел двойное списание. Баг нашли только после падения метрик списаний. Фикс — генерация уникального ключа на клиенте и проверка на стороне Stripe.
- Uber (2017): при повторной попытке оплаты из-за таймаута система не проверяла уникальность запроса — списала деньги дважды.

**Технические детали**
- Симуляция повторного запроса (через curl с одним `Idempotency-Key`):
  ```bash
  curl -X PUT https://api.stripe.com/v1/payment_intents/pi_123 \
    -H "Idempotency-Key: test_key_001" \
    -H "Authorization: Bearer sk_test_..." \
    -d "amount=1000"
  ```
  Выполнить дважды — второй раз должен вернуть тот же объект без нового charge.
- Проверка без idempotency_key: повторный PUT с теми же данными — ожидать либо ошибку 409, либо новый объект (зависит от реализации).
- Чек-лист:
  1. Эндпоинт принимает заголовок `Idempotency-Key`.
  2. Повтор с тем же ключом → idempotent response.
  3. Повтор с другим ключом → дубль не создаётся (если не предусмотрено иначе).
  4. Эмуляция плохого соединения: прервать запрос (ctrl+C в curl), затем retry с тем же ключом.

### В идеале кэш должен работать как швейцарские часы, но на практике CDN отдаёт старый JSON после обновления цены — как написать тест, который поймает такое до релиза?

**Links & Sources**
- [AWS CloudFront: Invalidating Files](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/Invalidation.html) — для принудительной инвалидации кэша нужно отправить запрос на инвалидацию по пути, иначе CDN будет отдавать старый контент до истечения TTL.
- [5 Common CDN Caching Mistakes](https://www.keycdn.com/blog/cdn-caching-mistakes) — ошибка №1: слишком большой `max-age`, из-за которого новые версии не доходят до пользователей.

**Багы и реальные кейсы**
- Крупный e‑commerce: после обновления цены на сервере CloudFront не инвалидировали кэш, TTL был 24 часа. Пользователи видели старую цену целый день. Баг нашли по жалобам — автоматический тест на актуальность данных отсутствовал.
- API новостного агрегатора: CDN (Fastly) кэшировал JSON с заголовком `Cache-Control: public, max-age=3600`. При каждом обновлении статьи в течение часа выдавался старый контент, хотя на origin данные менялись.

**Технические детали**
- Проверка заголовков кэширования:
  ```bash
  curl -I https://cdn.example.com/api/price
  ```
  Искать `Cache-Control`, `Expires`, `ETag`, `Age`.
- Сравнение ответов через CDN и напрямую к origin (минуя CDN через host header или IP):
  ```bash
  # запрос к origin напрямую
  curl -H "Host: api.example.com" http://origin-server/api/price
  # запрос через CDN
  curl https://cdn.example.com/api/price
  ```
- Тест на инвалидацию:
  1. Обновить цену на origin (например, через PUT на `/admin/price`).
  2. Дождаться выполнения инвалидации (если используется ручная инвалидация — выполнить её через AWS CLI: `aws cloudfront create-invalidation --distribution-id ... --paths /api/price`).
  3. Сделать запрос через CDN — тело ответа должно содержать новую цену.
- Чек-лист:
  1. Cache-Control не содержит `no-cache` или `must-revalidate` → проверить.
  2. После обновления данных TTL не превышает разумное время (для цен — 0 или минимальное).
  3. Инвалидация срабатывает принудительно — тест должен вызывать её и проверять результат.
  4. Разные географические регионы (если CDN распределённый) не отдают старый кэш.
