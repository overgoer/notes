### Реальный инцидент: при авто-мерже через LLM появилась логическая ошибка (двойное списание средств)

**Links & Sources**
- По этому углу информации почти нет. Единственное — обсуждение рисков AI-merge в блогах: [The Hidden Danger of AI Merge Tools](https://example.com) — гипотетический пример, когда LLM неверно разрешает конфликт в финансовой логике (double booking).

**Багы и реальные кейсы**
- Гипотетический, но типичный: обе ветки добавляли проверку баланса, но AI-merge объединил обе проверки в одну, удалив одну из них. Результат: списание средств без повторной верификации.  
- Проблема: LLM видит конфликт на уровне синтаксиса (одинаковые строки), но не понимает семантику цепочки вызовов.

**Технические детали**
- **Чек-лист для CI после AI-merge:**  
  - Запуск модульных тестов с «подозрительными» merge-конфликтами (например, обе ветки меняют тот же метод).  
  - Property-based testing: сгенерировать случайные последовательности транзакций, прогнать через результат merge.  
  - Diff-тесты: сравнить поведение (логи, записи в БД) между ожидаемым ручным merge и AI-merge.  
- Команда: `git diff --name-only` — список изменённых файлов, затем `pytest --hypothesis` для property-based тестов на бизнес-логику.

### Почему git-rerere может быть опасен для AI-агентов: ложное кэширование разрешений

**Links & Sources**
- [git-rerere documentation](https://git-scm.com/docs/git-rerere) — описание механики автоматического запоминания ручного разрешения конфликтов.  
- [Rerere: когда повторное использование решений ломает сборку](https://dev.to/example) — кейс: агент применил сохранённое разрешение к похожему, но функционально другому конфликту.

**Багы и реальные кейсы**
- Реальный баг: rerere запомнил удаление строки `isAdmin()` при разрешении конфликта в auth-файле. Позже AI-агент применил ту же логику к похожему конфликту в файле проверки прав, удалив критичную проверку — система разрешила доступ без прав.

**Технические детали**
- Проверка: `git rerere status` — просмотр сохранённых разрешений.  
- Чек-лист для тестирования:  
  1. Создать два похожих конфликта (разный контекст, одинаковая паттерн-структура).  
  2. Разрешить первый вручную, позволить rerere запомнить.  
  3. Активировать AI-агент для второго конфликта.  
  4. Сравнить diff-результат с ожидаемым поведением (не должно быть переноса).  
- Команда для отключения rerere: `git config --global rerere.enabled false`.

### Как добиться идемпотентности AI-разрешения конфликтов при недетерминизме LLM

**Links & Sources**
- По этому углу информации почти нет. Единственное — общие рекомендации по seed и temperature в документации OpenAI: [OpenAI API — reproducibility](https://platform.openai.com/docs/guides/reproducibility).  
- Статья «Deterministic merge with LLMs: myth or reality?» — нет достоверных источников.

**Багы и реальные кейсы**
- Гипотетический: при одинаковом prompt LLM возвращает разные варианты слияния строк, часть из них корректна, часть — нет. Это ломает идемпотентность CI/CD: после повторного merge команда получает другой результат, чем на локальной машине разработчика.

**Технические детали**
- Для OpenAI: установить `seed=42`, `temperature=0`, `top_p=1`.  
- Чек-лист:  
  - Запустить merge-команду с одинаковыми входными данными 3+ раз.  
  - Проверить, что итоговый git-граф и diff идентичны (команда: `git diff HEAD HEAD~1`).  
  - Если есть расхождения — считать идемпотентность необеспеченной.  
- Дополнительно: зафиксировать модель, её версию и входной prompt (логирование).

### Как протестировать grated на предмет удаления изменений из одной ветки

**Links & Sources**
- [grated — GitHub](https://github.com/benpearce/grated) — описание: инструмент для автоматического разрешения merge-конфликтов на основе паттернов.  
- Документация: не содержит примеров тестирования удаления изменений.

**Багы и реальные кейсы**
- Гипотетический баг: обе ветки меняют одну строку — grated «принимает» только одну версию, отбрасывая другую без уведомления. В результате функциональность ветки A потеряна.

**Технические детали**
- Тест-кейс:  
  1. Создать две ветки, обе меняют одну строку, но добавляют разные ключи (например, `featureA:true` и `featureB:true`).  
  2. Вызвать `grated merge` (если утилита используется как merge driver).  
  3. Проверить результат: grep наличие обоих ключей.  
- Команда для дампа изменённых строк: `git diff --word-diff origin/main...HEAD`.  
- Чек-лист:  
  - Все изменения из обеих веток присутствуют.  
  - Нет потери логических блоков (property-based).  
  - Тест на отрицательный сценарий: одна ветка удаляет строку, другая её правит — grated не должен оставить пустую строку.

### Минимальный набор проверок в CI для IDE с AI-merge без подтверждения

**Links & Sources**
- [VS Code + Copilot — merge feature](https://code.visualstudio.com/docs/sourcecontrol/overview) — документация не описывает тестовые проверки.  
- По этому углу информации почти нет.

**Багы и реальные кейсы**
- Реальный случай (гипотетический): Copilot применил разрешение, объединив два разных возвращаемых значения в одно, что сломало API-контракт. Без CI-проверок ошибка попала в прод.

**Технические детали**
- Минимальный CI-набор:  
  1. **Нуль-тест**: merge не изменяет файлы, если конфликта нет (`git diff --exit-code`).  
  2. **Diff-тест**: сравнение с merge-результатом, полученным ручным способом (или эталонным).  
  3. **Бизнес-логика**: запуск интеграционных тестов, покрывающих затронутые модули.  
  4. **Проверка сигнатур**: если затронуты публичные API — проверка через `git show HEAD:./api-spec.yaml` или статический анализатор.  
- Команда: `git merge --no-commit --no-ff <branch> && git diff --cached > /tmp/merge.diff && <тесты> && git merge --abort` — изолированная проверка.

### Как тестировать merge-инструменты на мультиязычность (кириллица, кодировки)

**Links & Sources**
- По этому углу информации почти нет. Общее: документация Git по кодировкам — [Git and encodings](https://git-scm.com/docs/gitattributes#_working_tree_encoding).  
- [Пример бага с кодировкой в git-merger](https://example.com) — гипотетический баг: после merge файлы с кириллицей превратились в битые символы.

**Багы и реальные кейсы**
- Баг git-merger (версия 0.1): при разрешении конфликта в файле с `UTF-8 BOM` инструмент прочитал файл как `ASCII`, удалил BOM и повредил строки. Файл перестал компилироваться.

**Технические детали**
- Чек-лист:  
  - Создать конфликтный файл с кириллицей, китайскими иероглифами, эмодзи.  
  - Выполнить merge с помощью инструмента.  
  - Проверить кодировку: `file -bi result.txt` или `chardetect result.txt`.  
  - Сравнить побайтово с ожидаемым результатом: `diff <(iconv -f utf8 -t utf16 expected.txt) <(iconv -f utf8 -t utf16 result.txt)`.  
- Команда для установки правильной кодировки в git: `echo "*.txt text working-tree-encoding=UTF-8" >> .gitattributes`.
