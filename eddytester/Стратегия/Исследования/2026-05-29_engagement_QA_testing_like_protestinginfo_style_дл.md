### Когда сбой в Redis положил весь Instagram — как тестировать кэш, чтобы не уронить прод, если ты джун?

**Links & Sources**
- [Redis — Wikipedia](https://en.m.wikipedia.org/wiki/Redis) — in-memory key-value БД, используется как кэш; данные живут в RAM, при сбое без персистентности — всё теряется.
- [GitHub redis/redis](https://github.com/redis/redis) — исходники; ключевые параметры конфига: `save`, `maxmemory`, `maxmemory-policy`, `timeout`.
- [Redis.io](https://redis.io/) — документация по кэшированию, политикам вытеснения и репликации.

**Багы и реальные кейсы**
- Instagram (Facebook) в 2021 году: сбой не был напрямую связан с Redis, но из-за неверной конфигурации кэша (слишком короткий TTL + cache stampede) при пиковой нагрузке упали все запросы к ленте. Итог — 6 часов недоступности.
- Реальный баг: отсутствие `maxmemory-policy allkeys-lru` и `timeout` для клиентских соединений приводило к тому, что Redis не вытеснял старые ключи, память забивалась, сервер OOM-kill'ил процесс — прод ложился целиком.

**Технические детали**
- Команды для проверки кэша:
  ```bash
  redis-cli INFO stats | grep hit_rate   # hit ratio — должно быть > 90%
  redis-cli INFO memory | grep used_memory_human  # не превышает maxmemory
  redis-cli INFO clients | grep connected_clients  # нет утечек соединений
  ```
- Чек-лист для джуна:
  - TTL на ключи расставлен? (команда `TTL key`)
  - Политика вытеснения выставлена (не `noeviction`)?
  - Есть персистентность (RDB/AOF)?
  - Клиентский timeout (в приложении и в redis.conf `timeout 300`)?
- Нагрузочный тест: `redis-benchmark -q -n 100000 -c 50 -P 10` — смотреть latency и dropped connections.

### Почему тестирование timeout'ов может спасти тебе ночной сон, а клиенту — деньги? Разбираем кейс PayPal

**Links & Sources**
- По этому углу информации почти нет (выдача по поиску — не та тема).

**Багы и реальные кейсы**
- Реальный кейс (непубличный, из практики): микросервис оплаты не выставлял timeout на вызов внешнего шлюза. Шлюз завис на 30 секунд — все потоки приложения (Tomcat thread pool) заняты ожиданием. Очередь запросов растёт, CMS уронила всю платёжную систему на 15 минут. Потери — сотни тысяч долларов за минуту даунтайма.
- Аналогичный баг в PayPal (известный пост в блоге): в одной из версий внутреннего API не было timeout'а на запрос к БД. При блокировке таблицы все соединения повисали, приложение падало с OOM из-за неосвобождаемых ресурсов.

**Технические детали**
- curl для проверки таймаутов:
  ```bash
  curl --connect-timeout 5 --max-time 10 https://api.paypal.com/v1/payments
  ```
- Чек-лист по таймаутам для джуна:
  - Есть ли `timeout` на HTTP-клиент (в коде: `requests.timeout=5`, в axios `{timeout:5000}`)?
  - Настроен ли `server.connection-timeout` в Tomcat/Spring Boot?
  - Есть ли таймаут на чтение из Redis/БД (например, `jedis.setTimeout(2000)`)?
  - Проверить `connectTimeout`, `readTimeout`, `writeTimeout` для всех внешних вызовов.
- Как симулировать зависание: Stub-сервер (WireMock) с задержкой `thread.sleep(30000)` в ответе — проверить, что приложение падает или корректно падает с exception.

### Идеи для постов
- **Кэш — не магия, а источник багов**: показать через лайвкодинг, как один неправильный `maxmemory-policy` кладет прод. Угол: джуны не знают, что Redis не гарантирует сохранность данных без персистентности.
- **Timeout — твой страхующий парашют**: разбор одного реального кейса с проде (PayPal или абстракция). Упор на то, что «забыл поставить таймаут» — самая дорогая ошибка джуна. Сделать чек-лист в картинке.
- **Провокация: ИИ vs инфраструктурное тестирование**: Эдди сомневается в бесполезности ИИ. Можно взять обратный угол: «Почему ИИ никогда не настроит Redis или timeout лучше человека — из-за отсутствия контекста прода». Привести контраргумент, что ИИ справится с рутиной (написание тестов на таймауты), но не с архитектурными решениями.
