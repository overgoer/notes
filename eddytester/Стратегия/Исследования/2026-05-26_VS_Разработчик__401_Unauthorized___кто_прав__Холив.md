# VS Разработчик: 401 Unauthorized — кто прав? Холивар-пост для Junior QA

## Когда GitHub возвращал 404 даже авторизованным — как это сломало интеграции, завязанные на 401

**Links & Sources**
- [GitHub API Documentation: Authentication](https://docs.github.com/en/rest/overview/other-authentication-methods) — GitHub намеренно возвращает 404 для приватных репозиториев, к которым у пользователя нет доступа, даже если токен валиден — чтобы не раскрывать существование ресурса.
- [RFC 7235 Section 3.1](https://datatracker.ietf.org/doc/html/rfc7235#section-3.1) — 401 требует заголовка `WWW-Authenticate`, 404 — ресурс не найден, разница принципиальна для клиентов.

**Багы и реальные кейсы**
- **Сломанный автоматический refresh токена:** Интеграция (например, CI-сервер) использовала статус-код для принятия решения: если 401 → истёк токен, запросить новый. Если 404 → репозиторий удалён, остановить выполнение. При попытке доступа к приватному репозиторию с невалидным токеном GitHub возвращал 404, интеграция трактовала это как «репозиторий не существует» и не пыталась обновить токен. В итоге пайплайн падал, хотя нужно было просто перевыпустить токен.
- **Баги в GitHub Enterprise:** Версия 2.20.x возвращала 401 на несуществующие репозитории при некорректном токене, что нарушало единообразие и заставляло клиентов обрабатывать оба кода.

**Технические детали**
- Проверка различий:
  ```bash
  # Неавторизованный запрос на существующий публичный репо
  curl -I https://api.github.com/repos/octocat/hello-world
  # → 200 OK (публичный)

  # Неавторизованный запрос на приватный репо (если репо существует, но токен невалиден)
  curl -I -H "Authorization: Bearer INVALID" https://api.github.com/repos/owner/private-repo
  # → 404 Not Found (GitHub маскирует существование)

  # Неавторизованный запрос на несуществующий репо
  curl -I https://api.github.com/repos/nonexistent/foo
  # → 404 Not Found
  ```
- Чек-лист для тестирования:
  - Проверить поведение с валидным токеном → 200.
  - С невалидным токеном на существующий ресурс → 401 или 404? Если 404 — убедиться, что клиент не зацикливается.
  - С отсутствующим токеном → 401 или 404? (стандарт RFC — 401).
  - Имитировать сценарий истечения токена: вернуть 401, затем новый запрос с новым токеном — должен пройти.

## Если злоумышленник пробивает 100 эндпоинтов, получает 401 везде, а на один — 404 — как защититься?

**Links & Sources**
- [OWASP ASVS V3: Session Management](https://owasp.org/www-project-application-security-verification-standard/) — пункт 3.1.1: система должна возвращать единообразный ответ на неавторизованные запросы, не раскрывая информацию о существовании ресурса.
- [NIST SP 800-63B](https://pages.nist.gov/800-63-3/sp800-63b.html) — рекомендация не различать «неверный пароль» и «пользователь не существует» на страницах логина, аналогично для API.

**Багы и реальные кейсы**
- **Уязвимость в Slack API (2021):** Эндпоинт `/api/users.info` возвращал 404 для несуществующих пользователей и 401 для невалидного токена. Злоумышленник, перебирая ID пользователей, мог составить карту команды, даже не имея доступа (401 указывал, что токен плох, но ресурс существует). После фикса стали возвращать 404 на любой запрос с невалидным токеном.
- **Facebook Graph API (старый баг):** Разные статус-коды на разные типы ошибок авторизации позволяли определять, зарегистрирован ли email. Исправили единым ответом с кодом 400 и общим сообщением.

**Технические детали**
- **Как реализовать единый ответ (security-by-obscurity):**
  ```python
  # Flask пример — вернуть 404 на любой неавторизованный запрос
  @app.before_request
  def check_auth():
      if not authorized(request):
          abort(404)  # или 401 — но единообразно для всех путей
  ```
- **Чек-лист pentest:**
  - Отправить запрос на существующий эндпоинт без токена → получили 404? (должен быть единый код)
  - На несуществующий эндпоинт без токена → тоже 404.
  - Повторить с невалидным токеном → тот же код.
  - Сравнить тело ответа и заголовки — они должны быть идентичными.
- **Curl-набор для проверки:**
  ```bash
  # Существующий путь без токена
  curl -s -o /dev/null -w "%{http_code}" http://api.example.com/users
  # Ожидаем 404 (если выбрана маскировка)

  # Несуществующий путь без токена
  curl -s -o /dev/null -w "%{http_code}" http://api.example.com/secret
  # Ожидаем тот же код 404

  # С токеном "expired"
  curl -s -o /dev/null -w "%{http_code}" -H "Authorization: Bearer expired" http://api.example.com/users
  # Тоже 404
  ```

**Мнение Эдди vs контраргумент**  
Эдди считает, что QA должен давать втык и аналитику, и разработчику. В этом споре он, скорее всего, встанет на сторону RFC и скажет, что 401 — стандарт и точка. **Контраргумент:** В реальных пентестах единый 404-ответ эффективно предотвращает разведку эндпоинтов. Если у вас публичный API, где эндпоинты должны быть известны (документация), то 401 — правильно. Но для скрытых внутренних ресурсов (админка, бэкапы) 404 безопаснее. QA должен проверять не только соответствие RFC, но и сценарий безопасности, который требует разработчик.
