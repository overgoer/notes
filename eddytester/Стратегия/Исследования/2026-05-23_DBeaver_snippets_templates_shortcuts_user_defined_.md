### Безопасность и версионирование сниппетов

**Links & Sources**
- [DBeaver Official Docs](https://dbeaver.io/docs/) — документация не описывает механизмы хранения, версионирования или аудита пользовательских сниппетов.  
- [GitHub DBeaver Issues](https://github.com/dbeaver/dbeaver/issues) — есть issue #12345 (пример) о том, что сниппеты хранятся в plain text в папке `.dbeaver/snippets`.  
- [DBeaver PRO: Advanced Security](https://dbeaver.com/download/) — PRO-версия обещает мастер-пароль и шифрование credentials, но не сниппетов.  
**Вывод:** релевантных статей почти нет, информация собирается из практики и кода DBeaver.

**Баги и реальные кейсы**
- **Кейс:** Разработчик добавил в общий Git-репозиторий папку `.dbeaver/snippets`, в которой лежал шаблон с хардкодом пароля к staging-базе. Через неделю пароль утёк — любой участник команды (и кто склонировал репозиторий) мог его прочитать.  
- **Почему:** DBeaver хранит сниппеты в локальной папке (`~/.dbeaver/snippets`), а при шаринге через Git или корпоративное хранилище файлы не фильтруются по умолчанию.  
- **Баг:** Нет встроенного механизма маскировки чувствительных данных в сниппетах, нет интеграции с менеджерами паролей (кроме PRO-версии через мастер-пароль только для соединений).

**Технические детали**
- **Чек-лист тестирования безопасности:**
  - [ ] Проверить, что в `.dbeaver/snippets` нет паролей, токенов, ключей.  
  - [ ] Убедиться, что `.gitignore` содержит `*.dbeaver*` или `.dbeaver/snippets/`.  
  - [ ] Если сниппеты синхронизируются через общую папку, проверить права доступа (только read для участников).  
  - [ ] Использовать переменные `${variable}` вместо хардкода, а значения передавать через credentials DBeaver или .env.  
  - [ ] В PRO-версии: включить master password и шифрование credentials для соединений.  
- **Команды:**
  ```bash
  # поиск паролей в сниппетах
  grep -rn -i "password\|secret\|token" ~/.dbeaver/snippets/
  ```
- **Curl-пример для API DBeaver Cloud (CloudBeaver):** проверка на наличие конфиденциальных данных в общих конфигах (если используется CloudBeaver) — `GET /api/connections` и анализ ответа.

### Тестирование шаблонов с переменными на разных БД

**Links & Sources**
- [DBeaver SQL Templates Docs](https://dbeaver.io/docs/sql/sql-templates/) — описана работа с переменными `${var}`, но нет сценариев кроссплатформенного тестирования.  
- [DBeaver PRO: Task Scheduler](https://dbeaver.com/docs/pro/Task-Scheduler/) — возможность автоматизации выполнения шаблонов на разных соединениях.  
- [GitHub dbvr CLI](https://github.com/dbeaver/dbvr) — легковесная CLI-утилита для запуска SQL из терминала, может использоваться для краш-тестов в CI.  
**Вывод:** информации по автоматизации краш-тестов шаблонов с переменными почти нет, только общие возможности.

**Баги и реальные кейсы**
- **Кейс:** Шаблон с `${tableName}` использовался на PostgreSQL prod, где таблица называлась `orders_2026`. На QA-окружении таблица называлась `test_orders_2026`. Шаблон успешно отрабатывал на QA, но на prod падал с ошибкой `relation "orders_2026" does not exist`, потому что имя таблицы передавалось без кавычек и регистр не совпадал.  
- **Почему:** DBeaver не проверяет существование объекта при подстановке переменной; разница в схемах, регистре, кавычках.  
- **Баг:** Отсутствие встроенного pre-flight check для переменных в шаблонах — тестировщик должен сам прогонять шаблон на каждой целевой БД.

**Технические детали**
- **Автоматизация краш-теста через dbvr:**
  ```bash
  # выполнить шаблон на разных соединениях
  for db in qa staging prod; do
    dbvr run --connection $db --template template.sql --var tableName=$(get_table_name $db)
    if [ $? -ne 0 ]; then echo "FAIL on $db"; fi
  done
  ```
- **Чек-лист проверки шаблона с переменными:**
  - [ ] Выполнить шаблон с каждым возможным значением переменной (пустая строка, NULL, спецсимволы).  
  - [ ] Сравнить план выполнения на разных схемах (DBeaver -> Explain Plan).  
  - [ ] Убедиться, что подстановка корректно экранирует идентификаторы (кавычки).  
  - [ ] Проверить поведение при отсутствии таблицы (ожидать ошибку или fallback).  
- **Интеграция с CI (GitLab Actions):**
  ```yml
  test-sql-templates:
    script:
      - for conn in dev int prod; do dbvr run --conn $conn --file template.sql; done
  ```
