### Когда в GitHub API из-за отсутствия корректных заголовков Cache-Control пользователи получали устаревшие данные о статусе билдов — как QA проверить, что кэш не маскирует реальные ошибки на бэкенде?

**Links & Sources**
- [cli/cli issue #7482](https://github.com/cli/cli/issues/7482) — пользователь заметил, что встроенный кэш `gh api --cache` не документирован, а при повторных запросах возвращает закэшированный ответ даже после изменения данных на сервере (требуется принудительная инвалидация).
- [Next.js issue #27129](https://github.com/vercel/next.js/issues/27129) — сборка в GitHub Actions использует кэш `.next/cache`, ключ которого завязан на `package-lock.json`; без изменения зависимостей кэш никогда не обновляется, строчки остаются старыми — реальная ошибка бэкенда не воспроизводится 30% билдов.

**Багы и реальные кейсы**
- **GitHub Actions + Next.js**: сборка падала из-за ошибки в коде, но кэш возвращал старый успешный артефакт — деплой проходил с устаревшей версией. Причина — ключ кэша не учитывал изменения в исходниках.
- **GitHub CLI `--cache`**: тестировщик запускал `gh api repos/owner/repo/actions/runs` с `--cache 5m` и получал один и тот же список раннов, хотя на бэкенде запускались новые — кэш не инвалидировался по `Cache-Control` с сервера.

**Технические детали**
- **curl — проверка заголовков кэширования**:
  ```bash
  curl -I https://api.github.com/repos/owner/repo/actions/runs
  # Смотрим Cache-Control, ETag, Last-Modified
  ```
  Ожидаемые заголовки от GitHub: `Cache-Control: private, max-age=60, stale-while-revalidate=30`. Если их нет или max-age > разумного — кэш маскирует изменения.

- **Симуляция устаревшего кэша** (чтобы увидеть, вернётся ли свежий ответ после изменения данных):
  1. Получить ETag: `curl -sI ... | grep etag`
  2. Запрос с `If-None-Match`:
     ```bash
     curl -H "If-None-Match: \"старый-тег\"" https://api.github.com/...
     ```
     Должен вернуть 304 Not Modified. Если потом на сервере изменить данные, повторный запрос с тем же ETag должен вернуть 200 с новыми данными — иначе кэш не инвалидируется.

- **Чек-лист для QA**:
  - [ ] Проверить наличие `Cache-Control` и `Expires` на всех эндпоинтах, где ожидается актуальность.
  - [ ] Убедиться, что после POST/PUT/PATCH на ресурсе его GET-ответ меняется (проверить ETag).
  - [ ] Прогнать сценарий: получить данные, подождать TTL, изменить данные через другой клиент, повторить GET — ответ должен быть свежим.
  - [ ] Проверить, что заголовок `Vary` корректно учитывает авторизацию и accept-типы.

*Заметка по Эдди:* его скепсис к AI здесь не прям, но если AI-агент будет генерировать тесты, игнорирующие кэш-заголовки — баги уйдут в прод. Контраргумент: AI умеет анализировать ответы и подставлять ETag, если QA задаст правильные промпты.

### Почему тестирование API без учёта rate limiting часто превращается в «мой тест упал, потому что меня забанили», а не из-за бага?

**Links & Sources**
- [Postman: API Rate Limiting Guide](https://blog.postman.com/api-rate-limiting/) — подробное описание алгоритмов (sliding window, token bucket) и что делать при 429: проверять `Retry-After`, ждать.
- [Reddit: Discord API блокирует после превышения лимитов](https://www.reddit.com/r/discordapp/comments/jqosi0/) — пользователь получил временный бан из-за частых тестовых запросов, хотя считал себя единственным клиентом.
- [Facebook post: 429 ошибка в тестах](https://www.facebook.com/groups/1577315533418837/posts/1580730099744047/) — тестировщик жаловался на «падающие тесты», оказалось — rate limit сработал на локальной машине.

**Багы и реальные кейсы**
- **Discord API**: разработчик писал бота, запускал тесты локально — после 30 запросов за секунду IP попадал в бан на 10 минут. Тесты показывали 429, но бага в коде не было — проблема в отсутствии backoff.
- **Twitter API (исторически)**: тесты авто-регрессии падали в пятницу вечером из-за того, что к концу недели накапливалось превышение лимита по аккаунту. Команда QA не учитывала скользящее окно.

**Технические детали**
- **curl — симуляция rate limit**:
  ```bash
  for i in {1..100}; do curl -s -o /dev/null -w "%{http_code}\n" https://api.example.com/endpoint; done | sort | uniq -c
  ```
  Должен появиться 429. Проверить `Retry-After`:
  ```bash
  curl -v https://api.example.com/endpoint 2>&1 | grep -i retry-after
  ```

- **Тест с ожиданием (пример на Python с использованием time.sleep)**:
  ```python
  import requests, time
  for _ in range(5):
      r = requests.get("https://api.github.com/rate_limit")
      if r.status_code == 429:
          wait = int(r.headers.get("Retry-After", 60))
          time.sleep(wait)
  ```  
  Это симулирует правильное поведение клиента, а не «забань меня».

- **Чек-лист для QA**:
  - [ ] Выяснить лимиты: кол-во запросов на окно, алгоритм (фиксированное окно / скользящее окно / token bucket).
  - [ ] Написать тест, который превышает лимит в 2 раза и проверяет, что приходит 429 + headers (RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset или Retry-After).
  - [ ] После 429 дождаться сброса и проверить, что запрос снова работает (200).
  - [ ] Проверить, что лимит сбрасывается в пересчёт на разные аккаунты/токены (если есть аутентификация).
  - [ ] В автотестах: внедрить exponential backoff + отслеживание `X-RateLimit-Remaining` — если оно = 0, тест должен пропуститься (skip), а не упасть.

*Мнение Эдди про AI:* он считает, что тестировщики недооценивают замену AI. Здесь контраргумент: AI-агенты тоже упираются в rate limits — если тесты не проектируют учёт лимитов, AI-агент будет бесконечно долбить API и баниться. Хороший QA должен закладывать это в дизайн тестов, а не надеяться на AI.
