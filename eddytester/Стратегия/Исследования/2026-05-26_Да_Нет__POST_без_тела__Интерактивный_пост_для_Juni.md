### POST без тела: 200 OK или 400 Bad Request? Разбор для Junior QA

### Как Stripe однажды принял POST без тела — почему это сломало баланс

**Links & Sources**
- [Stripe Wikipedia](https://en.wikipedia.org/wiki/Stripe,_Inc.) — общая страница компании, но **конкретного кейса с пустым POST нет ни в одном найденном источнике**.
- [GitHub обсуждение: Stripe API idempotency](https://github.com/stripe/stripe-python/issues/123) (гипотетический пример) — если не проверять тело, идемпотентный ключ может создать дублирующий объект с дефолтными полями.

**Багы и реальные кейсы**
- По этому углу информации почти нет. В открытых источниках (включая Stripe Changelog и блоги) не задокументировано, что Stripe создавал пустой платёжный объект на пустой POST. Вероятно, это внутренний баг / выдуманный кейс для обучения.
- **Как смоделировать**: если сервер не валидирует наличие обязательных полей (`amount`, `currency`, `source`), пустой POST (или POST с `{}`) может создать объект с `amount=0`. Это приведёт к списанию 0 валюты, но зачислению комиссии или созданию `balance transaction` с нулём, что ломает баланс при агрегации.

**Технические детали**
- Проверить: `curl -X POST https://api.stripe.com/v1/charges -u sk_test_... -d ""`
- Чек-лист:
  - POST без тела → должен быть `400 Bad Request` + `{"error": {"type": "invalid_request_error", "message": "Missing required param: amount"}}`
  - Если вернул `200` или `201` → проверить объект на наличие `amount=0` и `status`
  - Создать две транзакции без тела с одинаковым `Idempotency-Key` — если вторая вернёт тот же объект (с пустыми полями) — баг идемпотентности

### Сценарии, которые ломает 200 OK на POST без тела

**Links & Sources**
- По этому углу информации почти нет. Ссылки из результатов поиска (Pakistan Post, New York Post) не относятся к теме API тестирования.
- [REST API Tutorial — POST Method](https://restfulapi.net/http-methods-post/) — RFC 7231: POST должен создавать ресурс на основе тела запроса. Пустое тело ≠ валидный ресурс.

**Багы и реальные кейсы**
- **Создание "пустышек"**: POST `/api/users` без тела → `200 OK`. В базе появляется запись `users` с `{name: null, email: null}`. При попытке вывести список пользователей — null поля падают в UI или ломают сортировку.
- **Дубли при повторном запросе**: если нет идемпотентности, повторный пустой POST создаёт второго "null-юзера". С точки зрения бизнеса — мусорные записи без возможности их различать.
- **Нарушение бизнес-логики**: триггеры (например, "после создания юзера отправить приветственное письмо") срабатывают на пустые данные → письмо на null email падает с ошибкой, но пользователь уже считается созданным.

**Технические детали**
- Проверить:
  ```bash
  curl -X POST https://api.example.com/api/users -H "Content-Type: application/json" -d ""
  # vs
  curl -X POST https://api.example.com/api/users -H "Content-Type: application/json" -d "{}"
  ```
- Ожидание: `400 Bad Request` с описанием "empty body" или "missing field: name, email".
- Если сервер вернул `200`:
  - `GET /api/users/{id}` — должен быть null во всех обязательных полях
  - Проверить, что `user.id` не совпадает при повторном POST (если нет идемпотентности)
  - Проверить логи: должны быть `WARN: создан пользователь с пустыми полями`

- **Практика "поймай баг Stripe"**: даём curl и скриншот ответа — юзеры голосуют "400" или "200". Правильный ответ: 400 Bad Request, но с разбором что будет, если ошибиться (дубли, баланс). Хорошо заходит в канале про QA.
