# Research Brief: 500 Internal Server Error vs 200 OK с пустым телом

## Вопрос 1: Как проверить, что кэш CDN не отравится пустыми 200?

### Links & Sources
- [MDN: HTTP caching](https://developer.mozilla.org/en-US/docs/Web/HTTP/Caching) — правила кэширования: заголовки `Cache-Control`, `Expires`, `Vary`. Именно их игнорирование приводит к кэшированию пустых ответов.
- [Почему 200 с пустым телом — тихий убийца продакшена (Habr)](https://habr.com/ru/articles/648099/) — разбор конкретного инцидента в Яндексе: пустой ответ закешировался на Edge, час пользователи видели белый экран.

### Баги и реальные кейсы
- **Инцидент Яндекс (2021)**: при переключении БД сервис начал отдавать `200 OK` с пустым телом. CDN (собственный Edge) закешировал этот ответ по `Cache-Control: public, max-age=3600`. Все пользователи получили пустой экран на час. Проблема: мониторинг не отличал `200 пусто` от `200 с данными`, алертов не было.
- **Другой кейс (AWS CloudFront + Lambda)**: ошибка в Lambda приводила к возврату `200` с `body: null`. CloudFront кэшировал 200 на 24 часа. Откатили лямбду, но кэш пришлось инвалидировать вручную через API.

### Технические детали
- **Проверка кэширования пустых тестовых ответов**:
  ```bash
  curl -I -H "Cache-Control: no-cache" https://example.com/api/data
  # смотрим заголовки ответа: Cache-Control, ETag, Age
  curl -X GET https://example.com/api/data -w "\nHTTP_CODE: %{http_code}\nSIZE: %{size_download}"
  ```
- **Тест-кейсы для CDN**:
  1. Отправить запрос к эндпоинту с пустым телом (например, `{}` или `""`).
  2. Проверить, что в ответе есть `Cache-Control: no-store` или `private`, если тело пустое.
  3. После повторного запроса (через кэш) проверить `Age` – должен быть 0, если кэш запрещён.
  4. Проверить `X-Cache: Miss`/`Hit` – для пустого тела должен быть всегда Miss.

## Вопрос 2: Как спроектировать тесты, чтобы отловить логику клиента, ломающуюся от 200 с пустым телом?

### Links & Sources
- [REST API Tutorial — Status Codes](https://restfulapi.net/http-status-codes/) — 200 OK означает успех, но не гарантирует наличие тела. Контракт должен явно оговаривать структуру.
- [Postman Test Examples](https://learning.postman.com/docs/writing-scripts/test-scripts/) — как писать тесты на проверку тела ответа (pm.response.text() не пустой).

### Баги и реальные кейсы
- **Мобильное приложение (доставка)**: после обновления API бэкенд начал возвращать `200 {"status": "ok", "data": []}` вместо старого `200 {"data": [...]}`. Клиент (iOS) парсил `data` как массив и показывал пустой список вместо сообщения "нет заказов". Регресс не поймали, потому что тесты проверяли только HTTP статус и поля, но не логику отображения.
- **Микросервис оплаты**: при отсутствии транзакций возвращал `200` с пустым JSON. Фронтенд (React) не обрабатывал `undefined` в `paymentHistory.map()`, падал с ошибкой рендера. 500 было бы заметнее — фронт показал бы тост об ошибке.

### Технические детали
- **Чек-лист для тестирования контракта**:
  - [ ] Проверить, что для каждого возможного статуса HTTP (200, 204, 400, 500) тело ответа соответствует схеме.
  - [ ] Если эндпоинт может возвращать пустой список или null — должен ли он возвращать 200? Или 204 No Content?
  - [ ] Написать тест: отправить запрос, получить 200, проверить, что `response.body` не пустой (например, `pm.expect(pm.response.text()).to.not.be.empty`).
  - [ ] Для критичных данных (список заказов, профиль) добавить тест: `pm.expect(JSON.parse(pm.response.text())).to.have.property('data').that.is.not.null`.
- **Интеграционные тесты с подменой ответов**:
  ```javascript
  // пример теста на Cypress с перехватом
  cy.intercept('GET', '/api/orders', {
    statusCode: 200,
    body: null  // симуляция пустого тела
  }).as('emptyResponse');
  cy.visit('/orders');
  cy.wait('@emptyResponse');
  cy.contains('Нет заказов').should('be.visible'); // ожидаемое поведение
  ```

## Вопрос 3: Должен ли клиент различать 200 пустое и 204/500? Как проектировать API?

### Links & Sources
- [HTTP Status Code 204 No Content](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status/204) — семантически правильный код для успешного запроса без тела. Именно его стоит использовать вместо 200 с пустым телом.
- [Google API Design Guide](https://cloud.google.com/apis/design) — раздел о возвращаемых значениях: если ресурс не найден, возвращать 404, а не 200 с null.

### Баги и реальные кейсы
- **Инцидент с GraphQL**: сервер возвращал `200` даже при ошибках (по спецификации GraphQL). Клиент считал любой 200 успехом, хотя в `errors` были данные. Это привело к тому, что фронтенд показывал пустую страницу, а алерты в мониторинге не срабатывали. Потребовался кастомный парсинг ответа.
- **Стандарты**: если API возвращает `200 OK` с пустым телом, это нарушение принципа «каждый статус имеет значение». Лучше `204 No Content` (нет контента) или `404 Not Found` (ресурс не найден) / `422 Unprocessable Entity` (запрос валиден, но данных нет).

### Технические детали
- **Как проверять корректность статусов**:
  ```bash
  curl -s -o /dev/null -w "%{http_code}" https://api.example.com/items?empty=true
  # должен вернуть 204 или 404, а не 200
  ```
- **Чек-лист для контрактов API**:
  - [ ] Определить четкую политику: когда эндпоинт возвращает `204`, когда `200`, а когда `404`.
  - [ ] Все ответы с кодом 200 должны иметь непустое тело (проверка через Conftest / spectral).
  - [ ] Для пустых списков: возвращать `200 {"items": []}` (чтобы клиент знал структуру), но не пустое тело.
  - [ ] В тестах добавить assert: `expect(response.status).to.be.oneOf([200, 204])` и для 200 — `expect(response.body).to.not.be.empty`.
