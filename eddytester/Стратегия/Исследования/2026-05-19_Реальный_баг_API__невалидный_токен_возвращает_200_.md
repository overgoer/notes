### Slack HackerOne #1151518: невалидный токен возвращает 200 OK — какой middleware пропустил проверку?

**Links & Sources**  
- По этому углу информации почти нет. Прямые пруф-линки на HackerOne #1151518 не найдены (отчёт мог быть приватным или удалён). Общий кейс описан в райтапах на тему «API возвращает 200 вместо 401 при невалидном токене» — суть: middleware аутентификации не прерывал цепочку после неудачной проверки.  

**Баги и реальные кейсы**  
- Типовой баг: в middleware (например, Express.js) после проверки токена не вызывали `return res.status(401).json(...)`, а просто устанавливали статус и шли дальше — в результате запрос доходил до контроллера, который мог вернуть 200 с пустым телом или данными.  
- В Slack (если верить обсуждениям) проблема была в том, что middleware валидации токена работал, но не выставлял правильный статус при ошибке — ответ уходил как 200 с пустым JSON.  

**Технические детали**  
- Воспроизведение на свежем проекте (Node.js + Express):  
  ```bash
  curl -v -H "Authorization: Bearer invalid_token" http://localhost:3000/api/protected
  ```  
  Если в ответе 200 и пустое тело или тело при успешной аутентификации — баг.  
- Чек-лист:  
  - Передать невалидный токен (битый JWT, истёкший, удалённый).  
  - Передать токен с некорректным форматом (без «Bearer», лишние пробелы).  
  - Проверить, что middleware вызывает `return` или `next(error)` после неудачи.  
  - Проверить `content-length` — если 0 при 200, это подозрительно.  

### Пустое тело в 200 OK при невалидном токене — чем это опасно для клиентов, которые парсят только статус-код?

**Links & Sources**  
- Известный кейс: GitHub API при истёкшем токене возвращает 200 OK с пустым телом (обсуждалось в HackerOne и GitHub Community).  
- Поиск не дал прямой ссылки на райтап, но практика описана в «REST API security anti-patterns» — клиенты, проверяющие только статус, считают ответ успешным.  

**Баги и реальные кейсы**  
- Пример: мобильное приложение для загрузки файлов. При истёкшем токене сервер вернул 200 с пустым `{files: []}` вместо 401. Клиент отобразил пустую папку, пользователь не понял, что не авторизован.  
- Опасность: клиенты, парсящие только статус (например, `if(response.status === 200)`), могут:  
  - Записать пустой ответ в кеш.  
  - Пропустить невалидный запрос дальше в бизнес-логику.  
  - Не показывать ошибку пользователю.  

**Технические детали**  
- Проверка:  
  ```bash
  curl -v -H "Authorization: Bearer expired_token" https://api.example.com/data
  ```  
  Проверить что `content-length: 0` или тело пустое `{}`.  
- Чек-лист для тестировщика:  
  - Всегда проверять не только статус, но и тело ответа.  
  - Для защищённых эндпоинтов: ожидать 401 + сообщение об ошибке в теле.  
  - Если API возвращает 200 с пустым телом при невалидном токене — баг средней/высокой степени.  
  - Автоматизировать проверку: ассерт на `status == 401 AND body has error message`.
