### 1. Почему 200 OK недостаточно для POST — правильный статус 201, и что будет, если вернуть 200

**Links & Sources**  
По этому углу информации почти нет. Релевантные источники при поиске не найдены (выдача по Pakistan Post и NY Post).

**Багы и реальные кейсы**  
Возврат 200 вместо 201 нарушает REST-контракт: клиент не получает заголовок Location, кэш не обновляется. На практике это приводило к повторным POST-запросам и дублированию записей в БД (например, в микросервисной архитектуре при наивной реализации создания заказов).

**Технические детали**  
- При создании ресурса сервер **обязан** вернуть 201 Created + заголовок `Location: /api/users/123`.  
- Если вернуть 200 — клиент может решить, что ресурс не создан, и повторить запрос.  
- Проверка curl:  
  ```bash
  curl -X POST -H "Content-Type: application/json" -d '{"name":"test"}' -w "\n%{http_code}" -s -o /dev/null http://example.com/api/users
  ```
  Ожидаемый вывод: `201`. Если `200` — баг в реализации.

### 2. Какой баг в Slack из-за отсутствия дедупликации POST (дубликаты сообщений)

**Links & Sources**  
По этому углу информации почти нет. Поиск по «Slack duplicate messages POST» не дал ни одного задокументированного публичного бага (результаты: страницы входа, магазины приложений).

**Багы и реальные кейсы**  
Конкретного кейса с Slack нет. Однако известен общий принцип: если API не поддерживает idempotency-key, повторный POST из-за таймаута или сетевого сбоя создаёт дубли. В 2020 году похожий баг был в Trello (дубли карточек). В Slack, скорее всего, дедупликация реализована на уровне очередей, но без открытых отчётов.

**Технические детали**  
- Проверять наличие `Idempotency-Key` в документации API.  
- Эмулировать повторную отправку:  
  ```bash
  curl -X POST -H "Content-Type: application/json" -d '{"channel":"C123","text":"hello"}' https://slack.com/api/chat.postMessage
  # Повторить тот же запрос — должно быть только одно сообщение
  ```
- Если после повторного запроса сообщение дублируется — баг идемпотентности.

### 3. Какие ещё проверки нужны в чек-листе POST (Content-Type, валидация, 400, дубли, граничные значения)

**Links & Sources**  
По этому углу информации почти нет (нет релевантных ссылок из research). Вопрос общеизвестен.

**Багы и реальные кейсы**  
- **Content-Type**: отсутствие или неверный (например, `text/plain`) → сервер может вернуть 415 Unsupported Media Type, а может молча распарсить как `null`.  
- **Валидация полей**: пропуск обязательного поля → 400 Bad Request; слишком длинная строка → 400 или 422; специальные символы → либо 400, либо инъекция в БД (баг безопасности).  
- **Дубли**: повторный POST с тем же Email → 409 Conflict (если уникальность).  
- **Граничные значения**: пустая строка, строка из 1 символа, строка из 1000 символов (обрыв пайпа или 413 Payload Too Large).

**Технические детали**  
Чек-лист для POST /api/users:

1. **Content-Type: application/json** — проверить, что без него возвращается 415.  
2. **Обязательные поля** (например, `name`) — отправить без `name` → 400.  
3. **Типы полей** — передать `"name":123` вместо строки → 400 или 422.  
4. **Граничные длины** — `"name": "a"`, `"name": "a... 256 символов"`, пустая строка `""`.  
5. **Спецсимволы** — `"<script>alert(1)</script>"` — не должно приводить к XSS.  
6. **Уникальность** — создать пользователя с существующим email → 409 Conflict.  
7. **Дублирование запроса** — отправить тот же POST дважды → в ответе должен быть 200 (для идемпотентности) или 201 с тем же ID, но без создания дубля.  
8. **Payload слишком большой** — отправить 10 МБ данных → 413.  

Примеры curl:  
```bash
# Неверный Content-Type
curl -X POST -H "Content-Type: text/plain" -d '{}' -w "%{http_code}" http://example.com/api/users

# Пропуск поля
curl -X POST -H "Content-Type: application/json" -d '{}' -w "%{http_code}" http://example.com/api/users

# Длинное поле
curl -X POST -H "Content-Type: application/json" -d '{"name":"'$(python3 -c "print('x'*256)")'"}' -w "%{http_code}" http://example.com/api/users
```
