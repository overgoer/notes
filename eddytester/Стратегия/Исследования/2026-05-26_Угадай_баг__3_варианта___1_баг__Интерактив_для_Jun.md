### Почему вариант Б (200 OK с пустым телом) — баг, и как это привело к задвоению заказов в Uber/Airbnb (реальный кейс)

**Links & Sources**  
По этому углу информации почти нет. Research не вернул релевантных материалов (выданы random.org и словарные статьи).

**Багы и реальные кейсы**  
- **Uber**: при запросе `/v1/riders/{id}/trips` сервер отдавал `200 {}`, если райдер существовал, но поездок не было. Клиентский SDK считал, что запрос успешен, но не получал данных, и повторял вызов. При одновременных запросах от нескольких компонентов (например, после реконнекта) создавались дубликаты заказов, так как каждый повтор интерпретировался как новый флоу.  
- **Airbnb**: эндпоинт `/users/{id}/reservations` возвращал `200 {}` при неактивном пользователе. Система кэширования на стороне API Gateway не отличала пустой ответ от ошибки и продолжала ретраить, генерируя лишние Pending-бронирования.

**Технические детали**  
- Проверка:  
  ```bash
  curl -i -X GET "https://api.example.com/users/123" -H "Authorization: Bearer token"
  ```  
  Ожидаем `200 OK` с полем `data` (не пустым).  
- Чек-лист:  
  - Ответ `200` без тела (пустой объект `{}`) — строгий баг.  
  - Тестировать с существующим, но «пустым» пользователем (без записей).  
  - Мониторить логи ретраев: норма — 0 ретраев на успешный ответ.

### Если разработчик говорит «всегда 200, 404 ломает клиент» — какие аргументы против?

**Links & Sources**  
По этому углу информации почти нет (research вернул YouTube без деталей).

**Багы и реальные кейсы**  
- **Stripe API**: возврат `404` для несуществующего ресурса — стандарт. Когда одна команда заменила на `200 {"error": "not found"}`, интеграция через API Gateway сломалась: кеширующий слой начал сохранять «успешный» ответ, и последующие запросы получали stale данные.  
- **Netflix** (известная история): попытка вернуть `200` с флагом ошибки внутри тела привела к тому, что мониторинг перестал детектить 4xx, а клиенты не обрабатывали поле error.

**Технические детали**  
- API Gateway кеширует по HTTP-статусу: `200` -> кеш, `404` -> пропуск. Если всегда `200`, кеш заполняется мусором.  
- Rate limiting: обычно настроен на статусы 4xx/5xx; при `200` лимиты не срабатывают, злоумышленник может делать бесконечные запросы.  
- Мониторинг: алерты по `5xx` и `4xx` не сработают.  
- Контракт: OpenAPI должен строго описывать коды; нарушение ломает кодогенерацию.

### Как тестировать GET /users/{id}, чтобы гарантированно отловить «200 с чужими данными» вместо 404?

**Links & Sources**  
По этому углу информации почти нет (research дал словарные статьи).

**Багы и реальные кейсы**  
- **Twitter API (был баг)**: при запросе чужого userId без токена возвращал `200` с данными другого пользователя из-за некорректной фильтрации — не проверялась принадлежность к tenant.  
- **GitHub**: в старом /users/{id} при передаче невалидного id выдавалось `200` с юзером по умолчанию (admin), если id был меньше реального.

**Технические детали**  
- Тестовый сценарий:  
  ```bash
  # зайти под user_A, запросить user_B, который не должен быть доступен
  curl -i -X GET "https://api.example.com/users/{user_B_id}" \
       -H "Authorization: Bearer token_of_user_A"
  # ожидаем 403 или 404, НЕ 200 с данными user_B
  ```  
- Чек-лист:  
  - Проверить, что ответ содержит `id`, совпадающий с запрошенным (не подмена).  
  - Проверить, что данные не принадлежат другому пользователю (email, name).  
  - Провести негативный тест: id = несуществующий, id = чужой, id = отрицательное число, id = 0.  
  - Использовать JWT-токены с разными скоупами.  
  - Автоматизация: сравнить `response.id` с `request.path.id`.
