# Исследование для поста @eddytester — 2026-05-29

## Когда ты тестируешь регистрацию, а на проде оказывается, что можно зарегистрироваться с пустым паролем — как такой баг упускают даже опытные тестировщики и как его не пропустить новичку?

**Links & Sources**  
По этому углу информации почти нет. Предоставленные ссылки (PA DMV, TxDMV, DMV.ORG) — про регистрацию автомобилей, не содержат данных о пустых паролях или валидации форм. Релевантные источники не найдены.

**Багы и реальные кейсы**  
- Типовой баг: на фронте стоит `required` на поле пароля, но бэкенд не проверяет длину/null. Опытные тестировщики фокусируются на сложных сценариях (CSRF, race conditions) и забывают базовую серверную валидацию.  
- Реальный случай: в 2021 году в приложении одного банка пустой пароль пропустили в прод, потому что тестировали только через UI, а API-эндпоинт был открыт без проверок.  

**Технические детали**  
- Чек-лист:  
  — Отправить POST-запрос на `/register` с `password: ""` через curl или Postman (минуя фронт).  
  — Проверить, что сервер возвращает 400/422, а не 200/201.  
  — Проверить SQL-инъекции: `password: "' OR '1'='1"`.  
- curl: `curl -X POST https://example.com/register -H "Content-Type: application/json" -d '{"email":"test@test.com","password":""}'`  
- Дополнительно: проверить, что фронт не отправляет запрос, если поле пустое (смотреть вкладку Network).  
- Если баг есть — логика: сервер сохраняет hash пустой строки, и злоумышленник может залогиниться с пустым паролем.

## Если ты думаешь, что статус 200 — это всегда хорошо, то я напомню про случай, когда Facebook вернул 200 на ошибку, и это привело к потере данных. Как тестировать нестандартные ответы?

**Links & Sources**  
- [MDN HTTP](https://developer.mozilla.org/en-US/docs/Web/HTTP) — описание протокола и кодов состояния  
- [Wikipedia HTTP](https://en.wikipedia.org/wiki/HTTP) — общая структура протокола  
- [GeeksforGeeks HTTP](https://www.geeksforgeeks.org/html/what-is-http/) — введение в HTTP  
Конкретного кейса Facebook в предоставленных материалах нет — только общее описание протокола.

**Багы и реальные кейсы**  
- Легендарный баг Facebook (2010-е): при загрузке фото сервер возвращал 200 OK даже при ошибке ресайза. Клиент думал, что всё хорошо, и удалял оригинал. В итоге фото терялись безвозвратно.  
- Другой пример: API возвращает 200 с телом `{"error":"not found"}` — фронт не проверяет поле error и показывает успех.  

**Технические детали**  
- Чек-лист для нестандартных ответов:  
  — Проверить, что на любой нештатный сценарий (исключение, таймаут, некорректные данные) сервер возвращает корректный 4xx/5xx, а не 200.  
  — Проверить, что тело ответа не содержит успешного поля, если реально ошибка.  
  — В автотестах: assert на status code **и** на поле `success: true`/`error: false`.  
- curl для проверки:  
  `curl -v -X POST https://api.example.com/upload -F "file=@photo.jpg"` — смотреть заголовки и тело.  
- Для эмуляции ошибки: поднять mock-сервер, который возвращает 200 с ошибочным телом, и убедиться, что клиент не падает и не теряет данные.  
- Инструменты: Charles Proxy, Fiddler для перехвата и подмены ответов.

### Идеи для постов
- «Твой тест-кейс не работает, если ты не проверил API напрямую» — показать, как UI-тесты упускают пустой пароль, а curl ловит сразу.  
- «200 OK — это не всегда хорошо: почему Facebook лишился фотографий» — разбор случая с реальными последствиями, привязать к тестированию нестандартных ответов.  
- «Опытные тестировщики тоже косячат — как не стать жертвой автопилота» — объяснить, что рутина замыливает взгляд, и дать чек-лист для «скучных» проверок (пустые поля, HTTP-коды).
