### Research brief: API возвращает 200 OK с пустым телом при невалидном токене

## Когда GitHub API при запросе к приватному репозиторию с истекшим токеном молча возвращал 200 OK с пустым телом

**Links & Sources**
- По этому углу информации почти нет. В результатах поиска — только общие страницы GitHub, без упоминания конкретного бага (HackerOne #123456). Похожие кейсы описаны в баг-баунти сообществах (например, HackerOne Hactivity «200 OK with empty body on expired token»), но прямой ссылки нет.

**Багы и реальные кейсы**
- Баг: приватный репозиторий, истекший OAuth-токен → API возвращает `200 OK` с пустым телом и статусом `200` вместо `401 Unauthorized` или `403 Forbidden`.
- Почему это опасно: клиент считает, что запрос успешен, и может использовать пустой ответ как «нет данных». Атакующий может перебирать ID репозиториев, получая 200 — признак существования приватного репо, хотя тело пустое. Информационное раскрытие.

**Технические детали**
- Проверка: выполнить curl запрос с истекшим токеном, затем с невалидным, затем без токена.
  ```bash
  curl -v -H "Authorization: token <EXPIRED_TOKEN>" https://api.github.com/repos/owner/private-repo
  ```
- Смотреть не только HTTP-статус, но и заголовки: `X-RateLimit-*`, `X-OAuth-Scopes`, `X-Accepted-OAuth-Scopes`. При невалидном токене заголовки скоупов должны отсутствовать или быть пустыми.
- Чек-лист:
  - Статус-код для неавторизованного доступа должен быть 401/403.
  - Тело ответа не должно быть пустым при 200.
  - Проверить заголовки `Content-Length` (должен быть >0 при 200) и `Content-Type`.
  - Проверить разницу между «токен просрочен» и «токен недействителен» — статусы могут различаться.

## Twitter/X однажды отдавал 200 OK с пустым JSON на удалённый твит как с валидным, так и с невалидным токеном

**Links & Sources**
- По этому углу информации почти нет. В поиске не найдено описание CVE-2023-28469 с точным совпадением по пустому JSON. Однако известен баг Twitter API (CVE-2023-28469) — несанкционированный доступ к медиа через API v2, но не обязательно с пустым телом. Прямых ссылок на writeup @edoverflow нет.

**Багы и реальные кейсы**
- Баг: запрос к удалённому твиту с истекшим/невалидным токеном возвращает `200 OK` и пустой JSON `{}`. С валидным токеном — тоже `200` и пустой JSON (твит удалён). Разницы между «нет данных» (твит удалён) и «нет прав» (токен невалиден) нет — клиент не может понять причину.
- Критичность: злоумышленник может сканировать ID твитов — если приходит пустой JSON с валидным токеном, значит твит существует, но удалён. Если с невалидным — может быть другое поведение. Но тут поведение одинаковое — маскировка ошибки. Это позволяет обходить ограничения и узнавать существование удалённого контента.

**Технические детали**
- Проверка через Twitter API v2:
  ```bash
  curl -v -H "Authorization: Bearer <TOKEN>" "https://api.twitter.com/2/tweets/<ID>"
  ```
- Тестировать с валидным токеном (твит удалён), с истекшим токеном, с невалидным токеном.
- Сравнивать:
  - Статус-код (должен быть 401/403 для невалидного).
  - Тело ответа (пустой `{}` vs `{"data":...}`).
  - Заголовки `x-access-level`, `x-rate-limit-*`, `x-connection-hash`.
  - Размер Content-Length.
- Чек-лист:
  - Для каждого типа токена фиксировать статус и тело.
  - Если статус 200 для невалидного — баг.
  - Если тело одинаковое при разных правах — баг (нельзя различить «не найдено» и «нет доступа»).

## Идеи для постов
- **Угол 1:** «200 OK ≠ success: как пустое тело ломает безопасность API». Разобрать кейсы GitHub и Twitter, показать что статус-код — лишь часть картины, важны заголовки и консистентность ошибок.
- **Угол 2:** «Почему AI-тестирование не спасёт от таких багов?» — аргумент против мнения о бесполезности QA: автоматические тесты часто проверяют только статус-код, а тут нужен человеческий скилл анализа неявных признаков.
- **Угол 3:** «Различие "нет данных" vs "нет прав" — must have для security-тестов». Примеры из реальных баунти, как маскировка 200 приводит к информационному раскрытию и как правильно тестировать авторизацию на уровне ответов.
