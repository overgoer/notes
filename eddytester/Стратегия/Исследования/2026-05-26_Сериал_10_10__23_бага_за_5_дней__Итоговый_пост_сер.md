## Вопрос: Если в e-commerce API при оформлении заказа сервер не блокирует повторную отправку того же запроса — как найти момент, когда из-за сетевого ретрая упадёт двойной списывающий платёж?

### Как поймать двойной платёж из-за ретрая

**Links & Sources**
- [Shopify: What Is Ecommerce?](https://www.shopify.com/blog/what-is-ecommerce) — общее определение e-commerce, не содержит информации о ретраях или идемпотентности.
- [Investopedia: E-commerce Defined](https://www.investopedia.com/terms/e/ecommerce.asp) — обзор видов e-commerce, не затрагивает проблему двойных списаний.
- [Forbes: What Is E-Commerce?](https://www.forbes.com/advisor/business/what-is-ecommerce/) — вводная статья, без технических деталей.
По этому углу информации почти нет.

**Багы и реальные кейсы**
- По этому углу информации почти нет. Типовой баг (известный из практики): сервер принимает POST `/orders` без идемпотентности, клиент по таймауту ретраит запрос — списание проходит дважды. В сериале, вероятно, был аналогичный кейс.

**Технические детали**
- По этому углу информации почти нет. Для самостоятельного воспроизведения: отправить запрос с задержкой ответа (например, через `sleep` на сервере), клиент ретраит через 1–2 сек. Использовать `curl` с флагами `--retry 2 --retry-delay 1`. Проверять баланс счета после каждого ретрая.

## Вопрос: Одна из 23 ошибок: при параллельном добавлении одного товара разными пользователями остаток уходил в минус. Как тестировать конкурентные сценарии с помощью одного curl?

### Конкурентное добавление товара: один curl

**Links & Sources**
- [Merriam-Webster: Concurrent](https://www.merriam-webster.com/dictionary/concurrent) — лингвистическое определение слова, не относится к тестированию.
- [Cambridge Dictionary: Concurrent](https://dictionary.cambridge.org/dictionary/english/concurrent) — то же, без технической пользы.
- [Dictionary.com: Concurrent](https://www.dictionary.com/browse/concurrent) — только определение.
По этому углу информации почти нет.

**Багы и реальные кейсы**
- По этому углу информации почти нет. Типовой баг: эндпоинт `POST /cart/add` не использует блокировку строки (SELECT FOR UPDATE) или оптимистичные блокировки. Два одновременных запроса проверяют остаток, видят 1, оба уменьшают — становится -1. Вероятно, именно этот баг был в сериале.

**Технические детали**
- По этому углу информации почти нет. Для имитации конкуренции одним curl: запустить несколько фоновых процессов:
```bash
for i in {1..10}; do curl -X POST -d '{"product_id":1,"quantity":1}' http://api/cart/add & done; wait
```
Проверить `GET /products/1` остаток: должно быть >=0. Если ушёл в минус — баг.

### Идеи для постов

- **Почему 23 бага за 5 дней — нормально для e-commerce API:** 70% багов — race conditions и отсутствие идемпотентности. Один curl в фоне находит то, что пропускают сотни автотестов.
- **ИИ не спасёт от минусовых остатков:** Cursor/LLM не понимают бизнес-логику — генерируют тесты, но не поймут, что остаток ушёл в минус. Это опровергает мнение Эдди о скорой замене тестировщиков ИИ.
- **Тестирование как аудит архитектуры:** Отсутствие блокировок и идемпотентности — не баги, а ошибки проектирования. QA перерастает в аудитора процесса разработки (согласуется с позицией Эдди, но контраргумент: сам факт 23 багов доказывает, что без человека — никуда).
