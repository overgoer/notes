# Шпаргалка: 3 curl-команды для проверки API. Практический гайд для Junior QA

## Вопрос 1: Как отловить баг с отсутствующим Content-Type, если тестировали только через Postman, а в продакшене сервис упал?

**Links & Sources**
- [cURL man page](https://curl.se/docs/manpage.html) — curl не добавляет Content-Type автоматически для JSON; Postman — добавляет, что маскирует баг.

**Багы и реальные кейсы**
- В одном проекте разработчик писал curl-запрос для теста, но забыл указать `Content-Type: application/json`. В Postman всё работало (он сам проставляет `application/json` по умолчанию для JSON-тела). При деплое в продакшн клиенты (не Postman) слали `text/plain`, сервер отвечал 415 Unsupported Media Type, падала вся цепочка платежей. Тестировщик не проверил через чистый curl — баг ушёл в прод.
- Ещё кейс: сервер ожидал `Content-Type: multipart/form-data` для загрузки файла, а тестировщик проверял только через Postman — он сам добавляет границы и Content-Type. В продакшене мобильное приложение присылало просто `application/octet-stream` — сервер крашился.

**Технические детали**
- Проверять **обязательно через curl без Postman**:
  ```bash
  curl -X POST https://api.example.com/resource \
    -H "Content-Type: application/json" \
    -d '{"key":"value"}' \
    -v
  ```
- А затем **без заголовка** (чтобы увидеть, как отреагирует сервер):
  ```bash
  curl -X POST https://api.example.com/resource \
    -d '{"key":"value"}' \
    -v
  ```
- Чек-лист:
  - `Content-Type` в запросе должен соответствовать формату тела (JSON → `application/json`, XML → `application/xml`, form-data → `multipart/form-data`).
  - Проверить, что сервер не завязан на автоматическое определение типа (некоторые серверы требуют явный заголовок).
  - Использовать `curl -H "Content-Type: application/json"` всегда в тестовых скриптах, но также тестировать без него.
  - В `-v` смотреть ответ сервера: если 415 или 400 — скорее всего проблема в Content-Type.

## Вопрос 2: Почему при проверке через curl -I (HEAD) можно пропустить баг с медленным ответом, если не смотреть на Age и X-Cache?

**Links & Sources**
- [cURL man page](https://curl.se/docs/manpage.html) — флаг `-I` выполняет HEAD-запрос, не возвращая тело; заголовки `Age` и `X-Cache` показывают, пришёл ли ответ из кэша и как давно он был сгенерирован.

**Багы и реальные кейсы**
- В продакшене иногда падает производительность бэкенда: ответы генерируются не за 100 мс, а за 5 секунд. Но при использовании `-I` тестировщик видит быстрый ответ (5 мс), потому что запрос попал в кэш (CDN). Настоящее время генерации скрыто. Если не смотреть `Age` (показывает возраст кэша в секундах) и `X-Cache` (HIT/MISS), можно не заметить, что бэкенд деградировал.
- Реальный случай: в CDN настроили агрессивное кэширование, HEAD-запросы всегда уходили в кэш. QA проверял только HEAD с `-I`, получал ответ мгновенно. Через неделю после апдейта бэкенда упали новые трафик-лимиты — никто не заметил, потому что кэш маскировал проблему.

**Технические детали**
- Всегда проверять заголовки `Age`, `X-Cache`, `X-Cache-Hits`, `Cache-Control`:
  ```bash
  curl -I https://api.example.com/resource
  ```
- Для принудительного пропуска кэша:
  ```bash
  curl -I -H "Cache-Control: no-cache" https://api.example.com/resource
  ```
- Сравнить время ответа с `-I` (с кэшем) и с `-w "%{time_total}"` (с телом, через `-X GET`):
  ```bash
  curl -X GET -o /dev/null -s -w "Общее время: %{time_total}s\n" https://api.example.com/resource
  ```
- Чек-лист:
  - `Age: 0` — ответ свежий из кэша (сгенерирован только что).
  - `Age: > 0` — ответ кэширован, реальное время генерации бэкенда может быть больше.
  - `X-Cache: MISS` — запрос дошел до бэкенда.
  - `X-Cache: HIT` — ответ из кэша, не проверяет актуальность бэкенда.
  - Всегда дополнять `-I` запросом с телом (`-X GET`) и таймингом.

**Примечание по мнению Эдди**: Эдди считает, что AI не заменит тестировщиков, а инструменты вроде curl остаются базой. Однако нельзя полностью полагаться на автоматизацию или готовые клиенты (Postman) — нужно понимать подкапотные детали. В посте можно подсветить, что curl даёт нефильтрованный взгляд на поведение сервера, и это умение остаётся востребованным даже в эру AI. Контраргумент к тем, кто считает, что AI скоро вытеснит ручное тестирование: AI может сгенерировать curl-запрос, но он не поймёт, что заголовок Content-Type пропущен — ответственность всё ещё на тестировщике.
