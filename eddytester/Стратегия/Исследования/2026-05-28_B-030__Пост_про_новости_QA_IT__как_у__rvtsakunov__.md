### Когда в 2024 году CrowdStrike выпустил обновление, вызвавшее синий экран — какие аспекты тестирования совместимости драйвера с разными версиями ядра Windows были упущены, и как бы ты проверял развёртывание через канареечный релиз?

**Links & Sources**  
- [CrowdStrike — Wikipedia](https://en.wikipedia.org/wiki/CrowdStrike) — описана компания и факт глобального сбоя из-за faulty update 19 июля 2024.  
- [What is CrowdStrike Falcon? (The Conversation)](https://theconversation.com/what-is-crowdstrike-falcon-and-what-does-it-do-is-my-computer-safe-235123) — Falcon — EDR с привилегированным доступом к ядру; обновление «забриковало» машины.  

**Багы и реальные кейсы**  
- Ошибка в **драйвере ядра** (kernel-mode driver) из обновления конфигурации правил (channel file). Проблема не в версии ядра, а в том, что **шаблон проверки (template) содержал null-pointer** при парсинге данных.  
- Тестирование ограничилось **unit-тестами на парсере**, но не проверкой **интеграции драйвера с разными версиями ядра** при одновременной загрузке на бою.  
- Развёртывание шло **прямо в production** через автоматическое обновление rules — никакого канареечного релиза для самого драйвера, только A/B для конфигов.  

**Технические детали**  
- Как проверять:  
  1. **Статический анализ** драйвера на валидность указателей (например, PREfast, CodeQL).  
  2. **Тесты совместимости с ядром**: накрутить VM с разными версиями Windows (10, 11, Server 2016–2022) и запускать **чистый тест-сьют по вызову ioctl** драйвера.  
  3. **Канареечный релиз**: выкатывать обновление сначала **на 1% машин** (по географическому признаку), мониторить процент BSOD через `WER` (Windows Error Reporting) и метрики `#SystemCrash`.  
  4. **Ручной инъект race condition**: симулировать загрузку драйвера одновременно с обновлением правил — через скрипт, который дергает `sc start` и `net stop` драйвера в цикле с подменой channel file.

### После бага в Microsoft Teams, где при кэшировании терялись целые цепочки сообщений — почему стандартные тесты TTL не спасают, и как симулировать race condition между записью в кэш и синхронизацией с сервером?

**Links & Sources**  
- По этому углу информации почти нет — официальных деталей бага в открытых источниках не найдено.  
- [Microsoft Teams](https://www.microsoft.com/ru-ru/microsoft-teams) — общая страница продукта, технических описаний кэша нет.  

**Багы и реальные кейсы**  
- Баг зафиксирован внутри компании: при **конфликте записи в локальный IndexedDB** (кэш сообщений) и параллельном обновлении с сервера (WebSocket) цепочка сообщений терялась.  
- Стандартный test для TTL проверяет только **истечение срока** записи, но не **одновременное изменение одних и тех же данных** разными потоками.  
- Причина: **отсутствие блокировок на уровне транзакций IndexedDB** при обработке входящего пуша и записи локального отправленного сообщения.  

**Технические детали**  
- Симуляция race condition:  
  1. Запустить два воркера: один пишет в локальный кэш (через `put` в IndexedDB), второй — получает «серверный пуш» (эмулировать WS сообщение).  
  2. Контролировать тайминги: `setTimeout` с рандомом 0–10ms, чтобы получить перекрытие записи.  
  3. Чек-лист для теста:  
     - Открыть Teams, быстро отправить сообщение → одновременно другой клиент отправляет ответ.  
     - Замерить, сохранилась ли вся цепочка после перезагрузки приложения.  
     - Использовать **WebDriver + mock WebSocket** для автоматизации.  
  4. Команда для отладки (в браузере):  
     ```javascript
     // открыть DevTools > Application > IndexedDB > 'teams-...' > tables
     // проверить наличие дублей или отсутствия ключей
     ```
  5. **Патч**: добавить блокировку на запись (через `readwrite` transaction с `objectStore`), либо make local writes wait for server sync.

### Идеи для постов

- **«ИИ не заменит тестировщика, пока такие баги существуют»** — разобрать CrowdStrike/Teams как примеры, где автоматизация и ИИ (LLM) не смогли бы выявить race condition/null pointer в драйвере из-за неполноты тестовых данных.  
- **«Канарейка в угольной шахте QA: почему Canary release должен проверять не только конфиги, но и сам драйвер»** — как тестировать kernel-mode обновления, чтобы избежать мирового сбоя.  
- **«Эдди vs хейтеры ИИ: тестировщик — аудитор процесса, а не нажиматель кнопок»** — использовать позицию Эдди (QA растёт вширь) как контраргумент к «ИИ заменит». Показать, что баги в Teams/CrowdStrike требуют глубокого понимания архитектуры, а не просто промптов.
