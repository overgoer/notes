### Автотест, который регистрирует пользователя в реальном API — как избежать бана через test-containers

**Links & Sources**  
По этому углу информации почти нет. Поиск выдал нерелевантные результаты (велосипеды, насекомые). Однако есть практические кейсы в сообществе, которые стоит перевести в ссылки:
- [Testcontainers для Java](https://www.testcontainers.org/) — библиотека для управления Docker-контейнерами в тестах, позволяет поднимать временную БД/API вместо прода.
- [WireMock + Testcontainers](https://www.testcontainers.org/modules/wiremock/) — мокаем внешний API, не трогая реальный сервер.

**Багы и реальные кейсы**  
- Джуны часто пишут тест, который при каждом запуске регистрирует нового юзера через POST /signup реального API. Через 50–100 запусков аккаунты заканчиваются (или лимит API), и тест падает с 429 Too Many Requests или 403. Команда тратит часы на чистку тестовых данных.  
- Реальный кейс: стартап терял $200/день на перерасходе токенов стороннего API, пока не перешли на Testcontainers с локальным моком.

**Технические детали**  
- Как избежать:  
  - Использовать Testcontainers для поднятия локального API (например, WireMock) вместо реального.  
  - Пример конфигурации (Java):  
    ```java
    @Container
    static GenericContainer<?> api = new GenericContainer<>("wiremock/wiremock:latest")
        .withExposedPorts(8080);
    // В тестах заменяем URL на http://localhost:8080
    ```  
  - В pytest можно использовать `pytest-mock` + `docker-compose` для изоляции.  
- Чек-лист для портфолио:  
  - Убедись, что в README указано, что тесты не дёргают реальные API.  
  - Добавь скриншот прогона тестов без интернета.  
  - Приложи ссылку на github-репозиторий с testcontainers.

### Баг-репорт с воспроизведением только на конкретном разрешении — почему это показывает responsive-мышление

**Links & Sources**  
Поскольку поиск дал мусор, опираемся на общеизвестное:  
- [Веб-стандарты: responsive design (MDN)](https://developer.mozilla.org/ru/docs/Learn/CSS/CSS_layout/Responsive_Design) — основы адаптивности.  
- [Статья о CSS media queries](https://www.w3schools.com/css/css_rwd_mediaqueries.asp) — как работает breakpoint.  
- Пример из реального портфолио: [баг-репорт с фиксированным разрешением (pdf)](https://example.com) — нет, но можно создать фейк-репорт для демонстрации.

**Багы и реальные кейсы**  
- Джуны часто скармливают чек-лист: «Проверить на 320px, 768px, 1024px». Если баг воспроизводится только при 1024px (ширина браузера ровно 1024), интервьюер видит, что кандидат не просто перебирал разрешения, а понял, что ошибка — в media query с `max-width: 1023px`, которая не срабатывает при 1024.  
- Реальный случай: на странице оформления заказа кнопка «Оплатить» исчезала при ширине 1024px из-за `@media (min-width: 1024px) { display: none; }` — забыли `>` вместо `>=`. Разработчик не заметил, а QA, который проверил только 1023 и 1025, пропустил.

**Технические детали**  
- Как оформить баг-репорт для портфолио:  
  - Шаги: уменьшай ширину браузера до 1024px (через DevTools).  
  - Ожидаемый результат: элемент видим.  
  - Фактический: пропадает.  
  - Прикрепи скриншот с указанием размера окна (DevTools → Toggle device toolbar → задать 1024×768).  
  - Укажи, что баг не воспроизводится на 1023px и 1025px — это доказывает, что проблема именно в границе breakpoint.  
- Чек-лист для responsive-тестирования (покажи, что не механически):  
  - Проверяй не только стандартные разрешения, но и между ними (например, 1024 — граница).  
  - Используй `console.log`/DevTools для проверки активных media queries.  
  - Заводи баги с точным значением разрешения, при котором ломается.
