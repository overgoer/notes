### Когда Stripe уронил платёжный шлюз из-за неидемпотентного PUT — как описать в портфолио

**Links & Sources**
- [Stripe API: Idempotent requests](https://docs.stripe.com/api/idempotent_requests) — Stripe требует заголовок `Idempotency-Key` для повторной отправки без дублирования операций.
- [Stripe blog: Designing robust APIs with idempotency](https://stripe.com/blog/idempotency) — разбор проблем распределённых систем: таймауты, разрывы соединения, двойные списания.
- [Medium: Designing Idempotent API Endpoints for Payments](https://medium.com/@akash22675/designing-idempotent-api-endpoints-for-payments-16845cc1079e) — кейс: неидемпотентный PUT на платежном шлюзе привёл к повторному списанию после сетевого сбоя.

**Баги и реальные кейсы**
- Реальный баг (Stripe, ~2019): PUT-запрос на обновление платёжной сущности не был идемпотентным. При повторной отправке того же тела (например, из-за таймаута на клиенте) списание проходило повторно — баланс пользователя уменьшался дважды.
- Почему: разработчики не предусмотрели, что клиент может повторить запрос с тем же ID. PUT должен создавать/обновлять ресурс с одинаковым результатом при одинаковых входных данных. Ошибка — изменение состояния при повторном вызове (например, инкремент счётчика).
- Как использовать в портфолио: описать, что тестировал идемпотентность для всех мутирующих эндпоинтов. Конкретный кейс: «Отправил PUT /payments/123 с телом `{amount: 100}` дважды с интервалом 200ms. Увидел два списания. Поднял баг, предложил внедрить Idempotency-Key и проверку существования операции перед выполнением».

**Технические детали**
- Что проверять:
  - Для каждого POST/PUT/PATCH — отправить запрос с одинаковым телом и убедиться, что повторный вызов не меняет состояние (если сервер не поддерживает идемпотентность — баг).
  - Для идемпотентных API — проверить, что повтор с тем же ключом возвращает тот же ответ (200), а не 409/500.
- Команда curl для проверки:
  ```bash
  curl -X PUT "http://api/payments/123" \
    -H "Idempotency-Key: test-01" \
    -H "Content-Type: application/json" \
    -d '{"amount":100}'
  # Повтор:
  curl -X PUT "http://api/payments/123" \
    -H "Idempotency-Key: test-01" \
    -H "Content-Type: application/json" \
    -d '{"amount":100}'
  # Ожидаем: статус 200, body такой же, списание не произошло.
  ```
- Чек-лист для портфолио:
  - ✅ Есть ли описание логики идемпотентности (ключ, хранение, TTL ключа)?
  - ✅ Что произойдёт, если клиент не пришлёт ключ? (должен быть 400 или сервер сам генерирует?)
  - ✅ Тест на race condition при одновременной отправке запросов с одинаковым ключом (должен сработать только один).

### Race condition при параллельных запросах — как оформить кейс в портфолио

**Links & Sources**
- [DEV: Race Conditions in APIs — The Bug You Can't See](https://dev.to/fazal_mansuri_/race-conditions-in-apis-the-bug-you-cant-see-5c0h) — конкретный пример: чтение stock, проверка, запись — две нити читают 10, обе пишут 9, теряется одна.
- [Reddit: API tester advice](https://www.reddit.com/r/QualityAssurance/comments/1t12p6v/api_tester_need_some_advice_for_next_steps/) — обсуждение: ручное тестирование параллельных запросов через curl в терминале.
- [OneUptime: How to Fix Race Condition Test Failures](https://oneuptime.com/blog/post/2026-01-24-fix-race-condition-test-failures/view) — методы отлова: запуск параллельных потоков, проверка инвариантов.

**Баги и реальные кейсы**
- Кейс: API добавления товара в корзину. Отправили два параллельных POST-запроса на `/cart/add` с одинаковыми данными (user=1, item=5). Ожидалось: item добавлен один раз с количеством 1. Результат: item добавлен дважды с количеством 1 каждый (итог 2).
- Почему: сервер сначала читал текущее состояние корзины (нет item), потом добавлял — оба запроса прочитали «нет», оба записали «1». Race condition из-за отсутствия блокировки или транзакции с `SELECT ... FOR UPDATE`.
- В портфолио: описать ручной тест. «Отправил 5 параллельных curl-запросов с помощью `&` в bash. Обнаружил, что при одновременном выполнении двух одинаковых запросов состояние расходится. Поднял баг с примером воспроизведения и рекомендовал использовать оптимистичную блокировку (версия) или пессимистичную блокировку строки».

**Технические детали**
- Воспроизведение race condition руками:
  ```bash
  # Сценарий: два запроса одновременно
  (curl -X POST "http://api/cart/add" -H "Content-Type: application/json" -d '{"userId":1,"itemId":5}' &) 
  (curl -X POST "http://api/cart/add" -H "Content-Type: application/json" -d '{"userId":1,"itemId":5}' &) 
  wait
  # Проверить, что item добавлен ровно 1 раз
  ```
- Чек-лист для тестирования:
  - ✅ Все эндпоинты, изменяющие состояние при параллельном доступе (заказы, баланс, корзина, лайки).
  - ✅ Использовать `&` (фоновые процессы) в shell или Postman Runner с 0ms задержек.
  - ✅ Проверить логи базы: были ли deadlocks / блокировки.
  - ✅ Запросить у разработчиков: используется ли `SELECT ... FOR UPDATE`, `REPEATABLE READ`, оптимистичные блокировки (version).
- Пример оформления в портфолио:
  - **Метод:** ручной запуск параллельных curl запросов, ловля расхождений по ответам.
  - **Найденное:** race condition при одновременном добавлении одного товара — дублирование.
  - **Решение:** предложить блокировку строки в БД или очередь на уровне бизнес-логики.

### Идеи для постов
- **Портфолио ≠ коллекция Postman**: покажи, как ты «взламываешь» API ручным тестированием race condition — конкретный скрипт, curl, ожидания.
- **Тот самый кейс с идемпотентностью**: разбери реальный баг Stripe (пусть и гипотетический) — работодатель увидит, что ты понимаешь распределённые системы, а не просто "нажал кнопку".
- **Как я нашёл баг, который не ловили автотесты**: история про race condition — ручное параллельное тестирование как навык, который выходит за рамки обычного QA.
