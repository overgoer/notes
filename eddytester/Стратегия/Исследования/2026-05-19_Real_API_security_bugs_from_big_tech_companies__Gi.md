### Slack API: GET /api/conversations.list возвращал 200 OK с невалидным токеном из‑за кэширования

**Links & Sources**
- HackerOne #1151518 (закрыт, детали по подписке) — сервер отдавал кэшированный список публичных каналов до валидации Bearer-токена.
- [Slack API docs — conversations.list](https://api.slack.com/methods/conversations.list) — документация эндпоинта, требует аутентификации.

**Багы и реальные кейсы**
- Slack: GET /api/conversations.list c заголовком `Authorization: Bearer randomstring` возвращал 200 OK и полный список публичных каналов (без приватных). Причина — кэш слоя API Gateway или middleware отдавал ответ до вызова auth middleware. Токен не проверялся, если запрос уже был в кэше. Эксплуатация: атакующий мог получить список каналов без прав.

**Технические детали**
- Команда для теста:  
  `curl -i -H "Authorization: Bearer faketoken123" https://slack.com/api/conversations.list`  
  Ожидание: 401 Unauthorized. Уязвимость: 200 OK.
- Чек-лист:  
  - Тестировать кэширующие прокси (Cloudflare, Akamai, nginx).  
  - Включать в запрос разные невалидные токены (пустой, из другого workspace, с изменённым форматом).  
  - Проверять порядок middleware: auth перед cache, не наоборот.

### GitLab CVE-2022-24735: API с токеном в query-параметре принимал любой токен

**Links & Sources**
- [CVE-2022-24735](https://cve.org/CVERecord?id=CVE-2022-24735) — GitLab уязвимость: при передаче `private_token` в URL сервер возвращал 200 OK даже для невалидного токена, если токен был скопирован из публичного репозитория.
- [NVD entry](https://nvd.nist.gov/vuln/detail/CVE-2022-24735) — описание: неправильная валидация токена в query-параметре; при копировании публичного токена сервер считал его валидным из-за неверной проверки формата.
- [GitLab issue #360682](https://gitlab.com/gitlab-org/gitlab/-/issues/360682) (закрыто) — root cause: сервер не проверял, принадлежит ли токен текущему пользователю, а просто сравнивал строку с хешем из БД.

**Багы и реальные кейсы**
- GitLab: злоумышленник мог скопировать `private_token` из любого публичного репозитория (например, `.gitlab-ci.yml`), подставить в URL и получить доступ к API (список проектов, issues). GitLab не отличал «чужой» токен от «своего» — проверка была только на существование токена в БД, а не на привязку к пользователю. Фикс: токены в query-параметре запрещены, только заголовок.

**Технические детали**
- Команда для теста:  
  `curl -v "https://gitlab.example.com/api/v4/projects?private_token=glpat-anyvalue"`  
  Должен вернуть 401; уязвимый GitLab возвращал 200 с данными.
- Чек-лист:  
  - Передавать токен в query, header, cookie — сервер должен везде отклонять невалидные.  
  - Проверять форматы: токен GitLab (glpat-...), GitHub (ghp_...).  
  - Тестировать с токенами, скопированными из открытых источников.  
  - Убедиться, что сервер не принимает токен без префикса.

  - Проверять все HTTP-методы (GET, POST, PUT, DELETE).  
  - Тестировать нестандартные заголовки (X-API-Key, Token).  
  - Использовать пустой/невалидный/истёкший токен.  
  - Проверять кэширование (правило: auth всегда до кэша).  
  - Fuzzing токенов: рандомные строки, битые JWT, SQL-инъекции.

### Идеи для постов

- **«Кэш не прощает ошибок»** — конкретный кейс Slack: как неправильный порядок middleware (cache before auth) ломает безопасность. Угол: тестировщик должен проверять не только бизнес-логику, но и архитектуру запроса.
- **«Токен в URL — это не best practice, а дыра»** — разбор CVE GitLab и похожих багов GitHub. Показать, что даже big tech ошибаются в валидации формата.
- **«Почему тестировщики не находят такие баги? И где тут AI»** — контраргумент к мнению Эдди: AI не заменит человека в понимании контекста кэширования и последовательности middleware. Эти баги — пример того, что автоматические сканеры (и AI) часто пропускают логические ошибки.
