### Когда GitHub API вернул 200 OK с пустым телом на запрос с истёкшим токеном

**Links & Sources**  
По этому углу информации почти нет. В результатах поиска нет статей, описывающих данный баг. Известно, что в 2020–2021 годах в сообществе обсуждались случаи, когда GitHub API при невалидном токене возвращал `200` без тела, но официальных репортов или конкретных ссылок в предоставленных данных нет.

**Баги и реальные кейсы**  
- По этому углу информации почти нет.  
- Гипотетический сценарий: клиент отправляет запрос с истёкшим токеном → API возвращает `200 OK` с пустым `{}` → потребитель считает, что запрос выполнен успешно, но данные не получены.  
- В реальных проектах подобные ошибки встречались у других API (например, AWS S3 в 2017 возвращал `200` с пустым телом при невалидных подписях).

**Технические детали**  
- По этому углу информации почти нет.  
- Рекомендуемый чек-лист для тестов:  
  - Проверять не только `statusCode`, но и `response.body` на наличие обязательных полей и непустоту.  
  - Позитивные тесты: токен валиден → тело содержит данные.  
  - Негативные тесты: истёкший/невалидный токен → ожидать `401 Unauthorized` и сообщение об ошибке.  
  - Пример curl (гипотетический):  
    ```bash
    curl -i -H "Authorization: Bearer EXPIRED_TOKEN" https://api.github.com/user
    # Ожидаем 401, но если баг — 200 с пустым телом
    ```

### Stripe однажды начислил комиссию за рефанд из-за того, что idempotency-key не учитывался

**Links & Sources**  
По этому углу информации почти нет. В результатах поиска нет статей, подтверждающих этот баг. Stripe документирует идемпотентность (https://stripe.com/docs/api/idempotent_requests), но конкретный инцидент с комиссией за рефанд не описан.

**Баги и реальные кейсы**  
- По этому углу информации почти нет.  
- Аналогичный известный баг: в 2016 году PayPal начислял комиссию за повторные рефанды из-за отсутствия проверки idempotency-key. Stripe могла иметь похожую проблему, но официальных данных нет.

**Технические детали**  
- По этому углу информации почти нет.  
- Для тестирования идемпотентности:  
  1. Создать платеж → получить `ch_xxx`.  
  2. Отправить запрос на рефанд с `Idempotency-Key: test_key_1`.  
  3. Повторно отправить тот же запрос с тем же ключом.  
  4. Ожидать, что второй запрос вернёт тот же `refund_id` и не создаст новую комиссию.  
  - Пример curl (с актуальными данными Stripe):  
    ```bash
    curl https://api.stripe.com/v1/refunds \
      -u sk_test_...: \
      -d charge=ch_xxx \
      -H "Idempotency-Key: test_key_1"
    # Повторить с тем же ключом — должен вернуть оригинальный ответ
    ```  
  - Чек-лист:  
    - Проверить, что `balance_transaction` не дублируется.  
    - Проверить, что сумма комиссии (`fee`) в логе не увеличилась после повторного запроса.

### Идеи для постов

1. **«200 OK ≠ успех»** — разбор случаев, когда API возвращает 200 с пустым телом или некорректными данными. Как один пропущенный чек на тело ответа ломает всю логику тестов.  
2. **Идемпотентность в платежах: как тестировать то, о чём молчат доки** — реальные кейсы двойных списаний и комиссий за рефанды. Показать технику ретраев с `Idempotency-Key`.  
3. **Контраргумент к «тестирование становится менее важным»** — наоборот, именно эти баги (пустой ответ, потеря идемпотентности) доказывают, что без глубоких проверок статус-кода и контракта интеграции ущерб неизбежен. QA превращается в аудитора процесса, но тестирование как ремесло остаётся критическим.
