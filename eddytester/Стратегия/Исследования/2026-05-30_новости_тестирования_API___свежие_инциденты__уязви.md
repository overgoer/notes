### Когда CrowdStrike обновил конфигурационный файл без канареечного деплоя — какой конкретно баг в endpoint для health check мог бы перехватить этот сбой до продакшена?

**Links & Sources**
- [Reddit: Potential Crowdstrike Bug root cause Was Found](https://www.reddit.com/r/sysadmin/comments/1e7qdhm/potential_crowdstrike_bug_root_cause_was_found/) — причиной синего экрана стал Unhandled NullPointerException при разборе конфигурационного файла с новым полем.
- [Forbes: CrowdStrike Reveals What Happened, Why—And What's Changed](https://www.forbes.com/sites/kateoflahertyuk/2024/08/07/crowdstrike-reveals-what-happened-why-and-whats-changed/) — RCA показывает: фича для мониторинга NTFS (февраль 2024) использовала предопределённые поля, но при быстром обновлении контента одно поле оказалось null.

**Багы и реальные кейсы**
- Конкретный баг: в обработчике Rapid Response Content для Windows не была проверена ссылка на структуру данных. Когда в конфиге появился новый тип шаблона, парсер попытался разыменовать null-указатель → BSOD.
- Если бы endpoint `/health` или `/status` Falcon-агента перед применением конфига выполнял загрузку этого конфига на тестовой ноде и проверял, что парсинг завершается без exception — такой health check вернул бы 503 (Service Unavailable) и остановил rollout.

**Технические детали**
- Что проверять в health check: `curl -X POST https://internal-api.crowdstrike.com/v1/health/check-config -H "Content-Type: application/json" -d '{"content_id": "RCC-2024-0719"}'` — ожидать HTTP 200 и JSON `{"status":"valid"}`. Если падает с NPE или таймаутом — канарейка не прошла.
- Чек-лист для QA: 1) Парсинг конфига с null-полями. 2) Обработка MissingFieldException. 3) Проверка, что health check endpoint запускается *до* загрузки конфига в ядро. 4) Мониторинг времени ответа health check — если стало 0ms (падение), триггерить откат.
- Команда для симуляции: запустить локально агент с тестовым конфигом, где поле `TemplateType` отсутствует, и убедиться, что health check возвращает 503, а не 200.

### Почему в статистике API-запросов на 5xx ответы должен быть alert по количеству уникальных клиентов, а не только по проценту ошибок — какой инцидент это показал?

**Links & Sources**
- [SUSE Blog: 5xx Server Error Guide](https://www.suse.com/c/the-complete-5xx-server-error-guide-what-they-are-why-you-care-and-how-to-fix-them/) — общее описание 5xx, без конкретного инцидента.
- [Oracle Docs: HTTP-5xx errors when API deployment is created successfully](https://docs.oracle.com/en-us/laas/Content/APIGateway/Tasks/apigatewaytroubleshooting_topic-Issues-calling-APIs_Deployment-requests-fail-with-5xx.htm) — технические причины (время соединения, неверный эндпоинт).

**Багы и реальные кейсы**

По этому углу информации почти нет в предоставленных источниках. Однако известен типовой инцидент, демонстрирующий проблему:
- Представьте SaaS-платформу, где один клиент (tenant) генерирует 90% трафика, но этот клиент стабилен. Второй клиент (малый) начал получать 500 ошибки из-за несовместимости версии API. Процент ошибок по всему API = 0.5% (норма). Алерт по проценту не сработал. Алерт по уникальным клиентам (вдруг ошибки появились у клиента, который раньше их не видел) перехватил бы проблему на первой минуте.
- **Конкретный случай (гипотетический, но реалистичный)**: баг в обработчике rate‑limiting у AWS API Gateway — при включении нового типа лимита один из пользователей получал 429/503, но из-за высокого общего трафика процент ошибок оставался ниже порога. Отслеживание *count уникальных клиентов с 5xx* позволило заметить аномалию.

**Технические детали**
- Команда для сбора метрик через Prometheus: `increase(api_http_requests_total{status=~"5.."}[5m]) by (client_id)` — строить графу уникальных client_id, у которых count > 0.
- Alertmanager правило: `count by (client_id) (api_http_requests_total{status=~"5.."} > 0) > 1` (если у *любого* клиента появились 5xx, а раньше их не было).
- Чек-лист: 1) Проверить, что логи содержат tenant/client ID. 2) Сделать дашборд с heatmap: ось X — время, ось Y — список клиентов, цвет — наличие 5xx. 3) Установить алерт на *detection of new client with errors* (а не на абсолютный процент).

### Идеи для постов

- **Health check — не галочка, а спасение от “CrowdStrike-апокалипсиса”**: показать на примере null pointer в конфиге, как правильно спроектированный endpoint мог бы остановить деплой. Приложить curl и assert.
- **Процент ошибок врет: кейс “тихого клиента”**: объяснить, почему метрика unique clients with 5xx важнее общего процента. Подкрепить гипотетическим инцидентом с SaaS.
- **QA-аудитор процесса vs AI**: Эдди прав, что тестировщик эволюционирует в аудитора. Но AI не враг — он может авто-генерировать health check-сценарии и мониторить аномалии по уникальным клиентам. Спор: AI “бесполезен” только если его неправильно применяют. Внедрение AI-аналитики логов health check — шаг к автоматизации аудита, а не к замене человека.
