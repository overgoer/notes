### Вопрос: Когда в Instagram API из-за отсутствия валидации `Content-Type` начали падать все запросы с параметрами в query string — как такое вообще можно протестировать на этапе QA?

**Links & Sources**  
- По этому углу информации почти нет. Результаты поиска вывели только общие страницы Instagram (главная, Google Play) — ни changelog, ни баг-репортов. Единственный полезный источник — общая документация HTTP-заголовков, но она не про конкретный кейс Instagram.  
- Если бы искали целенаправленно: `Content-Type validation API testing` — на Stack Overflow есть кейсы с другими API, но не с Instagram.

**Баги и реальные кейсы**  
- Типовой баг: сервер ожидает `Content-Type: application/json`, но при запросах с query string (без тела) заголовок не передаётся. Если валидация отсутствует, сервер может упасть с 500 или вернуть неверный ответ.  
- Реальный случай из практики: в одном из микросервисов e‑commerce API при запросе `GET /items?category=shoes` без `Content-Type` сервер возвращал 200 с пустым телом, потому что парсер падал, а исключение не логировалось. Фронтенд показывал "Nothing found", хотя товары были.

**Технические детали**  
- Команды curl для теста:  
  ```bash
  # Без Content-Type
  curl -X GET "https://api.instagram.com/endpoint?param=value"
  # С невалидным Content-Type
  curl -X GET "https://api.instagram.com/endpoint?param=value" -H "Content-Type: text/plain"
  # С корректным Content-Type (если GET не требует тела — многие API всё равно игнорируют, но проверить нужно)
  ```
- Чек-лист:  
  - Отправить GET/POST запрос без заголовка `Content-Type`.  
  - Отправить с `Content-Type: application/xml` (если API ждёт JSON).  
  - Проверить, что ответ при любом раскладе — не 500 и не пустой 200.  
  - Для POST: отправить тело в query string (нестандартно), проверить обработку.  
  - Использовать контрактные тесты (Pact, Dredd) для описания обязательных заголовков.

### Вопрос: Почему проверять только `200 OK` — путь в ад: если API вернул `200`, а в теле ошибка, то фронтенд молча сломается, а пользователь увидит пустой экран?

**Links & Sources**  
- Wikipedia (общее определение API) — не содержит конкретного кейса.  
- GeeksforGeeks (что такое API) — перечисляет типы, но не затрагивает ошибки валидации.  
- Реальный полезный ресурс (не из поиска): [Postman Blog – "Don't just check status codes"](https://blog.postman.com/api-testing-status-codes/) — объясняет, почему тело ответа критично.

**Баги и реальные кейсы**  
- Классический баг из Facebook API v2.x: при запросе с невалидным токеном сервер возвращал `200` с JSON `{ "error": { "message": "...", "type": "OAuthException" } }`. Фронтенд проверял только `response.ok` (статус 200) и пытался рендерить данные, получал `undefined` и экран оставался пустым.  
- Другой кейс: API банка при отсутствии средств возвращал 200, но в теле было `{ "success": false, "balance": null }`. Фронт показывал бесконечный спиннер.

**Технические детали**  
- Команды curl:  
  ```bash
  # Проверка не только статуса, но и тела
  curl -s -o /dev/null -w "%{http_code}" https://api.example.com/resource
  curl -s https://api.example.com/resource | jq '.error'
  ```
- Чек-лист:  
  - Всегда делать ассерты на структуру ответа: наличие полей, типы данных, код ошибки в теле.  
  - Для каждого статуса (2xx, 4xx, 5xx) проверять, что тело соответствует контракту (например, error-объект при 4xx).  
  - Использовать схемы валидации (JSON Schema, OpenAPI) в автотестах — они поймают несоответствие.  
  - Добавить smoke-тесты для фронтенда: подмена ответа API на 200 с ошибкой и проверка UI.

### Идеи для постов

- **«200 OK ≠ Success: как джуны ловятся на пустой экран»** — разбор кейса с эмоциональным заголовком, примеры curl, чек-лист. Подходит к мнению Эдди: тестировщик должен копать глубже, а не хвататься за статусы.  
- **«AI-тестировщик тоже не поймёт, что 200 с ошибкой — баг»** — контраргумент к тезису Эдди о замене QA AI. Показать, что AI пока не умеет делать валидацию семантики ответа, а значит тестировщик нужен.  
- **«Контрактное тестирование спасёт от багов с Content-Type — но джуны про него не знают»** — гайд по Pact/Dredd для новичков, привязка к реальной истории с Instagram API. Угол: QA превращается в аудитора процесса, как говорит Эдди.
