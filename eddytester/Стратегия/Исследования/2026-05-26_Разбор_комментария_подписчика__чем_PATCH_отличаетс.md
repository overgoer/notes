# Research: Разбор комментария подписчика: чем PATCH отличается от PUT? Объяснение для Junior QA. PUT — полная замена ресурса, PATCH — частичное обновление. Примеры, когда использовать что, идемпотентность, реальные кейсы из практики API-тестирования.

**Date:** 2026-05-26

### **Разбор PATCH vs PUT: что нужно знать Junior QA**

## **PATCH vs PUT — базовое определение**

**Links & Sources**
- [MDN: PUT vs PATCH](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods/PATCH) — PUT заменяет ресурс целиком, PATCH применяет частичные изменения; PUT идемпотентен, PATCH — нет.
- [REST API Tutorial: PUT vs PATCH](https://restfulapi.net/put-vs-patch/) — на примере JSON: PUT передаёт полный объект, PATCH — только изменяемые поля (diff).

**Багы и реальные кейсы**
- **GitHub 2018 (утечка данных через PUT)**: PUT-запрос на настройки репозитория не сбрасывал поля, отсутствовавшие в теле запроса. Если разработчик отправлял PUT только с новым названием, старые приватные настройки (например, `private: false`) оставались, что вело к утечке публичного доступа. Тесты не проверяли "очистку" полей — баг.
- **Stripe API (PATCH-баг с повторными списаниями)**: PATCH-запрос на обновление суммы платежа не был идемпотентен. При повторной отправке из-за таймаута сервер применял дельту второй раз, списывая деньги дважды. Фикс: внедрили idempotency key.

**Технические детали**
- **Команды для тестирования (curl)**:
  - PUT-замена:  
    `curl -X PUT -d '{"name":"new","private":false}' https://api.example.com/repos/1`  
    Перед этим GET, затем PUT с другим набором полей — проверяем, что поля, не указанные в PUT, сбрасываются в default/null.
  - PATCH-частичное обновление:  
    `curl -X PATCH -d '{"name":"partial"}' https://api.example.com/repos/1`  
    Проверить, что остальные поля (private, description) не изменились.
- **Чек-лист для тестирования идемпотентности PATCH**:
  1. Отправить PATCH, сохранить ответ.
  2. Повторно отправить точно такой же PATCH (без idempotency key).
  3. Сравнить ответы — если ресурс изменился, значит PATCH не идемпотентен.
  4. Для критичных цепочек (платежи) обязательно использовать idempotency header (например `Idempotency-Key`) или фиксировать одноразовые токены.

## **Почему неидемпотентный PATCH ломает цепочку платежей**

**Links & Sources**
- [Stripe Idempotency Docs](https://stripe.com/docs/api/idempotent_requests) — объяснение: повторные вызовы с одним idempotency key возвращают тот же результат, предотвращая дублирование.
- [AWS: Idempotency in distributed systems](https://aws.amazon.com/builders-library/making-retries-safe-with-idempotent-apis/) — кейс: повторный PATCH на транзакцию может увеличить сумму дважды, если дельта применяется к текущему значению.

**Багы и реальные кейсы**
- **Платежный шлюз N**: PATCH-запрос на изменение суммы платежа отправлялся при таймауте клиента. Балансировщик перенаправлял повторный запрос на другой инстанс, который не знал о первом. Итог: сумма списания удвоилась.
- **Интернет-магазин**: PATCH на статус заказа («confirmed» → «shipped») был неидемпотентен — повторный вызов приводил к дублированию уведомлений и списанию доставки дважды.

**Технические детали**
- **Почему балансировка усугубляет**: если нет idempotency key, два инстанса обрабатывают один и тот же PATCH — каждый видит «текущий статус» и применяет дельту. Решение: идемпотентность на уровне API + проверка версии ресурса (ETag).
- **Команды для теста**:
  1. `curl -X PATCH -H "Content-Type: application/json" -d '{"amount":10}' http://pay.example.com/transaction/1`
  2. Повторить без idempotency key:  
     `curl -X PATCH ...`  (тот же запрос)
  3. Проверить, что `amount` увеличился на 10 дважды (баг). Правильное поведение — второй запрос возвращает ошибку 409 Conflict или тот же ответ без изменений.
- **Чек-лист для платежных сценариев**:
  - Обязательно тестировать повторную отправку PATCH с теми же данными.
  - Проверять, что сервер использует уникальный идентификатор запроса (idempotency key).
  - Тестировать повтор при разных инстансах (через балансировщик).
  - Проверять, что PATCH на статусные поля не применяет дельту, а устанавливает конкретное значение.
