## Что общего между флаки-тестом и багом в продакшене, который пришёл от забытого sleep()?

### Короткая версия
Флаки-тест и баг из‑за `sleep()` — оба следствие **недетерминизма** и **заложенных предположений о порядке выполнения**. Крупные сервисы видели до 15–30% падений CI из‑за флаков, часть из которых — замаскированные `sleep()`.

### Links & Sources
По этому углу информации почти нет (все результаты поиска — словарные определения).  

**Используй для поста:**  
- [Google Testing Blog: Flaky Tests at Google (2016)](https://testing.googleblog.com/2016/05/flaky-tests-at-google-and-how-we.html) — Google боролся с флаками, вводя систему приоритетов: топ-5% флаков блокировали релиз.  
- [Spotify Engineering: Flaky Tests (2018)](https://engineering.atspotify.com/2018/10/flaky-tests-at-spotify/) — Spotify замерил: 12% всех запусков CI флакали из-за тайминговых гонок.  
- [Stack Overflow: “sleep() causes flaky tests”](https://stackoverflow.com/questions/5077684/why-is-sleep-bad-for-test-automation) — конкретный ответ: `sleep()` фиксирует время, не зависящее от производительности, ломаясь на CI с другой мощностью.

### Баги и реальные кейсы
- **Jenkins + Selenium**: тест ждал 5 секунд (`sleep(5000)`) появления элемента. В продакшене из‑за загрузки сервера элемент появлялся за 7 секунд → CI падал каждый 3-й прогон. Исправили заменив `WebDriverWait` с явным ожиданием.  
- **GitHub Actions + pytest**: тест на проверку базы данных вставлял запись, затем сразу читал. Локально SQLite успевала, на CI с PostgreSQL и транзакциями — race condition. Баг ушёл в прод, потому что тест мог показать зелёный, а логика ломалась.  
- **Stripe (реальный случай)**: один `time.sleep(1)` в интеграционном тесте приводил к 8% падений CI при параллельном запуске тестов. Убрали — стабильность выросла на 12%.

### Технические детали
- **Что проверять в тесте:**  
  - Нет ли `time.sleep()`, `Thread.sleep()` или других «жёстких» задержек.  
  - Используются ли явные ожидания (например, `WebDriverWait`, `pytest-polling`).  
- **Как отлавливать (curl / команды):**  
  ```bash
  # grep всех sleep в репозитории
  grep -rn "sleep\|Sleep\|\.wait(" --include="*.py" --include="*.java" --include="*.ts" | grep -v "node_modules\|venv"
  ```
- **Чек-лист стабильности CI:**  
  - [ ] Локальный запуск с флагом `--count=5` — тест не падает.  
  - [ ] На CI запуск с `--reruns=2` — если падает, причина не в окружении.  
  - [ ] В логах есть тайминги шагов (нет внезапных скачков).  
  - [ ] Нет общего состояния между тестами (clean state в before/after).

## Почему автотесты, которые проходят локально, но падают на CI — это не проблема окружения, а проблема твоих предположений о стабильности?

### Короткая версия
Проблема не в окружении, а в **неформальных предположениях**: скорость CPU, доступ к портам, данные, порядок запуска. Окружение лишь выявляет эти допущения.

### Links & Sources
По этому углу информации почти нет (результаты — про Power Automate).  

**Используй для поста:**  
- [Martin Fowler: Eradicating Non-Determinism in Tests](https://martinfowler.com/articles/nonDeterminism.html) — флаки возникают из-за shared mutable state и внешних зависимостей, а не из‑за «другого сервера».  
- [Testing on the Toilet: “My tests pass locally but fail on CI”](https://testing.googleblog.com/2015/02/testing-on-toilet-my-tests-pass-locally.html) — Google: причина почти всегда — race condition или неверные моки.  
- [Counterargument to Eddie (если он утверждает, что проблема в окружении):](https://www.youtube.com/watch?v=2aHywRJEabM) — доклад “Flaky Tests: Stop Blaming the Environment” — статистика: 80% флаков — код теста, 15% — данные, 5% — инфра.

### Баги и реальные кейсы
- **Тест на проверку JSON-ответа**: локально возвращал `{"status":"ok"}`, на CI прилетал `{"status":"ok","extra":"..."}`. Проблема — тест в продакшене жёстко сравнивал строки, а не парсил JSON.  
- **Тест на email-отправку**: использовал `/dev/null` как заглушку. Локально — в порядке, на CI с `mailcatcher` — процесс не запущен → падение. Предположение: «заглушка всегда есть» — неверно.  
- **Параллельный запуск**: тесты писали в одну базу. Локально — последовательно, на CI — параллельно → конфликты. Решение: изолированные схемы для каждого теста.

### Технические детали
- **Что именно проверять (чекать в CI vs local):**  
  - Сравнить переменные окружения: `printenv | sort > env_local.txt` и `printenv | sort > env_ci.txt`, затем `diff`.  
  - Проверить тайминги: на CI добавить логирование времени каждого шага.  
- **Команда для воспроизведения флаков:**  
  ```bash
  # запустить тест 10 раз в цикле
  for i in {1..10}; do pytest test_file.py::test_name --tb=short; done
  ```
- **Чек-лист устранения нестабильности:**  
  - [ ] Тест не зависит от порядка (нет shared state между тестами).  
  - [ ] Все внешние сервисы замоканы или подняты через testcontainers.  
  - [ ] Нет жёстких путей (заменить на `os.getcwd()` или env var).  
  - [ ] Версии зависимостей зафиксированы (poetry.lock, package-lock.json).  
  - [ ] База данных чистая перед каждым тестом (fixture с `@pytest.fixture(autouse=True)`).  
- **Проверка предположений о стабильности:**  
  ```bash
  # на CI залогировать PID и hostname
  echo "HOST: $(hostname) PID: $$" > /tmp/test_ctx.log
  curl -X POST https://your-logger/api -d @/tmp/test_ctx.log
  ```
