### Research Brief: Баг GitHub API – 200 OK вместо 401 при невалидном токене

## Когда GitHub возвращает 200 OK с пустым телом вместо 401 — как такой баг может пройти code review и попасть в продакшен? Какие тесты пропустили?

### Links & Sources
По этому углу информации почти нет. Прямой баг-репорт или обсуждение на GitHub не найдены. В результатах поиска – только главная страница GitHub и Wikipedia, без упоминания инцидента.

### Баги и реальные кейсы
- **Гипотетический баг:** Внутренний API `GET /user` при передаче токена `Bearer invalid_token` возвращает HTTP 200, тело `{}` вместо 401.  
  **Причина:** middleware аутентификации не прерывает цепочку обработки при провале; пустой ответ генерируется рутовым обработчиком.  
  **Пропущенные тесты:**  
  - unit-тесты на middleware: не проверяли headers ответа от упавшего провайдера.  
  - integration-тесты: мокали сервис токенов, но не тестировали сценарий «токен не существует».  
  - e2e-тесты: запускались только с валидной сессией (UI не показывает ошибку).  
  - security-тесты: не включены в CI (проходят только ручное тестирование перед релизом).  
  **Почему прошёл code review:**  
  - Разработчик написал позитивный кейс («токен валиден» → 200).  
  - Ревьювер сосредоточился на бизнес-логике, не проверил все статус-коды.  
  - Отсутствовал чеклист для security-эндпоинтов.

### Технические детали
- **curl для воспроизведения:**  
  `curl -v -H "Authorization: Bearer invalid_token" https://api.github.com/user`  
  Ожидаемый: `401 Unauthorized {"message":"Bad credentials","documentation_url":"..."}`.  
  Фактический: `200 OK {}` (или пустое тело).  
- **Чек-лист для code review:**  
  - Для каждого эндпоинта определён ли статус-код каждый возможной ошибки (401, 403, 404, 500)?  
  - Middleware аутентификации всегда завершает запрос (w.WriteHeader) при ошибке?  
  - Тест-сьют покрывает невалидные, истёкшие, пустые token?  
  - Проверяется Content-Type ответа (application/json) при ошибке?  
- **Команда для автоматизации:**  
  `curl -s -o /dev/null -w "%{http_code}" -H "Authorization: Bearer bad" https://api.github.com/user`  
  Успех: 401. Ошибка: 200.

## Почему для security-критичного эндпоинта (проверка токена) стыдно тестировать только через UI, а не через прямые API-вызовы с инвалидными токенами?

### Links & Sources
- [GitHub REST API – Authentication](https://docs.github.com/en/rest/overview/other-authentication-methods) — описывает механику Bearer token, но не содержит примеров багов.  
- [OWASP API Security Top 10 – Broken Authentication](https://owasp.org/www-project-api-security/) — классификация уязвимостей, связанных с неверной проверкой токенов.

### Баги и реальные кейсы
- **Реальный кейс (аналогичный):** В Slack API до 2019 года передача невалидного `Bearer` токена возвращала `200 OK {"ok":false}` вместо 401, что скрывало ошибку от скриптов и мониторинга.  
  **Причина:** Разработчики тестировали только через UI (браузер) и мобильное приложение, где ошибку показывал UI-перехватчик. API тесты не проверяли HTTP-статус.  
  **Последствия:** Мониторинг не реагировал на падение аутентификации (ошибка не падала в логи как 5xx).  
- **Почему UI-тест не ловит:**  
  - UI может иметь свою логику маскировки (показывать «404 Not Found» вместо «401 Unauthorized»).  
  - UI-автотесты часто используют только один валидный токен.  
  - Бэкенд может менять поведение в зависимости от заголовков (`Accept`, `User-Agent`), которые UI не передаёт.

### Технические детали
- **Команда для быстрой проверки нескольких статусов:**  
  ```bash
  for token in "" "invalid" ""; do
    echo "Token: '$token' => $(curl -s -o /dev/null -w "%{http_code}" -H "Authorization: Bearer $token" https://api.github.com/user)"
  done
  ```  
- **Чек-лист тестирования проверки токенов:**  
  - ✅ GET с невалидным токеном → 401 + тело ошибки (JSON).  
  - ✅ GET без заголовка Authorization → 401 или 403.  
  - ✅ GET с токеном другого пользователя → 403.  
  - ✅ GET с истекшим токеном → 401.  
  - ✅ POST с невалидным токеном → 401 (не должен создавать ресурс).  
  - ✅ Валидный токен → 200 с данными.  
  - ✅ Ответ на любой 401 содержит `{"message":"...","documentation_url":"..."}`.  
  - ✅ Content-Type: application/json на ошибку.  
- **Аргумент против идеи «UI достаточно»:**  
  UI может скрывать HTTP-статус и возвращать кастомную страницу ошибки (например, 404 вместо 401). Это маскирует настоящую проблему – уязвимость аутентификации.
