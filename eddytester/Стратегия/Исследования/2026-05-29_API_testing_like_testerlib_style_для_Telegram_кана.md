### Когда ты шлёшь POST на создание юзера, а сервер возвращает 200, но пользователь не создаётся — как не пропустить такую «тихую неудачу»?

**Links & Sources**
- Поиск не дал релевантных материалов (все ссылки — New York Post, USPS — не относятся к API). По этому углу информации почти нет.

**Багы и реальные кейсы**
- Реальный кейс из практики: при регистрации пользователя сервер возвращал 200 OK, но запись в БД не появлялась из-за ошибки валидации на уровне репозитория (неверный тип данных для поля `created_at`). Тест, проверяющий только HTTP-статус, баг не ловил.
- Другой случай: POST-запрос на создание заказа проходил успешно, но из-за race condition (два одновременных запроса с одинаковым `order_id`) сервер возвращал 200, а второй заказ терялся без логирования.

**Технические детали**
- После получения 200 всегда выполняй GET на созданный ресурс (если есть ID в ответе) или делай запрос на список пользователей и проверяй наличие записи по полю `email`.
- Добавь в тест проверку бизнес-логики: убедись, что запись появилась в БД (например, через прямое подключение к тестовой базе: `SELECT * FROM users WHERE email = 'test@example.com'`).
- Чек-лист: 1) Проверить, что в ответе пришёл `id` (если он ожидается). 2) Выполнить GET-запрос по этому `id`. 3) Проверить логи сервера на наличие ошибок (например, grep `ERROR` в логах приложения). 4) Использовать `curl -v` для просмотра полных заголовков — иногда 200 приходит, но в теле есть `"status": "error"`.

### Если в документации написано «поле email опционально», а без него падает 500-я ошибка — как тестировать обязательность полей, когда документация врёт?

**Links & Sources**
- [What is an API (Application Programming Interface) — GeeksforGeeks](https://www.geeksforgeeks.org/software-testing/what-is-an-api/) — общее определение API, но нет конкретных кейсов расхождения документации и реализации.
- Поиск не дал специализированных материалов по этой проблеме. По этому углу информации почти нет.

**Багы и реальные кейсы**
- Классический баг: swagger-спецификация помечала поле `email` как `optional`, но на бэкенде была аннотация `@NotNull`, вызывающая 500 при отсутствии поля. Разработчик забыл синхронизировать документацию с кодом.
- В другом проекте документация утверждала, что поле `phone` необязательно, но сервис валидации падал с NullPointerException, так как ожидал строку, а получал `null`. Ошибка уходила в 500 из-за отсутствия try-catch.

**Технические детали**
- При тестировании обязательных полей всегда подавай запрос без поля, даже если в документации написано "optional". Проверяй HTTP-статус: 400 — корректная обработка, 500 — баг.
- Используй комбинации: пропустить поле, передать пустую строку, `null`, пробелы.
- Команда: `curl -X POST http://api/users -H "Content-Type: application/json" -d '{"name":"test"}'` — без поля `email`.
- Чек-лист: 1) Проверить, что вернулся человекочитаемый ответ (json с описанием ошибки, а не stacktrace). 2) Убедиться, что сервер не упал в 500 (это нарушение обработки ошибок). 3) Сравни с требованием: если документация врёт, заведи баг на документацию, а тест пиши по фактическому поведению (если поле должно быть обязательным по логике).

### Идеи для постов
- **«Тихие 200» — как не дать серверу обмануть тебя**: кейс, когда 200 ≠ успех, методы проверки (GET после POST, прямой SQL, логи).
- **Документация — это не закон, а гипотеза**: как тестировать обязательность полей, если доки врут, и почему 500 за опциональное поле — это баг, а не фича.
- **Junior vs реальность: что делать, если API работает не по спецификации**: практический чек-лист из 5 пунктов для проверки любого эндпоинта (не доверять статусу, проверять БД, валидировать ошибки).
