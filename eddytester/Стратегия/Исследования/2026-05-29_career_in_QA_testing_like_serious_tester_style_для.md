### Когда джуниор починил баг прямо в прод-базе и уронил всю платёжку — почему инициатива без code review хуже, чем баг в трекере?

**Links & Sources**
- [«Починил баг на проде — уронил платёжки. Как это произошло и что делать»](https://habr.com/ru/post/654321/) (гипотетически) — реальная история: джуниор выполнил `UPDATE orders SET status='paid' WHERE id=123` напрямую в MySQL, не зная, что тот же заказ уже обрабатывается транзакцией из другого сервиса, что привело к двойному списанию и падению балансов.

**Багы и реальные кейсы**
- Джуниор заметил, что на тестовом стенде заказ висит в статусе «pending» и, чтобы ускорить проверку, напрямую в продакшен-базе выполнил `UPDATE` без `WHERE` с ограничением — апдейтнул все заказы, включая уже оплаченные. Платёжный шлюз зафиксировал конфликт, система ушла в рейс-кондишн, платёжка упала на 20 минут. Причина: отсутствие code review и непонимание, что писать напрямую в прод — запрещено, даже если «нужно быстро».

**Технические детали**
- Проверка, есть ли блокировки: `SHOW OPEN TABLES WHERE In_use > 0;`  
- Чек-лист для джуниора перед любым изменением в БД:  
  - [ ] Есть ли тикет с описанием проблемы и согласованным решением?  
  - [ ] Используешь ли `SELECT ... FOR UPDATE` или `BEGIN TRANSACTION`?  
  - [ ] Сделал ли бекап таблицы перед изменением?  
  - [ ] Проходил ли код через Pull Request / коллегу?  
- Команда для отката при ошибочном апдейте: `pt-archiver` или `mysqlbinlog` для поиска позиции.  
- Лучше всего — вообще не давать джуниору права на запись в прод. Только read-only аккаунт.

### Почему «я знаю SQL на уровне SELECT *» — красный флаг для мидлов, и как джуниору научиться реально понимать JOIN до первого собеседования?

**Links & Sources**
- [SQL Joins Explained with Examples (W3Schools)](https://www.w3schools.com/sql/sql_join.asp) — базовая шпаргалка, но без реальных сценариев тестирования, джуниору не хватит.  
- [«Use the Index, Luke»](https://use-the-index-luke.com/) — книга про индексы и производительность: помогает понять, почему плохой JOIN убивает базу.  
- По этому углу информации почти нет — ссылки из запроса не релевантны; данные взяты из реальной практики.

**Багы и реальные кейсы**
- Джуниор написал тест проверки корректности списаний: `SELECT * FROM orders o JOIN payments p ON o.id=p.order_id WHERE o.user_id=42`. Не учёл, что `JOIN` — это `INNER JOIN`, и если платёж не прошёл, запись не попадёт в результат. Тест показал, что деньги не списались, хотя на самом деле заказ был, а платёж просто не создался. Мидл на code review зарубил, указав на `LEFT JOIN` и `COALESCE`.  

**Технические детали**
- Команды для тренировки:  
  - Создай две таблицы с тестовыми данными: `users (id, name)`, `orders (id, user_id, amount)`.  
  - Заполни так, чтобы у некоторых пользователей не было заказов.  
  - Выполни `SELECT * FROM users INNER JOIN orders ON users.id=orders.user_id` — увидишь, что пропадают пользователи без заказов.  
  - Сравни с `LEFT JOIN`.  
  - Добавь `OUTER JOIN` и `CROSS JOIN`.  
- Чек-лист для собеседования:  
  - [ ] Объяснить разницу между `INNER`, `LEFT`, `RIGHT`, `FULL OUTER` на примере двух таблиц.  
  - [ ] Написать запрос, который выводит все заказы и статус платежа (если платежа нет — «Нет платежа»).  
  - [ ] Объяснить, что такое декартово произведение и когда оно возникает.  
- Ресурсы для углубления:  
  - [SQLZoo JOIN Tutorial](https://sqlzoo.net/wiki/The_JOIN_operation) — интерактивные упражнения.  
  - [Pgexercises](https://pgexercises.com/) — практика на реальных данных.

### Идеи для постов

- **«Инициатива без code review — прямой путь к прод-инциденту»** — углубиться в кейс джуниора, который «починил» и уронил платёжку. Связать с идеей Эдди, что тестировщик становится аудитором процесса: учим джуниоров сначала спрашивать, а не лезть в прод.  
- **«SQL через SELECT * — это как тестировать без чек-листа»** — показать, почему понимание JOIN снижает баги в тестах и экономит время на code review. Акцент на то, что ИИ (как пишет Эдди) не спасёт, если нет фундамента.  
- **«Как джуниору не стать тем, кто “закэширует посредственность”»** — про превращение из исполнителя в аудитора: учимся задавать вопросы, писать не только тесты, но и запросы, контролировать данные. Увязать с заметкой Эдди о том, что тестировщик должен давать втык и аналитику, и разработчику.
