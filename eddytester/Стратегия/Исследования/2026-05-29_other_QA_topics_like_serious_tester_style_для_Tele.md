### Когда дважды отправляешь POST на `/create-order` — как проверить идемпотентность?

**Links & Sources**
- [Wikipedia: Idempotence](https://en.wikipedia.org/wiki/Idempotence) — формальное определение: операция идемпотентна, если многократное применение даёт тот же результат, что и однократное.
- [FreeCodeCamp: Idempotence Explained](https://www.freecodecamp.org/news/idempotence-explained) — реальные примеры: кнопка лифта, HTTP GET/PUT/DELETE, паттерны с очередями сообщений для обеспечения идемпотентности.
- [TheDataOps: Idempotency Guide](https://www.thedataops.org/idempotency/) — практика: как использовать Idempotency-Key в заголовке, как сервер должен возвращать один и тот же ответ (200 vs 201) при повторном запросе с тем же ключом.

**Багы и реальные кейсы**
- Stripe: без идемпотентности повторный POST `/create-payment-intent` списывает деньги дважды. Ошибка возникала при сетевых таймаутах — клиент ретраил, а сервер обрабатывал каждый запрос как новый.
- Тестовый API заказа: разработчик забыл проверять дубликаты по ключу идемпотентности → в БД упало два одинаковых заказа с разными ID. Пользователь получил две посылки, бизнес — возврат и репутационные потери.
- PayPal Idempotency Bug (исторический): в 2018 году из-за отсутствия идемпотентности на стороне мерчанта при повторной отправке подтверждения оплаты создавались дубликаты транзакций.

**Технические детали**
- Проверка идемпотентности через curl:
  ```bash
  # первый запрос
  curl -X POST https://api.example.com/create-order \
    -H "Idempotency-Key: uuid-123" \
    -H "Content-Type: application/json" \
    -d '{"item": "book"}'
  
  # второй запрос с тем же ключом
  curl -X POST https://api.example.com/create-order \
    -H "Idempotency-Key: uuid-123" \
    -H "Content-Type: application/json" \
    -d '{"item": "book"}'
  ```
- Ожидаемый результат: первый → 201 Created, второй → 200 OK (или 409 Conflict если не идемпотентно; но правильное поведение — 200 с теми же данными).
- Чек-лист:
  - Проверить с разными значениями Idempotency-Key (валидный UUID, пустая строка, null, спецсимволы).
  - Проверить поведение при частичном выполнении (сервер упал после записи в БД, но до отправки ответа) — ключ должен быть сохранён, повторный запрос вернёт сохранённый результат.
  - Убедиться, что после таймаута с тем же ключом не создаётся дубликат (common pitfall: сервер не чистит временные блокировки).
  - Нагрузочный тест: 1000 параллельных POST с одинаковым ключом → все должны получить один и тот же ответ, без дубликатов в БД.

### Почему тестеру важно различать `Cache-Control: no-cache` и `no-store`?

**Links & Sources**
- Предоставленные ссылки (Microsoft Edge, Google Support, ComputerHope) описывают только очистку кэша браузера, а не разницу заголовков. По тонкостям Cache-Control информации в них нет.
- [RFC 7234 — HTTP Caching](https://datatracker.ietf.org/doc/html/rfc7234) (не в предоставленных данных, но обязателен): `no-cache` разрешает кэширование, но требует каждый раз проверять свежесть у сервера (conditional GET). `no-store` запрещает любое сохранение ответа.

**Багы и реальные кейсы**
- Банковское приложение: на странице баланса поставили `Cache-Control: no-cache`. Это привело к тому, что прокси-сервер сохранял ответ, но при каждом запросе делал conditional GET. Баг: при офлайн-режиме пользователь видел старый баланс (потому что прокси отдавал сохранённую копию при ошибке соединения). Нужно было `no-store`.
- E‑commerce: статичные картинки товаров отдавались с `no-store` → каждый заход генерировал полноценный запрос к CDN, убивая производительность (TTFB вырос в 3 раза). Исправление: `Cache-Control: public, max-age=31536000, immutable`.
- API для авторизации: сервер отдавал токен с `Cache-Control: no-cache`. Браузер честно проверял свежесть, но если сервер отвечал 304 Not Modified — токен не обновлялся, и пользователь вылетал через час. Должно быть `no-store` + `private`.

**Технические детали**
- Проверка через curl:
  ```bash
  curl -I https://example.com/page
  ```
  Смотрим заголовок `Cache-Control`. Для no-cache также ищите `Pragma: no-cache` (устаревший аналог).
- Нагрузочный тест: разница между no-cache и no-store видна на повторных запросах. Используйте JMeter или k6:
  - С no-cache: каждый запрос делает conditional GET (если есть ETag или Last-Modified) → нагрузка на сервер меньше, но latency выше (из-за round-trip для проверки).
  - С no-store: каждый запрос полноценный → нагрузка на сервер максимальна.
- Чек-лист для тестирования кэширования:
  - Проверить, что `no-store` не кэшируется ни на одном уровне (браузер, CDN, прокси).
  - Для `no-cache`: проверить, что при отсутствии сети браузер **не** отдаёт кэш (должен показать ошибку, а не закэшированную версию).
  - Инструменты: Chrome DevTools → Network → Disable cache (только для dev, не отражает реальное поведение с no-cache).
  - Проверить, что сервер возвращает валидные `ETag` / `Last-Modified` для поддержки conditional GET при `no-cache`.

### Идеи для постов
- **«Idempotency-Key — не магия, а твой щит от дублей»** — показать на примере Stripe vs своего API: как один заголовок спасает деньги и нервы. Угол: тестировщик должен знать не только UI, но и протоколы (пересекается с «аудитор процесса»).
- **«Cache-Control: no-cache — это не “не кэшировать”, а “кэшировать, но проверять”»** — разбор типовой ошибки джунов, когда блокировка кэша ломает производительность. Контраргумент к мнению «QA не нужно знать HTTP-заголовки».
- **«AI в тестировании: замена или инструмент? Разбираем на багах»** — взять кейс с идемпотентностью и показать, как AI (например, автотесты на базе LLM) может генерировать проверки, но не гарантирует отсутствия дублей. Спор с Эдди: AI не бесполезен, но требует ручного контроля и понимания контекста.
