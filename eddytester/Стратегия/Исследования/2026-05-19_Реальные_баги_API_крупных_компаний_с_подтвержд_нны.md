### Как отловить регрессию 401 на 0.1% запросов (GitHub, 2018)

**Links & Sources**
- [Incident report: API authentication (GitHub Blog)](https://github.blog/2018-11-01-incident-report-api-authentication/) — после деплоя часть валидных токенов не обновлялась в кеше → 401; баг проявлялся на ~0.1% запросов из-за race condition между инвалидацией и загрузкой нового токена.

**Багы и реальные кейсы**
- Дефект: сервер кешировал результат проверки токена; при смене версии аутентификационного модуля часть кеша не сбрасывалась. Пользователи со старым типом токена получали 401, хотя токен был валидным.
- Почему сложно: сценарий проявлялся только при совпадении момента запроса и состояния кеша — нагрузочное тестирование с постоянным потоком не гарантирует воспроизведение.

**Технические детали**
- **Чек-лист**: 1) нагрузочный тест с ротацией 1000+ уникальных токенов; 2) параллельно мониторинг ошибок 401 в логах API-гейта; 3) canary-деплой с постепенным переключением трафика и сравнением процента 401 с предыдущей версией; 4) тест на «холодный» кеш — полная очистка кеша перед деплоем, затем первый запрос с каждым токеном.
- **Команды**: `wrk -t4 -c100 -d60s --latency --header "Authorization: token $RANDOM_TOKEN" https://api.github.com/user` → плюс парсинг статусов. Для canary — `curl -X POST -H "Authorization: Bearer ..." -d '{"percentage":5}' ...`.

### Stripe: ошибка идемпотентности с разным телом и одним ключом (2016)

**Links & Sources**
- [Idempotency (Stripe Blog)](https://stripe.com/blog/idempotency) — баг: повтор POST с тем же `Idempotency-Key`, но изменёнными данными (`amount`, `currency`) создавал новый charge вместо возврата первого ответа.

**Багы и реальные кейсы**
- Инцидент: Stripe хранил результат по ключу, но не проверял хеш тела запроса. Если клиент шёл с тем же ключом и другим телом — сервер обрабатывал как новый запрос, создавая двойные списания.
- Почему опасно: клиенты могли случайно отправить повторно с изменённой суммой (например, из-за таймаута и повторной генерации формы) — и списать деньги дважды.

**Технические детали**
- **Curl-тест**:  
  1. `curl -X POST https://api.stripe.com/v1/charges -H "Idempotency-Key: test1" -d "amount=1000¤cy=usd"` → получили charge_id.  
  2. `curl -X POST https://api.stripe.com/v1/charges -H "Idempotency-Key: test1" -d "amount=2000¤cy=usd" ` → ожидаем 200 с тем же charge_id и той же суммой.  
- **Чек-лист**: 1) проверка, что повтор с другим телом не дублирует операцию; 2) проверка, что ответ сервера идентичен (статус, тело) первому успешному; 3) тест с истекшим ключом (TTL) — повтор должен создать новый ресурс.

### Идеи для постов
- **«Почему ИИ не отловит GitHub-баг 401 на 0.1%»** — даже крутой автотест с 100% coverage не найдёт race condition в кеше без знания архитектуры и золотой пули «canary + мониторинг».
- **«Idempotency testing: скилл, который выделяет сеньора»** — пример Stripe показывает, что проверка одного ключа без хеша тела — дыра в безопасности; контраргумент Эдди: ИИ мог бы сгенерить такие тесты, но он не понимает контекст бизнеса (двойное списание).
- **«Как тестировать то, что невозможно воспроизвести в лаборатории»** — инструментарий: chaos engineering, feature flags, мониторинг процентилей ошибок.
