# Research: автоматизация тестирования API — ручное тестирование vs автотесты, что можно автоматизировать в бэкенде, простые техники

**Date:** 2026-05-30

---

### Когда 100% автотестов пропустили баг в заголовке — какие проверки нельзя автоматизировать?

**Links & Sources**
- [GeeksforGeeks: API Testing](https://www.geeksforgeeks.org/software-engineering/api-testing-software-testing/) — автотесты покрывают стандартные коды и JSON, но не проверяют кастомные заголовки, если их нет в спецификации.
- [Postman: What is API Testing](https://www.postman.com/api-platform/api-testing/) — ручное тестирование выявляет баги в недокументированных полях ответа, которые не заложены в автотестах.
- [Qase: API Testing Best Practices](https://www.qase.io/blog/a-comprehensive-guide-to-api-testing-best-practices-tools-challenges-and-more/) — причина: автотесты пишутся под известные сценарии, а заголовки часто остаются «серой зоной».

**Багы и реальные кейсы**
- В REST API сервер возвращал `Cache-Control: no-store` вместо `no-cache`. Кеширующий прокси начал отдавать устаревшие данные. Автотест проверял только status code и body, баг нашли через пару недель ручным просмотром заголовков.
- Ещё случай: при ошибке авторизации API возвращал некорректный заголовок `WWW-Authenticate` (отсутствовал realm). Клиенты не могли корректно обработать 401. Автотесты все проходили — они смотрели только код 401.

**Технические детали**
- Чек-лист для заголовков ответа: `Content-Type`, `Cache-Control`, `Access-Control-Allow-Origin`, `WWW-Authenticate`, `X-Request-ID`, `X-RateLimit-Remaining` и кастомные.
- Команда curl для ручной проверки всех заголовков:  
  `curl -I https://api.example.com/endpoint`  
  `curl -D - https://api.example.com/endpoint 2>&1 | grep -i "cache"`
- В автотестах обязательно добавлять ассерты на заголовки:  
  `pm.response.headers.get('Cache-Control')` (Postman)  
  `assert response.headers['Cache-Control'] == 'no-cache'` (Python requests)
- Почему автотесты часто пропускают: заголовки считаются «инфраструктурными», их редко закладывают в тест-кейсы. Нужно включать их в регресс как обязательные проверки.

### Стратегия: 80% багов находит ручное исследование — как не упустить критичное при автотестах?

**Links & Sources**
- [Atlassian: Types of Software Testing](https://www.atlassian.com/continuous-delivery/software-testing/types-of-software-testing) — manual vs automated: автоматизация даёт скорость, но не заменяет исследовательское тестирование новых сценариев.
- [BrowserStack: Different Types of Testing](https://www.browserstack.com/guide/types-of-testing) — risk-based testing: автотесты покрывают регресс, ручное — критические пути и граничные случаи.
- [Medium: Backend Testing Techniques](https://medium.com/theymakedesign/what-is-backend-testing-techniques-and-tools-8c9718d71177) — практика: автотесты ловят 20% багов, остальное — ручная проверка бизнес-логики и недокументированных состояний.

**Багы и реальные кейсы**
- На проекте автотесты покрывали 95% API, но релиз упал — забыли обработать случай, когда база данных возвращает `null` вместо массива. Ручное исследование (session-based test) сразу выявило этот кейс, а автотест использовал `assert response.json()['items']` — падал только при пустом массиве.
- Другой кейс: ручное тестирование нашло, что при превышении лимита запросов API возвращает `429` с телом `{"error":"rate limit"}` — ни один автотест не проверял этот сценарий, потому что его не было в требованиях.

**Технические детали**
- Стратегия: **risk-based + session-based testing**.
  - Автотесты: только критические эндпоинты (login, payment, core CRUD) + регресс. Писать их **после** того, как ручное исследование подтвердило корректное поведение.
  - Ручное: 2-4 часовые сессии на каждую новую фичу. Инструменты: Postman, curl, скрипты на Python для генерации тестовых данных.
- Пример чек-листа для ручного исследования:
  - Все HTTP-методы GET/POST/PUT/DELETE на ресурс.
  - Пустое тело, невалидный JSON, SQL injection в параметрах.
  - Сбои БД (отключить сервис, вернуть таймаут).
  - Заголовки авторизации (отсутствуют, просрочены, невалидный токен).
- Команда для имитации сбоя:  
  `curl -X POST https://api.example.com/orders -H "Content-Type: application/json" -d 'invalid'`  
  `curl -X DELETE https://api.example.com/orders/999999` (несуществующий ID)

### Идеи для постов
- **«Автотесты — это не магия: когда 100% coverage не спасает»** — реальные баги в заголовках, таймаутах, нулевых данных. Почему нужно оставлять ручные проверки на граничные сценарии.
- **«Стратегия для бэкенд-тестировщика: 20% автотестов + 80% ручного исследования»** — аргументировать, что автоматизация регресса не заменяет исследовательские сессии, особенно при неполных требованиях. Контраргумент к мнению «ручное тестирование умрёт».
- **«Как не пропустить баг, который не заложен в тест-кейсе»** — техники session-based testing, risk-based подход, чек-листы для ручного тестирования API (заголовки, коды ответа, обработка ошибок). Без воды, только конкретные команды curl и примеры.
