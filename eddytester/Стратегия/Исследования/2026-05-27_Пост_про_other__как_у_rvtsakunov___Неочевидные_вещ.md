### Когда разработчик говорит «это фича» — как доказать опасность до падения прода

**Links & Sources**
- По данному углу информации почти нет. (Можно было бы сослаться на "Technical Debt Quadrants" Фаулера, но без прямой привязки к legacy.)

**Багы и реальные кейсы**
- Кейс: legacy-функция форматирования даты возвращала "01.01.1970" при null. Разработчик: «так и задумано». Через год при миграции null стали частыми — прод упал с ошибкой валидации. Баг не считался багом, пока не стал критичным. Решение: property-based тест подсветил нарушение контракта (должно бросать исключение или пустую строку).

**Технические детали**
- Чек-лист доказательства:
  1. Зафиксировать спецификацию (если нет — написать ожидаемое поведение).
  2. Написать модульный тест на краевые случаи (null, некорректный формат).
  3. Добавить мониторинг аномальных входных данных в проде (OpenTelemetry, feature flag).
  4. Оформить баг-репорт с результатами теста и ссылкой на логи (если доступны).
- Команды для поиска: `grep -rn "formatDate" --include="*.php"` + `php -r 'include "legacy.php"; echo formatDate(null);'`
- Assertion для отладки: `assert(formatDate(null) === '');`

### Если баг не воспроизводится в тестовом окружении — как написать тест, который поймает его без логов прода

**Links & Sources**
- По данному углу релевантных источников нет (ссылки из поиска — про насекомых).

**Багы и реальные кейсы**
- Кейс: гонка данных при бронировании билетов — на проде раз в неделю дубликат primary key. В тестовой среде нагрузка низкая, не воспроизводится. Решение: добавлен unique constraint в БД (фикс), а для теста — параллельные потоки с `CountDownLatch` (JUnit). Логи SQL помогли найти момент совпадения ключей.

**Технические детали**
- Включение логов MySQL (осторожно, нагрузка):
  ```sql
  SET GLOBAL general_log = 'ON';
  SET GLOBAL log_output = 'TABLE';
  SELECT * FROM mysql.general_log WHERE argument LIKE '%INSERT INTO bookings%';
  ```
- Чек-лист теста:
  1. Определить критические участки (возможные race conditions).
  2. Использовать библиотеки параллельных тестов (pytest-xdist, kotlinx.coroutines).
  3. Запускать тест 1000 раз с разными задержками.
  4. Эмулировать нагрузку JMeter/Gatling (5 потоков, 10 итераций).
  5. Чистить состояние БД между прогонами.
- Если логов нет — добавить временный дебаг-лог через feature flag:
  ```python
  import logging
  logging.debug(f"Start booking user {user_id} at {time.time()}")
  ```

### Идеи для постов
- **Legacy-археология как суперсила джуна**: разбираться в старом коде и доказывать опасность «фич» — навык, выделяющий на фоне. Пересечение с мнением Эдди: QA перерастает в аудитора процесса — legacy требует аудита.
- **Баг не воспроизводится? Сделай тест, который его нервирует**: property-based и параллельные тесты для гонок. Контраргумент к идее Эдди про замену тестировщиков AI: именно такие эвристические тесты AI проектирует плохо.
- **Психология бага: как переубедить разработчика**: методика «доказательство от метрик» — не "это баг", а "вот тест, ловящий NPE на проде". В противовес хайпу AI: реальные проблемы решает дисциплина и документация, а не нейросети.
