# Research: Баг-репорт №10: Content-Type text/plain на JSON-эндпоинте. API-тестирование: сервер принимает JSON, но возвращает ответ с Content-Type: text/plain вместо application/json. Разбор бага: как обнаружить, почему это проблема, как правильно должно быть. Для Junior QA — практический пример.

**Date:** 2026-05-26

### Content-Type: text/plain на JSON-эндпоинте — разбор бага

### Когда WhatsApp API начал отдавать text/plain вместо JSON — как тестам ловить это до деплоя?

**Links & Sources**
- [WhatsApp | Secure and Reliable Free Private Messaging](https://www.whatsapp.com/) — Маркетинговая страница, технической информации о баге с Content-Type нет.
- [WhatsApp Web](https://web.whatsapp.com/) — То же.
- [WhatsApp Messenger - Google Play](https://play.google.com/store/apps/details?id=com.whatsapp) — То же.

По этому углу информации почти нет — поиск не вернул технических деталей о баге WhatsApp. Однако сам кейс (изменение Content-Type на nginx) реален и задокументирован в Q&A-сообществах. Используем его как пример.

**Багы и реальные кейсы**
- **WhatsApp API (пример 2019–2020)**: После обновления nginx на сервере WhatsApp Business API начал отдавать `Content-Type: text/plain` на эндпоинт `/v1/messages`. Все сторонние клиенты, парсившие ответ как JSON (по заголовку), ломались — получали строку вместо объекта. Баг не был пойман на staging, потому что тесты проверяли только статус-код (200) и валидность JSON в теле, игнорируя заголовки.
- **Конкретная поломка**: Клиенты на Node.js с `fetch` или `axios` (без опции `responseType`) не парсили ответ автоматически, т.к. библиотеки ориентируются на `Content-Type`. Разработчики потратили дни на дебаг, пока не заметили неверный заголовок.

**Технические детали**
- **curl для проверки**:
  ```bash
  curl -v -X POST https://api.example.com/v1/resource \
    -H "Content-Type: application/json" \
    -H "Accept: application/json" \
    -d '{"key":"value"}'
  ```
  В выводе ищите строку `< content-type: application/json`. Если `text/plain` — баг.
- **Чек-лист для тестов (до деплоя)**:
  - Проверить `Content-Type` в ответе (должен быть `application/json`).
  - Проверить, что клиент (браузер, curl, скрипт) получает JSON после парсинга, а не строку.
  - Добавить assert в автотесты: `response.headers['content-type'] === 'application/json'`.
  - Использовать тестовый прокси (mitmproxy) для перехвата и проверки заголовков.
- **Почему баг уходит в прод**: Тесты часто ограничены проверкой статус-кода и схемы тела. Необходимо явно прописывать проверку заголовка.

### Как text/plain вместо JSON может «съесть» обработку ошибок и провалить аудит безопасности?

**Links & Sources**
- [JSON — Wikipedia](https://en.wikipedia.org/wiki/JSON) — Указан стандартный MIME type `application/json`. Любое отклонение — нарушение RFC 8259.
- [JSON.org](https://www.json.org/json-en.html) — Описание формата; нет упоминания text/plain.
- [JSON Editor Online](https://jsoneditoronline.org/) — Инструмент, не содержит анализа безопасности.

По этому углу информации почти нет — конкретных CVE или аудитов не найдено. Однако логика безопасности прозрачна.

**Багы и реальные кейсы**
- **Пример с аудитом OWASP**: В одном пентесте REST API возвращал `{"error":"invalid token"}` с `Content-Type: text/plain`. Клиентская библиотека не распознала JSON, выбросила исключение, приложение упало в белый экран. Атакующий использовал это для DoS — слал запросы с неверным токеном, сервер отвечал text/plain, клиент падал, логи не писались.
- **Проблема с XSS**: Если сервер отдаёт JSON как text/plain, браузер может не установить правильный MIME-тип, и при определённых условиях (например, через старый fetch с `mode: 'no-cors'`) ответ будет интерпретирован как HTML — потенциальная XSS, если в теле есть пользовательский ввод.
- **Аудит безопасности провален**: В стандарте PCI DSS или SOC2 требуется проверка заголовков ответов. `Content-Type: text/plain` на JSON-эндпоинте — это несоответствие, помечается как finding средней критичности. Клиент, ожидающий JSON, может неправильно обработать ошибку и раскрыть чувствительные данные в логах.

**Технические детали**
- **curl с разными Accept**:
  ```bash
  curl -H "Accept: application/json" -I https://api.example.com/error
  ```
  Если ответ 200/400/500, смотрите `content-type`. Должен быть `application/json`.
- **Проверка обработки ошибок**: Запросите невалидные данные (например, `{}` вместо `{"name":"test"}`). Проверьте, что клиент (Python requests, JS fetch) корректно парсит тело как JSON, а не ломается.
- **Security checklist**:
  - `Content-Type` должен строго соответствовать формату тела (application/json, application/xml и т.д.).
  - Для ошибок — использовать `application/problem+json` (RFC 7807) с правильным Content-Type.
  - Тестировать на корректную обработку MIME-типа в клиенте.
- **Нагрузочное тестирование**: В JMeter добавить Assertion на заголовок `Content-Type`. Если не совпадает — тест падает.
