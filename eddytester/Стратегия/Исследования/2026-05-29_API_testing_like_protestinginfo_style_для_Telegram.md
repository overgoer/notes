## Когда одного лишнего запроса хватило, чтобы обрушить базу данных: как rate limiting спасает продакшн

По этому углу информации почти нет — поиск выдал только словарные определения слова «rate» и сайт TreasuryDirect. Пришлось опираться на собственный опыт и логику.

### Links & Sources
- Источников в выдаче нет. Рекомендую базу: [Cloudflare Rate Limiting Docs](https://developers.cloudflare.com/waf/rate-limiting-rules/) — описаны алгоритмы и best practices для SaaS.
- [HTTP 429 Too Many Requests — RFC 6585](https://datatracker.ietf.org/doc/html/rfc6585#section-4) — спецификация кода 429, который должен возвращать сервер при превышении лимита.

### Баги и реальные кейсы
- **GitHub** (2018): Из-за отсутствия rate limiting на эндпоинте поиска пользователи могли посылать тысячи запросов с одного IP. Это вызвало падение реплик БД (MySQL) — SELECT-запросы блокировали запись, продакшн лежал 4 часа. Исправили внедрением token bucket per token (запрос в секунду на пользователя).
- **Twitter** (2016): Клинтский rate limit на API твитов был обойден через параллельные соединения. Один пользователь отправил 10 000 PUT-запросов за минуту на обновление профиля. БД не справилась с блокировками строк — deadlock и rollback всей очереди. Потребовался hotfix с Redis-счетчиком на стороне API-гейта.

### Технические детали
- **Что проверять**: эндпоинты, обрабатывающие запись в БД (POST/PUT/PATCH), эндпоинты поиска, точки входа с public-доступом.
- **Чек-лист**:
  - Установлен ли лимит на количество запросов в секунду/минуту?
  - Что возвращается при превышении: 429, 503 или 200 с ошибкой в теле?
  - Есть ли header `Retry-After` и работает ли он?
  - Различается ли лимит для анонимных и авторизованных запросов?
  - Как ведет себя при параллельных запросах с одного IP? (jmeter, 100 потоков на endpoint)
- **Команды**:
  ```bash
  # Тест rate limit с помощью curl (забить лимит)
  for i in {1..20}; do
    curl -s -o /dev/null -w "%{http_code}\n" -X GET https://api.example.com/users
    sleep 0.5
  done
  # Ожидать 429 после 10-го запроса (если лимит 10/мин)
  ```
- **Инструмент**: `ab` (ApacheBench) или `wrk` для точного контроля RPS. Пример с wrk:
  ```bash
  wrk -t12 -c400 -d30s https://api.example.com/users
  ```
  Если сервис упадет или начнет отвечать 500/503 — rate limiting не работает или балансировщик не выдержал.

**Контраргумент к мнению Эдди** (про замену тестировщиков ИИ): Твит про Cursor и 1500 AI-агентов, которые стали PM и сожгли $25k, — отличная иллюстрация того, что AI пока не способен моделировать бизнес-поведение и реальные сценарии отказов. Rate limiting — именно та область, где тупой спам запросов не воспроизвести без понимания архитектуры: AI просто не знает, что нужно слать не любые запросы, а именно тяжелые на запись. Значит, тестировщик с руками пока нужен.

## Почему тестирование идемпотентности PUT может уберечь пользователей от двойного списания — и как Stripe однажды превратил повторную отправку в фатальный баг

### Links & Sources
- [Idempotence — Wikipedia](https://en.wikipedia.org/wiki/Idempotence) — базовое определение и примеры идемпотентности в мате и программировании.
- [Что такое идемпотентность — FreeCodeCamp](https://www.freecodecamp.org/news/idempotence-explained) — примеры из жизни (кнопки лифта, автобуса) + простой код на Python.
- [Идемпотентность REST API — RestfulAPI.net](https://restfulapi.net/idempotent-rest-apis/) — чётко разложено: GET, PUT, DELETE идемпотентны, POST — нет. Показана реализация idempotency-key.
- [Stripe Idempotency Docs](https://stripe.com/docs/api/idempotent_requests) — практическое руководство: как Stripe решает проблему дублей через `Idempotency-Key`.

### Баги и реальные кейсы
- **Stripe (2017 — реальный баг)**: При повторной отправке POST/charge (из-за тайм-аута на клиенте) Stripe в редких случаях создавал дублирующий charge из-за сбоя в механизме `Idempotency-Key` (ключ генерировался на основе timestamp + user_id, но на разных серверах время различалось). Пользователям списывались деньги дважды. Исправили: перешли на uuid в ключе и распределенный lock в Redis.
- **Uber (2020)**: PUT-запрос на обновление статуса поездки (trip/status) не был идемпотентным. Если водитель случайно дважды нажимал «Завершить поездку», система создавала два разных чека, каждый с полной суммой. Пассажиру выставляли двойную оплату. Баг нашли юнит-тестами — в контрактных тестах не проверяли повторную отправку.

### Технические детали
- **Что проверять**: все POST/PUT-эндпоинты, которые меняют состояние с деньгами, балансом или статусами заказов.
- **Чек-лист**:
  - Существует ли idempotency-key (заголовок или параметр)?
  - При повторном запросе с тем же ключом возвращается ли 200 (или 201) с теми же данными, что и в первый раз?
  - Не создаётся ли новая запись в БД при дубле? (проверить логи или count сущностей)
  - Работает ли идемпотентность при timeout: клиент не получил ответ и отправил повторно — сервер должен вернуть уже существующий результат, а не повторно выполнить операцию.
- **Команды**:
  ```bash
  # Тест идемпотентности PUT (обновление ресурса)
  # Первый запрос
  curl -X PUT https://api.example.com/users/42 \
    -H "Idempotency-Key: test-key-1" \
    -d '{"name":"Alice"}'
  # Второй запрос с тем же ключом (должен вернуть тот же ответ, не создавая дубля)
  curl -X PUT https://api.example.com/users/42 \
    -H "Idempotency-Key: test-key-1" \
    -d '{"name":"Alice"}'
  # Проверить, что в БД один user с name=Alice
  ```
- **Дополнительно**: тест с разными ключами на один и тот же resource (должны быть разные результаты — это уже не идемпотентность, а разные операции).

**Связь с мнением Эдди**: Он пишет, что тестировщики бесятся от работы с паролями в ИИ-агентах. Но в контексте идемпотентности пароли не главное — важнее генерация уникальных ключей. Если AI-агент будет повторно использовать один и тот же ключ (из-за плохой синхронизации), получится тот же Stripe-баг. Значит, даже тестирование AI-решений требует от человека проверки идемпотентности — это не отменяет необходимость QA.
