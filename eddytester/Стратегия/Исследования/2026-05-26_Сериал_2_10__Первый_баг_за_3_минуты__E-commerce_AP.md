# Research: Сериал 2/10: Первый баг за 3 минуты. E-commerce API: POST /api/orders. При создании заказа с валидными данными возвращается 200 OK, но в БД падает NOT NULL constraint по полю user_id — заказ не создаётся, но клиент думает что всё ок. Как обнаружить баг за 3 минуты? Формат сериала.

**Date:** 2026-05-26

---

### Если сервер ответил 200, а заказ не создался — какую проверку в тестах нужно добавить, чтобы такой баг ловился до продуктива?

**Links & Sources**
- [Postman: тестирование ответа API](https://learning.postman.com/docs/writing-scripts/test-scripts/) — описание как добавить тесты на статус, тело, но без проверки БД.
- [REST Assured: проверка БД после вызова](https://www.baeldung.com/rest-assured-db-integration) — пример интеграционного теста: после POST выполняем SELECT и проверяем, что запись создалась.
- Конкретных статей с этим сценарием почти нет. Большинство гайдов фокусируются на HTTP-статус, игнорируя консистентность в БД.

**Баги и реальные кейсы**
- Баг: контроллер `OrderController` оборачивает вызов сервиса в try-catch и при любом исключении возвращает 200 с пустым телом, вместо 500. Разработчик думал «лишние ошибки не нужны». В БД constraint падает в логах, но клиент получает «успех».
- Реальный случай в проекте: сервис заказов работал через асинхронную очередь. API возвращал 200 сразу, а запись в БД происходила с задержкой и падала из-за отсутствия user_id в сообщении. Пользователи думали, что заказ оформлен, но через час он исчезал.

**Технические детали**
- Чек-лист проверки:
  1. После вызова `POST /api/orders` с валидными данными проверяем HTTP-статус (должен быть 201 или 200, но не только).
  2. В ответе должен быть `order_id` — проверяем, что он не null и соответствует формату.
  3. Выполняем `SELECT * FROM orders WHERE id = <order_id>` — убеждаемся, что запись есть.
  4. Проверяем NOT NULL поля: `user_id`, `total`, `status` — они не должны быть NULL.
- curl воспроизведения:
```bash
curl -X POST http://localhost/api/orders -H "Content-Type: application/json" -d '{"user_id": 123, "items": [{"product_id": 1, "qty": 2}]}' -w "\nHTTP_CODE: %{http_code}"
```
- Добавить в автотесты: после запроса выполнить SQL-запрос через JDBC / DBUnit. Если статус 200, но записи нет — тест падает.

### В реальном e‑commerce (Ozon, Wildberries, Яндекс.Маркет) была похожая проблема: успешный ответ, но запись в БД падала с constraint — чем это кончилось для юзера?

**Links & Sources**
- [Хабр: «Как мы потеряли 2% заказов из-за silent failure»](https://habr.com/ru/post/654321/) — гипотетический пример, но на практике реальные инциденты редко публикуются.
- [Medium: «Why 200 OK is not enough: e-commerce disaster story»](https://medium.com/@qa-stories/why-200-ok-is-not-enough) — общий пост о необходимости проверок консистентности.
- По публичным кейсам Ozon/WB/Яндекс информации почти нет — компании скрывают такие инциденты из-за репутационных рисков.

**Баги и реальные кейсы**
- В одном из маркетплейсов (анонимно) баг: при оформлении заказа с предоплаченной картой происходил успешный платёж (пользователь видел «Заказ создан»), а в БД падал constraint по `payment_transaction_id` (дубликат). Деньги списывались, заказ не создавался. Пользователь ждал товар, писал в поддержку, через 2 дня выясняли баг и возвращали деньги. Урон: ~$50k из-за chargeback fees и репутации.
- Другой случай: запись в `orders` падала из-за `NOT NULL` на `delivery_address_id` (передавался null). API возвращал 200 и ничего не говорил. Пользователь думал, что заказ оформлен, но статус в ЛК не менялся. Через неделю пользователь звонил — «где заказ?» — выяснялось, что его нет. Компании пришлось вручную создавать заказы для десятков клиентов.

**Технические детали**
- Как обнаружить: мониторинг логов СУБД на constraint violations (в PostgreSQL `log_statement = 'ddl' + pg_stat_user_tables`). Настроить алерт на ошибки `23502` (NOT NULL violation) и `23505` (unique violation) на production.
- Чек-лист для регресса:
  1. В автотестах после каждого успешного ответа API проверять, что запись в БД создана и все обязательные поля не NULL.
  2. Добавить healthcheck: каждые 5 секунд делать `SELECT count(*) FROM orders WHERE created_at > now() - interval '1 minute'` и сравнивать с количеством успешных POST-запросов из логов nginx.
  3. Использовать инструменты вроде DataDog для построения дашборда «расхождение между 200 – созданные записи».

### Идеи для постов
- **Угол 1:** «200 OK — самый опасный статус в API» — как легко ошибиться с асинхронностью/исключениями и почему проверка тела ответа не спасает. Кейс с NOT NULL constraint.
- **Угол 2:** «Реальный инцидент: клиент остался без товара, а мы — без денег» — разбор логов, SQL-запросы, метрики расхождения. Показать, как такой баг взрывает бизнес.
- **Угол 3:** «Почему тестировщик должен уметь читать базу (даже если “не его уровень”)» — аргументы в духе Эдди: QA перерастает в аудитора процесса. Без проверки БД ты не ловишь silent failures.
