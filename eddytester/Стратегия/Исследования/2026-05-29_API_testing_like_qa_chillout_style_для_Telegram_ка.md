### Когда API отдаёт закешированный ответ, хотя данные уже изменились — как проверить, что кэш сбросили, а не сделали вид?

**Links & Sources**  
- [MDN HTTP Caching](https://developer.mozilla.org/en-US/docs/Web/HTTP/Caching) — полный разбор заголовков Cache-Control, ETag, Last-Modified, условных запросов и политик кэширования.  
- [MDN If-None-Match](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/If-None-Match) — как принудительно проверить, изменился ли контент, не скачивая тело.  
- [MDN Cache-Control](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Cache-Control) — описание no-cache, no-store, max-age и принудительного сброса.  
- [RESTful API Caching – Best Practices](https://restfulapi.net/http-caching/) — примеры правильной инвалидации кэша при изменении ресурса.  

**Багы и реальные кейсы**  
- **Баг: игнорирование инвалидации по ETag** — при PATCH обновлении профиля сервер менял данные в БД, но не обновлял ETag. GET с If-None-Match проходил 304, хотя контент изменился. Разработчик «сбрасывал кэш» по таймеру, а не по факту изменений.  
- **Баг: CDN-кэш с длинным TTL** — после редактирования статьи через API клиент получал старую версию, пока не истечёт TTL (1 час). Команда утверждала, что кэш сброшен, но на самом деле CDN не присылал принудительной purge-команды.  
- **Кейс: «липовый» сброс через обновление timestamp** — сервер менял Last-Modified, но возвращал те же данные (из-за бага в логике). Клиент считал, что кэш актуален, и не перезапрашивал.  

**Технические детали**  
- `curl -I -X GET <url>` — просмотр заголовков Cache-Control, Age, ETag, Last-Modified.  
- `curl -H "Cache-Control: no-cache" <url>` — принудительный запрос без кэша.  
- `curl -H "If-None-Match: \"prev_etag\"" -o /dev/null -w "%{http_code}" <url>` — если ответ 304, значит кэш не сброшен.  
- `curl -X PUT -d "new data" <url> && curl -I -X GET <url>` — проверять, что ETag изменился после обновления.  
- Чек-лист:  
  1. Есть ли у ресурса ETag или Last-Modified?  
  2. Меняется ли ETag после успешного POST/PUT/PATCH?  
  3. Возвращается ли 304 при If-None-Match до изменения?  
  4. Вернётся ли 200 и новые данные после изменения при If-None-Match с тем же ETag?  
  5. Принудительный сброс через заголовок Cache-Control: no-cache — работает?

### Если сервер возвращает 200 OK, но тело ответа пустое — это баг или фича, и как правильно тестировать такие случаи?

**Links & Sources**  
- [RFC 7231 §6.3.1 – 200 OK](https://tools.ietf.org/html/rfc7231#section-6.3.1) — спецификация: 200 подразумевает тело, если не указано иное (исключения: HEAD, CONNECT, DELETE с пустым телом).  
- [Microsoft REST API Guidelines – Empty Responses](https://github.com/microsoft/api-guidelines/blob/vNext/Guidelines.md#93-response-formats) — рекомендация: 204 No Content при успешном запросе без тела.  
- [HTTP Status Codes – 200 vs 204](https://httpstatuses.com/204) — разбор, когда использовать 204 вместо 200 с пустым телом.

**Багы и реальные кейсы**  
- **Баг: 200 с пустым телом вместо 204** — сервер удалял сущность, возвращал 200 OK без тела. Клиент парсил JSON и падал с исключением. Разработчик не знал про 204 и считал «так проще».  
- **Баг: пустой ответ из-за ошибки сериализации** — сервер ловил исключение и отдавал пустой ответ с 200 вместо 500. Клиент получал успешный статус, но все поля null.  
- **Кейс-фича: GET /items?filter=empty** — возвращает 200 с телом `[]` (пустой массив). Если же вернуть пустое тело (не массив), это баг — клиент ожидает массив.  
- **Кейс-норма: HEAD-запрос** — 200 с пустым телом — корректно (заголовки есть, тела нет).  

**Технические детали**  
- `curl -v <url>` — смотреть статус и Content-Length.  
- `curl -o /dev/null -w "%{http_code} %{size_download}" <url>` — проверить размер тела (0 = пусто).  
- `curl -X DELETE <url> -w "%{http_code}"` — DELETE должен возвращать 204 (без тела) или 200 с телом, но не пусто.  
- Чек-лист:  
  1. Статус 200 — есть ли тело? Если нет, то это может быть баг (кроме HEAD, CONNECT).  
  2. Какой тип запроса? DELETE, HEAD — допускается. GET, POST, PUT, PATCH — обычно тело должно быть.  
  3. Соответствует ли контракту API? (OpenAPI spec — проверять required response body).  
  4. Проверить Content-Type и Content-Length: если 0 — тело отсутствует.  
  5. Попробовать парсить ответ — будет ли исключение?

### Идеи для постов  
- **«200 OK с пустым телом: баг или норма? Разбор на реальных случаях»** — плюс чек-лист, когда это ок, а когда надо бить в колокола.  
- **«Как не попасться на лживый сброс кэша: конкретные curl-команды и ETag-проверки»** — практическая инструкция для junior.  
- **«Почему HTTP-статусы — это не просто “200 — хорошо, 500 — плохо”: кейсы, которые ломают прод»** — с упором на критическое мышление и проверку политик кэширования.  

*Примечание к позиции Эдди*: его скепсис к ИИ здесь не противоречит — сложные кейсы с кэшем и пустыми телами требуют понимания бизнес-логики и протокола, а не просто сгенерированного теста. Тестировщик, который разбирается глубже статусов, — тот самый «аудитор процесса» (как он пишет). Контраргумент: если ИИ научится анализировать спецификации и корректно верифицировать кэш, замена возможна — но сейчас такие задачи не решает ни один популярный инструмент (например, Playwright/POSTMAN с AI-генерацией не проверяют инвалидацию кэша).
