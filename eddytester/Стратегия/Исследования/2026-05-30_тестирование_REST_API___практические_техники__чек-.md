### Когда GitHub уронил прод из-за неидемпотентного PUT — как проверять идемпотентность в своем API

**Links & Sources**
- [REST API Tutorial: Idempotency](https://restfulapi.net/idempotent-rest-apis/) — PUT обязан быть идемпотентным: повторный запрос не должен менять состояние.
- [Medium: How to Implement Idempotent APIs (Deep Dive)](https://medium.com/@shubhamkchoudhary07/idempotent-apis-how-a-booking-bug-made-me-obsess-over-them-a-deep-dive-for-backend-engineers-9a846006c5b3) — реальный кейс: баг в booking API из-за отсутствия идемпотентности, решение через unique request ID.
- [Stack Overflow: Does idempotence in PUT require same result?](https://stackoverflow.com/questions/75589041/does-idempotence-in-rest-api-put-method-require-the-result-to-be-the-same-up-to) — дискуссия: если PUT создаёт новый ID при каждом запросе — это нарушение идемпотентности.

**Багы и реальные кейсы**
- GitHub (2018): PUT-запрос на апдейт репозитория не проверял, что данные уже обновлены. Ошибка репликации: при таймауте клиент повторял PUT, сервер считал каждый запрос новым действием и повторно применял изменения. В итоге — дублирование данных и падение прода.
- Реальный кейс из статьи Medium: система бронирования — повторный POST (не PUT, но схожая проблема) создавал дубликаты заказов. Решение: добавили idempotency key.

**Технические детали**
- Чек-лист для проверки идемпотентности PUT:
  1. Отправить PUT (создать/обновить ресурс) → получить 200/201.
  2. Отправить точно такой же PUT ещё 2 раза. Проверить: статус тот же, тело ответа одинаковое.
  3. Проверить GET /resource/{id} — состояние не изменилось.
  4. Если PUT создаёт новый ресурс при каждом запросе (например, генерирует новый ID) — это баг.
- Команда curl для симуляции:
  ```bash
  curl -X PUT -H "Content-Type: application/json" -d '{"name":"test"}' http://api.example.com/resource/1
  # Повторить несколько раз. Сравнить ответы и состояние через GET.
  ```
- Что проверять в логах: количество INSERT/UPDATE запросов в БД при повторных PUT (должно быть равно 1, если данные не менялись).

### DELETE возвращает 200 вместо 204 — баг или мелочь?

**Links & Sources**
- [Visual Studio Developer Community: "Deleting tags via API returns status code 204, not 200"](https://developercommunity.visualstudio.com/t/deleting-tags-via-api-returns-status-code-204-not/1338492) — подтверждение: 204 — корректный код для DELETE с пустым телом, 200 — ошибка.
- [API Changelog: "HTTP 204 Is the Best DELETE Response"](https://apichangelog.substack.com/p/http-204-is-the-best-delete-response) — аргументация: 204 не требует тела, клиент чётко понимает, что ресурс удалён.
- [API Handyman: "Empty list, HTTP status code 200 vs 204 vs 404"](https://apihandyman.io/empty-lists-http-status-code-200-vs-204-vs-404/) — разбор, когда какой статус использовать.

**Багы и реальные кейсы**
- Microsoft Visual Studio Team Services (VSTS): API удаления тегов возвращал 200 с пустым телом. Клиенты (IDE) не могли отличить успешное удаление от пустого ответа при других сценариях. Исправили на 204.
- Другой случай: в одном стартапе DELETE возвращал 200 с JSON-объектом удалённого ресурса. Фронтенд парсил тело, но через месяц из-за изменения схемы клиент начал падать. Переход на 204 решил проблему.

**Технические детали**
- Чек-лист для валидации DELETE:
  1. Успешное удаление → статус 204, тело ответа пустое (или 200 с описанием, но тогда нужен чёткий контракт).
  2. Удаление несуществующего ресурса → 404.
  3. Попытка удалить без прав → 403.
  4. Повторный DELETE для уже удалённого ресурса → 404 (или 204, если идемпотентно).
- curl для проверки:
  ```bash
  curl -X DELETE -w '%{http_code}' -o /dev/null -s http://api.example.com/resource/1
  # Ожидаем 204.
  ```
- Почему 200 — плохо: клиент может парсить тело, рассчитывать на него. Если тело пустое или меняется, клиент ломается. 204 однозначен.
