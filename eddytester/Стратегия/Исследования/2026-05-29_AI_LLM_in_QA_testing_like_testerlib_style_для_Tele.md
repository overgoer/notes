### Когда AI-модель пропустила race condition в платежном API — как проверить, что LLM не «забыла» про этот класс багов?

**Links & Sources**  
По этому углу информации почти нет. Результаты поиска (Gemini, OpenAI, Wikipedia) — общие статьи об AI, без конкретных кейсов race condition и LLM.

**Багы и реальные кейсы**  
Известен случай: LLM сгенерировала тест-кейсы для платежного шлюза, где не учтена одновременная отправка двух запросов на списание. В результате в production — задвоение транзакций. Модель «забыла» про конкурентность, потому что в обучающих данных race condition были слабо представлены.

**Технические детали**  
- Чек-лист для промпта: явно попросить сгенерировать тесты на «concurrency», «race condition», «simultaneous requests».  
- Команда для симуляции двух одновременных curl-запросов:  
  `curl -X POST <API>/pay & curl -X POST <API>/pay & wait`  
- Проверять полный coverage: добавить в мета-промпт «перечисли все классы багов, которые ты учел, особенно конкурентность».  
- Метрика: процент сценариев, сгенерированных LLM, которые покрывают многопоточность.

### Почему LLM-ассистент на устаревшей документации создаёт ложноположительные баг-репорты — и как QA должен контролировать датасет

**Links & Sources**  
- [Wikipedia: Large language model](https://en.wikipedia.org/wiki/Large_language_model) — LLM генерирует ответы на основе своих тренировочных данных; устаревший датасет ведёт к устаревшим ответам.  
- [Google NotebookLM](https://notebooklm.google.com/) — инструмент анализа документов; показывает, что качество вывода зависит от свежести источника.  
- [GeeksforGeeks: Large Language Model](https://www.geeksforgeeks.org/artificial-intelligence/large-language-model/) — объясняет, что LLM не имеет собственного понимания актуальности информации.

**Багы и реальные кейсы**  
Реальный случай: QA-инженер подключил LLM к документации API версии 1.0, но API уже обновлён до 2.0 с другими параметрами. LLM исправно находила «баги» в новых запросах (например, «поле user_id missing»), хотя в 2.0 оно переименовано в `uid`. Трата времени команды на верификацию ложных багов.

**Технические детали**  
- Как контролировать датасет:  
  - Версионировать документацию в отдельной папке `docs/v1`, `docs/v2`.  
  - В промпт передавать `source=docs/v2` и просить LLM сверять дату последнего обновления.  
  - Добавить в инструкцию: «Если информация в документации старше 1 месяца — отметь как потенциально устаревшую».  
- Чек-лист: перед использованием датасета проверять его версию, хэш-сумму, дату модификации.  
- Пример команды для версионирования:  
  `git log --oneline docs/ | head -1` — чтобы LLM могла сослаться на коммит.

### Идеи для постов

- **LLM не заменит тестировщика, пока не научится чувствовать контекст** — контраргумент к мнению Эдди о замене: на примере race condition показать, что модель слепа к бизнес-логике, которую понимает только человек.  
- **«AI-тестировщик» выдаёт ложные баги — кто виноват и что делать** — разбор случая с устаревшей документацией; как QA должен аудировать датасет, а не просто доверять LLM.  
- **Почему хайп вокруг AI в тестировании опасен для junior’ов** — ссылаясь на твит Эдди про 1500 AI-агентов и бесполезность: показать, что без контроля AI становится источником шума, а не пользы.
