### Как проверить, что POST /users не перетирает чужих данных при id=1?

**Links & Sources**
- [VirtuosoQA: 60 Test Cases For API Testing](https://www.virtuosoqa.com/post/api-testing-test-cases) — postcondition — ключевой пропуск в тестах API: HTTP 201 не доказывает, что ресурс реально сохранён или не перезаписан.
- [StackOverflow: Consequences of POST not being idempotent](https://stackoverflow.com/questions/13991904/consequences-of-post-not-being-idempotent-restful-api) — POST по дизайну неидемпотентен, но его могут использовать как PUT — риск повторного создания или перетирания.
- [Reddit: If UI test cases exists are API test duplicates recommended?](https://www.reddit.com/r/QualityAssurance/comments/cqhow0/api_testing_strategy_if_ui_test_cases_exists_are/) — UI-тесты не покрывают проверку идемпотентности и изоляции ресурсов на уровне API.

**Багы и реальные кейсы**
- Баг в production: при создании пользователя сервер возвращал `id=1`, но из-за сброса автоинкремента после бекапа новый пользователь перезаписал старого. Тест по postcondition (GET /users/1) показал несоответствие полей.
- Кейс: два параллельных POST с одинаковым телом. Сервер ответил 201 для обоих, но второй запрос на самом деле обновил запись первого, т.к. id генерировался не уникально (баг в логике генерации).

**Технические детали**
- Чек-лист:
  1. Отправить POST /users с уникальными данными, сохранить ответ (id).
  2. Выполнить GET /users/{id} и сверить все поля.
  3. Отправить второй POST с другими данными, получить новый id.
  4. Проверить, что первый пользователь не изменился (GET /users/{firstId}).
  5. Проверить, что второй id не равен первому.
- curl:
  ```bash
  curl -X POST http://api/users -H "Content-Type: application/json" -d '{"name":"Alice"}'
  # response: {"id":1,"name":"Alice"}
  curl -X GET http://api/users/1
  curl -X POST http://api/users -H "Content-Type: application/json" -d '{"name":"Bob"}'
  curl -X GET http://api/users/2
  curl -X GET http://api/users/1  # name должен остаться Alice
  ```
- Дополнительно: проверка на одновременные запросы (используйте `parallel` или `xargs -P 2`).

### Как найти тихую мутацию PUT /items/{id} — игнорирование лишних полей?

**Links & Sources**
- [Imperva: Black Box Testing](https://www.imperva.com/learn/application-security/black-box-testing/) — чёрный ящик: тестировщик не знает кода, но обязан проверять все поля и граничные случаи.
- [CloudNativeNow: Black Box Testing APIs in Microservices](https://cloudnativenow.com/contributed-content/black-box-testing-apis-in-microservices-why-your-tests-pass-but-your-system-still-fails/) — тесты проходят, когда моки расходятся с реальностью; PUT с лишними полями может молча отбрасываться без ошибки — баг в прод.
- [SoftwareTestingEdu: Black Box Testing in Manual Testing Explained](https://softwaretestingedu.com/black-box-testing-in-manual-testing-explained/) — метод эквивалентного разбиения: лишние поля — это некорректный класс, который должен вызывать 400 или игнорироваться с предупреждением, но часто молча пропускается.

**Багы и реальные кейсы**
- Реальный баг: PUT /items/5 пришёл с полем `color: "red"`, которого нет в схеме. Сервер вернул 200, поле `color` не сохранилось, но и ошибки не было. Позже фронтенд начал слать это поле (добавили новый функционал), и данные терялись.
- Кейс: PUT с полем `status: "archived"` (недокументированное) — статус не изменился, а пользователь думал, что архив успешен.

**Технические детали**
- Чек-лист:
  1. Отправить PUT с одним лишним полем (например, `"extra":"value"`).
  2. Проверить HTTP-статус: ожидается либо 400 (Bad Request) с сообщением об ошибке, либо 200 и игнорирование поля (но это должно быть задокументировано). Если 200 и поле просто исчезло — баг.
  3. GET /items/{id} и убедиться, что лишнего поля нет, а остальные не изменились.
  4. Проверить, что поле `extra` не попало в ответ (если сервер возвращает все полученные поля).
- curl:
  ```bash
  curl -X PUT http://api/items/1 -H "Content-Type: application/json" -d '{"name":"new name","unknown_field":"should not appear"}'
  curl -X GET http://api/items/1 | grep unknown_field
  ```
- Дополнительно: проверить, что изменение без лишних полей работает корректно (контрольный тест).
