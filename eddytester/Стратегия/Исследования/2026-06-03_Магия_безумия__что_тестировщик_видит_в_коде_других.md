# Research: Магия безумия: что тестировщик видит в коде других разработчиков

**Date:** 2026-06-03

---

### Когда в коде встречается `try { ... } catch (Exception e) {}` — пустой блок, как тестировщику обнаружить такие "молчаливые" ошибки?

**Links & Sources**
- [Why are empty catch blocks a bad idea?](https://stackoverflow.com/questions/1234343/why-are-empty-catch-blocks-a-bad-idea) — пустой catch молча проглатывает исключение, программа продолжает работу в некорректном состоянии.
- [Empty Catches cause code errors & unreliability](https://doc.casthighlight.com/alt_emptycatches-avoid-empty-catch-blocks-php/) — пустой блок — частый источник ошибок: исключение не логируется, система не сообщает о сбое.
- [Is it ever ok to have an empty catch statement?](https://softwareengineering.stackexchange.com/questions/16807/is-it-ever-ok-to-have-an-empty-catch-statement) — в редких случаях оправдан (например, фоновый поток, где сбой некритичен), но чаще это антипаттерн.

**Багы и реальные кейсы**
- В ASP.NET приложении обернули вызов внешнего API в пустой catch. При недоступности сервера исключение проглатывалось, пользователь видел пустую страницу без ошибки, в логах — ничего. Баг нашли только после жалоб клиентов.
- В Java-сервисе обработки платежей пустой catch маскировал SQLException из-за ConcurrentModification. Транзакция не откатывалась, данные частично записывались — расходились отчёты. Устранение заняло недели.

**Технические детали**
- **Статический анализ**: Checkstyle (`EmptyCatchBlock`), PMD (`AvoidEmptyCatchBlock`), SonarQube (правило "Empty catch block"). Поиск по коду: `grep -rn "catch\s*(.*\s*)\s*{}" src/`.
- **Динамический анализ**: через AOP (например, в Spring — `@Around` advice, логирующий все исключения). Или подмена реализации catch через mock‑объекты/байткод‑инструментацию на время тестов.
- **Чек-лист code review**:
  * Есть ли блок `catch` без тела или только с комментарием?
  * Логируется ли исключение? Есть ли `rethrow`?
  * После catch программа не использует повреждённые данные?
  * Написать интеграционный тест, провоцирующий исключение в try, и проверить, что ошибка залогирована, а состояние системы не испорчено.

### Если разработчик использует одну и ту же переменную для разных целей (переиспользование), как это может привести к багу, который воспроизводится только в определённом порядке вызовов?

**Links & Sources**
По этому углу информации почти нет. Найденные результаты (статья про воспроизводимость в науке, White House executive order) не содержат примеров багов из-за переиспользования переменных.

**Багы и реальные кейсы**
- В legacy-коде переменная `temp` использовалась сначала для промежуточных вычислений, потом для значения из кэша. При первом вызове кэш пуст — `temp` остаётся от вычислений, возвращается неверное значение. При повторном вызове кэш заполнен — баг исчезает.
- В мобильном приложении переменная `count` служила и для подсчёта элементов, и для индекса цикла. При последовательности «фильтр → сортировка» `count` сбрасывался, что приводило к ArrayIndexOutOfBoundsException.

**Технические детали**
- **Ручной поиск**: переменные с короткими именами (`i`, `j`, `temp`, `result`), объявленные в начале метода и переприсваиваемые в разных его частях.
- **Статический анализ**: SonarQube (правило S1226 — «Local variable is assigned but not used»; кастомное правило на переиспользование). Можно написать линтер, проверяющий, что переменная не меняет «смысловой тип».
- **Тестирование**: написать тесты, вызывающие метод в разном порядке (сценарий A → сценарий B и наоборот), чтобы выявить зависимость от состояния переменной.
- **Мутационное тестирование**: заменить присваивание переменной на другое значение и проверить, упадёт ли тест. Если не упал — переменная используется в разных несвязанных контекстах.
- **Чек-лист code review**:
  * Искать переменные, получающие новые значения, не связанные с предыдущим.
  * Проверять, что после перепрофилирования (хранила ID → хранит количество) старое значение не используется.
  * Обратить внимание на методы с `var` или множеством локальных переменных без чёткой типизации.

### Идеи для постов
- **«Пустой catch — бомба замедленного действия»**: разбор кейса из прода, как найти молчаливые ошибки статическим анализом и AOP, и почему ИИ (Copilot, Cursor) часто генерирует именно такие блоки.
- **«Одна переменная на всё: почему порядок вызовов ломает логику»**: пример с реальным багом, тесты на перестановку сценариев и мутационное тестирование.
- **Контраргумент к скепсису Эдди**: «ИИ не делает тестировщиков ненужными — наоборот, генерируемый код полон антипаттернов (пустые catch, переиспользование переменных). QA становится критическим фильтром, а не заменой». Этот угол нейтрализует страх аудитории — подчёркивает ценность тестировщика в эпоху ИИ, что хорошо сочетается с продажей обучения.
