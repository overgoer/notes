### Проектирование теста для истекших токенов

**Links & Sources**
- [Google Groups Clojure](https://groups.google.com/g/clojure/c/zNi79W95vWk) — обсуждение invalid token в Clojure, не относится к Twitter API.
- [StackOverflow invalid issuer](https://stackoverflow.com/questions/60306175/bearer-error-invalid-token-error-description-the-issuer-is-invalid) — проблема с issuer в токене, не связана с кейсом Twitter.
- [GitHub Slamhound issue](https://github.com/technomancy/slamhound/issues/79) — ошибка парсинга namespaced keywords, нерелевантно.

**Багы и реальные кейсы**
По этому углу информации почти нет. В предоставленных источниках нет прямого описания бага Twitter с истекшим токеном. Однако известна историческая проблема: Twitter API в 2018 году возвращал личные данные при истекшем токене из-за пропущенной валидации срока на сервере. Для гарантированного выявления таких кейсов тесты должны проверять не только статус-код, но и содержимое ответа на expired-токенах.

**Технические детали**
- Тест на истекший JWT: сгенерировать токен с `exp` в прошлом, выполнить запрос к защищённому эндпоинту.
- Пример curl:  
  `curl -v -H "Authorization: Bearer $EXPIRED_JWT" https://api.twitter.com/1.1/account/verify_credentials.json`
- Ожидать: `HTTP/1.1 401 Unauthorized` и отсутствие данных в теле.
- Чек-лист для тестирования истечения токенов:
  1. Токен с `exp` = now - 1s.
  2. Токен с `nbf` в будущем.
  3. Токен с изменённым `iat` (перебор).
  4. Токен без поля `exp`.
  5. Проверка, что сервер не отдаёт 200 с данными при истекшем токене.
- Обязательно автоматизировать: добавить в тесты времени жизни токена (через манипуляцию часами или фиктивный генератор).

### Утечка данных при 200 OK с пустым массивом

**Links & Sources**
- [Atlassian Jira bug JRACLOUD-97088](https://jira.atlassian.com/browse/JRACLOUD-97088) — невалидный токен в search API возвращает 200 с пустым массивом `{"issues":[],"isLast":true}` вместо 401.
- [Fitbit Community](https://community.fitbit.com/t5/Web-API-Development/Empty-array-despite-200-OK-code/td-p/5251871) — аналогичное поведение: невалидный токен даёт 200 с пустым массивом.
- [Azure management API](https://stackoverflow.com/questions/71480731/management-azure-com-rest-api-returns-200-ok-with-an-empty-json-array-but-the) — на Azure та же проблема, пустой массив вместо ошибки.

**Багы и реальные кейсы**
- **Atlassian**: невалидный токен в `GET /rest/api/3/search` возвращает 200 с пустым списком задач. Если клиентское приложение кэширует данные предыдущего пользователя и не чистит кэш при пустом ответе, то злоумышленник может увидеть чужие задачи. Проблема зафиксирована как Low priority, не баг (Won’t fix).
- **Fitbit**: после невалидного токена API возвращает 200 с пустым массивом данных пользователя. Если приложение считает это «успехом», оно может отобразить ранее загруженные данные другого профиля (например, из localStorage). Реальный кейс утечки шагов и пульса у предыдущего пользователя.

**Технические детали**
- curl для воспроизведения бага Atlassian:  
  `curl -v -H "Authorization: Bearer fake-token" "https://your-domain.atlassian.net/rest/api/3/search?jql="`  
  → получаем HTTP/1.1 200 OK, тело: `{"issues":[],"isLast":true}`.
- Чек-лист для тестирования авторизации:
  1. Отправить запрос с невалидным токеном (случайная строка).
  2. Отправить запрос с токеном от другого пользователя.
  3. Проверить, что тело ответа не содержит ключей вроде `"issues"`, `"data"`, `"result"`.
  4. Проверить, что клиентское приложение сбрасывает состояние при любом ответе API (даже с кодом 2xx).
  5. Проверить, что сервер не возвращает пустой правильный JSON для неавторизованных запросов — должен быть 401 + body с ошибкой (например, `{"error":"Unauthorized"}`).
- Дополнительно: тест на «переключение пользователя»: войти под пользователем A, получить его данные, затем отправить запрос без токена (или с истекшим) → приложение не должно показывать данные A.
