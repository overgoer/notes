**Ссылки и источники**  
- По этому углу информации почти нет. Доступные веб-результаты (Xbox Cloud Gaming) не относятся к багу GitHub.

**Баги и реальные кейсы**  
- Подтверждённого инцидента с датой и описанием не найдено. В аналитической заметке упоминается «Анализ реального бага GitHub (200 OK вместо 401)», но без конкретики.

**Технические детали**  
- Информации недостаточно для формулировки команд или чек-листов. Рекомендуется изучить архив issues GitHub API и постмортемы за 2020 год.

### Stripe double charge из-за idempotency-key (2015)

**Ссылки и источники**  
- По этому углу информации почти нет. Результаты поиска (stripe.com, Wikipedia) не содержат описания инцидента.

**Баги и реальные кейсы**  
- Подтверждённого случая двойного списания из-за idempotency-key с датой не обнаружено. Известно, что Stripe публиковал постмортемы по race condition, но они не попали в предоставленные данные.

**Технические детали**  
- Нет конкретных curl-запросов или чек-листов. Для тестирования idempotency следует гонять параллельные POST с одинаковым `Idempotency-Key` и проверять, что сервер возвращает один и тот же ответ без повторного выполнения операции.

### Twitter silent failure на дублирующийся твит (2019)

**Ссылки и источники**  
- По этому углу информации почти нет. Результаты поиска (twitter.com, x.com) не содержат описания бага.

**Баги и реальные кейсы**  
- Подтверждённого инцидента с молчаливым 200 OK при дублирующемся твите в 2019 году не найдено. Возможно, баг не был задокументирован в открытых источниках.

**Технические детали**  
- Нет конкретных примеров. Для контрактного тестирования следует проверять, что API возвращает корректный статус (например, 409 Conflict) при попытке повторной публикации с тем же контентом.

### Идеи для постов

- «Тихие неудачи» опаснее 5xx: как клиенты игнорируют код ответа и что с этим делать — разбор на примерах GitHub, Twitter, Stripe.
- Race condition в идемпотентности: почему повторные запросы с одинаковым ключом могут привести к двойному списанию (даже если официальных постмортемов нет).
- Как Эдди прав: тестирование API становится аудитом процессов — нужно проверять не только код, но и согласованность тела ответа с контрактом. Контраргумент: без конкретных кейсов сложно убедить команду в необходимости таких проверок.
