### Когда AI-агент отправляет подтверждение бронирования через SendGrid, а письмо попадает в спам из-за отсутствия DKIM — как тестировать доставляемость, чтобы не пропустить критическое сообщение пользователю?

**Links & Sources**
- [Mailtrap: 6 Best Email APIs for AI Agents Compared [2026]](https://mailtrap.io/blog/best-email-api-for-ai-agents/) — ключевые метрики: SPF, DKIM, DMARC, проверка репутации отправителя; Mailtrap позволяет тестировать письма до отправки в продакшен.
- [Artsyl: Email Platforms Supporting AI Agents](https://www.artsyltech.com/blog/email-infrastructure-for-ai-agents-2026) — агентам нужен не только email: если письмо не дошло, резервный канал (SMS/WhatsApp).
- [SendGrid Support: Mastering Email Deliverability](https://support.sendgrid.com/hc/en-us/articles/9477130010011-Mastering-Email-Deliverability) — блокировка из-за попадания IP в блоклисты; проверка через MXToolbox.

**Баги и реальные кейсы**
- Прод-баг: агент бронирования отправил подтверждение с DKIM=pass, но SPF failed из-за новой subdomain. Письмо ушло в спам, пользователь не увидел → потеря заказа.
- Кейс: после добавления DKIM письмо всё равно попадало в спам — оказалось, DMARC политика (p=quarantine) настроена жёстко, при пересылке через forwarder ломалась подпись.

**Технические детали**
- Проверка DNS: `dig TXT yourdomain.com` для SPF, `dig TXT default._domainkey.yourdomain.com` для DKIM, `dig TXT _dmarc.yourdomain.com` для DMARC record.
- Команды: `curl --ssl-reqd --mail-from ... --mail-rcpt ...` для теста SMTP. Использовать `swaks` для отправки тестового письма.
- Чек-лист:
  1. SPF record (include:spf.sendgrid.net).
  2. DKIM (CNAME для SendGrid, 2048-bit ключ).
  3. DMARC (p=none, p=quarantine, p=reject).
  4. Прогон письма через Mailtrap Email Testing – спам-скор.
  5. Проверка репутации IP на MXToolbox blacklist.

### В Matrix Protocol нет центрального сервера, но при перегрузке одного инстанса сообщения могут теряться — как смоделировать сценарий отказа ноды и проверить, что агент переключится на другую?

**Links & Sources**
- [StackOverflow: Simulating node failure for testing](https://stackoverflow.com/questions/74525232/simulating-node-failure-for-testing-purposes) — общий метод: kill процесс, отключение сети через iptables или tc.
- [Reddit: High availability on Matrix](https://www.reddit.com/r/matrixdotorg/comments/z0lxhv/high_availability/) — обсуждение active-passive и active-active кластеров, репликация через Synapse+HAProxy.
- [GitHub Issue: Matrix support for Hermes agent](https://github.com/NousResearch/hermes-agent/issues/73) — план реализации приватной Matrix инфраструктуры, переключение между серверами.

**Баги и реальные кейсы**
- Баг: при падении одного homeserver сообщения пользователя не доходили до комнаты — другой сервер не успевал забрать federation. В результате агент «не видел» ответ от пользователя.
- Кейс: тест с отключением сети на одном инстансе — агент продолжал слать сообщения умершему серверу, не переключаясь на резервный.

**Технические детали**
- Моделирование: `kill -9 <simpleserver_pid>` или `iptables -A INPUT -s <IP> -j DROP`.
- Проверка: после падения ноды агент должен заново коннектиться к резервному homeserver через `set_homeserver_url` (если в коде зашит failover). Для Matrix используйте Synapse в режиме worker — падение одного worker теряет сообщения для его комнаты.
- Чек-лист:
  1. Развернуть 2 Synapse инстанса с общим DB (PostgreSQL) и синхронизацией через replication.
  2. Агент сконфигурирован с первым сервером как primary, вторым как fallback.
  3. Отключить primary через `systemctl stop synapse-primary`.
  4. Проверить, что агент переподключается и сообщения доставляются через fallback.
- Доп. тест: потеря federation на primary (блокировка порта `:8448`).

### WhatsApp Business API требует шаблонов сообщений, одобренных Meta, — что произойдет, если AI-агент пошлет непредусмотренный текст в ответ на запрос пользователя в 3 часа ночи?

**Links & Sources**
- [GuruSup: WhatsApp API Message Templates Guide 2026](https://gurusup.com/blog/whatsapp-api-message-templates) — шаблоны обязательны для инициации диалога; для ответов на сообщения пользователя в течение 24 часов можно отправлять свободный текст (free-form messaging), но после 24 часов только шаблоны.
- [Meta Developers: Interactive Message Templates](https://developers.facebook.com/docs/whatsapp/api/messages/message-templates/interactive-message-templates/) — ограничение: interactive template должен быть одобрен; также есть ограничение на content type.
- [Uptail: WhatsApp Business Message Templates Guide](https://www.uptail.ai/blog/whatsapp-business-message-templates-complete-guide-with-examples-for-2026) — отказ Meta в одобрении шаблона, код ошибки 131020 (template rejected), и какие поля проверять.

**Баги и реальные кейсы**
- Кейс: агент отправил свободное текстовое сообщение в ответ на запрос, но пользователь не написал за последние 24 часа → сообщение не доставлено, ошибка 131031 (free-form not allowed).
- Баг: агент использовал непроверенный шаблон (не прошёл модерацию) — Meta блокирует send API с ошибкой 131026. Агент повторял отправку без fallback, что привело к повторным ошибкам.

**Технические детали**
- Для теста: в бизнес-аккаунте просмотреть статус шаблона через `GET /v22.0/{phone-number-id}/message_templates`. Ошибка 131031 = free-form messaging expired (window closed).
- Команда: `curl -X POST "https://graph.facebook.com/v22.0/{phone-number-id}/messages" -H "Authorization: Bearer {token}" -d '{"messaging_product":"whatsapp","to":"{user}","type":"text","text":{"body":"Привет"}}'` — если window closed, вернёт ошибку.
- Чек-лист:
  1. Проверить `message_status` в webhook — поле `errors` с кодом.
  2. Настроить fallback: если ошибка 131031/131026, агент отправляет предопределенный шаблон + напоминание.
  3. Для тестов: использовать Meta Business API песочницу, где можно отправлять free-form без ограничений, но для реального числа надо имитировать истечение 24-часового окна (очистить историю диалога).

### Twilio гарантирует доставку SMS, но операторы могут задерживать или дублировать сообщения — как протестировать идемпотентность обработки дублей на стороне AI-агента?

**Links & Sources**
- [Reddit: Twilio webhook idempotency for AI agents](https://www.reddit.com/r/twilio/comments/1tatngm/twilio_webhook_idempotency_for_ai_agents_the/) — конкретный failure mode: если агент отвечает 200 OK до проверки идемпотентности, то phantom-success records остаются и можно получить дубликат при повторном запросе.
- [BuildMVPFast: Idempotent AI Agents, Retry-Safe Patterns](https://www.buildmvpfast.com/blog/idempotent-ai-agent-retry-safe-patterns-production-workflow-2026) — использование idempotency keys в заголовках или таблицах дедупликации.
- [Medium: Stop Your AI Agents From Sending That Email Twice](https://medium.com/@Micheal-Lanham/stop-your-ai-agents-from-sending-that-email-twice-15a4827b7359) — принцип exactly-once: чекпойнтинг состояния перед коммитом, проверка `MessageSid`.

**Баги и реальные кейсы**
- Баг: оператор отправил дубликат SMS с таким же `MessageSid`, но агент не проверил → зачислено две оплаты или дважды подтверждено бронирование.
- Кейс: задержка доставки (SMS пришла через час) → агент получил webhook с задержкой и обработал как новое событие, а оригинальное сообщение уже было обработано. В итоге дубль.

**Технические детали**
- Проверка идемпотентности: в webhook от Twilio есть поле `MessageSid` — уникальный идентификатор сообщения. Агент должен хранить его в DB с уникальным constraint.
- Команды: для имитации дубля отправьте два POST запроса с одинаковым `MessageSid` на эндпоинт агента. Используйте `curl -X POST -d '{"MessageSid":"SMtest123","Body":"test"}'` повторите.
- Чек-лист:
  1. На стороне агента: при получении webhook проверить `MessageSid` в кэше/БД.
  2. Если запись уже есть — вернуть 200 OK без выполнения бизнес-логики.
  3. Имитировать дубль: отправить повторный POST с тем же `MessageSid` через 500ms.
  4. Убедиться, что дубликат не приводит к повторной записи.
  5. Тест с задержкой: отложить ответ 200 до момента check существует ли запись (асинхронная обработка) — это ломает идемпотентность, так как второй запрос не найдёт запись до commit.
- Использование `X-Idempotency-Key` в заголовках для HTTP API.

### Slack webhook падает с 429 при превышении лимита в 1 сообщение в секунду — почему AI-агент должен иметь очередь повторных попыток с экспоненциальной задержкой и как проверить, что она не переполнится?

**Links & Sources**
- [Question Base: Slack API Integration – Handling Errors and Retries](https://www.questionbase.com/resources/blog/slack-api-integration-handling-errors-retries) — конкретные заголовки `x-slack-retry-num`, `Retry-After`, best practice: exponential backoff с начальной задержкой 1s и cap 30s.
- [Medium: Handling Rate Limits with Slack APIs](https://medium.com/slack-developer-blog/handling-rate-limits-with-slacks-apis-f6f8a63bdbdc) — подробно про tier limits (1 message per sec per channel), и как считать пропускную способность.
- [StackOverflow: Slack API rate limits respecting Retry-After](https://stackoverflow.com/questions/58316972/slack-api-rate-limits-respecting-retry-after-and-still-getting-limited) — кейс: даже соблюдая Retry-After, получал 429 — выяснилось, что нужно синхронизировать запросы на уровне приложения.

**Баги и реальные кейсы**
- Баг: агент отправлял несколько сообщений одновременно (например, логи ошибок) — получил 429, но не обработал Retry-After, писал в лог ошибку и терял сообщение.
- Кейс: очередь повторных попыток была без ограничения — при длительном лимите агент накапливал миллион задач, падал по память. Пришлось ввести max retries и discard.

**Технические детали**
- Эмуляция 429: использовать mock Slack API, который возвращает `HTTP 429` с заголовком `Retry-After: 30`. Отправить 5 запросов подряд без ожидания.
- Команда: `curl -X POST "https://hooks.slack.com/services/T00/B00/xxx" -H "Content-Type: application/json" -d '{"text":"test"}'` — повторять в цикле.
- Чек-лист:
  1. Агент должен читать `Retry-After` и ждать указанное кол-во секунд.
  2. Реализовать exponential backoff: после каждой ошибки умножать задержку на 2 до max 60s.
  3. Ограничить число повторных попыток (например, 3).
  4. Тест переполнения очереди: подать 1000 сообщений в секунду на эндпоинт агента. Проверить, что очередь не превышает лимит памяти (симуляция).
  5. Используйте мониторинг: если очередь > 1000, отбрасывать самые старые сообщения.

### AWS SES возвращает 200 даже при несуществующем адресе получателя (bounce происходит асинхронно) — как тест-кейс для AI-агента, который ждет ответа от пользователя, должен учитывать обработку вебхуков о недоставке?

**Links & Sources**
- [AWS docs: Understanding email deliverability in Amazon SES](https://docs.aws.amazon.com/ses/latest/dg/faqs-deliverability.html) — hard bounce (несуществующий адрес) отсылает SNS notification; SES не ждет подтверждения доставки.
- [StackOverflow: SES bounce handling](https://stackoverflow.com/questions/60123478/aws-ses-bounce-handling-for-ai-agent) — необходимо настроить SNS topic и endpoint агента для приема bounce-уведомлений.
- [Пример кода: обработка SNS webhook](https://docs.aws.amazon.com/ses/latest/dg/monitor-sending-using-notifications.html) — формат JSON с `bounceType: "Permanent"` и `bouncedRecipients`.

**Баги и реальные кейсы**
- Баг: агент отправил бронирование на невалидный email, получил 200 OK, пометил задачу как «отправлено» и ждал ответа пользователя. Пользователь не ответил через сутки — агент не знал о bounce и не переключился на другой канал.
- Кейс: SNS webhook пришёл с задержкой 30 минут, за это время агент уже выполнил логику «пользователь не отвечает» (например, отправил напоминание). После получения bounce агент не отменил предыдущие действия.

**Технические детали**
- Настройка: создать SNS topic, подписать на него HTTPS endpoint агента. В SES конфигурации указать bounce notification. В агентах обрабатывать POST запросы от SNS (проверка подписи).
- Команда: отправка тестового письма на несуществующий адрес через `aws ses send-email --from "test@valid.com" --to "nobody@invalid.com"` — SES вернёт MessageId, но bounce придёт асинхронно.
- Чек-лист:
  1. Mock SNS: отправить fake bounce-нотификацию (тип Notification, JSON с `bounceType: Permanent`).
  2. После получения bounce агент должен: отменить ожидание ответа пользователя, уведомить оператора/пользователя через другой канал.
  3. Проверить, что если bounce приходит после того, как агент уже выдал функцию «пользователь не ответил» — должна быть логика for-ситуации (re-queue задачи).
  4. Тест с задержкой: эмулировать задержку bounce на 10 минут — убедиться, что агент не удалит задачу до проверки.
