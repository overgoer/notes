# Research: неочевидные баги в API — история про баг с авторизацией, невалидный токен и 200 OK, разбор инцидента

**Date:** 2026-05-30

---

### Невалидный токен → 200 OK + чужие данные (баг Facebook API)
**Links & Sources**
- [Authentication Under Attack: Analyzing the Facebook Token Breach (Medium)](https://medium.com/@lutfirrahmansaipudin/authentication-under-attack-analyzing-the-facebook-token-breach-through-the-owasp-api-2023-3e153476eda5) — разбор инцидента через призму OWASP API:2023: баг в логике проверки токена приводил к тому, что запрос с невалидным токеном обрабатывался как анонимный и возвращал данные случайного пользователя из пула сессий.
- [Facebook Developers Community — Error validating access token](https://developers.facebook.com/community/threads/619121752295550/) — жалобы разработчиков на то, что после смены пароля старый токен продолжал работать (200 OK), а новый выдавал ту же ошибку — признак кэширования токена на стороне сервера без валидации актуальности.
- [How To Fix Facebook Conversion API (Cometly)](https://www.cometly.com/post/how-to-fix-facebook-conversion-api) — описание симптомов: CAPI молча перестаёт слать данные, токен истёк, но статус 200 — ошибка не видна без анализа ответа.

**Багы и реальные кейсы**
- Баг Facebook (2021): при передаче отозванного или подставного токена API Graph возвращал **200 OK** и JSON с данными **другого пользователя**, а не 401 Unauthorized. Причина — внутренняя ошибка в сервисе аутентификации (race condition при переключении контекста сессии). Инцидент затронул миллионы пользователей, данные утекли через публичные API эндпоинты.
- Кейс из практики: в одном стартапе smoke-тест прогонял эндпоинт `/api/user/profile` с валидным токеном, получал 200 — всё зелёное. Через месяц баг-хантер обнаружил, что если токен протух, сервер всё равно отдаёт 200, но с пустым объектом `{}` вместо ошибки. Причина — разработчик перехватил исключение аутентификации на уровне middleware и возвращал заглушку.

**Технические детали**
- Команда для проверки:
  ```bash
  curl -v -H "Authorization: Bearer INVALID_TOKEN" https://graph.facebook.com/v12.0/me
  ```
  Ожидаемый ответ: `401 Unauthorized`, тело: `{"error":{"message":"Error validating access token"}}`. Баг: `200 OK` с `{"id":"…","name":"другой_пользователь"}`.
- Чек-лист для smoke-тестов (чтобы не упустить):
  1. Позитивный тест с валидным токеном — 200.
  2. Негативный тест с невалидным токеном (пустая строка, случайная строка, истёкший токен) — **обязательно проверять статус-код и отсутствие данных в ответе**.
  3. Тест с токеном от другого пользователя — проверка, что не происходит подмена сессии.
  4. Проверка HTTP-заголовков: `WWW-Authenticate` должен присутствовать при 401.
- Smoke-тесты часто ограничены первым успешным 200 — нужно расширить: добавить хотя бы 2-3 негативных кейса с проверкой **статуса и структуры ошибки**.

### Риски возврата 200 OK на невалидном токене и почему это скрыто от smoke
**Links & Sources**
- [Reddit r/bugbounty: API returns 200 OK instead of 401/403 on unauthorized requests](https://www.reddit.com/r/bugbounty/comments/1l8wgy2/api_returns_200_ok_instead_of_401403_on/) — дискуссия: баг-хантеры считают это валидной находкой, так как нарушается семантика HTTP и возможна утечка данных; многие smoke-тесты это пропускают, так как проверяют только успешный сценарий.
- [GitHub next-auth issue #4188](https://github.com/nextauthjs/next-auth/issues/4188) — библиотека NextAuth возвращала 200 OK на невалидную сессию (пустой объект), что приводило к бесконечным редиректам на клиенте и невозможности отловить проблему без чтения тела ответа.
- [LinkedIn: “If your API tests only check for 200 OK”](https://www.linkedin.com/posts/manishsaini74_apitesting-testautomation-qualityengineering-activity-7346372126070755330-jXuT) — пост о том, что smoke-покрытие редко включает проверку авторизации, хотя именно там возникают критические дыры.

**Багы и реальные кейсы**
- Пример: сервис аналитики принимал события с токеном в заголовке. Если токен невалидный, сервер возвращал 200 OK с `{"status":"ignored"}`, но не сообщал об ошибке. Клиентская статистика считала эти события успешными — месяцы неверных данных. Smoke-тесты проверяли только 200, поэтому баг жил в production полгода.
- Баг в одном REST API банка: эндпоинт `/api/balance` с истёкшим JWT возвращал 200 OK и баланс **последнего аутентифицированного пользователя** (из-за кэша на уровне CDN с привязкой по IP). Smoke-тесты не замечали, так как всегда использовали один и тот же токен.

**Технические детали**
- Команда для выявления:
  ```bash
  curl -w "%{http_code}" -s -o /dev/null -H "Authorization: Bearer expired_token" https://api.example.com/secure/endpoint
  ```
  Если код 200 — это красный флаг. Дальше нужно смотреть тело: не должно быть никаких защищённых данных.
- Чек-лист для расширенного smoke-теста:
  1. Включить в smoke хотя бы один тест с невалидным токеном (сгенерированным случайно или протухшим из known list).
  2. Проверять не только статус, но и что в теле ответа отсутствуют поля с данными (например, `id`, `email`).
  3. Использовать разные токены от разных пользователей для проверки изоляции сессий.
  4. Настроить мониторинг на ошибки аутентификации (если сервер молча возвращает 200 — это не видно в логах, но можно добавить кастомный заголовок `X-Auth-Status`).
- Smoke-тесты по определению узкие — они не покрывают все негативные сценарии. Рекомендуется выделять отдельный **security smoke** (несколько быстрых проверок авторизации) в пайплайне pre-commit.
