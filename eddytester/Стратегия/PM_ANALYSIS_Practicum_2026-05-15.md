## 2. IMPROVEMENT ZONES (оценка по decision_criteria)

### ZONE A: Больше багов в API (HIGH PRIORITY)

**Проблема:** Сейчас в API ~23 BUG-аннотации. В боте — 4 задачи. Разрыв.
**Решение:** Добавить 10-15 новых багов в v0-test-api, синхронизировать с ботом.

**Какие баги добавить (конкретика):**

| # | Баг | Тип | Где | Сложность |
|---|-----|-----|-----|-----------|
| 1 | **Duplicate emails** — POST /users создаёт пользователя с уже существующим email (нет уникальности) | Валидация | БЭК | Лёгкий |
| 2 | **Age = 150 проходит** — граничное значение не проверяется сверху | Валидация | БЭК | Лёгкий |
| 3 | **DELETE возвращает 200 с пустым телом вместо 204** | HTTP-спецификация | БЭК | Лёгкий |
| 4 | **PUT не требует тело** — 200 OK с null вместо 400 Bad Request | Валидация | БЭК | Средний |
| 5 | **PATCH игнорирует поля** — обновляет только name, age остаётся старым | Логика | БЭК | Средний |
| 6 | **Content-Type: text/plain на DELETE** — как на GET, но на другом методе | Заголовки | БЭК | Лёгкий |
| 7 | **Cache-Control: no-store нет** — ответ кэшируется браузером | Безопасность | БЭК | Лёгкий |
| 8 | **Idempotency key не проверяется** — повторный POST с тем же ключом создаёт дубль | Идемпотентность | БЭК | Средний |
| 9 | **Offset пагинации с отрицательным значением** — -1 даёт 500 | Обработка ошибок | БЭК | Лёгкий |
| 10 | **Rate limiting отсутствует** — 1000 запросов в секунду проходят | Безопасность | БЭК | Средний |
| 11 | **CORS: Access-Control-Allow-Origin: *** — любой сайт может читать API | Безопасность | БЭК | Лёгкий |
| 12 | **JSON inject: name = `{"age": 999}`** — если name передать объектом, API съедает | Фаззинг | БЭК | Средний |
| 13 | **500 на специальные символы** — name = `<script>alert(1)</script>` → 500 | XSS/санитизация | БЭК | Средний |
| 14 | **PUT /users/:id — создаёт нового при несуществующем ID** вместо 404 | REST-спецификация | БЭК | Средний |
| 15 | **PATCH /users/:id — не возвращает обновлённый объект** (возвращает старый) | Логика | БЭК | Лёгкий |

**Добавлять порциями:** первые 5 багов — до запуска продаж. Остальные — по мере.

### ZONE B: Больше HTTP-методов (MEDIUM PRIORITY)

**Проблема:** Сейчас в API только CRUD (GET, POST, PUT, PATCH, DELETE). Нет продвинутых методов.

**Что добавить:**

| Метод | Endpoint | Что делает | Баг-потенциал |
|-------|----------|------------|---------------|
| **HEAD** | /api/head/users | Возвращает только заголовки без тела | Может не возвращать Content-Length — кэширующий баг |
| **OPTIONS** | /api/options/users | Возвращает разрешённые методы | Может врать — показывать PUT, но PUT не работает |
| **PURGE** | /api/purge/cache | Сбрасывает кэш (нестандартный, но есть в API-шлюзах) | Нет авторизации — любой сбрасывает |
| **TRACE** | /api/trace/echo | Возвращает запрос как есть (диагностика) | Может выдать заголовки авторизации — security hole |

**Приоритет:** HEAD и OPTIONS — простые, добавить до запуска. PURGE и TRACE — для Pro-тарифа.

### ZONE C: SQL-доступ к БД для пользователей (HIGH RISK, HIGH REWARD)

**Идея:** Дать пользователям Practicum возможность подключиться к БД PostgreSQL через read-only аккаунт и писать SQL-запросы. Проверять данные, которые они создали через API.

**Как это может выглядеть:**
- После покупки пользователь получает credentials к read-only SQLite/PostgreSQL
- Может подключаться через любой SQL-клиент (DBeaver, psql, DataGrip)
- Видит только свои данные (WHERE created_by = user_id)
- Может писать SELECT, но не INSERT/UPDATE/DELETE

**Что это даёт:**
- Уникальное УТП: "не только API, но и БД"
- Навык: тестировщик видит данные на уровне БД — сильный скилл для продакшна
- Дополнительные баги: данные в API и в БД могут расходиться (например, API показывает age=25, а в БД age=null — баг синхронизации)

**Риски:**
| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| SQL-инъекция через read-only доступ | Низкая | Read-only + ограниченные права + отдельная изолированная БД |
| Нагрузка на сервер | Средняя | SQLite для каждого юзера или отдельный инстанс Postgres |
| Сложность настройки | Высокая | Выделить день на devops |
| Путаница у новичков | Высокая | Только для Pro-тарифа + гайд "как подключиться" |

**Решение:** Добавить SQL-доступ **только для Pro-тарифа** (3990₽). Как апсейл после базового. Базовый — API + видео. Pro — API + видео + SQL-доступ + чат.

**Архитектурно:**
```
Юзер → API (создаёт данные) → PostgreSQL
Юзер → Read-only SQL user → те же данные через SELECT
```
Раз в N минут — проверка: совпадают ли данные из API и из БД. Если нет — это баг (и это фича).

**Что есть у Русова (конкурент):** Telegram-игра с 25 секретными кодами, спрятанными в курсе.

**Что может быть у тебя:**
- **Система уровней в боте:** "Новичок" (нашел 3 бага) → "Охотник" (10 багов) → "Мастер" (все баги)
- **Ачивки:** "Первый баг", "Баг-хантер" (нашёл баг, который пропустили другие), "SQL-мастер" (выполнил первый запрос)

- **"Баг дня":** каждый день новый скрытый баг в API — кто найдёт первым, получает значок

**Приоритет:** Низкий до запуска. Добавить через месяц после старта продаж.
