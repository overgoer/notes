# POST /orders — один из трёх ответов баг. Какой? 🔥

Отправляю POST /orders с купоном. Получаю три разных ответа.

**Вариант A**
Сумма заказа = 0. Купон на 100% — сработал. Заказ создан (201).
Товар есть. Денег нет.

**Вариант B**
Товар удалили из каталога. Я отправляю заказ — 201 Created.
Товара в базе нет. Отгрузить нельзя.

**Вариант C**
Всё ок. Заказ создан. Но HTTP-статус — 200.
Заголовка Location нет.

---

**Какой из трёх вариантов — баг?**

🤔 A — сумма 0
🤔 B — заказ с мёртвым товаром
🤔 C — 200 вместо 201

Пиши свой вариант 👇

🔥 если работали с купонами
👍 если ловили хотя бы один из этих кейсов

---

**Разбор (для нетерпеливых):**

Баги — ВСЕ три. Это не викторина «найди один», а ловушка.

Но самый опасный — **вариант C**.
200 вместо 201 = платёжный шлюз может решить, что запрос не прошёл, и списать деньги повторно. Двойной платёж, возвраты, поддержка горит.

А) Сумма 0 — проблема бизнес-логики. Сервер не проверяет мин. сумму. Нужен 422.
Б) Товара нет — проблема валидации. Сервер не проверяет актуальность. Тоже 422.
В) 200 вместо 201 — нарушение RFC. Post создаёт ресурс → должен быть 201 + Location.

Какой вариант казался нормой? Пиши 👇
