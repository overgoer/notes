## 📋 РАЗБОР (ответ через час)

### POST без тела → 500. Разбираю с доказательствами.

Час прошёл. Давайте разбирать.

Правильный ответ: **А) Бэкенда — должен отдавать 400 Bad Request.**

Почему? Разбираю по шагам.

---

### Шаг 1. Воспроизвожу

Запрос №1 — пустое тело:

```bash
curl -v -X POST https://api.practicum.com/users \
  -H "Content-Type: application/json" \
  -d ''
```

Ответ:
```
HTTP/1.1 500 Internal Server Error
```

Запрос №2 — пустой JSON:

```bash
curl -v -X POST https://api.practicum.com/users \
  -H "Content-Type: application/json" \
  -d '{}'
```

Ответ:
```
HTTP/1.1 500 Internal Server Error
```

---

### Шаг 2. Смотрю в логи сервера

```
ERROR: NullPointerException
at com.practicum.UserController.createUser(UserController.java:42)
Cause: 'name' is null
```

Сервер **падает с ошибкой**, потому что не ожидает пустых данных. Это не обработка — это авария.

---

### Шаг 3. Что говорит RFC?

RFC 7231, раздел 6.5.1:

> **400 Bad Request** — сервер не может обработать запрос из-за синтаксической ошибки или невалидных данных.

Пустое тело → невалидный запрос → **должен быть 400**, а не 500.

500 = «я сломался, но не знаю почему».  
400 = «ты прислал фигню, пересылай».

Разница принципиальная.

---

### Шаг 4. Как должно быть

Правильная обработка:

```python
def create_user(request):
    if not request.body:
        return {"error": "Body is required"}, 400

    data = json.loads(request.body)
    if not data.get("name"):
        return {"error": "Name is required"}, 400

    # создаём пользователя
    user = User.create(data)
    return user.to_json(), 201
```

Сервер должен:
1. Проверить, что тело есть и它不是 пустое
2. Проверить обязательные поля
3. Только потом создавать ресурс

---

### Шаг 5. Почему вариант «Б» (клиент идиот) — неверный

Клиент может быть любым:
- Мобильное приложение с багом
- Postman с ручным запросом
- Curl из скрипта
- Другой микросервис со своим форматом

**Ты не контролируешь клиента.** Сервер — последняя линия защиты.

Если сервер падает на пустом теле — это баг бэкенда. Точка.

---

### Шаг 6. Почему вариант «В» (норма) — тоже неверный

500 = Internal Server Error = «всё сломалось, я не знаю что делать».

Если это «нормальная реакция» — значит сервер постоянно в панике. Это:
- 📉 Падает uptime (мониторинг дёргается)
- 📉 Нет понятного ответа клиенту
- 📉 Непонятно, что исправлять

500 — это всегда исключение. Для **ожидаемых** плохих запросов есть 400.

---

### Итог: чек-лист для тестирования POST

При тестировании любого POST-запроса проверяй:

| Сценарий | Ожидаемый статус |
|----------|-----------------|
| 🔴 Тело пустое (no body) | 400 |
| 🔴 Тело = пустой JSON `{}` | 400 |
| 🔴 Обязательное поле отсутствует | 400 |
| 🔴 Невалидный тип поля | 400 |
| 🟢 Всё корректно | 201 Created |
