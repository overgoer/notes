# Music Server — план реализации и здоровье сервера

## Железные правила

> **Главное:** ни один существующий сервис не должен пострадать. Музыкальный стек — гость на сервере, а не хозяин.

### 1. Изоляция

- Всё — только в Docker-контейнерах. Никаких установок пакетов в систему, никаких npm/pip install в глобальном окружении.
- Контейнеры используют свои порты. Сверяться с уже занятыми (проверять `ss -tlnp` или `docker ps`).
- Папка `/music` — отдельная, не пересекается с `/root/`, `/var/www/` и другими рабочими директориями.
- Не менять конфиги существующих сервисов (Nginx, Apache, PM2, Hermes, BSA).

### 2. Ресурсы

- **RAM:** Текущее потребление ~3.8GB. Добавляем не более 500MB на весь музыкальный стек. Если какой-то контейнер начинает жрать больше — выключаем.
- **Swap:** Уже почти полный (1.9GB/2GB). Если после запуска стека swap начал расти — значит не хватает памяти. Надо либо уменьшать нагрузку, либо докупать RAM.
- **CPU:** Должен быть Idle >50% большую часть времени. Если нагрузка стабильно >80% — проблема.
- **Диск:** Следить за `/music/library` — не дать ему сожрать весь диск. Установить лимит через Docker volume.

### 3. Правила развёртывания

- Каждый компонент запускается и **тестируется отдельно**. Никогда не запускаем всё сразу и не надеемся что заработает.
- На каждый контейнер — `docker run` или docker-compose. Перед запуском — проверка, что порт свободен.
- Если компонент не запустился за 30 секунд — разбираемся, не запускаем следующее.
- Если что-то пошло не так, а откат невозможен — останавливаюсь и зову Эда.

### 4. Критерии, что сервер здоров

| Показатель | Норма | Тревога |
|------------|-------|---------|
| RAM usage | <80% (6.2GB) | >90% (7GB) — жму тормоза |
| Swap usage | <500MB | >1.5GB — память в дефиците |
| CPU load average | <2.0 (на 4 ядрах) | >3.5 — перегруз |
| Disk usage | <80% | >90% — срочно чистить |
| Docker контейнеры | Все running | Есть unhealthy/exited |
| Существующие сервисы | Доступны как и были | Любой перестал отвечать |

### 7. Откат

Удаление музыкального стека:
- `docker stop` всех контейнеров
- `docker rm` контейнеров
- Удалить `/music`
- Никаких следов в системе не остаётся. Это Docker — `rm -rf` работает идеально.
