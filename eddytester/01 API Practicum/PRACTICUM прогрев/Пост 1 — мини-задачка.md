# Пост 1 — мини-задачка с провокацией

**Статус:** ✅ Готово

**Цель:** Сбор вовлечения (комментарии, гипотезы), создание интриги

**Публикация:** Основной канал

---

## Текст поста

```
🔍 Мини-задачка для тех, кто работает с API

Вот фрагмент документации:
Эндпоинт: POST /api/v1/orders
Тело запроса: {"product_id": 123, "quantity": 1}
Заголовок: Authorization: Bearer abc123
Ответ при успехе: 201 Created

А вот реальный ответ сервера на этот запрос:
→ Статус: 200 OK
→ Тело: {"order_id": 456, "status": "failed", "token": "xyz789"}

❓ Что не так?

Пиши гипотезу в комменты. Не бойся ошибиться — разберём лучшие ответы.

Для тех, кто хочет проверить свою гипотезу на живом эндпоинте — 
завтра будет разбор + мини-практикум в боте.

P.S. Тем, кто не работает с API — следующий пост будет про 
фронтенд-тестирование. Не отписывайтесь 😌
```

---

## 🐛 Баги в задаче (для разбора)

| № | Баг | Уровень |
|---|-----|---------|
| 1 | 200 OK вместо 201 Created | Мидл+ |
| 2 | status: "failed" при 200 OK | Все |
| 3 | order_id: 456 при status: failed | Сеньор |
| 4 | Новый токен в ответе при failed | Сеньор (security) |

---

## 📊 Ожидаемые реакции

**Юниор:**
- «200 — значит успех, а status: failed — это странно»

**Мидл:**
- «Должен быть 201, status failed при 200 — баг»

**Сеньор:**
- «Почему новый токен выдали при failed? Это security issue! order_id не должен генерироваться при failed!»
