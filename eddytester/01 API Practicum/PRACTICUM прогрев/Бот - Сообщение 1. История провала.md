
Давай разберём на примере, с чем возникают сложности при поиске багов. Расскажу на своём опыте.

Один из первых рабочих багов:

Я тестировал метод создания заказа, отправлял данные о заказе в Postman и проверял их наличие в БД

На тесте все ок, проблем не вижу - веду задачу в релиз, ребята релизят, и мгновенно после раскатки на прод ... 0 заказов в работе! 100% новых заказов зависают сразу после создания и не идут по процессу дальше.

А всё дело в мелочи: заказ создается со статусом  `"NEW "` вместо `"NEW"` и просто игнорируется системой. Пользователь оплатил товар, но склад его не видит, письмо с чеком и деталями об успешной покупке не уходит клиенту. Заказы копятся в базе «мертвым грузом».

Я читал документацию.
Разработчик тоже читал.
Мы оба не подумали о пробелах, потому что привыкли работать в «стерильных» условиях учебных задач.

В чем урок?
На реальном проекте тебя не бьют по рукам сразу. Тебе дают выстрелить себе в ногу. Без практики на «кривых» API ты просто не ждешь подлянки там, где все горит зеленым. 

Только багованный API учит вниманию к деталям, которое отличает сильного QA от того, кто просто жмет кнопки. 

Жми «Дальше», чтобы я рассказал, как научиться видеть такие ловушки заранее.
