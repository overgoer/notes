## Цель курса (уточнено)

**Главная цель:** вау-эффект + ощущение «я разобрался». Студент приходит с нулём — уходит с чувством что он реально тестировал API, нашёл баги, и это было круто.

Образовательная ценность — важна, но подчинена эмоциональному отклику. Студент должен не столько ЗНАТЬ, сколько ЧУВСТВОВАТЬ: «я могу это делать».

## 5 проблем текущего курса

1. **8 уроков теории до первой практики.** Студент теряет мотивацию, теория без немедленного применения забывается.
2. **Размытые домашки.** «Бегло просмотреть документацию», «Изучить структуру HTTP» — нет конкретики: сколько тест-кейсов, какие значения, куда прислать.
3. **«Практика» = пассивный просмотр.** 90% времени Эдди показывает, 10% — «теперь сам». Нет моментов «открой Postman и делай вместе со мной».
4. **Тест-кейсы обсуждаются, но не требуются.** Студент не сдаёт тест-кейсы как артефакт — только отмечает найденные баги в чекбоксах.
5. **V2 используется один раз (l11) и забывается.** Сравнение v1/v2 — главная фишка курса — не используется в l12-l15.

## Акт I: Экипировка (уроки 1-4)

Цель: студент настроил инструменты, понимает что такое API, умеет мыслить тест-дизайном.

### Урок 1. Введение (10 мин)

- Экран: 3 скриншота — тест-кейс, баг-репорт, заполненный чек-лист /glearning
- «Вот что ты научишься делать. Вот что пришлёшь мне. Вот что будет в твоём портфолио.»
- **ДЗ:** открыть /glearning, ввести ключ, убедиться что работает

### Урок 2. Postman + первый запрос (20 мин)

- Скачать → импортировать коллекцию → авторизация → POST /users → ответ → Reset DB
- **ДЗ:** установить Postman, импортировать коллекцию, создать первого пользователя, очистить базу

### Урок 3. HTTP + API за один раз (25 мин)

- Только то что нужно тестировщику: структура запроса, структура ответа, методы, статус-коды
- API = договор: «сервер обещает, ты проверяешь»
- **ДЗ:** сделать 5 запросов разными методами, посмотреть на ответы

### Урок 4. Тест-дизайн для API (25 мин)

- Классы эквивалентности + граничные значения на КОНКРЕТНЫХ полях нашего API (age, name, id)
- Сразу в Postman: «age=17 → статус minor», «age=66 → retired»
- «Где прячутся баги: границы, неожиданные типы, пустые значения»
- **ДЗ:** составить таблицу классов для age, name и status

## Акт II: Полный цикл на GET /users (уроки 5-8)

Цель: студент проходит ПОЛНЫЙ путь один раз с максимальной поддержкой.

### Урок 5. GET /users — Теория + План (15 мин)

- Читаем доку вместе → составляем тест-план (шаблон на экране)
- Пункты: структура ответа, типы данных, фильтр status, заголовки, статус-коды
- **ДЗ:** заполнить шаблон тест-кейсов (минимум 10 штук)

### Урок 6. GET /users — Практика (25 мин) ★ КЛЮЧЕВОЙ УРОК

- **Один спойлер — Content-Type.** Полный цикл: нашёл → тест-кейс → баг-репорт
- «Я показал процесс на одном примере. Дальше — ты сам по плану из урока 5.»
- Экран показывает где смотреть, НЕ говорит что там не так
- **ДЗ:** пройти все тест-кейсы, найти баги, отметить на /glearning

### Урок 7. Баг-репорт (20 мин)

- Разбор шаблона на багах которые студент УЖЕ нашёл
- **ДЗ:** оформить баг-репорты на ВСЕ найденные баги в GET /users. Прислать.

### Урок 8. V2: Эталон (15 мин)

- Импорт v2, сравнение v1 vs v2
- «V2 — твой помощник. Всегда сравнивай.»
- **ДЗ:** импортировать v2, сравнить все ответы GET /users

## Акт III: Самостоятельная практика (уроки 9-12)

Прогрессия: образец → методика → подсказки → экзамен.

**Единый шаблон видео (~15-18 мин):**
1. Читаем доку (2 мин)
2. План тестирования — специфика метода (5 мин)
3. Показываю где смотреть, НЕ анализирую (5 мин)
4. Пауза «теперь ты» (1 мин)
5. Мост к V2 — «сравни» (2 мин)

**ДЗ для каждого:** 3 артефакта (тест-кейсы → результаты → баг-репорты).

### Урок 9. GET /users/:id (18 мин)
Прогрессия: **«делай по образцу»**

- Специфика: id как часть пути, не query-параметр
- Указатели без спойлеров: id=1, несуществующий, abc, -1, большое число

### Урок 10. POST /users (18 мин)
Прогрессия: **«делай с методикой»**

- Специфика: тело запроса, валидация, Content-Type запроса, 201 vs 200
- Указатели: границы age, спецсимволы в name, уникальность, лишние поля

### Урок 11. PATCH /users/:id (15 мин)
Прогрессия: **«почти сам»**

- Специфика: full update vs partial — СЮРПРИЗ «документация говорит partial, сервер требует все поля»
- Подсказки без прямой демонстрации

### Урок 12. DELETE /users/:id (15 мин)
Прогрессия: **«экзамен»**

- Специфика: soft delete, статус-код 204, безопасность
- Эдди почти ничего не показывает. Только список что проверять.

## Принципы перезаписи

### Про спойлеры
- **Один спойлер на весь курс** — в уроке 6 (GET /users, Content-Type)
- Остальные уроки: экран показывает данные, Эдди говорит ГДЕ смотреть, но НЕ говорит что там
- «Вот заголовки. В доке написано что должно быть. Наша задача убедиться. Пауза — проверь сам.»

### Про паузы
- В каждом практическом видео — явная остановка «теперь ты»
- Видео становится диалогом, не лекцией

### Про домашние задания
- Каждый практический урок → 3 артефакта:
  1. Тест-кейсы по шаблону
  2. Результаты тестирования
  3. Баг-репорты
- Отправка на eddythetest@gmail.com
- Формат: конкретные значения для проверки, количество тест-кейсов, куда слать

### Про V2
- V2 — постоянный инструмент сравнения, не отдельный урок
- В каждом практическом уроке: «сравни v1 и v2»
