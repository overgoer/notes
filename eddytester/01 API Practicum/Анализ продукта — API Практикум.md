# Анализ продукта — API Практикум

## 1. Что есть: карта кода и транскриптов

### Репозиторий API (v0-test-api)

**Сервер:** Express.js + PostgreSQL, PM2, порт 3000.

**Две версии роутов:**
- `/v1/api/users` — с багами (V1)
- `/v2/api/users` — без багов (V2)

**5 эндпоинтов на каждую версию:**

| Метод | Endpoint | V1 — что сломано | V2 — что починено |
|--------|----------|-------------------|-------------------|
| GET / | Список пользователей | Content-Type text/plain (вместо application/json), нет X-Content-Type-Options/X-Frame-Options, Cache-Control: public (должен быть приватным) | Content-Type правильный, заголовки безопасности есть |
| GET /:id | Пользователь по ID | id-1 вместо id (сдвиг на 1) | Правильный id |
| POST / | Создать | Нет валидации name (цифры проходят), возраст 17 принимается (должен быть >= 18), age возвращается строкой | validateName() и validateAge() корректны |
| PATCH /:id | Обновить | Нет валидации name, отсутствует проверка age < 0 (падёт с 500 fatal error), нет id в ответе, age как строка | Полная валидация |
| DELETE /:id | Удалить | **Не удаляет** — только SELECT (возвращает "user would be deleted but not really") | DELETE FROM возвращает "User deleted successfully" |

**Инфраструктура:**
- Swagger-документация по /documentation
- Middleware: логирование запросов, кэширование, валидация API-ключа (x-fix-bug header)
- Метод /v1/api/reset для очистки БД (DELETE всех записей по ключу)
- Тесты: один файл `tests/post-users.test.js` (асинхронный, axios)

**Документированные баги (из Баги.md и Единой карты):** 20 шт.

## 3. Путь ученика (User Journey)

### Шаг 0 — Лендинг / Введение
- Ученик видит обещания: "найдешь 15+ багов", "тестируешь реальное API с первого урока", "чек-лист для любого API"
- Оплачивает (практикум платный)
- Получает письмо с API-ключом, ссылкой на чат в Telegram, ссылкой на уроки

### Шаг 1 — Установка Postman (урок 2)
- Скачивает Postman, создаёт учётку
- Импортирует коллекцию (ссылка с Google/Яндекс Диска)
- **Проблема:** коллекция на внешнем диске — ссылка может устареть, у ученика не с первого раза получается импорт

### Шаг 2 — Первый запрос (урок 3)
- Делает POST /v1/api/users с {name: "Anna", age: 25}
- Получает 201 Created
- Выполняет GET /v1/api/users, видит созданного пользователя
- Знакомится с методом CLEAR DB для сброса данных

### Шаг 3 — Теория (уроки 4-7)
- HTTP, API, классы эквивалентности, граничные значения
- **Проблема:** 4 урока чистой теории подряд после первого касания практики — может отвалиться ученик

### Шаг 4 — GET: практика + баги (уроки 8-9)
- Ученик составляет тест-кейсы, сверяется с разбором в видео
- **Находит баг 1:** Content-Type = text/plain (урок 10)

### Шаг 5 — Баг-репорт (урок 10)
- Оформляет первый баг-репорт по шаблону
- **Сильный момент:** шаблон реально хорош (заголовок, шаги, ожидаемый/фактический результат)
- **Проблема:** первый баг найден на 5-м уроке практики (урок 10 из 16)

### Шаг 6 — V2 (урок 11)
- Импортирует коллекцию V2
- Проверяет, что Content-Type исправлен в /v2/api/users
- **Сильный момент:** эффект "Вау, баг правда починили!"

### Шаг 7 — Поиск остальных багов (уроки 12-15)
- GET by ID (урок 12) — задание для самостоятельной работы
- POST (урок 13) — несколько багов: name без валидации, age 17 проходит
- PATCH (урок 14) — баги: name, age < 0 → 500 error
- DELETE (урок 15) — баг: "не удаляет, только делает вид"

### Шаг 8 — Финал (урок 16)
- Поздравление
- Ссылка на чек-лист и форму обратной связи
- **Обещают:** "штук 10 багов нашли"

**Где находится бот:** воронка предполагает Practicum Bot (Telegram-бот с задачами) до API Практикума, но по состоянию бэклога — в боте реализованы только Задача 1 и Задача 2, остальное не готово.

## 4. Ответы на вопросы

### Вопрос 1: Зайдёт ли это ученикам?

**Кому да:**
- Абсолютным новичкам в тестировании, которые **никогда не работали с API** — для них сам факт "я отправляю запрос и получаю ответ" — открытие
- Тем, кто учится по видео (а не по тексту) — формат "смотрю за экраном" для них органичен
- Тем, кто ищет портфолио для джуна — "я нашёл 10 багов в реальном API" звучит весомо

**Кому нет:**
- Тем, кто хоть раз работал с API в реальном проекте — баги слишком учебные (Content-Type text/plain — это не "баг уровня прода", а ошибка настройки)
- Тем, кто ожидает интенсивного вовлечения — формат "посмотри видео, повтори" без обратной связи убивает
- Тем, кто хочет автоматизацию — в курсе только ручное тестирование
- Тем, кто быстро теряет интерес — первые 4 урока затянуты

**Основная проблема с аудиторией:** целевая аудитория автора (подписчики eddytester) вероятно состоит из тех, кто уже немного в теме — им курс покажется простым. А полным новичкам нужно больше поддержки. Продукт между двух стульев.

### Вопрос 2: Разрыв между обещанием и реальностью

**Главный разрыв: "минимум 15 багов" vs "штук 10"**

В введении (урок 1): "как минимум 15 багов или даже больше".
В финале (урок 16): "как минимум штук 10 багов".

В коде багов действительно много (20 задокументировано в карте), НО:
1. В видео реально показывается и оформляется только **1 баг** — Content-Type text/plain
2. Остальные баги ученик должен найти **самостоятельно**, без подсказок
3. Нет системы подсчёта найденных багов — ученик не знает, сколько он нашёл

**Другие разрывы:**
- "Практика с первого урока" — реально: первый запрос в уроке 3, первый баг — в уроке 10
- "Обратная связь от автора" — в реальности: асинхронный практикум, ученик пишет в чат и ждёт
- "Чек-лист для тестирования любых методов" — в финале сказано "у тебя остался чек-лист", но в транскрипте финала автор сам признаёт: *"что-то я заметил что ни в одном видео напрямую не упоминаю чек-лист для тестирования любого метода... кажется забыл его интегрировать"* — то есть автор сам знает об этом разрыве

### Вопрос 3: Что самое сильное в продукте?

1. **V1 → V2 переход.** Это блестящая идея: ученик находит баг в V1, заводит баг-репорт, переключается на V2 и видит, что баг починили. Это даёт реальное ощущение работы тестировщика. Полный цикл: баг → репорт → фикс → ретест.

2. **Живой API на PostgreSQL.** Не симуляция, не Postman mock — настоящий сервер с БД, с авторизацией, со Swagger-документацией. Ученик работает как в реальном проекте.

3. **Шаблон баг-репорта.** Качественный, профессиональный, с примером заполнения. Это то, что реально пригодится в работе.

4. **Метод CLEAR DB.** Умная идея — дать ученику инструмент для сброса тестовых данных. Показывает инженерную культуру.

5. **Документация в нескольких форматах:** Swagger, Postman-коллекция, Google Docs — ученик учится работать с разными источниками требований.

### Вопрос 4: Что самое слабое?

1. **Монотонность.** Структура всех уроков одинаковая: теория → разбор документации → действие в Postman. DELETE и PATCH уроки — почти копия POST, но с другим методом. Ученик заскучает к 12-му уроку.

2. **Нет вовлекающих механик.** 
   - Нет прогресса ("вы нашли 3/10 багов")
   - Нет достижений
   - Нет тестов на понимание
   - Нет интерактива внутри видео
   - Нет автоматической проверки результатов ученика

3. **Разрыв между видео и Telegram-ботом.** Бот (Practicum Bot) должен прогревать и сопровождать, но по факту — существует отдельно, задачи не синхронизированы с уроками, воронка не замкнута.

4. **Отсутствие системы.** Чек-лист есть как PDF, баг-репорт — как Google Doc, коллекция — как файл. Всё разрозненно, нет единого рабочего пространства ученика.

5. **Теория в чистом виде.** Уроки 4-7 (HTTP, API, классы, границы) — 4 видео без практики. Для новичка — слишком сухо. Для опытного — скучно.

6. **Обратная связь — асинхронный чат.** Автор обещает: *"Разбор твоих тест-кейсов и баг-репортов"*, но при 100+ учениках это невозможно без масштабирования. Непонятно, как эта работа будет поставлена.

### Вопрос 5: Что критически исправить до запуска? (оценка по времени)

| № | Что исправить | Почему критично | Время |
|:--|:-------------|:-----------------|:------|
| 1 | **Синхронизировать обещание 15 багов** — или добавить баги, или убрать цифру | Судебный риск (обман потребителей), репутационный риск, недовольство учеников | 1 час — отредактировать видео/текст |
| 2 | **Добавить систему трекинга прогресса** (минимум: сколько багов нашёл) | Без этого ученик не чувствует завершённости, не понимает "а всё ли я нашёл?" | 2 дня — простой чек-лист "баги для поиска" текстом |
| 3 | **Собрать все материалы в одном месте** (Google Classroom, Notion или PDF-пособие) | Разрозненность материалов — главная причина отвала учеников | 1 день |
| 4 | **Убрать хотя бы один чисто-теоретический урок** (HTTP и API объединить) | 4 урока подряд без практики — убивают вовлечение | 1 день — реструктурировать плейлист |
| 5 | **Проверить все ссылки** (коллекция на Google/Яндекс Диске, шаблоны, статья на Хабр) | Мёртвые ссылки = брошенный курс | 2 часа |
| 6 | **Интегрировать чек-лист в тело курса** (сейчас он существует как отдельный файл и не упоминается в видео) | Автор сам признал проблему в списке материалов | 2 часа — добавить страницу или PDF-напоминание |
| 7 | **Сценарий обработки "а если ученик не нашёл баг"** | Сейчас ученик может пройти весь курс и не найти ничего, потому что баги не подтверждены и не подсвечены | 1 день — база: просто список "вот какие баги есть, сверься" |
| 8 | **Добавить момент "проверь себя" после каждого метода** (1-2 вопроса) | Завершённость цикла обучения | 2-3 дня — сделать простые квизы в Google Forms |

**Итого критического:** ~5-7 дней работы одного человека. Без этого — высокий риск возвратов и негатива.

### Вопрос 6: Прогноз конверсии и удовлетворения (по 10-балльной шкале)

| Параметр | Оценка | Обоснование |
|----------|--------|-------------|
| **Конверсия лендинг → покупка** | 3/10 | Очень плохо. Продукт не сформулирован как ценность (что я получу на выходе?), цифра 15 багов не подкреплена, нет соцдоказательств, нет понятной структуры |
| **Конверсия покупка → начало (1-3 урок)** | 7/10 | Хорошо. Первые шаги простые: установить Postman, отправить запрос — это даёт быстрый выброс дофамина |
| **Конверсия начало → финал** | 3/10 | Крайне низкая. После 4-го урока начинается теория, нет мотивации продолжать, структура монотонна |
| **Удовлетворение прошедших** | 5/10 | Среднее с перекосом в минус. Кто прошёл — найдут 3-5 багов (из обещанных 15), получат 2 работающих файла (шаблон + чек-лист), но уйдут с осадком "ну, норм, но не о-го-го" |
| **Net Promoter Score (NPS)** | -10 (негативный) | Мало кто будет рекомендовать активно. Скорее "рекомендую новичкам посмотреть первую часть бесплатно" |
| **Вероятность возврата денег** | 40% | Каждый второй может запросить возврат из-за несоответствия ожиданий (15 багов → 3-5 самостоятельно найденных) |

**Общий вердикт:** продукт не готов к коммерческому запуску в текущем виде. Самая сильная часть (V1→V2, живой API) не упакована в работающий пользовательский опыт. Нет каркаса обучения — есть набор разрозненных видео и файлов. При текущем качестве рекомендуется сделать бесплатным или условно-бесплатным (100-300 руб), собрать обратную связь, доработать, и только потом запускать полноценный платный запуск.

## 5. Дополнительные наблюдения (по данным из кода и транскриптов)

### Что не совпадает в документации и коде
- Swagger говорит: `name` с pattern `^[A-Za-z\\s-]+$` (только латиница). В видео (урок 13) автор говорит: "кириллица тоже принимается". Код V2 использует `validateName()` — реализацию не видели, но между сваггером и словами автора есть несоответствие.
- POST: возраст должен быть ≥ 0 по документации. Но V1 пускает 17 лет, V2 — требует 18+. Граница сдвинута.
- DELETE: в документации метод DELETE, в коде V1 — SELECT (даже не DELETE).

### Баги, которые НЕ показаны в видео
Из 20 задокументированных багов, в видео реально демонстрируется и оформляется **только один** — Content-Type text/plain на GET /users. Остальные:
- id-1 (GET /:id) — не подсвечен явно
- age 17 проходит — упомянут в тестах, но не показан как баг
- DELETE не удаляет — не показан
- Age как строка — не показан
- PATCH: age < 0 → 500 fatal error — не показан
- Отсутствие заголовков безопасности — упомянуто в теории, но не как баг

### Про тесты
Единственный тестовый файл — `tests/post-users.test.js` — на самом деле не покрывает все баги. Он:
- Тестирует POST (v1 и v2) — хорошо
- НО: не тестирует GET, PATCH, DELETE
- Использует `test-key-123` как API key (хардкод)
- Работает локально (http://localhost:3000), рассинхронизирован с продакшеном

## 6. Итоговая рекомендация

Сильная идея (живой API с багами → фикс в V2) провалена слабой реализацией курса. 80% работы сделано в коде (классная инженерная часть), 20% — в продуктовой упаковке (плохо).

### Минимальный "превращаем в продукт":
1. **Чек-лист багов** — "вот 10 багов, отметь, какие нашёл"
2. **Единая рабочая тетрадь** — Notion/Google Doc с поурочной структурой
3. **2-3 скринкаста с подсветкой багов** (где автор показывает: "а вот ещё один")
4. **Синхронизация бота** — каждое видео сопровождается сообщением от бота
5. **Убрать/заменить "15 багов"** на честное "10+ багов" или реально добавить баги

Без этого — продукт для самых мотивированных self-learners, а не для массового платного курса.
