Тестирование API на практике: Задача 1.

Типичный запрос на создание заказа в интернет-магазине. Упрощенный, конечно же.
Ввели данные, отправили, сервер вернул `200 OK`.
Всё как я люблю: минимум логики, максимум противоречий. Что не так с этим ответом? Пиши в комментариях количество багов, которые видишь.
А их здесь минимум вот столько: 4 (под спойлером)

Запрос:

> `POST /api/v3/orders/create`
> 
> **Body:**
> 
> JSON
> 
> ```
> {
>   "item_id": 404,
>   "quantity": 0,
>   "promo_code": "WINTER2026"
> }
> ```

**Ответ:**

> **Status:** `200 OK`
> 
> **Body:**
> 
> JSON
> 
> ```
> {
>   "success": true,
>   "order_id": null,
>   "message": "Quantity must be grater than 0",
>   "server_time": "2026-09-11T12:00:00Z"
> }
> ```

---

**Разбор (для тебя):**

1. **Баг №1 (Главный):** Статус-код `200 OK` при фактической ошибке валидации. Должен быть `400 Bad Request`. Это ломает автоматизацию и мониторинг.
    
2. **Баг №2:** Поле `order_id` в ответе со значением `null`. Если заказ не создан, этого поля вообще не должно быть в схеме, либо структура должна отличаться.

### Пост №2: «Тайны личного кабинета»

**Тип бага:** Утечка данных (PII Leak) + Математика.

**Уровень:** Средний / Внимательность.

**Текст поста:**

Задача на внимательность. Тестируем ручку получения данных профиля пользователя. Разработчик клянется, что всё по ТЗ.

Найди здесь критический баг безопасности. Подсказка: смотри не только на то, что пришло, но и на то, ЧТО ИМЕННО пришло. 🕵️‍♂️

**Запрос:**

> `GET /api/v2/profile?user_id=8841`
> 
> **Headers:**
> 
> `X-Auth-Token: admin_token_99`

**Ответ:**

> **Status:** `200 OK`
> 
> **Body:**
> 
> JSON
> 
> ```
> {
>   "id": 8841,
>   "username": "tester_pro",
>   "email": "test@example.com",
>   "balance": 1500.00,
>   "internal_password_hash": "sha256$uI89!jd#k9",
>   "roles": ["user"],
>   "is_active": true
> }
> ```

---

**Разбор (для тебя):**

1. **Баг №1:** Утечка чувствительных данных. Поле `internal_password_hash` никогда не должно отдаваться в API, это огромная дыра в безопасности.
    
2. **Баг №2 (Бонусный):** Формат баланса. В финансовых API использование `float` (с точкой) — риск потери точности. Обычно используют целые числа (в копейках/центах).

### Пост №3: «Магия фильтрации»

**Тип бага:** Нарушение логики фильтрации / Инвентаризация.

**Уровень:** Продвинутый.

**Текст поста:**

Баттл: QA против бэкенд-логики. Мы запросили список активных товаров со скидкой. Система выдала результат.

Где система нас обманула? На кону — звание «Зоркий глаз». 🔍

**Запрос:**

> `GET /api/products?status=active&has_discount=true&limit=2`

**Ответ:**

> **Status:** `200 OK`
> 
> **Body:**
> 
> JSON
> 
> ```
> {
>   "items": [
>     {
>       "name": "Механическая клавиатура",
>       "status": "active",
>       "price": 5000,
>       "old_price": 7000
>     },
>     {
>       "name": "Игровая мышь",
>       "status": "archived",
>       "price": 2500,
>       "old_price": 2500
>     }
>   ],
>   "total_count": 45
> }
> ```

---

**Разбор (для тебя):**

1. **Баг №1:** Фильтр `status=active` не сработал — второй товар в статусе `archived`.
    
2. **Баг №2:** Фильтр `has_discount=true` не сработал для второго товара — цена и старая цена совпадают (скидки нет).
    
3. **Баг №3:** Лимит. Мы просили `limit=2`, нам пришло 2 товара, но `total_count: 45` намекает, что пагинация может быть сломана, если при таких фильтрах товаров физически меньше. Но первые два — явные провалы.

### Пост №4: «Дырявая корзина»

**Тип бага:** Отрицательные значения / Типы данных.

**Уровень:** Вау-эффект.

**Текст поста:**

Финальный босс! Нам нужно обновить количество товара в корзине. Мы решили немного похулиганить и отправить странные данные. Бэкенд ответил `200 OK`.

Почему этот ответ — повод для экстренного тикета с приоритетом Blocker? 🔥

**Запрос:**

> `PATCH /api/cart/items/55`
> 
> **Body:**
> 
> JSON
> 
> ```
> {
>   "quantity": -1
> }
> ```

**Ответ:**

> **Status:** `200 OK`
> 
> **Body:**
> 
> JSON
> 
> ```
> {
>   "cart_id": "abc-123",
>   "item_id": 55,
>   "new_quantity": -1,
>   "total_price": -499.00
> }
> ```

---

**Разбор (для тебя):**

4. **Баг №1:** Отрицательное количество. Система позволила установить `-1`.
    
5. **Баг №2:** Отрицательная итоговая стоимость заказа. Поздравляю, теперь магазин должен денег покупателю за то, что тот «забрал» товар. Это классический баг на логику, который новички часто не проверяют.
