**3 заголовка, которые я проверяю первыми (и вам советую)**

Я не смотрю тело ответа, пока не проверил шапку.

Звучит как странный ритуал, но за 5+ лет я привык: баги сидят не в JSON, а в том, как сервер думает про твой запрос.

---

**🔥 Content-Type**

Content-Type в запросе - это униформа. Одел `application/json` - значит, "я JSON". Одел `text/plain` - значит, "я просто текст".

Большинство фреймворков вообще не проверяют униформу. Express, Spring, Flask парсят тело по содержимому, а не по заголовку. Прислал JSON с `text/plain`? Ок, спарсили, 200 OK.

И вот здесь открывается дыра.

Злоумышленник может слать запросы с `Content-Type: text/plain`, а внутри - валидный JSON. Сервер его съест. А защита, которая завязана на заголовок, - не сработает:

🔘 Валидация схемы висит на `application/json` - с `text/plain` она просто не запустится
🔘 Rate limiting считает `text/plain` запросы отдельно - можно долбить API в обход лимитов
🔘 Security-логи смотрят заголовок, а не тело - в логах безобидный текст, по факту атака

Проверяется за 10 секунд:
```
curl -X POST /api/users \
  -H "Content-Type: text/plain" \
  -d '{"role": "admin"}'
```

200 OK вместо 415 Unsupported Media Type = дверь открыта.

**⚡️ Cache-Control**

Самый коварный заголовок - баг плавающий.

Обновил данные - API отвечает старыми. Перезашёл - новыми. "Магия". На самом деле - сервер закэшировал ответ и радует пользователей вчерашними новостями.

Я такое ловил на проде: пользователи жалуются, что данные не обновляются. Девопсы говорят "всё чистили". А ответ всё равно летит из кэша, потому что Cache-Control не настроен.

Как проверять:
- POST на создание → GET на чтение. Если ответ пришёл без `Cache-Control: no-cache` - с вероятностью 90% там кэш.
- После PUT/PATCH - проверь, что GET отдаёт свежие данные, а не закэшированное старьё.
- Если баг «плавает» - кинь в URL параметр `?v=1`. Баг исчез? Поздравляю, ты локализовал проблему.

👉 У меня был большой разбор кэширования с примерами и чек-листом: https://t.me/eddytester/398

**🛡 CORS**

Любимая отмазка фронтендеров: "CORS - это не наша проблема".

На самом деле CORS - это акт доверия. Сервер говорит браузеру: "этому домену я доверяю". Если CORS настроен через `*` - доверять можно всем. Включая сайт, который украдёт твои данные.

Как проверять: шлёшь запрос с `Origin: https://evil.com`. Смотришь `Access-Control-Allow-Origin` в ответе. Если там `*` или твой `evil.com` - у тебя проблемы.
