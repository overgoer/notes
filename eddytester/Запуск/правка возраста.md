# Спецификация и пошаговый план: Исправление бага 17 лет (POST /users)

> Этот документ описывает устранение рассинхронизации между фактическим поведением API, автотестами, документацией и кабинетом практикума на сайте.

---

## 🔍 Описание проблемы

В курсе заложена важная бизнес-логика назначения статусов на основе возраста:
* **Младше 18 лет (0-17 включительно):** автоматически получают статус `"minor"`.
* **Регистрация кандидатов младше 18 лет разрешена** (это не валидационная ошибка, блокирующая создание, а обычный процесс создания с особым статусом).

### В чем рассинхрон сейчас:
1. **Код V1:** `if (age < 18) status = 'minor'`. То есть 17-летние регистрируются со статусом `"minor"`. Логика отрабатывает **правильно**.
2. **Документация (`docs/bugs.md` и `BUG_POST_USERS_003`):** написано, что в V1 присутствует баг — 17-летние получают статус `"candidate"` вместо `"minor"`.
3. **Newman/Postman тесты:** ожидают, что в V1 17-летние получат `"candidate"`.
4. **Jest-тесты (`tests/`):** ошибочно ожидают, что в V2 возраст 17 лет должен отклоняться ошибкой 400 (хотя регистрация minor-кандидатов разрешена по ТЗ). Из-за этого Jest-тесты сейчас падают.
5. **Кабинет `/glearning` и бэкенд (`progress.js`):** баг `p_age_17` («Возраст 17 проходит») считается «реальным» багом в V1 (хотя проходить он должен по ТЗ), а баг `p_status_17` («17 лет получает статус minor») считается фейковым дистрактором.

---

## 🎯 Выбранное решение

Мы реализуем баг **`p_status_17`** в его истинном методическом виде:
* **Поведение V1 (Баг):** 17-летний кандидат успешно регистрируется, но получает неверный статус `"candidate"` (в коде делаем границу `< 17` вместо `< 18`).
* **Поведение V2 (Фикс):** 17-летний кандидат регистрируется и получает корректный статус `"minor"`.
* **Валидация:** Возраст 17 лет является абсолютно легальным для регистрации в обеих версиях (отклонять его по 400 ошибке в V2 — не нужно).

* Создать и переключиться в рабочую ветку для правок:
  ```bash
  git checkout -b feature/age-status-fix
  ```

### Шаг 2: Правка роута V1 (Реализация бага в коде)
* **Файл:** `src/routes/v1/users.js`
* **Изменение:** В блоке определения статуса меняем `age < 18` на `age < 17`.
  ```javascript
  let status = 'candidate';
  if (age < 17) status = 'minor'; // Баг: 17 лет получит 'candidate'
  ```

### Шаг 3: Правка валидации в V2 (Устранение избыточного блока)
* **Файл:** `src/routes/v2/users.js`
* **Изменение:** Убедиться, что `validateAge(age)` пропускает возраст 17 (в сервисе валидации уже стоит `age >= 0 && age <= 120` — пропускает корректно). В самом роуте статус 17-летнего должен назначаться как `"minor"`.

### Шаг 4: Обновление конфигурации уроков на бэкенде
