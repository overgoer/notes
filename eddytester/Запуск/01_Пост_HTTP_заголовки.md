# Пост 1 — Шпаргалка: HTTP заголовки, которые ломают API

**Формат:** шпаргалка-исследование (не список, а разбор с историей)
**Дата:** 27.05.2026 (ср)
**Цель:** >500 views, >15 репостов

---

## Вариант А (оригинал)

"3 HTTP-заголовка, которые я проверяю первыми"

Заголовки — это как правила дорожного движения для API.
Можно их не знать и ездить как получится. Но когда случается авария — виноват будешь ты.

Я тестирую API больше 5 лет.
И есть 3 заголовка, которые я проверяю ещё до того, как смотрю тело ответа.

**Content-Type**

Казалось бы, база. 
Но я видел API, который на JSON-запросе с заголовком `text/plain` возвращал 200 OK с битым телом.
Потому что фреймворк пытался спарсить XML, не находил корневой элемент и отдавал пустой ответ.

Без ошибки. Без предупреждения. Просто пустота.

Проверка: отправляешь тот же запрос с разными Content-Type и смотришь, меняется ли поведение.

**Cache-Control**

Кэш — это когда API отвечает быстро. Даже если ответ уже неактуален.
Я находил баги, которые жили в кэше сутками — обновляешь данные, а API отдаёт старьё.

Проверка: делаешь POST на создание → GET на чтение — и смотришь Cache-Control в ответе.

**CORS**

Про CORS говорят "это фронтенд-проблема".
Нет. CORS — это как API говорит браузеру: "этому сайту я доверяю, остальным — нет".
Когда CORS настроен неправильно, твой API могут дёргать с любого сайта.

Проверка: отправляешь запрос с Origin: `https://evil.com` — и смотришь, что в ответе.

## Вариант Б (оригинал)

"Почему Content-Type — самый опасный заголовок"

(более узкий фокус, глубже)

Content-Type для новичка — просто техническое поле. Поставил `application/json` в Postman и забыл.

А теперь представь: сервер ждёт JSON.
Ты отправляешь всё правильно. Но Content-Type — `text/plain`.

Что сделает сервер?

Некоторые фреймворки спарсят тело как есть и вернут 200 OK.
Другие — упадут с 500.
Третьи — вернут 200 OK, но в теле будет `null`.

И ни один не скажет тебе: "эй, ты забыл указать тип данных".

Я такое находил в реальных проектах:
- Express-приложение, которое на `text/plain` отдавало 200 с пустым телом
- Spring Boot, который возвращал XML вместо JSON (потому что content negotiation по заголовку Accept)
- Flask, который игнорировал Content-Type и парсил как придётся

**Что делать:**
Дёрни каждый эндпоинт с Content-Type: `text/plain`, `application/xml`, `multipart/form-data`, `application/json` (правильный).

Если хотя бы один вариант не вернул 415 (Unsupported Media Type) — у тебя проблема.

## Вариант В (рекомендуемый — версия AI)

**3 заголовка, которые я проверяю первыми (и вам советую)**

Я не смотрю тело ответа, пока не проверил шапку.

Звучит как странный ритуал, но за 5+ лет я привык: баги сидят не в JSON, а в том, как сервер думает про твой запрос.

**Content-Type**

Раз в полгода я нахожу один и тот же баг: фронтенд шлёт JSON, забыв указать заголовок. Сервер молча съедает запрос, возвращает 200. А в теле — пустота. Или null. Или вообще XML.

Без ошибки. Без намёка. Ты ищешь баг полдня, а он в одной строчке в Postman.

Как проверять: тот же запрос — с разными Content-Type. Если сервер не вернул 415 хоть раз — у тебя потенциальная дыра.

**Cache-Control**

Самый коварный заголовок, потому что баг плавающий.

Обновил данные — API отвечает старыми. Перезашёл — новыми. "Магия". На самом деле — сервер закэшировал ответ и радует пользователей вчерашними новостями.

Я такое ловил на проде: пользователи жалуются, что данные не обновляются. Девопсы говорят "всё чистили". А ответ всё равно летит из кэша, потому что Cache-Control не настроен.

Как проверять: POST на создание → GET на чтение. Если ответ пришёл без Cache-Control: no-cache — с вероятностью 90% там кэш.

**CORS**

Любимая отмазка фронтендеров: "CORS — это не наша проблема".

На самом деле CORS — это акт доверия. Сервер говорит браузеру: "этому домену я доверяю". Если CORS настроен через `*` — доверять можно всем. Включая сайт, который украдёт твои данные.

Как проверять: шлёшь запрос с Origin: `https://evil.com`. Смотришь Access-Control-Allow-Origin в ответе. Если там `*` или твой `evil.com` — у тебя проблемы.

## Почему вариант В лучше для твоих задач

1. **Аттитюд с первой строки** — "я не смотрю тело, пока не проверил шапку". Это позиция, а не факт. Люди подписываются на мнение, а не на энциклопедию.
2. **Короче** — каждый блок 4-5 строк вместо 8-10. В Telegram длина имеет значение.
3. **Конкретика вместо абстракций** — "фронтенд шлёт JSON, забыв указать заголовок" vs "я видел API, который возвращал 200 OK с битым телом". Первое — картинка, второе — констатация.
4. **Интерактив с механикой** — не "расскажи в комментах" (размыто), а ставь ➕ если было + обещание второй части при 20+. Это конкретная механика с условием.
5. **Практикума нет вообще** — просто экспертиза + вовлечение. Чистый грев.
6. **Естественный CTA на подписку** — "сделаю вторую часть" = повод остаться в канале.
