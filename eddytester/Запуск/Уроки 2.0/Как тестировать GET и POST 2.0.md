## 🛡️ Раздел: Требования к формату

### 📌 Контракт (Что написано в доке):
- Все запросы выполняются по **HTTPS**.
- Формат данных всегда **application/json** (в запросах и ответах).
- Заголовок **X-Fix-Bug** с API-ключом обязателен для всех запросов. Без него — `401 Unauthorized`.
- Ответ обязан содержать заголовки безопасности: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Cache-Control: no-cache`.

---
### 🧪 Список кейсов:
- 🔓 **Запрос без API-ключа:** Убери заголовок `X-Fix-Bug`. *(Ожидаем: 401)*.
- 🔑 **Запрос с невалидным ключом:** Отправь `X-Fix-Bug: invalid_key`. *(Ожидаем: 401)*.
- 📦 **Проверка Content-Type:** Проверь заголовок ответа `Content-Type`. *(Ожидаем: application/json. Наш баг v1: возвращает text/plain)*.
- 🛡️ **Сверка безопасности:** Проверь заголовки безопасности в ответе. *(Ожидаем: nosniff, DENY, no-cache. Наш баг v1: они удалены, а Cache-Control изменен на public)*.

---
## 🔁 Раздел: GET /v1/api/users

### 📌 Контракт (Что написано в доке):
- Возвращает список пользователей.
- Имеет необязательный параметр **`status`** (`minor`, `candidate`, `retired`).
- Параметр принимает **только одно** значение.
- Значение **не чувствительно** к регистру.
- Невалидный статус -> ошибка **`400 Bad Request`**.
- Если пользователей по фильтру нет -> **`200 OK`** и пустой массив **`[]`**.

---
### 🧪 Список кейсов:
- 🔠 **Регистронезависимость:** Запроси `?status=Candidate`. *(Ожидаем: успешный поиск. Наш баг v1: фильтр регистрозависим и вернет пустой список)*.
- 🧹 **Грязный статус:** Запроси `?status=banana`. *(Ожидаем: 400 и JSON ошибки. Наш баг v1: проглатывает banana и возвращает 200 [])*.
- 🔗 **Перечисление статусов:** Запроси `?status=minor,candidate`. *(Ожидаем: ошибку 400, так как разрешено только одно значение)*.
- 🕳️ **Пустой поиск:** Удалить пользователей и запросить список. *(Ожидаем: 200 []).*

## 🧱 Раздел: POST /v1/api/users

### 📌 Контракт (Что написано в доке):
- Создает нового кандидата с уникальным именем.
- **`name`** — строка (только латиница, пробелы, тире).
- **`age`** — целое число ≥ 0.
- Возраст определяет статус:
  - `0–17` → status: `minor`
  - `18–65` → status: `candidate`
  - `66+` → status: `retired`
- Автоматически создает **один** скрининговый пропуск (Pass) на 1 час.

---
### 🧪 Список кейсов (Валидация JSON):
- 🕳️ **Пропуск полей:** Отправь тело без `name` или без `age`. *(Ожидаем: 400 "Valid name/age is required")*.
- 🎨 **Запрещенные символы в имени:** Отправь `name: "John123"` или `name: "Иван"`. *(Ожидаем: 400. Наш баг v1: сохраняет цифры и спецсимволы)*.
- 👯 **Дубликат имени:** Отправь одно и то же имя дважды. *(Ожидаем: 400 или 409 "User with this name already exists")*.
- 🧬 **Дробный возраст:** Отправь `age: 25.5`. *(Ожидаем: 400)*.

---
### 🧪 Список кейсов (Классы и границы возраста):
Применяем технику граничных значений для полей возраста:
- 📌 **Диапазон minor (0-17):** Отправь возраст `0`, `1`, `17`. *(Ожидаем: статус minor. Наш баг v1: возраст < 17 заблокирован, а 17 лет создает minor)*.
- 📌 **Диапазон candidate (18-65):** Отправь возраст `18` и `65`. *(Ожидаем: статус candidate)*.
- 📌 **Диапазон retired (66+):** Отправь возраст `66`. *(Ожидаем: статус retired. Наш баг v1: возраст > 65 полностью заблокирован)*.
- 📌 **Негативный тест:** Отправь возраст `-1`. *(Ожидаем: 400 "Valid age is required")*.

---
### 🧪 Список кейсов (Побочные эффекты):
- ⛓️ **Количество пропусков:** Создай юзера, затем сделай `GET /v1/api/passes`. *(Ожидаем: ровно 1 пропуск. Наш баг v1: сервер плодит два пропуска на одного юзера)*.
- ⏰ **Время действия:** Проверь поле `expires_at` в созданном пропуске. *(Ожидаем: ровно +1 час от времени создания)*.
- 📝 **Формат кода:** Проверь поле `pass_code`. *(Ожидаем: SCR- и 8 заглавных hex символов. Наш баг v1: генерирует scr_ и строчные буквы)*.

---
## 📝 Раздел: PATCH /v1/api/users/{id}

### 📌 Контракт (Что написано в доке):
- Обновляет имя и возраст пользователя.
- Метод реализован как **полный апдейт** — в теле запроса **обязательно** указывать оба поля: `name` и `age`.
- Если пользователя нет -> **`404 Not Found`** и `{"message": "User not found"}`.

---
### 🧪 Список кейсов:
- ✂️ **Частичное обновление (Баг по ТЗ):** Отправь тело только с полем `name` или только с `age`. *(Ожидаем: 400, так как ТЗ требует указания обоих полей при PATCH)*.
- 👤 **Обновление несуществующего:** Отправь PATCH на ID, которого нет в базе. *(Ожидаем: 404)*.
- 🔢 **Типы данных в ответе:** Проверь типы данных измененного объекта. *(Ожидаем: age — число, id — присутствует. Наш баг v1: возвращает age как строку, а поле id вообще вырезано из ответа)*.

---
## ❌ Раздел: DELETE /v1/api/users/{id}

### 📌 Контракт (Что написано в доке):
- Удаляет пользователя по ID.
- Если ID не найден -> **`404 Not Found`**.
- Возвращает сообщение об успехе и удаленный объект.

---
### 🧪 Список кейсов (Сквозная проверка):
- 🕵🏼‍♂️ **Сквозное удаление:**
  1. Вызови `DELETE /v1/api/users/{id}`. *(Ожидаем: 200 и данные удаленного юзера)*.
  2. Вызови `GET /v1/api/users/{id}` с этим же ID. *(Ожидаем: 404 User not found, подтверждающий удаление. Наш баг v1: DELETE возвращает успех, но запись из БД не удаляет!)*.

# 🚀 Итог: Навык работы с контрактом

Тестирование по документации — это не просто вызовы запросов:
1. Мы берем **правило** из доки.
2. Подбираем под него **набор значений** (границы, типы, обязательность).
3. Смотрим в **ответ сервера** (статус-код, типы данных в JSON, заголовки ответа).
4. Проверяем **смежные эндпоинты** (побочные эффекты).
