## 1. Query-параметры (GET)

Параметры запроса— это переменные в URL-адресе после знака **`?`**,<br> разделенные знаком **`&`**. <br>Используются для фильтрации, сортировки и пагинации данных.
>
> *Пример:* `/v1/api/users?status=candidate&limit=10`

---
<div align="left">

### 🧪 Список проверок (Query-параметры):

- **Регистр значений:** Отправь `?status=Candidate` и `?status=CANDIDATE`.<br><br>
- **Грязный ввод (Invalid Value):** Отправь `?status=banana`.<br><br>
- **Пустое значение:** Отправь `?status=`.<br><br>
- **Перечисление (Comma-separated):** Отправь `?status=minor,candidate`.<br><br>

---
## 2. Заголовки ответов (Response Headers)

### 📌 Понятие:
> **Response Headers** — служебные метаданные, которые сервер присылает вместе с ответом. Они сообщают клиенту формат данных, правила безопасности и правила кэширования.

---
### 🧪 Список проверок (Заголовки ответов):

- 📦 **Content-Type (Формат):** Проверь, равен ли заголовок `application/json` при возврате JSON-данных.
- 🔒 **Cache-Control (Кэширование):** Личные данные пользователей должны иметь заголовок `no-store` или `private`.
- 🛡️ **Заголовки безопасности:** Проверь наличие заголовков `X-Content-Type-Options: nosniff` и `X-Frame-Options: DENY`.

---
## 3. Тело запроса JSON (POST Body)

### 📌 Понятие:
> **Request Body** (Тело запроса) — данные, отправляемые клиентом внутри запроса (обычно в формате JSON). Используется методом POST для создания новых ресурсов.
>
> *Пример:* `{"name": "Ivan Petrov", "age": 25}`

---
### 🧪 Список проверок (Тело JSON):

- 🕳️ **Обязательность полей:** Отправь запрос без поля `name` или без `age`.
  *(Ожидаем: статус-код 400 Bad Request с понятной ошибкой).*
- 🔀 **Лишние/Неописанные поля:** Отправь `{"name": "Ivan", "age": 25, "role": "admin"}`.
  *(Ожидаем: сервер проигнорирует поле "role" или выдаст ошибку).*
- 🎨 **Спецсимволы в строках:** Отправь `{"name": "John123!@#", "age": 25}`.
  *(Ожидаем: валидацию имени на стороне сервера. В v1 её нет).*

---
## 4. Граничные значения (POST Boundaries)

### 📌 Понятие:
> **Граничные значения** — проверка поведения бизнес-логики на стыках допустимых диапазонов (классов эквивалентности). Чаще всего баги прячутся именно на границах.
>
> *Классы по ТЗ:* `minor` (0-16), `candidate` (18-65), `retired` (66+).

---
### 🧪 Список проверок (Границы чисел):

- 📌 **Нижняя граница:** Возраст `0` и `1`.
  *(Ожидаем: успешное создание со статусом minor. Наш баг v1: возраст < 17 полностью заблокирован).*
- 📌 **Точки переключения классов:** Возрасты `16`, `17`, `18`, `65`, `66`.
  *(Ожидаем: переключение логики статусов. Наш баг v1: возраст 17 создает minor, а возраст > 65 заблокирован).*
- 📌 **Негативные тесты:** Возрасты `-1` и `150`.
  *(Ожидаем: ошибку 400 Bad Request).*

---
## 5. Типы данных в JSON

### 📌 Понятие:
> **Типы данных** — строгое соответствие формата передаваемого значения схеме API (число к числу, строка к строке, булево к булеву).

---
### 🧪 Список проверок (Типы данных):

- 🔢 **Строка вместо числа:** Отправь `"age": "25"` (в кавычках).
  *(Ожидаем: ошибку 400 или приведение к числу 25. Наш баг v1: сервер сохраняет строку и возвращает её в ответе).*
- 🔠 **Число вместо строки:** Отправь `"name": 12345` (без кавычек).
  *(Ожидаем: ошибку валидации).*
- 🧬 **Булево значение:** Отправь `"age": true`.
  *(Ожидаем: ошибку 400 Bad Request).*

---
## 6. Побочные эффекты (Side Effects)

### 📌 Понятие:
> **Побочные эффекты** — сопутствующие изменения в системе при создании ресурса (записи в смежных таблицах БД, генерация уникальных токенов, отправка писем).

---
### 🧪 Список проверок (Побочные эффекты):

- ⛓️ **Количество связанных ресурсов:** После POST-запроса на создание пользователя сделай GET-запрос в эндпоинт `/passes`.
  *(Ожидаем: ровно 1 созданный пропуск. Наш баг v1: сервер ошибочно создает 2 пропуска).*
- 📝 **Формат автогенерации:** Проверь формат сгенерированного кода пропуска.
  *(Ожидаем: код вида SCR-... по контракту. Наш баг v1: генерирует scr_... в нижнем регистре).*

---
## 7. Сквозной тест (End-to-End)

### 📌 Понятие:
> **Сквозная проверка (или сравнение)** — сверка целостности данных путем последовательного вызова методов POST (создание) и GET (чтение).

---
### 🧪 Список проверок (Сквозной тест):

1. **Создай** пользователя через **POST /users** (запомни `id` из ответа).
2. **Запроси** этого пользователя через **GET /users/:id** по его `id`.
3. **Сравни:**
   - Все ли поля совпадают по значениям?
   - Не изменились ли типы данных? (например, число на строку).
   - Присвоился ли верный статус по бизнес-логике?
