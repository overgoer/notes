# 📊 Полезные SQL-запросы для тестировщика
## Урок: Как проверять API напрямую через базу данных
<div align="left">

---
## 🗺️ План урока

| Что разберем на практике | Какой SQL-инструмент используем |
| :--- | :--- |
| 🔍 **Проверка создания записи** | Простой `SELECT` с сортировкой и лимитом. |
| ⚖️ **Авто-аудит бизнес-логики** | Сложное условие `WHERE` с логическими `OR` / `AND`. |
| 👯 **Поиск дубликатов в БД** | Группировка `GROUP BY` и фильтр `HAVING COUNT`. |
| ⛓️ **Сквозная проверка связей** | Объединение таблиц через `JOIN`. |
| ⏰ **Поиск зависших статусов** | Сравнение дат со системным временем `NOW()`. |
| 📈 **Аудит статистики** | Агрегация данных через `COUNT(*)`. |

## 1. Проверка создания записи

### 💻 SQL-запрос:
```sql
SELECT * FROM users
ORDER BY id DESC
LIMIT 5;
```

---
### 📌 Зачем это важно:
> Проверяет, что отправленный через `POST /v1/api/users` кандидат действительно записался в базу данных, и его поля (имя, возраст, статус) сохранились без искажений.

### 🧪 Когда полезно:
- Сразу после вызова POST-методов для подтверждения успешной вставки.
- Для проверки того, что автоинкрементный `id` увеличивается корректно.

---
## 2. Авто-аудит бизнес-логики (Поиск несоответствий)

### 💻 SQL-запрос:
```sql
SELECT id, name, age, status FROM users
WHERE (age < 18 AND status != 'minor')
   OR (age BETWEEN 18 AND 65 AND status != 'candidate')
   OR (age >= 66 AND status != 'retired');
```

---
### 📌 Зачем это важно:
> Этот один запрос автоматически проверяет всю таблицу на соответствие ТЗ! Если база вернула хотя бы одну строку — вы нашли баг в логике назначения статусов бэкендом.

### 🧪 Когда полезно:
- Для проверки правильности автоматического назначения статуса в зависимости от переданного возраста.
- Позволяет за секунду протестировать всю базу данных после нагрузочного тестирования.

---
## 3. Поиск дубликатов в БД

### 💻 SQL-запрос:
```sql
SELECT name, COUNT(*) FROM users
GROUP BY name
HAVING COUNT(*) > 1;
```

---
### 📌 Зачем это важно:
> Проверяет уникальность имён. По ТЗ имя кандидата должно быть уникальным. Если этот запрос вернул строки, значит, API пропустило дубликаты и в базе образовался мусор.

### 🧪 Когда полезно:
- При проверке валидации уникальности полей в методе `POST /v1/api/users`.
- Позволяет доказать разработчику наличие бага "Race Condition" или отсутствие уникального индекса в БД.

---
## 4. Сквозная проверка связей (JOIN)

### 💻 SQL-запрос:
```sql
SELECT u.name, u.status, p.pass_code, p.status AS pass_status
FROM users u
JOIN screening_passes p ON u.id = p.user_id;
```

---
### 📌 Зачем это важно:
> Проверяет побочные эффекты. При создании пользователя ему должен автоматически выдаваться скрининговый пропуск. `JOIN` связывает пользователя и его пропуск, позволяя увидеть всю картину на одном экране.

### 🧪 Когда полезно:
- Для проверки интеграционных сценариев (создался ли пропуск, тот ли у него `user_id`, совпадает ли привязка).

---
## 5. Поиск зависших статусов (Просрочка)

### 💻 SQL-запрос:
```sql
SELECT id, pass_code, expires_at, status 
FROM screening_passes
WHERE expires_at < NOW() AND status = 'active';
```

---
### 📌 Зачем это важно:
> Если время действия пропуска истекло (`expires_at < NOW()`), но его статус всё ещё равен `'active'` — это логический баг! Либо не отработал фоновый скрипт очистки (Maintenance), либо сломалась логика обновления статуса.

### 🧪 Когда полезно:
- При тестировании жизненного цикла пропусков и работы фоновых задач очистки (Maintenance Service).

---
## 6. Аудит статистики и дашбордов

### 💻 SQL-запрос:
```sql
SELECT status, COUNT(*) FROM users
GROUP BY status;
```

---
### 📌 Зачем это важно:
> Быстрая сверка статистики. Если на фронтенде отображается дашборд с количеством кандидатов по статусам, этот запрос мгновенно покажет, совпадают ли цифры на экране с реальными данными в БД.

### 🧪 Когда полезно:
- При тестировании аналитики, дашбордов и сверке общих счетчиков в веб-интерфейсе.

# 🚀 База данных — твой главный свидетель

> ☝🏻 Помни: база данных никогда не врет, в отличие от ответов API.  
> 
> Ответ от сервера может вернуть код `200 OK`, но только прямой запрос в БД покажет, сохранились ли данные правильно.

### Сверяй каждый запрос к API с состоянием твоей базы данных!
