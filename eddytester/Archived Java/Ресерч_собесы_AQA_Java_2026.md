## 1. Т-Банк — критическое мышление > формальное покрытие

**Что спрашивают:** Дают метод, который никогда не вызывается (dead code), просят написать тест.

**Что проверяют:**
- Не умение писать тесты "для галочки"
- А готовность **задать вопрос** и проявить критическое мышление
- Ожидание: "Этот точно вызывается? Если нет — предлагаю удалить или добавить флаг фичи"

**Реальный кейс:** Разработчик добавил `calculateRisk()` для будущей фичи. QA написал юнит-тест, чтобы пройти код-ревью. Через месяц баг (деление на ноль) упал в прод — тест не проверял логику, а только факт вызова.

**Чек-лист твоего ответа:**
- [ ] Уточнить: "Этот метод используется в продакшене?"
- [ ] Если нет — предложить удалить или `@Disabled` с комментарием
- [ ] Если настаивают на тесте — параметризованный тест с граничными значениями

## 2. Кэши: Redis + Cache-Control: no-store

**Что спрашивают:** «Как тестировать Redis-кэш, если сервер шлёт `Cache-Control: no-store`, но данные всё равно кэшируются?»

**Суть бага:** Расхождение заголовка HTTP и реального поведения. `no-store` запрещает кэширование, но Redis/Nginx может игнорировать заголовок.

**Инструменты проверки:**
```bash
# Первый запрос — проверить заголовок
curl -I https://api.example.com/items | grep -i cache-control

# Второй запрос — проверить, изменились ли данные
sleep 5 && curl -s -o /dev/null -w "%{http_code}" https://api.example.com/items

# TTL в Redis
redis-cli TTL cache:items:list
```

**Чек-лист:**
- [ ] Проверить `proxy_ignore_headers` в Nginx
- [ ] В Java: проверить `@Cacheable` — не игнорирует ли заголовки ответа
- [ ] Тест: изменить данные в БД, повторить GET — если ответ старый, баг

## 3. Яндекс — пагинация и таймауты

**Что спрашивают:** «API возвращает 200 для списка из 10 000 элементов — автоматизируй, чтобы не упал по таймауту и не сжёг CI».

**Реальный кейс:** Тест делал один запрос без пагинации, сервер отвечал 25 секунд, CI-раннер убивал процесс. Решение: 10 запросов по 1000 элементов, каждый с таймаутом 5 секунд.

**Java-решение (RestAssured):**
```java
@Test(timeout = 30_000)
public void testLargeList() {
    ExecutorService executor = Executors.newFixedThreadPool(5);
    List<Future<Response>> futures = new ArrayList<>();
    for (int i = 0; i < 10; i++) {
        int page = i;
        futures.add(executor.submit(() ->
            given()
                .param("page", page)
                .param("size", 1000)
                .when()
                .get("/api/items")
                .then()
                .statusCode(200)
                .extract().response()
        ));
    }
    int total = futures.stream()
        .map(f -> f.get(5, TimeUnit.SECONDS))
        .mapToInt(r -> r.jsonPath().getList("items").size())
        .sum();
    assertThat(total).isEqualTo(10_000);
}
```

**Чек-лист:**
- [ ] `connectTimeout(2000)` + `socketTimeout(5000)` в RestAssured
- [ ] Пагинация или параллельные запросы (≤5 потоков)
- [ ] Отдельный CI-раннер с тегом `@StressTest`

## 4. Ozon — ConcurrentHashMap в контексте тестов

**Что спрашивают:** «Зачем API-тестировщику знать про `ConcurrentHashMap` и `synchronized`?»

**Почему это важно:** Параллельные тесты в CI используют общие пулы тестовых данных. Без синхронизации — race condition и flaky-тесты.

**Разница:**
| HashMap | ConcurrentHashMap | synchronizedMap |
|---------|-------------------|-----------------|
| Не thread-safe | Thread-safe | Thread-safe |
| Быстрый | Почти как HashMap | Медленный (блокирует весь мап) |
| Падает в CI | Стабилен | Тормозит при большом кол-ве тестов |

**Для собеса:** Объяснить, что используешь `ConcurrentHashMap` для thread-safe тестовых данных при параллельном прогоне. `synchronized` — слишком грубая блокировка.

## 5. Авито — асинхронность без Thread.sleep

**Что спрашивают:** «Тест упал из-за того, что `Thread.sleep` не сработал — как тестировать асинхронные вызовы?»

**Почему `Thread.sleep` — антипаттерн:**
- Ненадёжный (разные тайминги в CI)
- Замедляет тесты
- Не даёт гарантии, что асинхронная операция завершилась

**Инструмент: Awaitility**
```java
await().atMost(5, SECONDS)
    .untilAsserted(() -> {
        assertThat(response.getBody()).contains("expected");
    });
```

**Продвинутые кейсы:**
- `catchUncaughtExceptions` — ловит исключения из других потоков
- Работа с `CompletableFuture` через `await().until(() -> future.isDone())`
- Тестирование WebSockets/SSE

## 6. Сбер — System Design для QA

**Что спрашивают:** «Как спроектировать стенд, который симулирует 1000 RPS, но не ложит продакшен-базу?»

**Главная ловушка:** Не техническая, а архитектурная. Ожидают, что скажешь: «Нельзя грузить прод-базу напрямую».

**Правильный ответ:**
- Изолированный тестовый стенд с отдельной БД (клон/слепок прода)
- Rate-limiting на API Gateway
- Моки внешних сервисов
- Инструменты: k6 / Gatling / JMeter
- Мониторинг: Prometheus + Grafana (latency, error rate, CPU/memory)
- Паттерны: Circuit Breaker, Bulkhead, Retry with backoff

## 7. CompletableFuture + race conditions в CI

**Что спрашивают:** «Тест проходит локально, но падает в CI — как ловить гонки данных?»

**Причины:**
- Разное окружение (CPU, память)
- Непредсказуемый порядок выполнения потоков
- Отсутствие синхронизации в тестовых данных

**Чек-лист:**
- [ ] Использовать `CountDownLatch` в тестах для синхронизации потоков
- [ ] `Awaitility.await().untilAsserted()` вместо sleep
- [ ] Проверить общие тестовые данные на thread-safety
- [ ] В CI: увеличить таймауты, уменьшить параллельность
