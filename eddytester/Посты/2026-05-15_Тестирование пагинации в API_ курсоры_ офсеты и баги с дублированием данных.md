# Тестирование пагинации в API: курсоры, офсеты и баги с дублированием данных
**Date:** 2026-05-15
**Angle:** 
**Status:** черновик

---

Есть классический баг пагинации — дубликаты записей при OFFSET/LIMIT. Причина простая: между запросами страниц вставили новый элемент в начало списка. Первая страница взяла позиции 0–9, вторая — 10–19. Но из-за сдвига на одну новую запись часть старых записей попала в обе страницы. Пользователь видит один и тот же пост дважды.

## Как воспроизвести руками

1. Вставь 11 записей в таблицу. Запроси page1: `LIMIT 10 OFFSET 0`.
2. Вставь ещё одну запись (имитация параллельной активности).
3. Запроси page2: `LIMIT 10 OFFSET 10`.
4. Собери все ID с обеих страниц. Если `len(set(ids)) != len(ids)` — дубликат найден.

Проверить одной командой:

```bash
curl "https://api.example.com/items?limit=10&offset=0" > p1.json
# параллельная вставка
curl "https://api.example.com/items?limit=10&offset=10" > p2.json
python -c "import json; ids=[]; [ids.extend([i['id'] for i in json.load(open(f'p{x}.json'))['results']]) for x in [1,2]]; print(len(set(ids)) == len(ids))"
```

`False` — баг.

## Почему так происходит

OFFSET-пагинация работает по физическому смещению. Если между запросами появляются новые строки — индексы сдвигаются, и границы страниц «плывут». При удалении — наоборот, образуются пропуски (пользователь теряет записи).

## Когда и что использовать

**Данные меняются редко** (каталог товаров, справочники): OFFSET с сортировкой по уникальному полю (`ORDER BY id`) почти безопасен.

**Данные динамичные** (лента новостей, комментарии, сообщения): используй курсорную пагинацию (keyset). Принцип: `WHERE id > ? ORDER BY id LIMIT 10`. Курсор фиксирует позицию и не сдвигается при вставках.

Важные нюансы:
- Нужен стабильный ключ — например, `(created_at, id)` или просто `id`.
- Если курсор использует только `timestamp` без tiebreaker — Twitter в 2017 году наступал на эти грабли и показывал дубликаты.
- Если курсор указывает на удалённый объект — Stripe падал в 500.
- OFFSET без `ORDER BY` — GitHub API отдавал дубли при таком подходе.

## Что проверить в тестах

1. Параллельная вставка между страницами — нет дублей.
2. Параллельное удаление между страницами — нет пропусков.
3. `next_cursor` / `next_page` пуст на последней странице — иначе клиент уйдёт в бесконечный цикл.
4. Курсор на удалённый объект — сервер не падает, а возвращает корректный ответ.

## Вывод

Не верь OFFSET-пагинации для динамических данных. Если не хочешь ловить дубли на проде — тестируй сценарий «вставка между страницами». Для статичных данных OFFSET норм, для динамики — курсоры.

**Реалы:** видел, как тестировщики проверяют пагинацию одним запросом: «первая страница работает, ок». Потом на проде юзеры жалуются. Ещё смешнее: бекендер использует OFFSET с автоинкрементным ID, но сортировка по дате меняет порядок — дубли всё равно вылезают.

Тест на дубликаты — один из самых простых в написании, но один из самых частых пропусков в регрессии. Не пропускай.
