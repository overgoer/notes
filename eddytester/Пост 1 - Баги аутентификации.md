## Post Draft (Channel Agent)

**Situation**
Ты тестируешь логин, регистрацию, сброс пароля. Всё работает. Но под капотом — дыры, которые превратят твой API в решето. Вот 5 багов аутентификации, которые реально встречаются в продакшене.

**Analysis**

**Баг #1: OAuth redirect_uri без валидации порта**
Атакующий регистрирует клиент с `redirect_uri = http://127.0.0.1:9999/callback`. Провайдер не проверяет порт. Жертва авторизуется — code уходит на локальный порт атакующего.
*Как проверить:* отправить запрос с `redirect_uri` на `localhost` или `127.0.0.1` со своим портом. Если сервер не ругается — баг.
`curl "https://oauth-provider.com/auth?response_type=code&client_id=...&redirect_uri=http://127.0.0.1:9999/callback"`

**Баг #2: JWT exp на год, logout не инвалидирует токен**
Access token живёт год. При logout сервер удаляет refresh token, но access остаётся. Атакующий, перехвативший токен, пользуется им год.
*Как проверить:* получить токен → logout → вызвать API с тем же токеном. Если 200 — баг.

**Баг #3: Session fixation — ID не меняется после логина**
Сервер принимает session ID, переданный в URL или куке, и не генерирует новый после аутентификации. Атакующий даёт жертве свою ссылку с предустановленным ID — после логина получает доступ к сессии.
*Как проверить:* `curl -b "SESSIONID=attacker123" -X POST -d "user=test&pass=test" https://site.com/login` — если SESSIONID в ответе не изменился — баг.

**Баг #4: OAuth state = "123" (статический)**
Защита от CSRF в OAuth — параметр state. Если он статический (или отсутствует), атакующий может инициировать авторизацию жертвы и подменить code.
*Как проверить:* перехватить OAuth-запрос, проверить state. Если "123" или одинаковый каждый раз — баг.

**Баг #5: Refresh token rotation — race condition**
Идеальная реализация: каждый раз выдаётся новый refresh, старый аннулируется. Но если два запроса приходят одновременно — оба могут получить новые токены, а старый аннулируется только один раз. Второй запрос падает.
*Как проверить:* отправить два параллельных запроса на refresh, проверить, что оба вернули 200. Один из них должен упасть — если нет, ротация не работает.

**Verdict**
Это **баги**. Пять дыр, каждая — готовая CVE в твоём API. Что делать:
- OAuth: валидировать redirect_uri строго (схема + хост + порт)
- JWT: использовать чёрный список токенов (blacklist по jti) или короткий exp
- Session: регенерировать ID после логина
- OAuth state: генерировать случайный, проверять при callback
- Refresh: добавить grace period или distributed locking

**Takeaway**
Аутентификация — первый рубеж. Если он дырявый, остальная безопасность не имеет значения. Потрать 15 минут на эти 5 проверок перед каждым релизом.

**Реалы**
- Баг #2 (JWT после logout) — классика. Многие разработчики даже не знают, что access token нужно отзывать.
- Race condition на refresh — редкий зверь, но когда он стреляет — это катастрофа: все пользователи одновременно теряют сессии.
- Session fixation считается древней уязвимостью, но до сих пор встречается в новых проектах.

## Post Draft (Style Adapter)

Представь, что ты тестируешь логин. Всё работает — регистрация, вход, выход. Но под капотом пять дыр, которые превращают твой API в решето. Вот как их найти за 15 минут.

Первый и самый частый — OAuth redirect_uri. Сервер часто не проверяет порт в редиректе. Атакующий регистрирует клиент с `redirect_uri = http://127.0.0.1:9999/callback`, жертва авторизуется — и authorization code (одноразовый код для получения токена) улетает на локальный порт атакующего. Проверяется одним curl: `curl "https://provider.com/auth?response_type=code&client_id=...&redirect_uri=http://127.0.0.1:9999/callback"`. Если сервер не ругается — баг.

Второй — JWT живёт год после logout. Access token (токен доступа, который прикрепляется к каждому запросу) часто не имеет механизма отзыва. Сервер удаляет refresh token, но access остаётся. Перехватил токен — пользуешься год. Проверка: получить токен, сделать logout, дёрнуть API тем же токеном. Если 200 — баг.

Третий — session fixation. Сервер принимает session ID из URL и не меняет его после логина. Атакующий кидает ссылку `http://bank.com/?PHPSESSID=ABC123`, жертва логинится, ID не регенерируется — и атакующий заходит в аккаунт. Проверка: `curl -b "SESSIONID=attacker123" -X POST -d "user=test&pass=test" https://site.com/login` — если ID в ответе не изменился, всё, баг.

Четвёртый — OAuth state = "123". Параметр state защищает от CSRF-атак в OAuth-flow. Если он статический, атакующий может инициировать авторизацию жертвы и подменить code. Генерируй случайный state при каждом запросе.

Пятый — race condition на refresh token. Идеальная ротация: каждый раз выдаётся новый refresh, старый умирает. Но при двух одновременных запросах оба могут получить новые токены, и один из клиентов упадёт. Лечится distributed locking (блокировкой на уровне данных) или grace period (окном, в течение которого старый токен ещё принимается).

**Что делать:** проверяй эти 5 точек перед каждым релизом. Аутентификация — первый рубеж, и если он дырявый, остальное неважно.

**Полезно:** [PortSwigger: OAuth 2.0 vulnerabilities](https://portswigger.net/web-security/oauth), [OWASP: Session Fixation](https://owasp.org/www-community/attacks/Session_fixation)

## Research Brief

Тема: Authentication and authorization testing (JWT, OAuth, session-based)

Ключевые находки:
- OAuth redirect_uri: localhost без валидации порта — перехват authorization code
- JWT exp на год: logout не инвалидирует access token (нет blacklist)
- Session fixation: сервер не регенерирует session ID после логина
- OAuth state: статическое значение — CSRF в authorization code flow
- Refresh token rotation: race condition при параллельных запросах
- JWT algorithm confusion: CVE-2015-9235, библиотека принимает "alg":"none"
- Max-Age vs Expires: разница в интерпретации ведёт к уязвимости
