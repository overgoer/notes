# 🔥 Форматы №2 и №3: «API-шпаргалка» и «Тестировщик VS Разработчик»

Краткое описание, примеры из этого диалога и готовые темы для постов — без шаблонов, только реальные, технически точные, простые в исполнении.

## 📜 Формат №2: «API-шпаргалка: 3 строки, как в проде»

**Когда публиковать:** Каждый день, 2–3 раза в неделю. Быстро, без подготовки, с высокой отдачей.

**Что это:** Одна операция API → три команды `curl`/`jq`/`grep`/`httpie`, которые работают *прямо сейчас*, в prod-среде, без Postman и без настройки.

**Почему работает:**
- Аудитория получает **рабочий инструмент** — не теорию, а «скопируй и запусти».
- Это **не учебный пример**, а **то, что вы реально используете** (и что будет в практикуме).
- Нет «должен знать» — есть «вот, попробуй».

**Как писать:**
- Заголовок — конкретное действие (`Как проверить жив ли API`, `Как понять — баг или фича?`).
- Три команды — одна под другой, с кратким пояснением *что каждая делает*.
- Конец — призыв: `Кидайте свои`, `Скопируй и запусти`, `Добавь в шпаргалку`.

**Пример из этого диалога:**
> `curl -s https://api.example.com/health | jq '.status'` → Как быстро проверить, жив ли API.
> 
> `curl -v -X POST https://api.example.com/login -d '{"u":"a","p":"b"}' 2>&1 | grep -i "401\\|403"` → Как понять, что аутентификация сломалась (а не тело/заголовок).
> 
> `curl -s https://api.example.com/users/123 | jq '.id, .email' | head -n 5` → Как быстро получить ровно то, что нужно — из любого JSON.
> 
> Кидайте свои команды — соберу в «API-Шпаргалку 2026».

| 1 | Как быстро проверить жив ли API | `curl -s -o /dev/null -w "%{http_code}" https://api.example.com/health` | Всегда отдаёт код 200/503 без тела — быстро и надёжно |

| 5 | Как понять, возвращает ли API `401` или `403`, если страница белая | `curl -s -o /dev/null -w "%{http_code}" -X GET https://api.example.com/protected` | Белая страница в браузере — не статус. Это `curl` покажет реальный код |

| 8 | Как быстро проверить, поддерживает ли API `PATCH` | `curl -s -I -X PATCH https://api.example.com/endpoint | grep -i "allow"` | `Allow: GET,PUT,PATCH` — честный ответ от сервера |
| 9 | Как получить `Set-Cookie` из логина — и передать в следующий запрос | `curl -s -D - -X POST https://api.example.com/login -d 'u=a&p=b' | grep "Set-Cookie"` | Без этого — следующий запрос будет `401`. Это работает в 100% cookie-аутентификаций |
| 10 | Как проверить, возвращает ли API JSON или HTML (когда ожидается JSON) | `curl -s https://api.example.com/endpoint | head -c 50 | grep -q "<html" && echo "HTML" || echo "JSON"` | Решает проблему «почему у меня 500?» — на самом деле 200, но с HTML-ошибкой |

| 12 | Как проверить, кешируется ли POST (что не должно быть) | `curl -s -H "X-Test: 1" -X POST https://api.example.com/test | curl -s -H "X-Test: 2" -X POST https://api.example.com/test` | Если ответы одинаковые — кеш работает. Это ошибка. |
| 13 | Как быстро посмотреть `Access-Control-Allow-Origin` | `curl -s -I https://api.example.com/endpoint | grep -i "access-control"` | Без этого фронт не может делать запрос. Это первая причина «запрос не ушёл» |
| 14 | Как проверить, принимает ли сервер `gzip` | `curl -s -H "Accept-Encoding: gzip" -I https://api.example.com/endpoint | grep -i "content-encoding"` | Если `gzip` — сервер экономит трафик. Если нет — это баг. |
| 15 | Как проверить, есть ли в ответе JSON-поле `error` (а не `errors`) | `curl -s https://api.example.com/endpoint | jq -r '.error // empty'` | У многих API `error` — строка, `errors` — массив. Это частая причина `500` на фронте |
| 16 | Как проверить, разрешает ли CORS endpoint | `curl -s -H "Origin: https://example.com" -I https://api.example.com/endpoint | grep -i "access-control"` | Без этого `fetch()` в браузере падает. Это не «баг фронта» — это «баг бэкенда» |
| 17 | Как получить все параметры query из `curl`-лога (когда URL длинный) | `curl -s 'https://api.example.com/search?q=term&limit=10' -v 2>&1 | grep "GET"` | Показывает *точно*, что ушло в URL. Спасает при отладке `414 URI Too Long` |
| 18 | Как проверить, возвращает ли API `429 Too Many Requests` при ручном тестировании | `curl -s -w "%{http_code}" -X GET https://api.example.com/endpoint{1..5}` | Показывает, с какого запроса начинается `429`. Это не настройка — это прод-логика |
| 19 | Как проверить, есть ли в ответе `X-RateLimit-Limit` | `curl -s -I https://api.example.com/endpoint | grep -i "rate"` | Эти заголовки есть в 90% прод-API. Без них — вы не знаете лимиты |
| 20 | Как проверить, работает ли `If-None-Match` (ETag-кеширование) | `curl -s -I -H "If-None-Match: abc" https://api.example.com/endpoint` | Если `304` — кеш работает. Если `200` — нет. Легко и точно |

## 📜 Формат №3: «Тестировщик VS Разработчик: чья версия правда?»

**Когда публиковать:** 1–2 раза в неделю. Высокая вовлечённость, низкая подготовка.

**Что это:** Один кейс из прод/собеса/чата → версия тестировщика → версия разработчика → **как вы доказали, кто прав** (не спором, а данными).

**Почему работает:**
- Снимает синдром самозванца: «Я не глупый — я просто не видел данных».
- Это не про «кто плохой», а про «как докопаться до правды» — ваша суперсила.
- Каждый читатель мысленно подставляет свой кейс → и сразу пишет в комменты.

**Как писать:**
- Заголовок — конкретная ошибка/сообщение (`"401 Unauthorized"`, `"Ян: — Да-да. Пошёл я…"`, `"500 на логине"`).
- Тестировщик говорит: `«Это баг — нужно показывать понятное сообщение»`.
- Разработчик говорит: `«Это фича — клиент должен знать, что запрос неправильный»`.
- Вы — как пришли к правде: `curl`, `логи`, `git blame`, `база`, `снимок экрана`.
- Конец — призыв: `Кидайте свои кейсы`, `А у вас как?`, `В практикуме будет модуль "Как договориться"`.

**Пример из этого диалога:**
> «API возвращает `400 Bad Request`.  
> **Тестировщик говорит:** «Баг. Нужно показывать читаемое сообщение — что не так?»  
> **Разработчик говорит:** «Фича. Клиент должен знать, что запрос неправильный. Сообщение — в логах».  
> **Как мы пришли к правде?**  
> 1) Запросили `curl -v` — увидели, что сервер *не возвращает `message` в теле*.  
> 2) Посмотрели `git blame` на этот метод — оказалось, `message` убрали 2 года назад, но API не обновляли.  
> 3) Решили: `message` — нужен. Добавили в `master`.  
> А у вас такое было? Кидайте кейсы — в практикуме будет модуль «Как договориться с разработчиком».»
